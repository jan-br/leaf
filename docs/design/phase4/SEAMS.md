## ContractId derivation + collision policy (the stable cross-build identity `ContractId(u64)` = `contract_hash(&str)`, rea

**Decision:**

Pin ContractId to ONE canonical-path-string hash, derived identically by every reader. Concretely:

(1) HASH FUNCTION — `pub const fn contract_hash(canonical_path: &str) -> u64`, the SINGLE entry point. Algorithm fixed as **FNV-1a 64-bit** (offset basis 0xcbf29ce484222325, prime 0x100000001b3, hashing the UTF-8 bytes of `canonical_path`). FNV-1a (not Fx) because it is byte-order-independent, trivially `const fn` on stable, and identically reproducible across compiler builds/platforms — the cross-build invariant. `pub struct ContractId(pub u64)` wraps the result; the top bit is NEVER reserved/salted.

(2) THE CANONICAL PATH STRING — the ONE input is a deterministic string built as `format!("{crate}::{module_path}::{ident}")` using EXACTLY these three components and NOTHING else:
   - `crate` = the Cargo PACKAGE name with hyphens normalized to underscores (`env!("CARGO_PKG_NAME").replace('-', "_")`), i.e. the crate's extern-crate identifier — the same token a downstream `use leaf_redis::RedisAutoConfig` writes.
   - `module_path` = `module_path!()` with the leading crate segment STRIPPED (so `crate::foo::bar` → `foo::bar`); empty for a crate-root item (string becomes `{crate}::{ident}`).
   - `ident` = the bean's TYPE identifier (struct/enum name) for `#[component]`/stereotypes/`#[auto_config]`; for a `#[bean] fn` it is `{ConfigTypeIdent}::{method_ident}` (the config struct path + method name), so two `#[bean] fn pool()` on different config structs never collide.
   `declared_name`/`#[component(name=...)]`/aliases are DELIBERATELY EXCLUDED — they are mutable DX labels keyed by the name overlay, not identity; including them would make `leaf.autoconfigure.exclude = leaf_redis::RedisAutoConfig` unmatchable (the exclude path carries no bean-name) and would break a ContractId on a cosmetic rename.

(3) MARKERS/APPTYPE — `MarkerId(contract_hash("leaf::Component"))` and `AppType(contract_hash("web.servlet"))` feed a hand-authored stable FQN string THROUGH THE SAME `contract_hash`. The contract is therefore "any author-stable canonical string → FNV-1a"; the per-bean rule in (2) is the deterministic way the macro BUILDS that string for a Descriptor. One function, one algorithm, two string-construction call-sites (macro-built vs hand-authored), never two hash functions.

(4) COLLISION POLICY — the `freeze()` guard HARD-ERRORS, never salts. On any two distinct canonical paths producing the same u64, `RegistryBuilder::freeze()` returns `AssemblyError::ContractIdCollision { id: u64, a: &'static str, b: &'static str }` carrying BOTH canonical-path strings + both crate provenances. The two canonical-path strings (not just the descriptors) are retained as `&'static str` on the Descriptor specifically so the guard can name them. Salting is rejected because a salt is non-reproducible across builds, which would make a config-file exclude string match in one build and miss in the next — the exact silent-break this seam exists to prevent.

(5) THE STRING IS THE SEMVER SURFACE — the canonical path (crate + module-path + type/method ident) is documented as a PUBLIC, semver-governed contract: a crate rename, a `pub use` re-export that downstream code excludes-by, or moving a bean to a different module is a BREAKING change that changes the ContractId, and is the author's conscious responsibility exactly like renaming a public type. The exclusion typo-guard (`ExclusionError::NoSuchTarget`, naming what it considered) is the safety net that turns a stale exclude into a LOUD error rather than a silent no-op.

**Affects:** registry-core (ADR-02 / 01-registry-core): the freeze-time ContractId collision guard + by_contract index + Descriptor.contract field, discovery-codegen (ADR-03 / 02-discovery-codegen): the leaf-macros emit-site, contract_hash const fn in leaf-core, MarkerId/COMPONENT consts, stereotype! minting, the #[bean]-fn ident rule, conditions-autoconfig (ADR-09/ADR-11 / 05-conditions-autoconfig): ExclusionSet keys, #[leaf::main(exclude=[...])] path-to-ContractId lowering, leaf.autoconfigure.exclude config-string resolution, before/after OrderHint edges, the NoSuchTarget typo-guard, bootstrap-diagnostics (14): AppType(ContractId), all run-participant linkme streams keyed/tie-broken on ContractId + SequenceKey, environment-config (06): indirectly, via the leaf.autoconfigure.exclude relaxed-bound config string resolved to a ContractId, error-model (ADR-12): the new AssemblyError::ContractIdCollision variant + Integration{ContractId} arm already keying on it

**Rationale:** Byte-identical across readers is the whole point: today ADR-02 says (crate-stable-name :: ident :: declared-name), ADR-09/auto-config says (Cargo package + module-path + ident + declared-name), and ADR-03 says an opaque contract_hash(stable_ident). Three macros feeding three different strings produce three different u64s for the same bean — the freeze guard, MarkerId edges, and config exclude keys would silently disagree. Pinning ONE canonical-path-builder eliminates the divergence by construction (charter §2.11 unification).; declared-name MUST be excluded or the headline use-case breaks: the corpus's own exclusion call-site is contract_hash("leaf_redis::RedisAutoConfig") — a pure type path with no bean-name. If declared-name fed the hash, that exclude string could NEVER match the bean's ContractId. ADR-02's inclusion of declared-name directly contradicts ADR-09's exclude path; the contradiction is resolved in favor of the type path (the thing ops can actually write).; FNV-1a over Fx: cross-build/cross-platform reproducibility is the load-bearing invariant (TypeId was rejected precisely because it is not stable across builds). FNV-1a is endianness-independent, const-fn-trivial on stable, and specification-fixed; Fx is a rustc-internal hasher with no stability guarantee. Same reason salting is rejected — a non-reproducible id makes a config exclude string a coin-flip across rebuilds.; module-path is included (not just crate+ident) because two same-named types in different modules of one crate are distinct beans and must get distinct ids; the leading-crate-segment strip makes module_path!() agree with the downstream use path. The #[bean]-fn rule (config-ident::method) is required because bare method idents (pool, repo) collide trivially across configs.; Hard-error at freeze (not salt) is mandatory because a u64 FNV-1a collision is astronomically unlikely but not impossible, and a salt would trade a loud, fixable, build-time error for a silent cross-build-nonreproducible footgun — violating charter §2.9 (fail early, rich, actionable). The error names BOTH canonical strings so the author can disambiguate by renaming one.; Keeping ONE contract_hash(&str) entry point (not a tuple-typed API) is what lets MarkerId/AppType (hand-authored FQNs) and Descriptors (macro-built canonical paths) share the identical function — markers ARE ContractIds per ADR-10/02; a second derivation path for markers would re-fragment the very identity this seam unifies.

**Ripples:** registry-core (01 / ADR-02): change the ADR-02 comment from `const FNV/FxHash of (crate-stable-name :: ident :: declared-name)` to `FNV-1a 64-bit of the canonical path crate::module_path::ident (declared-name EXCLUDED)`. Add a `canonical_path: &'static str` field to Descriptor (alongside `contract: ContractId`) so the freeze guard can name colliders; the by_contract HashMap and freeze pass are otherwise unchanged. The collision guard becomes a hard AssemblyError::ContractIdCollision{id,a,b}.; discovery-codegen (02 / ADR-03): leaf-macros gains the canonical-path builder — emit `contract_hash(concat!(env!("CARGO_PKG_NAME")-normalized, "::", <stripped module_path!()>, "::", <ident or ConfigIdent::method>))` as the ONE expansion for Descriptor.contract AND the emitted canonical_path str. contract_hash stays `const fn(&str)->u64` but its doc-comment pins FNV-1a and 'never salt'. MarkerId/AppType const sites are unchanged (they already pass an FQN string). The hyphen→underscore crate-name normalization is specified so macro-built and hand-written paths agree.; conditions-autoconfig (05 / ADR-09/11): ExclusionSet::from_compile lowers each `#[leaf::main(exclude=[leaf_redis::RedisAutoConfig])]` path by re-running the SAME canonical-path builder on the referenced type's path → contract_hash; merge_runtime resolves a `leaf.autoconfigure.exclude` string by canonicalizing it identically. before/after OrderHint edges that name types resolve the same way. The NoSuchTarget typo-guard is retained verbatim as the loud safety net for stale paths. No change to the ladder (exclude>back-off>default).; bootstrap-diagnostics (14): AppType(ContractId) and every run-participant stream keep using contract_hash unchanged — they already feed hand-authored stable strings ('web.servlet') through the same function; this seam just confirms that is the canonical entry point, so no code change, only a cross-reference to the pinned contract.; error-model (ADR-12): add the closed variant AssemblyError::ContractIdCollision{ id: u64, a: &'static str, b: &'static str } to the freeze/seal Tier-1 error set; it renders both canonical paths + crate provenances via the one Diagnostic renderer. The existing Integration{ContractId} arm is unaffected.; documentation: a single 'ContractId is a semver surface' note is added to the public docs — crate rename / bean module-move / changing the re-exported type path is a breaking change that shifts the id; the exclusion typo-guard converts a stale exclude into a loud NoSuchTarget rather than a silent miss. This note is referenced identically by all four reader subsystems so none re-derives its own rule.

## @Fallback-role vs OnMissingBean candidate arithmetic — pinned as the single `FallbackDemote`-fused step inside the `prim

**Decision:**

PIN ONE rule. Representation: replace the two independent bool flags `is_primary`/`is_fallback` on `Cand` with the SINGLE source of truth `CandidateRole { Normal, Primary, Fallback }` already minted in leaf-core (TOPOLOGY/05) PLUS one orthogonal `is_fallback: bool` derived field — i.e. make role a 2-axis value, not a 3-way enum, because a `@Fallback` bean CAN also be `@Primary` (a starter shipping a default-primary among its own beans). Concretely: `Cand.candidate_role: CandidateRole` where `CandidateRole` becomes `struct CandidateRole { primacy: Primacy, fallback: bool }` with `enum Primacy { Plain, Primary }`. `Normal = {Plain,false}`, `Primary = {Primary,false}`, `Fallback = {Plain,true}`, `Fallback+@Primary = {Primary,true}`. The macro maps `#[bean(primary)]`→primacy=Primary, `#[bean(fallback)]`/auto-config-default→fallback=true. This is the IDENTICAL value read by the Selector, the DefinitionProbe, and condition-report; no second flag scheme.

The `primary_promote` layer runs as a THREE-step fused pass in this FIXED order (replacing the doc-03 two-pass):
- STEP A — FallbackDemote (NEW, runs FIRST): partition the incoming set into `regulars = {c : !c.fallback}` and `fallbacks = {c : c.fallback}`. If `regulars` is non-empty, the working set becomes `regulars` (drop all fallbacks). If `regulars` is empty (all-fallback), the working set stays the full fallback set. Rationale: "a Fallback loses to any non-fallback of the same contract" is enforced BEFORE primacy, so a `@Fallback @Primary` can never beat a plain user `Normal`.
- STEP B — PrimaryPromote (over the STEP-A working set, unchanged logic): unique min-`hierarchy_depth` `primacy==Primary` → `Unique`; ≥2 primaries at the same min depth → `Resolved::Ambiguous` (PrimaryConflict); local primary beats parent primary by min depth.
- STEP C — if zero primaries in the working set: `len==1 → Unique`; `len>1 → Narrowed(working set)` (hand to `name_match`); `len==0` is unreachable (STEP A guarantees ≥1). The old "all-fallback → Abstain" becomes: all-fallback set simply flows on as its own working set (a lone fallback wins as Unique; multiple fallbacks fall to name_match) — the demote is a no-op when there is nothing to demote to.

Worked truth table (per-type-view, single-valued point), all IDENTICAL everywhere:
- {Fallback + user-Normal} → STEP A drops Fallback → {Normal} → Unique(Normal). User wins. ✓
- {Fallback + Fallback} → STEP A keeps both → no primary → Narrowed → name_match; no name hit → terminal `len>1` → `NoUniqueBean`. Ambiguity, NOT silent. ✓
- {Fallback + Fallback + one @Primary-among-them} → STEP A keeps all (all fallback) → STEP B unique primary → Unique(that fallback-primary). A starter's own default-primary resolves among its peers. ✓
- {Fallback + Normal + Normal} → STEP A → {Normal,Normal} → no primary → Narrowed → name_match. ✓
- {Fallback + Normal(@Primary) + Normal} → STEP A → {Normal@Primary, Normal} → STEP B unique primary → Unique. ✓
- {Fallback@Primary + Normal} → STEP A drops the fallback (even though it is @Primary) → {Normal} → Unique(Normal). The "Fallback loses to any non-fallback" rule DOMINATES @Primary. ✓

OnMissingBean relationship (pin (b) of COHERENCE): `CandidateRole.fallback=true` is the DEFAULT soft override for auto-config beans (they register, then lose to any user Normal via STEP A). `#[on_missing_bean(T)]` is the OPT-IN hard all-or-nothing back-off (the candidate does not even register a slot when T already resolves Unique). They are NOT both applied by default; OnMissingBean is reserved for the "must not even register / would conflict at definition level" case. `on_missing_bean(T)` matches iff `would_resolve_unique(T) != Unique`; `on_bean`/`on_single_candidate` iff `== Unique` — and `would_resolve_unique` runs the EXACT three-step fold above over the definition view (no instantiation), so a lone-registered Fallback reads as `Unique` (correctly making `on_missing_bean` back off) and `Resolvability::Ambiguous(u16)` carries the post-STEP-A `working_set.len()` (the count after fallback demotion, never the raw by-type count).

Count/self-check arithmetic (pin (c)): the collection-injection expected-count self-check operates on REGISTERED rows (post-condition, post-exclusion) only. An excluded/OnMissingBean-backed-off candidate mints NO BeanId slot, so it is structurally absent from both the candidate set and the expected count — no subtraction needed. A demoted-but-registered Fallback HAS a slot and is COUNTED in the by-type set; but `primary_promote`/`FallbackDemote` is STRUCTURALLY bypassed for `Cardinality::Multiple`, so the expected-count for collections is the raw registered-and-condition-passing count with NullBean excluded (doc-03 rule unchanged). The anti-DCE expected-vs-found precondition compares ExpectedManifest rows against registered rows — it fires on a DCE-vanished crate (loud `SourceVanished`), NOT on a deliberate back-off (silent `Negative`) — these stay the four crisply distinct report classes.

**Affects:** injection-resolution (03) — the Selector primary_promote layer, conditions-autoconfig (05) — DefinitionProbe / OnMissingBean / CandidateRole representation, registry-core (01) — Descriptor.meta CandidateRole representation and the merged Cand view, autoconfig-starters-bom / TOPOLOGY — CandidateRole enum shape in leaf-core ABI, collection-injection (03) + anti-DCE expected-count self-check

**Rationale:** Single source of truth: the COHERENCE seam (lines 13-14, 26) flagged that three subsystems describe @Fallback arithmetic with NO single owner. Pinning it as ONE fused FallbackDemote+PrimaryPromote step in the Selector, read verbatim by the DefinitionProbe, satisfies invariant #11 (unification) and the doc-05 promise of 'one definition of unambiguous' shared by conditions AND App<Wired> validation.; Found a real representation bug: doc-03 uses TWO independent bools (is_primary/is_fallback) on Cand while doc-05/TOPOLOGY use a 3-way CandidateRole{Normal,Primary,Fallback}. A 3-way enum cannot represent a @Fallback that is also @Primary (a legitimate starter case: a default-primary among the starter's own beans). The two designs disagree. Pinning a 2-axis CandidateRole{primacy, fallback} reconciles them and is expressible identically in both the Selector and the probe.; Found an ordering bug: doc-03's two-pass runs PrimaryPromote (PASS 1) BEFORE the fallback partition (PASS 2). That ordering lets a @Fallback @Primary win PASS 1 over a plain user Normal — violating COHERENCE's stated rule 'a Fallback loses to any non-fallback of the same contract'. Pinning FallbackDemote to run FIRST (STEP A) makes the rule dominate primacy and produces the user-wins-always result the charter's least-surprise intent demands.; Charter invariant #9 (fail with rich actionable context, refuse to guess): {Fallback+Fallback} resolves to a loud NoUniqueBean via the terminal len>1 rule, never a silent pick. Resolved::{None,Ambiguous} is non-coercible (doc-03), so the no-guess property is a type, and the probe's Resolvability::Ambiguous(u16) count is the post-demote count so back-off and wiring give the SAME verdict.; Separation of NullBean / NoSuchBean / condition-not-met (doc-01, doc-05 four-class taxonomy) is preserved: excluded/OnMissingBean-backed-off candidates mint no slot (absent from set AND expected-count, no subtraction), demoted-but-registered Fallbacks keep their slot (counted, just lose selection), and a DCE-vanished crate is a loud SourceVanished — three distinct states that never alias.

**Ripples:** injection-resolution (03): rewrite the primary-fallback feature's PASS 1/PASS 2 prose into the FIXED three-step FallbackDemote→PrimaryPromote→len-rule order inside the single `primary_promote` layer; FallbackDemote runs FIRST. ## TypeRow upcast-row emission policy (registry-substrate `Descriptor.provides[]`) co-decided with the proxy newtype's mult

**Decision:**

ALL-DECLARED, ONE-PROXY, PER-METHOD-DISPATCH, DEDUP-BY-BeanId. Concretely:

(1) EMISSION SET. For a bean `B`, the thin macro emits one `TypeRow { view: TypeId::of::<dyn Tr>(), upcast }` into `Descriptor.provides[]` for EVERY service trait `Tr` that is BOTH (a) syntactically named in the bean's declaration site the macro can see — the `#[component] impl Tr for B` blocks, plus traits listed in an explicit `#[provides(dyn A, dyn B)]` attribute — AND (b) object-safe / dyn-compatible (a `where Self: Sized` or generic method makes the trait non-`dyn`; the macro silently omits such a trait from `provides[]`, never emitting an unupcastable row). The bean's own concrete `self_type` is always present as the `BeanId`-identity row but matches by exact `TypeId` only (a concrete handle is NOT a service-trait view). This is the "all-declared" arm: declared = "the union of every dyn-safe trait the macro sees at this expansion site", NOT "only the supertraits leaf observed some other crate inject" (the macro cannot see cross-crate use sites, so only-used is unsound across crates).

(2) ONE PROXY STRUCT. An advised multi-trait bean `B` exposing dyn-safe service traits `{T1..Tn}` generates exactly ONE proxy newtype `BProxy { chain: Arc<AdviceChain>, source: Arc<dyn TargetSource> }` that carries `impl Ti for BProxy` for ALL `i in 1..n` — the SAME set as `provides[]`, byte-for-byte (one shared generator computes the set once; emission and proxy-impl read it from the same list). Never one-proxy-per-trait. The single proxy `Arc<BProxy>` upcasts independently to each `Arc<dyn Ti>` (trait upcasting, stable 1.86), so every `provides[]` view resolves to the SAME allocation.

(3) PER-METHOD DISPATCH THROUGH ONE CHAIN. Every method of every `Ti` on `BProxy` delegates through the SAME `Arc<AdviceChain>` and SAME `Arc<dyn TargetSource>`, tagged with its own `Call::new::<{Ti::method}>(..)` carrying the method's `MethodKey`. The chain dispatches per-MethodKey; the trait view the caller happens to hold is irrelevant to which advice fires (pointcut matches on `JoinPointMeta.method`, not on the view). There is NO "the chain dispatches through trait Tk" — the chain is per-bean-global (the `after_init` wrap-once-globally rule already fixed); all views share it.

(4) DEDUP-BY-BeanId IN COLLECTIONS. Collection/Map injection over a supertype `dyn Super` queries `Registry::candidates(TypeId::of::<dyn Super>())` which returns `&[BeanId]`; a bean appears at most once under any single `view` key, and the materialized `Vec`/`HashSet`/`Map` is keyed/deduped by `BeanId` (the dense join key), NOT by `(BeanId, view)`. A bean providing N views injected into `Vec<dyn Super>` therefore contributes exactly ONE element (the proxied handle if advised, under the `dyn Super` upcast), never N. This is structural: distinct views are distinct TypeId keys, and a single supertype injection touches exactly one key.

DEFERRED-BUT-NOT-BLOCKING measurement: row-count bloat of all-declared is measured on a realistic large graph (the registry-core remaining-risk benchmark) BEFORE leaf-core freezes `provides[]`; the only sanctioned reaction to a bad measurement is a `#[provides(...)]` opt-OUT narrowing knob (author explicitly lists fewer views), never a silent only-used inference. The default stays all-declared.

**Affects:** registry-substrate / registry-core (01) — owner of Descriptor.provides[]/TypeRow and Registry.by_type, discovery-codegen (02) — component-stereotypes' Descriptor carries the provides[] rows; the thin macro emits them, proxy-interception (08) — the generated transparent newtype's impl-set and the ProxyPlan/after_init wrap, injection-resolution (03) — candidate-resolver's candidates(ty) keying + collection-injection dedup/count arithmetic over a supertype, binding-conversion (07) — collection/Map injection over a supertype shape (the consumer that flagged the over-supertype count), declarative-advice (09) — tx/cache/validation/retry/@Async on a multi-trait repository bean wrapped by one proxy, codegen-boundary / leaf-macros — the single shared generator that computes the dyn-safe declared-trait set once for both provides[] and the proxy impls

**Rationale:** Cross-crate safety forces all-declared. The macro expands at the bean's declaration site and cannot see how downstream crates inject the bean. 'Only-used' would require whole-program knowledge that does not exist at per-crate expansion; an undeclared-but-used supertrait would be a silent NoSuchBean in a consuming crate — exactly the silent-divergence bug class the charter's fail-fast/no-silent-fallback invariant forbids. All-declared is the only choice consistent with leaf-core being a stable per-crate-emitted ABI.; One-proxy is the ONLY shape that keeps Spring's transparency invariant (charter: advice is transparent dyn Svc; injection sees the SAME type). N proxies would mean a bean has N distinct advised identities; a Vec<dyn Super> and a Ref<dyn Other> on the same bean would hold DIFFERENT allocations with independent chains — breaking the per-bean-global wrap-once rule proxy-interception already fixed at after_init, and making self-consistency (advice_chain_of) meaningless. One struct impl-ing all dyn-safe traits is structurally feasible in Rust and upcasts to each view at zero alloc, so there is no cost to choosing it.; provides[] and the proxy impl-set MUST be the identical set or the system is unsound: the registry advertising a dyn-view the proxy cannot satisfy is an uncatchable runtime upcast failure (the after_init swap replaces the raw ErasedBean with the proxy; if the proxy lacks an impl the raw bean had, a later upcast to that view panics). Making one shared generator compute the dyn-safe declared set ONCE and feed both emission paths is the only way to guarantee they never drift — this is why the decision must be used IDENTICALLY in registry-core and proxy-interception, not co-incidentally.; Per-method dispatch (not per-trait) falls out of the already-fixed primitives: the chain is per-bean-global at after_init, pointcuts match on JoinPointMeta.method (a MethodKey), and Call carries the MethodKey. There is no coherent 'which trait the chain dispatches through' — the view is caller-incidental. This reuses the ONE comparator/ONE chain/ONE ProxyPlan without minting anything, honoring the charter's 'one wrap, four uses' unification.; Dedup-by-BeanId is required to keep collection-injection count arithmetic correct and is consistent with the fixed dense-BeanId join-key invariant. A single supertype injection queries one TypeId key, so a multi-view bean appears once by construction; deduping the materialized collection by BeanId (not by view) closes the only residual hazard (a bean reachable under two distinct super-views that a future merged query might union) and matches injection-resolution's existing collect-all-over-&[BeanId] shape. It also keeps the @Fallback/NullBean count-folding (co-decided elsewhere) operating on BeanIds, not view-tuples.; Dyn-safety filtering at emission is non-negotiable: a non-dyn-compatible trait cannot be a TypeRow view (no dyn TypeId) nor a proxy impl target. Silently omitting it (rather than hard-erroring) preserves 'concrete-only / non-dyn traits stay injectable by concrete type, just not as a service-trait view' — the same named divergence proxy-interception already took for advised beans needing a service trait. A bean whose ONLY traits are non-dyn and that is advised still hits the existing mandatory-service-trait hard-error.; Measured-for-bloat with a #[provides] opt-out (never silent only-used) keeps the default safe while giving an escape hatch at the author level — matching the charter's 'escape hatches at every level' and the toolkit's measure-before-build discipline. Crucially the escape is explicit author narrowing, so it can never reintroduce the cross-crate silent-miss that only-used inference would.

**Ripples:** registry-core (01): Keep `Descriptor.provides: &'static [TypeRow]` and `TypeRow{view,upcast}` exactly as drafted. PIN in the 'Injectable-as-supertype' resolved-question: provides[] = all dyn-SAFE declared service traits (object-safety filter added — was implicit), one row each, the row set is IDENTICAL to the proxy impl-set, dedup of collection candidates is by BeanId. The remaining-risk bullet changes from 'all-declared vs only-used open' to 'all-declared FIXED; bloat measured; #[provides] opt-out is the only narrowing lever'. by_type/candidates() are unchanged (already BeanId-valued and per-view-keyed).; discovery-codegen (02): The thin macro's per-bean emission gains a shared leaf-codegen helper `declared_dyn_views(item) -> &[TypeRow]` used by BOTH the Descriptor emit and (when advisable) the proxy emit, so a stereotype's Descriptor.provides[] can never disagree with its proxy. component-stereotypes' note 'Descriptor carries these rows' is now concrete: rows = the helper's output. Remaining-risk bullet (line 175) resolved to all-declared+measured.; proxy-interception (08): The generated newtype is renamed in spirit from 'per advisable service trait' to 'per advisable BEAN': ONE `BProxy` carrying `impl Ti for BProxy` for every Ti in the same dyn-safe declared set as provides[]. The line-58 remaining-risk ('one proxy implementing all vs one per trait; which trait dispatches') is resolved: one proxy, all views, per-MethodKey dispatch through the one shared chain. ProxyPlan/after_init unchanged (still per-BeanId, wrap-once-global). `register_proxy!(Concrete)` escape for generics unchanged. ErasedProxy fallback's MethodTable must cover methods across ALL views (one table, all MethodKeys).; injection-resolution (03): candidate-resolver unchanged in mechanism; PIN the collection-injection rule that materialized Vec/HashSet/Map dedups by BeanId (a multi-view bean is one element under a single supertype query). This composes with the already-decided NullBean count-folding and the @Fallback arithmetic (both operate on BeanIds). The advised-concrete-injection COHERENCE rejection still holds: a Concrete TypeMatch against an advised bean is a startup error — now reinforced because the concrete self_type row is explicitly NOT a service-trait view.; binding-conversion (07): collection/Map injection over a supertype (the over-supertype count it flagged) is answered: one element per contributing BeanId under the queried dyn-Super view; the proxy is seen under that view consistently. No change to the binder/CPS machinery — this is purely the injection-side count guarantee binding-conversion consumes.; declarative-advice (09): The line-179 remaining-risk is resolved — tx/cache/validation/retry/@Async on a multi-trait repository bean wrap the ONE BProxy; advice fires per-MethodKey regardless of which trait view the caller holds, so a @Transactional method on dyn Write and a @Cacheable method on dyn Read share one chain and one ProxyPlan entry. No resolution/chain/plan logic enters the macro — it only emits the const TypeRow list + the delegating impls.
