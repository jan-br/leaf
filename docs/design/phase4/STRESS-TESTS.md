## Scenario: `A service `OrderSvc` (concrete struct) that exposes a service trait `dyn Order`, is annotated `@Transactional` + `@Cacheable` on its methods, is declared `#[component(scope = REQUEST)]` (PerContextKey / request-scoped), AND is injected somewhere by its concrete type (`Ref<OrderSvc>` / `inject: OrderSvc` field) â€” the maximally-compound, intentionally-adversarial case. Two variants are traced: (V1) exactly as stated (concrete-type injection, which the fixed seam says must be rejected at App<Wired>), and (V2) the corrected form (injected as `Ref<dyn Order>`) which is the configuration that actually has to *run*, so the deeper tx+cache+request-scope composition is stressed rather than masked by the early rejection.`

**Verdict:** MOSTLY COMPOSES, with two genuine BLOCKERS that must loop back to Phase 3 before this scenario is sound. The compound case is a good stress test precisely because it forces the request-scope axis (bean-lifecycle/execution-context) and the advice axis (proxy/declarative-advice) to interleave at one bean, and that interleave exposes two places the design has NOT actually unified: (1) the after_init proxy-swap inside Engine::create and the scoped-proxy ScopeTarget install at the injection seam are two install paths that both fire for a request-scoped advised bean and are never reconciled â€” left as-is this is a double-advise (chain runs twice) or a raw-handoff (chain runs zero times), either of which silently breaks the core invariant; (2) the V1 'advised bean injected by concrete type' rejection that the seam decisions treat as FIXED has no specified mechanism â€” the injection-mechanics Selector cannot see 'is advised' because that lives in proxy-substrate's frozen ProxyPlan, which is not a declared input to App<Wired>::validate(). Both are fixable-in-place at the design level (skip after_init for PerContextKey + advised; make ProxyPlan an explicit validate() input and add the advised-concrete terminal check), but both require a Phase-3 edit because they change which subsystem owns the install/reject step â€” exactly the kind of cross-subsystem ownership seam COHERENCE.md flagged the empty container-lifecycle subsystem (gap #1) as the home for. The remaining three findings are friction/cosmetic: the no-active-scope failure is irreducibly runtime (needs a rich ResolveError + a doctor warn, not a validate-time guarantee the docs overclaim), and @Cacheable-on-a-scoped-bean sharing entries across scope instances is a real DX footgun that should be a loud lint rather than a silent surprise. None of these reach back to Phase 2 â€” the toolkit primitives (ErasedBean, Provider/Engine::create, ProxyPlan, ScopeTarget/InstanceStore, the App phase machine, cmp_order, one LeafError) are each individually correct and sufficient; the failures are all in the SEAMS between subsystems that Phase 4 exists to pin, and they land squarely on the container-lifecycle template and the advised-concrete/scoped-proxy install ownership that the Phase-4 agenda already lists as work items.
