# Loop-back Round 2 — Cluster C2 (C2 — config-binding / @Value validation-LOCUS reconciliation applied to PRISTINE)

## Design

CLUSTER C2 [C2-config-locus] pins ONE validate-time site for ALL config-value diagnostics and reconciles the validate-placement + R5 self-contradiction across the owning docs, on the fixed toolkit (zero Phase-2 reopen).

THE ONE SITE — App<Wired>::validate() (runs BEFORE refresh):
1. Config-properties bind+JSR-validation locus pinned to a single carve-out. validate() PRE-MATERIALIZES each config-properties bean — the ONLY bean class it builds, safe because its Provider::provide is pure-projection (&Env + BindHandler, no ResolveCtx). It invokes the binder-backed factory + the stock ValidationBindHandler (Validate::validate after each Object node), aggregates every BindError/Convert/Violation into the one Tier-2 AssemblyReport, and STORES the bound Published::Shared(Arc) into the bean's singleton OnceCell. So DB_POOL_SIZE=0 failing range(min=1) is a Tier-2 node, never an R5 fault.
2. @Value coercion locus pinned via a THROWAWAY validate-time dry-run sub-pass. For every eager bean's InjectionPlan, for each PointKind::Value row, validate() calls the SAME monomorphized resolve_into::<T>(&env, point, &ctx) and DISCARDS the value, collecting Convert/UnresolvedValue at Tier-2. Same static target T and same &Env+ConvertCtx as the real R5 call, so dry-run and real coercion are provably one code path (no second parser, no drift). order.max-retries=abc → u16 surfaces in the ONE report. The macro emits a per-bean fn(&Env,&ConvertCtx)->SmallVec<[LeafError]> dry-run thunk over the Value rows so validate() needs no T-erasure.

R5 SELF-CONTRADICTION FIXED. The refresh R5 eager bitset is edited to EXCLUDE the config-properties beans validate() already pre-bound (eager-EXCLUDED-because-PREBOUND: slot OnceCell already initialized) in addition to lazy/scoped/prototype; smart-init + Role::Infrastructure stay force-included. R5 publishes the already-bound Arc and NEVER re-runs Provider::provide — so a bind failure can only have surfaced at validate (RunState pre-refresh), never the R5 cancel-cascade (RunState→Failed). The prior reading (a singleton Application config bean matching the force-include rule and re-binding at R5) is explicitly closed. Stated identically in 13-doc R5, SEAMS seam-1 R5 (additive clarification, not a structural reopen), and binding-07.

VALIDATE-PLACEMENT FIXED. registry-core-01 line-32 prose + line-135 comment rewritten from validate-INSIDE-refresh (init context services → self-check → validate → eager) to validate-BEFORE-refresh that DEFERS to the 13-container-lifecycle/seam-1 R0..R8 template. Now all five — seam-1, 13-doc, bootstrap-14 step10/11, registry-01, binding-07 — agree: seal_environment → route_conditions → run_autoconfig → seal() → App<Wired>::validate() → Context::refresh(). RunState-at-failure and teardown obligation are now single-valued.

STRICTNESS LEVER. env-06's 'lenient @Value reads for parity' line DELETED and replaced with strict-collect-into-AssemblyReport wording (both prose + resolved-open-questions). BootstrapSettings gains the 8th field startup_validation: StartupValidation { Strict, Lenient, Skip } (Default=Strict), read ONCE at validate()'s head and threaded down into the placeholder sub-pass, the @Value dry-run ConvertCtx, and the config-properties bind. Strict aggregates all at Tier-2; Lenient downgrades ONLY the two value-shape facets (@Value/placeholder) to Severity::Warn; Skip (hash-matching cargo leaf prepare plan) elides the cold @Value re-walk COST only. THE SKIP/LENIENT CARVE-OUT folded into the lever definition: config-properties range/nested JSR checks have NO Tier-3 fallback (the bean is materialized exactly once at validate), so they stay HARD under both Lenient and Skip — never silently elided, honoring charter §1.7. Reconciles TOOLKIT.md line-93.

The fix is internally consistent across the corpus (markers and validate-before-refresh / eager-EXCLUDED-because-PREBOUND phrasing verified present in every owning doc) and meets the PRISTINE bar for the config-binding-path and diagnostics-tiers scenarios.

## Residual risks

- Skip soundness is co-owned with cargo leaf prepare: under Skip the config-properties bind+JSR still runs (no fallback) but the @Value cold re-walk is elided on the AOT plan's word — the plan-hash must actually cover the bound config; unverified until prepare ships (flagged in env-06 risk + the SEAMS lever clause)
- The @Value dry-run cost is a measure-before-build cold-walk over every eager PointKind::Value row; acceptable-cost is unproven without a real fixture (same class as the strict-collect placeholder pre-flight)
- A LazyValue<T> coercion remains the ONE intentional Tier-3 value facet (deliberately deferred, not dry-run at validate) — documented as by-design, not a regression
- This C2 fix is independent of the OTHER still-open round-2 loop-backs the VERDICT lists (propagating C3/C4/C5/C6/C7 into their owning docs, the SEAMS C6 stub, the C7 two-budget drain, the step-5 facility-quiesce-pre-LIFO ordering); those are out of C2's scope and remain for their respective reconciliations — leaf-core ABI cannot fully freeze until they land
- The config-properties pre-materialization assumes its Provider is genuinely pure-projection (&Env+BindHandler only); a future config type that smuggled a ResolveCtx dependency would break the 'safe to build before wiring is live' guarantee — already structurally enforced (the factory physically cannot autowire a collaborator) but relies on that structural invariant holding
