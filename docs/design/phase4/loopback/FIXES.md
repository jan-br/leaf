## Widen ScopeMismatch validation  `[C4-scopemismatch]`

### Corrected design
ONE coherent rejected-case taxonomy in the Selector's terminal fail-fast rule, owned by injection-mechanics (doc-03), consumed by App<Wired>::validate() (leaf-boot), expressed on the fixed toolkit primitives. No new mechanism, no Phase-2 reopen.

== 1. The lifetime-rank primitive (the missing total order over scope) ==
Mint ONE pure total helper in leaf-core `order` module, beside `cmp_order` but DISTINCT from it (cmp_order is @Order/RoleTier only; this is scope lifetime):
  `fn lifetime_rank(m: Multiplicity) -> u8` mapping the EXISTING `Multiplicity{Once, PerContextKey(ScopeKind), PerResolution}` axis to a longest->shortest ordinal:
    Once = 0 (singleton, container lifetime, longest)
    PerContextKey(ScopeKind) = ScopeKind.lifetime_rank() (SESSION=1, REQUEST=2; custom scopes declare their rank at `register_scope` — added field on the existing `ScopeDef`/`ScopeKind` registration, see newTypes)
    PerResolution = u8::MAX (prototype, per-call, shortest)
  `fn outlives(holder: Multiplicity, target: Multiplicity) -> bool = lifetime_rank(holder) < lifetime_rank(target)` (holder lives strictly longer than target => target is "shorter-scoped" relative to the holder). This is the ONE definition of "shorter scope" every site reads; it derives entirely from data already on the frozen `Descriptor.scope: ScopeDef`, so it is sync/pure over the frozen snapshot (no construction, no IO).

== 2. The single rejected-case enum (the taxonomy) ==
Replace the narrow prose "singleton constructor-injecting a request-scoped bean by value" with ONE closed enum carried inside `ResolveError` -> `LeafError`, aggregated into the one `AssemblyReport` at Tier-2. Add to the existing `ResolveError` (doc-03 lines 124-128) two sibling variants alongside NoSuchBean/NoUniqueBean/Cycle — keeping ResolveError the single rich fail-fast type:

  ResolveError::ScopeMismatch {
      point: InjectionPoint,                 // the offending by-value point
      holder: BeanKey, holder_scope: Multiplicity,
      target: BeanKey, target_scope: Multiplicity,
      class: ScopeMismatchClass,             // see below
      fix_hint: &'static str,                // "convert `w: Worker` to `Lookup<Worker>` (fresh per call) or `LazyRef<Worker>`"
  }
  ResolveError::AdvisedConcreteInjection {   // C3's terminal check, housed in the SAME taxonomy
      point: InjectionPoint, bean: BeanKey,
      concrete: TypeId, available_views: SmallVec<[TypeId;2]>,  // the provides[] dyn views
      fix_hint: &'static str,                // "inject `Ref<dyn Order>` not `Ref<OrderSvc>`"
  }

  pub enum ScopeMismatchClass {
      ShorterScopedByValue,   // PerResolution OR PerContextKey target injected by-VALUE into a longer-lived holder (the widened core)
      ScopeStoreLifetime,     // a longer-lived holder reaching a shorter-lived AmbientStore directly (store-lifetime mismatch, see §4)
  }

This is the "one coherent rejected-case taxonomy in the Selector" the cluster demands: ShorterScopedByValue subsumes both the old request-only case AND the prototype-into-singleton case AND request-into-session; AdvisedConcreteInjection is the third sibling; ScopeStoreLifetime is the fourth. The pre-existing NoSuchBean/NoUniqueBean/Cycle stay crisply distinct — these are NOT folded in.

== 3. The widened Selector terminal check (the load-bearing fix) ==
In `Selector::resolve_one`, AFTER the layer fold picks a winner Cand and BEFORE returning `Resolved::One`, run a single pure terminal `scope_and_advice_check(ip, winner, holder_scope, &proxy_plan)`. It fires three ordered, mutually-exclusive checks; the FIRST match wins so a point yields exactly one rejected-case (no double-report):

  (a) ADVISED-CONCRETE (C3, runs first because it is a hard type-shape error independent of scope):
      if winner.bean_id ∈ proxy_plan.by_bean  AND  ip.kind == PointKind::Bean
         AND ip.produced == winner.self_type (concrete identity row, NOT a provides[] TypeRow dyn view)
      => ResolveError::AdvisedConcreteInjection. (A `provides[]` dyn-view match is fine — it resolves to the transparent proxy.)

  (b) SHORTER-SCOPED-BY-VALUE (the WIDENED core — this is the bug fix):
      if ip.kind == PointKind::Bean (by-value ctor/field param, NOT Deferral/SelfRef)
         AND outlives(holder_scope, winner.scope.multiplicity)   // target strictly shorter-lived than holder
      => ResolveError::ScopeMismatch{ class: ShorterScopedByValue, .. }.
      This now covers: singleton<-prototype (PerResolution; the silent-effectively-singleton + Published::Owned-can't-be-stored case), singleton<-request, session<-request, and request<-prototype — every "by-value injection of a shorter-lived target" symmetric with the cycle rule. The point is REJECTED before any constructor runs.
      EXEMPT (no error — these are exactly the sanctioned escapes): PointKind::Deferral (`Lookup<T>`/`LazyRef<T>`/`Inject<T>`) and PointKind::SelfRef remove the construction edge and re-resolve per call; AND the case where `winner` resolves through a `provides[]` dyn view that the ProxyPlan/scoped-proxy turns into a transparent `ScopeTarget` proxy (a shorter-scoped target injected as `Ref<dyn Svc>` of a bean that has a service trait is served by the ScopeTarget lane, not by value) — i.e. (b) only fires when the holder would literally MOVE/store a shorter-lived instance.

  (c) SCOPE-STORE-LIFETIME (§4): a structural check on the resolved store axis (see below).

The check is sync/pure over the frozen Registry + frozen ProxyPlan; it costs only on the resolved-winner path and adds nothing to the warm len<=1 trace-free path beyond two integer comparisons and one HashMap probe.

== 4. Scope-store lifetime mismatch (StoreSource axis) ==
A by-value point is the multiplicity axis; the store axis has its own latent mismatch: a longer-lived holder that resolves a target whose `ScopeDef.store == StoreSource::AmbientStore(ScopeKind)` MUST go through a re-resolving handle/proxy, never capture the `&dyn InstanceStore` or a single `get_or_init` result, because that store is dropped at scope-end while the holder outlives it. validate() flags `ScopeMismatchClass::ScopeStoreLifetime` when a non-deferral, non-proxied point resolves an AmbientStore-backed target into a ContainerStore-backed (or longer-lived AmbientStore) holder. This is the structural, validate-time half. The IRREDUCIBLY-RUNTIME half (no active scope bound in the ambient Cx at call time — undecidable at validate per STRESS-TESTS line 27) stays a runtime `ResolveError` at first acquisition (the existing `cx.scope_store(scope.store)?` `?`-fail) PLUS a build-time `leaf doctor` warn ("request-scoped bean X has no scope opener in this app; a non-web entrypoint will fail on first call") — the doctor-warn the CONVERGENCE C4 line mandates. validate() does NOT over-claim to catch the no-active-scope case.

== 5. ProxyPlan as a declared validate() input (shared with C3) ==
`App<Wired>::validate()` inputs are pinned as (FrozenContainer + WiringPlan + ProxyPlan). ProxyPlan is already frozen at seal() one phase earlier (proxy-substrate doc-08 line 19), so this is a wiring change, not a new artifact: validate threads `&ProxyPlan` into `Selector::resolve_one`'s terminal check so check (a) can join `proxy_plan.by_bean[winner]`. == 6. Strictness lever interaction (honor seam-5) ==

## Proxy two-site install  `[C3-proxy-twosite]`

### Corrected design
PROBLEM (the gap the stub leaves open). The stress-test scenario (advised + REQUEST-scoped + concrete-injected OrderSvc) exposes that the seam's claim "the proxy is installed at ONE site (the after_init O(1) ProxyPlan lookup)" is FALSE for a scoped-AND-advised bean: there are structurally TWO install sites, and Phase 3 reconciled NEITHER. (a) bean-lifecycle's publish_pipeline runs `raw = run_after_init_decorators(raw)` UNCONDITIONALLY across all three Multiplicity branches (04 line 28/142), so when ScopeTarget::get -> InstanceStore::get_or_init -> Engine::create builds the per-request OrderSvc, after_init looks up ProxyPlan.by_bean[OrderSvc_id], sees it advised, and wraps the live per-request instance in an OrderProxy{FixedTarget}. (b) But the injection seam (08 line 29) ALREADY handed the singleton holder an OrderProxy{ScopeTarget}. Result: the outer ScopeTarget proxy's get() returns an INNER FixedTarget proxy => the tx+cache chain fires TWICE per call; OR, if one blindly "skips after_init for scoped", the doc never says so and the ScopeTarget path then yields a raw un-advised instance (the no-raw-advised-handoff invariant the whole seam protects, violated). This loop-back PINS the two-site rule, names the second installer, makes the advised-concrete rejection actually fire, gives advised prototypes the Owned lane, and makes request-scoped advised beans re-resolve correctly — all on the fixed toolkit, building on C6 (RoleTier/cmp_chain) and C4 (ScopeMismatch + ProxyPlan-as-validate-input + scope_and_advice_check).

=== (1) THE INSTALL-SITE PARTITION (the load-bearing fix) ===
The ProxyPlan is the SINGLE decision artifact (frozen once at seal()), but it has TWO install sites keyed by Multiplicity. Pin the partition as a closed function of the WINNER bean's `scope.multiplicity`, owned by container-lifecycle (the now-authored leaf-boot template) and consumed identically by both subsystems:

  /// THE one rule. Owned by container-lifecycle; read by Engine::create AND injection-resolution's resolve().
  /// Tells each site whether IT is the proxy installer for this bean.
  pub fn proxy_install_site(m: Multiplicity) -> ProxyInstallSite { match m {
      Multiplicity::Once          => ProxyInstallSite::AfterInit,      // singleton: FixedTarget at after_init
      Multiplicity::PerResolution => ProxyInstallSite::AfterInit,      // advised prototype: Owned lane at after_init (see §3)
      Multiplicity::PerContextKey(_) => ProxyInstallSite::Resolver,    // scoped: ScopeTarget at the injection seam (see §2)
  } }
  #[derive(Clone, Copy, PartialEq, Eq)]
  pub enum ProxyInstallSite { AfterInit, Resolver }

SITE 1 — after_init (Engine::create publish_pipeline), CORRECTED. `run_after_init_decorators` becomes Multiplicity-aware:
  fn run_after_init_decorators(&self, raw: ErasedBean, id: BeanId, scope: &ScopeDef, cx: &ResolveCtx) -> Result<ErasedBean, ResolveError> {
      // PerContextKey (scoped) beans are NEVER wrapped here — the InstanceStore stores the BARE live ErasedBean.
      if proxy_install_site(scope.multiplicity) != ProxyInstallSite::AfterInit { return Ok(raw); }
      match self.proxy_plan.advisors_for(id) {            // O(1); empty slice => not advised
          []        => Ok(raw),                            // un-advised: pass through unwrapped (eligibility gate)
          advisors  => Ok(wrap_fixed(raw, advisors, cx)?), // Once: OrderProxy{ chain, source: Arc<FixedTarget(raw)> }
      }
  }
This means: for PerContextKey, the per-request instance the InstanceStore caches is the UNWRAPPED OrderSvc — exactly what ScopeTarget::get must yield so the OUTER resolver-installed proxy is the sole proxy. (The PerResolution/advised-prototype case is §3, also at after_init but via the Owned lane, not FixedTarget.)

SITE 2 — the resolver (injection-resolution's resolve()), NEWLY NAMED. This is the second installer the stress-test says the design "doesn't name".   fn install_scoped_proxy(winner: &Cand, ip: &InjectionPoint, ctx: &ResolveContext) -> Result<ErasedBean, ResolveError> {
      let advisors = ctx.proxy_plan.advisors_for(winner.bean_id);   // already frozen, shared input
      let chain = ctx.scoped_proxy_cache.get_or_build(winner.bean_id, || {
          // make_interceptor for each advisor (resolves Ref<dyn TransactionManager>/Ref<dyn CacheManager>),
          // ordered by cmp_chain over each AdvisorRef.chain_key() (C6) — built ONCE per (holder, bean), memoized.

## Events get their own DispatchInterceptor  `[C5-events-dispatch]`

### Corrected design
PROBLEM (the type-false reuse the stub left open): events.md §event-multicaster (line 45/124/159) claims `PipelineMulticaster` REUSES `leaf_core::aop::Interceptor` VERBATIM — chain entries AsyncDispatch/ErrorIsolation/ContextProp/Metrics wrapping a `CoreDispatch` sink. STRESS-TESTS line 240 proves this is type-impossible: the proxy `Interceptor::intercept(call: Call<'a>, next: Next<'a>) -> BoxFuture<Result<ErasedRet, AdviceError>>` wraps ONE Call and yields ONE ErasedRet, whereas the multicaster fans out over N listeners and yields an aggregate `DispatchOutcome`. A chain entry that wraps the WHOLE dispatch future sees an opaque blob and cannot interpose PER LISTENER (the per-listener loop lives inside CoreDispatch, BELOW the chain), so an AsyncDispatch that must hand EACH of N futures to the Spawner, or an ErrorIsolation that must catch EACH listener's Err and continue, simply cannot be that trait. The fix authors a SECOND, structurally-distinct interceptor trait shaped for pub/sub that shares ONLY the comparator (cmp_chain) + the RoleTier ordering grade with proxy-substrate — never the Interceptor type.

=== (A) THE EVENTS DISPATCH MODEL — leaf-core::events::dispatch ===
The multicaster is a TWO-LEVEL pipeline, both levels governed by the ONE upstream `cmp_chain` (RoleTier-first) over the multicaster-pipeline order constants already pinned in leaf-core::order by the C6 loop-back (ASYNC_DISPATCH_ORDER=100, ERROR_ISOLATION_ORDER=200, CONTEXT_PROP_ORDER=300, METRICS_ORDER=400):

  // Per-LISTENER invocation chain entry — the events-own around shape (NOT proxy::Interceptor).
  pub trait DispatchInterceptor: Send + Sync {
      fn around<'a>(&'a self, ctx: &'a DispatchCtx<'a>, entry: &'a ListenerEntry, next: ListenerNext<'a>)
          -> BoxFuture<'a, Result<ListenerOutcome, DispatchError>>;
  }
  // The continuation the chain threads to reach the listener (the per-listener analogue of proxy Next):
  pub struct ListenerNext<'a> { /* remaining DispatchInterceptors + tail to the adapter call */ }
  impl<'a> ListenerNext<'a> { pub fn invoke(&mut self) -> BoxFuture<'a, Result<ListenerOutcome, DispatchError>>; }
  // Per-dispatch shared facts every chain entry reads (NOT a Call — no MethodKey/ErasedArgs/TargetSource):
  pub struct DispatchCtx<'a> { pub ev: &'a ErasedEvent, pub policy: DispatchErrorMode, pub origin: ContainerId }

`ListenerNext::invoke` is NOT replayable/skippable like proxy `Next::proceed` (no retry/cache semantics on the bus); it is a single forward call to the next per-listener interceptor or, at the tail, to `entry.adapter`. The CONTRAST with proxy is explicit: proxy `Next` is FnMut (replay) + skippable (cache short-circuit); `ListenerNext` is forward-only.

=== (B) THE MULTICASTER SEAM, RE-SHAPED FOR FAN-OUT ===
  pub trait Multicaster: Send + Sync {  // BeanKey::ByName("applicationEventMulticaster"), Role::Infrastructure
      fn dispatch<'a>(&'a self, ev: &'a ErasedEvent, seq: ListenerSeq<'a>) -> BoxFuture<'a, DispatchOutcome>;
  }
  // DEFAULT impl — the fan-out driver, NOT a proxy AdviceChain:
  pub struct PipelineMulticaster {
      chain: Box<[Arc<dyn DispatchInterceptor>]>,  // cmp_chain-sorted at refresh; per-listener around entries
      core:  CoreDispatch,                          // the per-listener iteration driver (the fan-out lives HERE)
  }
  pub struct CoreDispatch;  // for each entry in seq IN cmp_chain ORDER: build ListenerNext over `chain`, call around-pipeline ONCE per listener, collect ListenerOutcome::Emit for chained re-publish
  pub enum DispatchOutcome { Completed, Aborted(LeafError), PartiallyIsolated { errors: SmallVec<[LeafError;1]> }, Scheduled }

CRITICAL STRUCTURAL FIX: the per-listener `for entry in seq` loop lives in `CoreDispatch`, and for EACH listener CoreDispatch builds a fresh `ListenerNext` over the SAME `chain` and runs the per-listener around-pipeline. So `AsyncDispatchInterceptor` (hands THIS listener's future to the Spawner via the upstream `Spawner` + a Detach SpawnHandle), `ErrorIsolationInterceptor` (catches THIS listener's Err and routes to the sink, returns Ok so the loop continues), `ContextPropagationInterceptor` (CxDecorator capture/restore around THIS listener's async hop), and `MetricsInterceptor` interpose AROUND ONE listener — which is exactly what they could never do as a whole-dispatch wrapper. The "one substrate" claim is CORRECTED to "one comparator (cmp_chain) + one RoleTier grade across two structurally-distinct interceptor shapes" (matching the C6 loop-back's part-D commitment and the TypeRow seam's "two interceptor shapes" note).

=== (C) THE DEFERRAL (transactional-listener) RECONCILIATION WITH DECLARATIVE-ADVICE (tx) ===
Phase-deferral is a listener KIND resolved in the ListenerSeq (a `defer: Option<TxDeferral>` field on ListenerEntry), evaluated by CoreDispatch BEFORE any DispatchInterceptor can interpose — NOT a DispatchInterceptor itself (keeps the engine shape-agnostic; the deferral decision lives in the resolved descriptor). The @Async-spawn-site rule (resolving STRESS-TESTS blocker #2/line 244) is PINNED:
  • A `defer=Some` entry is STRUCTURALLY INELIGIBLE for `AsyncDispatchInterceptor`. CoreDispatch ALWAYS reaches a defer=Some listener inline FIRST (the deferral branch runs before the async hop can interpose) so it can read `Cx::current()?.get::<TxResourceKey>()`, and if an active tx is found, register a tx-sync callback on `TxState.synchronizations()` (the declarative-advice TxSyncRegistry) for the chosen phase, capturing the @Async marker.
  • For a defer=Some + @Async listener, the @Async marker is consumed by the DEFERRED-FIRE path: the spawn happens INSIDE the tx-sync callback at the phase boundary (commit), owned by the tx interceptor — NOT by the multicaster's AsyncDispatchInterceptor. This is the ONLY correct site: spawning at publish time would (i) lose the Isolate TxResourceKey (so CoreDispatch could never detect the active tx) and (ii) run the listener before commit, violating AFTER_COMMIT.
  • `AsyncDispatchInterceptor::around` therefore carries an eligibility predicate: `entry.defer.is_none()`. State identically in events.md (AsyncDispatch eligibility excludes defer=Some) and declarative-advice.md (the tx-sync callback owns the deferred @Async spawn).
  • The deferred-fire callback registers its SpawnHandle into the SAME facility-drain set the multicaster quiesce uses, so a tx-deferred @Async spawn is covered by shutdown drain accounting (fixing STRESS-TESTS friction line 250); host-bean destruction is ordered AFTER the facility drain completes.

=== (D) RETURN-VALUE-AS-NEW-EVENT (chaining) ===
`ListenerOutcome::{None, Emit(SmallVec<[ErasedEvent;1]>)}` is collected by CoreDispatch and re-published via `publish_erased`. Chaining is FORCED OFF for any listener that is async/spawn-dispatched (the events macro hard-errors at compile time if a chaining return is declared on an @Async/spawn listener — there is no caller to receive the emitted event). A defer=Some listener may chain only if it fires synchronously at its phase (defer + @Async ⇒ chains=false).

=== (E) CONDITION-TIMING — ONE CONTRACT (resolving STRESS-TESTS blocker #3/line 252) ===
PICK declarative-advice's contract: the listener condition (`CondExprFn = fn(&EvalCx)->Result<bool,ExprError>`, the shared expr-i18n sync-pure closure — NO bus-local matcher) is evaluated AT PUBLISH TIME over the publish-time ambient Cx for BOTH inline AND deferred listeners. Rationale: the event value is Arc-snapshotted at enqueue (see G), the tx is still open at publish so #root/tx context is RICHER there, and the deferred fire reads the SAME captured snapshot for consistency. For a defer=Some listener whose condition is false at publish, CoreDispatch does NOT register the tx-sync callback at all (cheap pre-filter). events.md line 34/40 is corrected to match (was the contradicting "re-evaluated at fire-time").

=== (F) CROSS-COHORT @Order (resolving STRESS-TESTS friction line 242) ===
@Order (cmp_chain order key) is meaningful ONLY WITHIN a phase cohort: inline (defer=None) listeners order among themselves; each TxPhase bucket orders among itself; cross-cohort @Order is structurally ignored (a deferred listener ALWAYS runs after an inline one regardless of order). The frozen ListenerSeq is PARTITIONED at refresh into per-cohort sub-sequences (`ListenerSeq { inline: Box<[ListenerEntry]>, deferred: BTreeMap<TxPhase, Box<[ListenerEntry]>> }`), each cmp_chain-sorted within itself, so the ordering contract is visible in the plan. A leaf doctor / App<Wired> diagnostic warns when an inline and a deferred listener on the same event type carry @Order implying a cross-cohort sequence that cannot hold.

=== (G) ADAPTER ABI — Arc-backed event (resolving STRESS-TESTS friction line 246) ===
`ErasedEvent.value` becomes `Arc<dyn Any + Send + Sync>` (was `Box<dyn Any+Send>`) so a deferred listener can capture an owned, cheaply-cloned, 'static handle for the tx-sync callback. ONE adapter ABI: `adapter: fn(ErasedBean, Arc<dyn Any+Send+Sync>) -> BoxFuture<'static, Result<ListenerOutcome, DispatchError>>` — the inline path borrows through the Arc (one extra clone, vs the previous Box), the deferred path moves the Arc into the 'static callback. This unifies inline + deferred adapter ABI and removes the Box/Arc + borrow/'static mismatch. Cost delta vs the chaining path (which already boxed) is one Arc vs one Box.

=== (H) ERROR POLICY ===

## Teardown in-flight drain + per-request tx finalization  `[C7-teardown-drain]`

### Corrected design
OWNERSHIP NOTE: the C1 upstream loop-back already authored docs/design/phase3/13-container-lifecycle.md and SKETCHED this cluster (context-close steps 1/2/5, the RequestScopeRegistry seam, the Refreshing×shutdown rule). C7's job is to make that sketch PRISTINE: pin the concrete object graph + protocol the sketch leaves implicit, close the one load-bearing wiring gap (which ledger the tx rollback rides), and split the single `grace` Deadline so a stuck finalizer cannot hang shutdown. The design below is authored STRICTLY on the fixed toolkit (Cx/AmbientStore/InstanceStore/TeardownLedger, the one Provider/Engine::create seam, the C1 RunState/TeardownLedger/cmp_chain, the tx advice's TxResourceKey/TxState) and reopens NO Phase-2 decision.

== 1. THE TWO LEDGER FAMILIES (pinned) ==
There are exactly two TeardownLedger FAMILIES of the SAME `leaf_core::ledger::TeardownLedger` type (no new ledger type minted):
(a) THE ONE container ledger — owned by ContainerCore, fed by `register_destroyer_if_shared` (bean-lifecycle), drained LIFO by shutdown() step 5 (reverse WiringPlan wave-order). Carries singleton destroyers, multicaster/projector/SchedulerCore quiesce, partial-destroy + cancel-cascade unwind.
(b) PER-REQUEST ledgers — one per request-scope `InstanceStore`, reached via `cx.scope_store(scope.store)?.ledger()` (the existing `InstanceStore::ledger() -> &TeardownLedger` already in bean-lifecycle). Each carries that request's per-key destroyers AND — the load-bearing pin — the @Transactional interceptor's async commit/rollback FINALIZE entry.
These never alias: the container ledger has a single serialization owner (ContainerCore); each per-request ledger is owned by its InstanceStore and lives in the ambient Cx, outside the container ledger entirely.

== 2. THE TX-FINALIZE-ON-THE-REQUEST-LEDGER PIN (closes break-7, the abandoned-tx hole) ==
The tx advice (declarative-advice 09 line 26/28) says async rollback is "an awaited step in the interceptor OR a TeardownLedger entry" but never names WHICH ledger. PIN: when the @Transactional interceptor's `begin()` binds `TxResourceKey: CxKey{Value=TxState, POLICY=Isolate}` into the request Cx, it ALSO pushes a `TxFinalizeEntry` onto the AMBIENT request-scope ledger — `cx.scope_store(REQUEST)?.ledger().push(TxFinalizeEntry{tx_state, manager: Weak<dyn TransactionManager>, must_rollback: Arc<AtomicBool>})`. Normal completion: the interceptor commits/rolls-back inline AND marks the entry finalized (so the ledger drain skips it — idempotent, finalize-once via a `OnceCell<Outcome>` inside the entry). Cancel path: the sync-Drop guard sets `must_rollback=true` (the existing must-rollback flag, declarative-advice item 5) but does NOT do the async rollback (no async Drop); the still-pending `TxFinalizeEntry` is what the in-flight-request drain runs. Run() of the entry = `if must_rollback { manager.rollback(tx_state).await } else { manager.commit(tx_state).await }`, fired AFTER_ROLLBACK/AFTER_COMPLETION syncs via the same TxSyncRegistry. If a request-scope bean is NOT advised (no tx), no entry is pushed — zero cost. This makes "no async Drop → ledger drains it" true at the request-scope/cancel intersection, where the relevant ledger is the request ledger, not the container ledger.

== 3. RequestScopeRegistry — the concrete seam (closes break-1) ==
`leaf_core::lifecycle::RequestScopeRegistry` is a Role::Infrastructure-installed bean (injectable like AvailabilityHandle), holding `outstanding: AtomicUsize`, a `Notify` woken on every decrement, and a lock-light `tracked: Mutex<SlotMap<ScopeId, TrackedScope>>` where `TrackedScope { store: Arc<dyn InstanceStore>, cancel: RequestCancel }`. The web/request layer, at the `Holder::scope(request_store, handler_fut)` boundary, calls `registry.enter(store: Arc<dyn InstanceStore>, cancel: RequestCancel) -> RequestScopeGuard` BEFORE running the handler. enter() does: `outstanding += 1`, inserts the TrackedScope, returns a `RequestScopeGuard{registry: Weak, id: ScopeId}`. RequestScopeGuard::Drop does `outstanding -= 1`, removes the slot, and `notify.notify_waiters()` — so a normal request completion (handler returns, scope closes, guard drops) decrements the barrier and the per-request ledger drains at scope-end as today (bean-lifecycle line 33, unchanged). The guard is purely a counting/registration handle; it carries NO blocking work in Drop (sync, cancel-safe).
`RequestCancel` is a cooperative cancel signal (a `watch<bool>` / `CancellationToken`-shape clone the web layer's handler future selects on at its await points) — NEVER a foreign `task::abort`. The request task observes cancel and unwinds cooperatively to its `Holder::scope` boundary, which drops the guard and drains its own ledger. The registry only HARD-drains a ledger directly (below) for tasks that do not cooperate within the finalize budget.

== 4. shutdown() context-close (C) — the PRISTINE step list ==
Reaffirms C1's step order, pins the Deadline split and the drain protocol:
0. CAS close-once (valid only from RunState==Running; Created|Refreshing → ShutdownRequested flag, see §6).
1. QUIESCE INTAKE FIRST: Readiness→RefusingTraffic AND DISARM SchedulerCore (no new scheduled body once RunState leaves Running). RunState→Stopping. (closes the scheduler-fire-races-shutdown friction: disarm here, in-flight-body drain stays a step-5 ledger entry.)
2. IN-FLIGHT-REQUEST DRAIN = `registry.drain(grace, finalize_budget).await -> DrainOutcome`. Two-phase, two budgets:
   (2a) QUIESCE-AWAIT: `select! { _ = await_outstanding_zero() => Quiesced, _ = grace.sleep() => timed_out }` — a reactive await on the Notify until `outstanding==0` (charter §2.4: notification, never poll), bounded by `grace: Deadline` (BootstrapSettings.shutdown.grace). If it reaches zero, every request closed gracefully and drained its own ledger via guard-Drop — return Quiesced.
   (2b) COOPERATIVE-CANCEL + HARD-DRAIN: for each still-`tracked` scope, fire its RequestCancel, then `select!` await the guard-Drop (cooperative path) RACED against a separate `finalize_budget: Deadline`. Any scope that has not self-drained when finalize_budget elapses is drained DIRECTLY by the registry: `tracked.store.ledger().drain(finalize_budget).await` — running the TxFinalizeEntry (rollback) + per-key destroyers under that budget. Return `CancelledAfterGrace{drained, ledgers_run, finalize_overruns}`. The two-budget split (grace for "wait for natural completion", finalize_budget for "now force the rollback") is the C7 fix to the C1 single-`grace` sketch: a wedged DB rollback consumes only finalize_budget and can never hang shutdown indefinitely; an overrun is recorded in ShutdownReport, never silently swallowed.
3. PUBLISH Closed{reason} on the live multicaster with IsolateEach dispatch (a Closed-listener fault is a LeafError into ShutdownReport, NEVER aborts teardown). Closed listeners run with NO request/tx scope bound (documented). (closes the Closed-dispatch friction.)
4. stop_all() DESC integer-Phase via JoinSet under per-phase Deadline (stop-before-destroy). RunState→Closing.
5. DRAIN THE ONE CONTAINER LEDGER LIFO under per-bean Deadline → ShutdownReport. Singleton destroyers (logical close()) run HERE — and because step 2 already cancelled+drained all in-flight stragglers, no live request holds a Ref that would call into a now-closed pool (closes break-6: Arc-strong-count = MEMORY soundness only; LOGICAL teardown ordering is enforced by the step-2-before-step-5 barrier). SchedulerCore in-flight-body drain + multicaster/projector quiesce are ledger entries here.
6. RunState→Closed. No async Drop anywhere; Arc strong-count governs the real memory free.

== 5. WHY THIS COMPOSES ==
The request task suspended at .await #2 holding the open tx: step 2a's grace expires (it never completes); step 2b fires its RequestCancel — it either cooperatively unwinds to Holder::scope (guard drops, its ledger drains, TxFinalizeEntry rolls back) OR, if wedged, the registry hard-drains `store.ledger()` directly, running the same TxFinalizeEntry rollback, under finalize_budget. Either way the tx is deterministically rolled back (committed only on the no-cancel path), the per-key destroyers run, and only THEN (step 5) does DbPool.close() fire. "No async Drop" holds; "stays sound" now means BOTH memory (Arc) and logical (drain-before-close).

== 6. Refreshing × shutdown() (closes break-4) ==
Reaffirms C1: a shutdown() CAS observed while `RunState.is_running_or_starting()` is true AND not yet Running (i.e. ∈{Created,Refreshing}) sets `ShutdownRequested{flag: AtomicBool, notify: Notify}` and returns a pending ShutdownReport that resolves when refresh() diverts. refresh() checks the flag at each R-step boundary; if set it enters the (B) cancel-cascade (cancel the in-flight R5 wave JoinSet, partial-destroy via the container ledger LIFO, RunState→Failed, publish StartupFailed). The container TeardownLedger has a SINGLE serialization owner (ContainerCore) so R5's concurrent pushes and a drain never race. In-flight REQUEST scopes cannot exist pre-Running (Readiness only flips to AcceptingTraffic at Ready, after R8), so the cancel-cascade has no request-drain obligation — the request-drain (step 2) is structurally reachable only from Running. This makes the two teardown directions disjoint and race-free.

### Residual risks
- The grace + finalize_budget Deadline DEFAULTS and the cooperative-cancel-vs-hard-drain threshold are tuning parameters that need an end-to-end web-load stress fixture (inherited from bean-lifecycle line-163 / execution-context line-164, now OWNED by step 2 but unmeasured). A finalize_budget set too low rolls back tx's that would have committed; too high delays process exit.
- A request task wedged in native blocking code (not at an .await) cannot observe RequestCancel cooperatively; the registry's direct hard-drain of store.ledger() runs the rollback on the SHUTDOWN task, but the wedged task still holds an Arc clone — memory-sound but the rollback races the wedged task's own (never-reached) finalize. The OnceCell finalize-once makes this deterministic (first writer wins) but the wedged task's connection is only reclaimed by the pool-timeout fallback. Needs the cancel-safety audit declarative-advice line-171 flagged.
- Nested request-in-session scopes: the registry tracks the outermost (request) scope; if a SESSION scope outlives many requests and is itself in the registry, drain ordering (session ledger after all its request ledgers) relies on leaf-first walk — correct by construction but untested for >2 nesting levels.
- Prototype async-teardown remains structurally absent (prototype pushes no destroyer); a stateful prototype holding an open resource at shutdown is a documented leaf doctor build-warning, not covered by the in-flight drain (request/session only) — unchanged from C1 doc.
- If StartupValidation::Skip is authorized by a cargo-leaf-prepare AOT plan, the R2 request-scope-opener-absent WARN still runs (it is a refresh-time doctor check, not a validate() pass) — confirmed not elided, but the interaction with a Skip plan that omitted the web crate's RequestScopeRegistry contributor is untested.
