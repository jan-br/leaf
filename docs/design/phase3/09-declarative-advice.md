# Declarative Cross-Cutting Advice  `[declarative-advice]`

_Every advice concern as a flat Interceptor-chain entry ordered by cmp_chain (RoleTier-first composite over the unchanged cmp_order; SEAMS C6): transaction demarcation, caching, method/bean validation, retry/resilience, @Async dispatch, persistence-exception translation, and transaction-phase-bound event listeners._

Every cross-cutting concern leaf ships — transaction demarcation, caching, method/bean validation, retry & concurrency-limit, @Async dispatch, persistence-exception translation, and transaction-phase-bound event listeners — is re-unified to ONE shape: a flat const `AdvisorDescriptor` row collected into the shared `ADVISORS` linkme slice, lowering to ONE `Arc<dyn Interceptor>` placed into a per-bean `AdviceChain` ordered by the single `cmp_chain` comparator (the RoleTier-first composite over the unchanged `cmp_order`; SEAMS C6 — all of these are `Role::Infrastructure`). There is NO second weaving mechanism, NO bespoke pipeline, NO feature-private registry: each concern is exactly the `proxy-substrate`'s replayable/skippable `Next::proceed()` primitive plus a small concern-specific service trait resolved through the ordinary `Provider`. The chain's correctness-grade ordering (validation → retry → @Async → cache → tx → translation innermost, with cache OUTSIDE tx so a hit avoids opening a tx and @Async positioned so a tx never spans the spawn hop) is the pinned const data on each descriptor (`order: i32`, `CACHE_ORDER=400 < TX_ORDER=500`) and inspectable via `Container::advice_chain_of` + `leaf doctor`. Transactional-events is the one member that is NOT an interceptor: it is a `ListenerDescriptor` carrying a `defer: Option<TxDeferral>` whose dispatch reroutes through the transaction interceptor's phase-synchronization registry — proving the same `Cx`-bound tx-resource primitive serves both the tx interceptor and the deferred listener. The unification thesis: tx owns the one tx-resource `Cx` key and the one synchronization protocol; everything else (cache short-circuit, retry replay, validation short-circuit, translation map-err, @Async detach, after-commit deferral) is a thin consumer of toolkit primitives that already exist.

## Shared machinery (how this rests on the toolkit)

The whole subsystem mints almost nothing; it rests on six fixed toolkit primitives and the unification is that all seven concerns share them rather than each owning a mechanism.

1. ONE call-routing primitive (proxy-substrate). Every concern is an `Interceptor` in the per-bean `AdviceChain`; its body wraps `next.proceed()`. The two hard requirements the substrate already built in are exactly the two these features need: `Next::proceed()` is REPLAYABLE (retry calls it N times — each a fresh future) and SKIPPABLE (a cache hit returns `Ok(ret)` without calling proceed). Validation short-circuits by returning `Err` before proceed; translation maps the `Err` of proceed; @Async detaches proceed onto the `Spawner`. No concern re-implements chain assembly, matching, or short-circuit.

2. ONE flat-descriptor discovery + anti-DCE defense (proxy-substrate, cross-crate-discovery). Each concern contributes exactly one (retry contributes two) const `AdvisorDescriptor { id: ContractId, order: i32, role: Role::Infrastructure, pointcut, condition: Option<&dyn Condition>, make_interceptor }` into the shared `ADVISORS` linkme slice via `::leaf_core` paths. `make_interceptor(&Container)` resolves the concern's service bean (`dyn TransactionManager`, `dyn CacheManager`, the translator chain) through the ordinary `Provider`, so advice injects collaborators without being a bean. The mandatory expected-vs-found self-check covers each advisor stream AND each execution facility: a DCE'd tx/cache/resilience/translation crate is a loud `AntiDceError::SourceVanished`, never a silent no-op (no silently-untranslated error, no silently-sync @Async, no silently-disabled @Cacheable). `enable_*!()`/auto-config emits the force-link reference + `ExpectedManifest` entry.

3. ONE comparator + frozen plan (injection-mechanics order module, proxy-substrate ProxyPlan). The chain sorts by `cmp_chain(&ChainKey,&ChainKey)` at CORRECTNESS grade — the canonical `leaf_core::order::cmp_chain` (RoleTier first, then the unchanged pure `cmp_order`, then `ContractId`; SEAMS C6), no second comparator. The cache-OUTSIDE-tx / translation-innermost ordering is the pinned `order: i32` const data (`CACHE_ORDER < TX_ORDER`), resolved once into the frozen `ProxyPlan` at `seal()`, decoration is an O(1) `BeanId` lookup. Activation is gated by the descriptor's `condition: CondExpr` evaluated at `App<Resolve>` (the opt-in-by-default `OnMissingBean` + `OnProperty` ladder is the standard auto-config gate; absent in the bare engine).

4. ONE ambient substrate with per-key Propagation (async-context-model Cx). The transaction resource lives in ONE `TxResourceKey: CxKey { POLICY = Isolate }` — never auto-captured across a spawn hop, realizing Spring's never-inheritable rule as a typed default. Nested data-access reads the ambient `Cx`; @Async's `CxDecorator` captures only `Inherit` keys (locale/trace), structurally excluding the tx resource. Transactional-events reads the same `TxResourceKey` to find the active tx's synchronization registry. This is the single place "the current transaction" lives — tx interceptor, nested DAO, and deferred listener all read one bundle, no ThreadLocal, no `AopContext.currentProxy()`.

5. ONE cancel-safety discipline, no async Drop (async-context-model 5d, proxy-substrate, C7). Around-advice releases sync-droppable resources on the cancel path via RAII `Drop` (the `ConcurrencyGate` `Permit`, a Cx scope guard, a "must-rollback" flag); genuinely-async finalization is an awaited ledger entry or explicit `.await` in the interceptor body, never Drop. C7 PINS WHICH LEDGER: a tx commit/rollback rides a `TxFinalizeEntry` pushed onto the PER-REQUEST `InstanceStore` ledger (the `RequestScopeGuard`'s ledger, reached via `InstanceStore::ledger()`), NOT the container TeardownLedger — so on SIGTERM mid-request the 13-container-lifecycle teardown step-2 in-flight-request drain (bounded `grace`, then cooperative-cancel + per-request-ledger drain under `finalize_grace`) runs the open tx's awaited rollback; cache write / multicaster quiesce / scheduler drain ride the container `TeardownLedger` drained at step-5. A future dropped mid-tx sets the rollback flag synchronously (sync Drop); the actual async rollback is the `TxFinalizeEntry` driven by the in-flight drain, with the pool-timeout fallback for a truly-abandoned connection. `TxFinalizeEntry` is finalize-once (an `OnceCell` guards double commit/rollback).

6. ONE error/diagnostic spine (error-model). All faults are `LeafError` nodes: rollback-rule matching, retry classification, and translation all match over `ErrorKind` / downcast over the typed `chain`. `DataAccessError` is `ErrorKind::Integration { kind_id: ContractId }` extended BY DATA (no `dyn DataAccessError` trait family, no core ABI bump per vendor). Detached @Async / after-commit listener panics route to the container `AsyncUncaughtFailureHandler` and enter via the open `Integration` arm. Mandatory-trait (service-trait-required-for-advised-bean) violations are Tier-0 `#[diagnostic::on_unimplemented]`; a dangling tx-manager qualifier or malformed cron is a Tier-2 `AssemblyReport` entry; the assembled chain order is a §2.9 inspectable diagnostic.

## Features

### `transaction-management`
ONE `AdvisorDescriptor` (Role::Infrastructure, order = `TX_ORDER=500`, INSIDE cache so a cache hit short-circuits before a tx opens — `CACHE_ORDER=400 < TX_ORDER`, SEAMS C6) whose `make_interceptor` resolves `Ref<dyn TransactionManager>` (by the `#[transactional(manager="…")]` qualifier when present, else primary). `#[transactional]` is a THIN attribute macro emitting one `const TxAttribute { propagation: Propagation, isolation: Isolation, read_only: bool, timeout: Option<Duration>, rollback_on: &'static [ErrorMatch], no_rollback_on: &'static [ErrorMatch], manager: Option<&'static str> }` plus the `AnnotationMetadata` presence flag the pointcut matches on. ONE async-first trait collapses Platform+Reactive: `trait TransactionManager: Send + Sync { fn begin(&self, def: &TxDefinition, cx: &ResolveCtx) -> BoxFuture<Result<TxState, LeafError>>; fn commit(&self, st: TxState) -> BoxFuture<Result<(), LeafError>>; fn rollback(&self, st: TxState) -> BoxFuture<Result<(), LeafError>>; fn synchronizations(&self, st: &TxState) -> &TxSyncRegistry; }`. RESOURCE BINDING is the crux and is resolved onto the toolkit: `begin()` installs the resource via a `TxResourceKey: CxKey { type Value = TxState; const NAME="tx"; const POLICY = Propagation::Isolate }` — the interceptor wraps `next.proceed()` in `Scoped` so the binding re-installs per poll and survives work-stealing; nested data-access calls `Cx::current()?.get::<TxResourceKey>()`. REQUIRES_NEW shadows via `Cx::with::<TxResourceKey>(new)` (one-directional child); the outer is structurally suspended (not the innermost value) and the prior `Cx` restores on the sync `Drop` of a stack guard. The interceptor body: read TxAttribute → begin/join per propagation → push a finalize-once `TxFinalizeEntry` onto the PER-REQUEST `InstanceStore` ledger (`RequestScopeGuard::ledger()`, the C7 ledger drained by 13-container-lifecycle teardown step-2's in-flight drain under `BootstrapSettings.shutdown.finalize_grace`) → `next.proceed().await` → on `Ok` commit (firing BEFORE_COMMIT then AFTER_COMMIT syncs), on `Err` matching a rollback rule rollback (firing AFTER_ROLLBACK), always AFTER_COMPLETION; the `TxFinalizeEntry`'s `OnceCell` marks the tx finalized so the shutdown drain does not double-finalize a tx the body already completed. A `Drop` guard flags un-finalized state for the cancel path; the explicit async rollback is driven either by the interceptor body OR — on SIGTERM mid-request — by the in-flight-drain step running the still-armed `TxFinalizeEntry`, with pool-timeout as the safe fallback (no async Drop).

**Resolved open questions:** Resource binding primitive: it IS the async-context-model `Cx` with a `TxResourceKey` whose `POLICY = Isolate` — not a ThreadLocal, not explicit threading, not a new scope SPI. Nested data-access looks up the ambient Cx; the never-inheritable rule is the typed Isolate default. R1's ambient model wins, realized on the fixed Cx primitive.; Cancellation without async Drop: explicit interceptor-driven rollback on the Err/early-return path + a sync-Drop 'must-rollback' flag + connection-pool-timeout reclaim for the truly-dropped-at-await case. The async finalization is never on Drop; it is an awaited step in the interceptor or a TeardownLedger entry — the ONE cancellation contract shared with retry/context-close.; Demarcation locus: per-bean-global at the frozen ProxyPlan (every external caller advised; self-invocation bypasses, diagnosed) — uniform with all advisors, no per-edge variant.; Rollback rules over Result: `ErrorMatch` matches over `ErrorKind` + downcast over the LeafError `chain` (reusing the error-model matching vocabulary). Default = any `Err` rolls back unless `no_rollback_on` matches. Panics are NOT a rollback channel (no catch_unwind).; One async-first TransactionManager trait (Platform+Reactive collapsed); the manager binds the resource inside begin() into the TxResourceKey, the interceptor drives commit/rollback/suspend and fires synchronizations.; Runtime manager-selection (R1), NOT a build-time TxPlan — it is condition/profile-friendly and the make_interceptor resolution rides the ordinary Provider; the typed-Tx-effect (R3-B) is dropped as viral and signature-changing (cuts DX#1).; Send/Sync: TxState rides a spawnable future so it is Send+Sync+'static (the uniform bound on ErasedBean); a !Send driver is a compile-tier diagnostic, no ?Send carve-out.

### `caching`
ONE `AdvisorDescriptor` (Role::Infrastructure, order = CACHE_ORDER, sorted OUTSIDE tx by default so a hit avoids opening a tx — the canonical line-1772 correctness concern, data-driven and inspectable) whose `make_interceptor` resolves `Ref<dyn CacheManager>`. Thin macros `#[cacheable]`/`#[cache_put]`/`#[cache_evict]` emit `const CacheOpMeta { op: CacheOp, names: &'static [&'static str], key_fn: fn(&ErasedArgs)->CacheKey, condition: Option<fn(&ErasedArgs)->bool>, unless: Option<fn(&ErasedRet)->bool>, sync: bool }` — key/condition/unless are typed closures lowered from typed args (no string interpreter; the expression-language neighbour's typed-closure realization). The interceptor: build key via the generated pack/unpack glue (typed, not per-arg box), eval condition; `@Cacheable` does `cache.get(key).await` and on HIT SKIPS proceed (returns `Ok(stored)` — the substrate short-circuit), on MISS `next.proceed().await`, eval unless, `cache.put`. Value erasure is resolved by keying the typed store on `(MethodKey, CacheKey)` and storing `Box<dyn Any+Send+Sync>` whose `TypeId` is checked at read (a mismatch is a `LeafError`, structurally prevented by the MethodKey in the key). `trait Cache: Send+Sync { fn get(&self,k:&CacheKey)->BoxFuture<Result<Option<StoredValue>,LeafError>>; fn put(..)->BoxFuture<..>; fn evict(..); fn clear(..); }`; `trait CacheManager { fn cache(&self,name:&str)->Option<Arc<dyn Cache>>; }`. `sync=true` single-flight is a per-cache in-flight map of shared (waker-driven) futures — NOT polling; a cancelled waiter does not cancel the computation (the computer owns the future, waiters hold a shared handle); if the computer is cancelled the next waiter is promoted.

**Resolved open questions:** Realization: RUNTIME interceptor over the shared chain (R1), NOT compile-time woven (R2) nor injected handle (R3) — so cache/tx ordering is chain-arbitrated by cmp_order and inspectable, and self-invocation behaves identically to tx (the consistency the cluster demands).; Shared named cache vs typed values: ONE heterogeneous physical store keyed by (MethodKey, CacheKey) with a TypeId-checked `Box<dyn Any+Send+Sync>` — cross-method `@CacheEvict` under the same name+key works because the store is one physical map per name; the MethodKey component prevents a wrong-type downcast.; Key/condition/unless: TYPED closures over real method args (compile-checked, erasure-free), reached through the proxy-substrate's per-method typed pack/unpack glue — `#result` for unless/key-on-result is the typed ErasedRet after proceed. No interpreter on the key path.; Single-flight (sync=true): a waker-driven shared-future in-flight map; cancelling one waiter never cancels the shared computation; the computer owns the future; if the computer is cancelled the next waiter is promoted or the wait fails with a LeafError. Shared structured-concurrency point with retry, solved once here.; Self-invocation: REPLICATED (consistent with tx), documented + diagnosed, never silently eliminated.; Origin-agnostic backends: `Arc<dyn CacheManager>`/`Arc<dyn Cache>` are ordinary beans (in-memory default, Caffeine/Redis integration crate); the anti-DCE self-check covers the caching advisor so a never-linked cache crate is a loud diagnostic, never a silent disable.; Value lifetime/eviction: Arc-shared StoredValue; TTL/LRU is the backend's concern; CacheErrorHandler policy (swallow-and-fall-through-to-proceed vs fail-fast) is a knob, a cache I/O error never corrupts the method result path.

### `validation`
TWO faces over ONE engine. ENGINE: `trait Validate { fn validate(&self, cx: &mut ValidationContext); }` — object-safe, SYNC, CPU-bound, collect-all (pushes `Violation { path, constraint_id, message_key, params, rejected_value_repr }` into an accumulator, never fail-first). `#[derive(Validate)]` is thin: per-field constraint checks + `#[validate(nested)]` → recursive `child.validate(cx.enter(seg))` for the @Valid cascade with a visited-set cycle guard; built-in constraints are plain `fn(&T,&Params)->Option<Violation>` in leaf-core; custom constraints are path-referenced fns; cross-field is `#[validate(with=fn)]` over `&Self`. FACE 2 (method validation): ONE `AdvisorDescriptor` (Role::Infrastructure, order = VALIDATE_ORDER, OUTERMOST so bad args never reach tx/cache) whose interceptor validates `#[valid]` args before `next.proceed()` (SHORT-CIRCUITS with `Err(ConstraintViolations)` — the real method never runs) and the return after; it reaches concrete arg types through the proxy-substrate's typed pack/unpack glue, dispatching `Validate::validate` directly (or a `dyn Validate` upcall at the erased boundary). FACE 3 (config binding): the binder calls the SAME `Validate::validate` after binding a typed config tree and maps each `Violation.path` back to the canonical property KEY + Origin, pushing into the SAME `ErrorSink` accumulator the binder/conversion use, aggregated into the Tier-2 `AssemblyReport` (fail fast at boot). Messages store key+params and resolve LATER at error-render time against messages-i18n + the ambient locale `Cx` key — keeping the sync validate path free of a bean lookup.

**Resolved open questions:** Where constraints live: COMPILED PER TYPE (R1) as a generated `Validate` impl that travels with the type — zero link-time collection, zero DCE hazard, generics-friendly (monomorphized per use). Only the method-validation ADVISOR rides the shared ADVISORS slice (its DCE risk covered by the standard self-check). The descriptor-registry model (R2) is rejected for the engine.; Constraint vocabulary: compiled-in, path-referenced checkers (R1) — reliable cross-crate, generics-friendly; a constraint must be named at the derive site (no runtime 'discover all constraints' query), with `with=fn` as the dynamic escape.; Validation vs conversion: SEPARATE pass over an already-converted value (Spring's deliberate separation), NOT fused into coercion — but both faces push into the ONE shared `ErrorSink` collect-all accumulator (the unification §2.11 win), so 'validate without converting' stays answerable.; Type info to the method-validation interceptor: the proxy-substrate's generated per-method typed pack/unpack glue delivers `&ConcreteArg`, so `Validate::validate` is called directly on the typed path; the erased `ErasedProxy` lane uses a `dyn Validate` upcall. Downstream of proxy-substrate, now concretely resolved.; Message resolution: DEFERRED to error-formatting time (store message_key+params, resolve when rendering against messages-i18n + the locale Cx key) — the sync validate hot path never touches a bean.; Collect-all accumulator: it IS the error-model `ErrorSink` shared by binder + conversion + validation, not a bespoke set — one 'collect all the things wrong across this typed boundary' primitive aggregated into AssemblyReport.; Property-path mapping: the cascade builds a Rust-field path; the config face maps it through relaxed-binding's key model + origin-tracking to the kebab-case key + Origin in the LeafError node.; Groups/group-sequences: deliberately dropped in v1 (awkward in the compile-everything derive); flagged as a remaining risk, not built.

### `retry-resilience`
TWO `AdvisorDescriptor`s (both Role::Infrastructure) over shared primitives, plus an imperative primitive. PRIMITIVE: `RetryTemplate { policy: RetryPolicy, backoff: Arc<dyn BackoffPolicy> }` with `fn execute<T>(&self, op: impl FnMut()->BoxFuture<'_,Result<T,LeafError>>) -> BoxFuture<'_,Result<T,LeafError>>` — a pure async loop that on a retryable error awaits `backoff.next_delay(attempt)` on the ONE shared reactive timer-wheel (the sanctioned cold-path poll, never busy-wait), exhausting into the last error or a recovery fn; usable container-free (the self-invocation escape hatch). `RetryPolicy { max_attempts, is_retryable: fn(&LeafError)->bool }`; `BackoffPolicy` impls Fixed/Exponential{base,mult,max,jitter}/None. RETRY ADVISOR (order = RETRY_ORDER, inside validation, outside tx — so each attempt is its own tx) wraps `RetryTemplate::execute(|| next.proceed())` — relying on the substrate's REPLAYABLE `Next` (each proceed() is a fresh future). CONCURRENCY-LIMIT ADVISOR (`#[concurrency_limit(n)]`) holds an `Arc<dyn ConcurrencyGate>` sized from the attribute (reusing async-context-model's gate — NOT a new limiter); its body: `let _permit = gate.acquire().await; next.proceed().await` — the `Permit`'s sync RAII `Drop` releases on completion AND cancellation (no leak, no async Drop); n=1 is the declarative instance lock. Error classification matches over `ErrorKind` + downcast over the typed chain.

**Resolved open questions:** Shape: TWO bespoke interceptors (R1) over a standalone RetryTemplate + the shared ConcurrencyGate — intent-faithful and independently interleavable with tx/cache by cmp_order. The unified layer-pipeline (R2) and policy-as-data (R3) are deferred; R2's extra layers (circuit-breaker/bulkhead/rate-limit) are a post-v1 growth path, not the default.; Error classification locus: a typed predicate `is_retryable(&LeafError)` matching over `ErrorKind` + downcast over the chain (the error-model vocabulary) — the erased seam still carries enough structure because the chain is the typed LeafError, not a bare `dyn Error`.; proceed() replayability: REQUIRED and already guaranteed by proxy-substrate's `Next` being REPLAYABLE (FnMut → fresh future). Retry is the named consumer that forced this into the substrate contract; resolved upstream, consumed here.; Concurrency-ceiling home: it IS async-context-model's `ConcurrencyGate` (Semaphore-backed bounded Permit with RAII release) — the ONE primitive shared by task-execution, @ConcurrencyLimit, and scheduled-fire overlap, not solved three times.; Permit across backoff sleeps: RE-ACQUIRED per attempt by default (the limit advisor sits OUTSIDE retry in the chain only if so configured; default order places concurrency-limit innermost-of-the-two so the permit is held per-attempt, not across all sleeps) — avoiding budget waste during backoff; the relative order of the two resilience advisors is `order:i32` data, inspectable.; Cancellation/drain: the cooperative-cancel + RAII-Permit-release contract — a dropped retrying/gated future abandons the loop and releases the permit on sync Drop; the ONE framework cancellation contract shared with tx/context-close/scheduling.; Anti-DCE: `enable_resilient_methods!()` force-links the resilience crate and the mandatory expected-vs-found self-check asserts BOTH advisor descriptors present, so a binary can never silently ship with resilience disabled.; Imperative-vs-declarative: RetryTemplate is a first-class container-free primitive (the annotation is thin sugar driving it); the monomorphized zero-box path is available off the AOP path as the Phase-3 escape.

### `async-execution`
@Async is ONE `AdvisorDescriptor` (Role::Infrastructure, ASYNC_ORDER positioned OUTSIDE tx so a transaction never spans the spawn hop — the deterministic-sort correctness the body-rewrite sacrifices) — explicitly a chain advisor, NOT a body-rewrite (rejected as default to keep self-invocation semantics uniform with @Transactional). `#[async_]` emits `const AsyncMeta { qualifier: Option<&'static str>, drop_policy: DropPolicy }`; `make_interceptor` resolves `Ref<ExecutionFacility>` (the primary Role::Infrastructure executor bean, by qualifier when present). The interceptor body, for a fire-and-forget method (returns `()`/`Result<(),_>`): capture `Cx` via `CxDecorator` (Inherit keys only — locale/trace ride; the `TxResourceKey` is Isolate and is structurally NOT captured, so a spawned task cannot inherit a tx connection and shatter isolation), then `spawner.spawn(proceed().scoped(captured_cx))` with `DropPolicy::Detach` and return immediately; for a future-returning method, return a handle. Detached-task panics route to the container `AsyncUncaughtFailureHandler` and enter the LeafError chain via the open `Integration{ContractId}` arm. SCHEDULING (`#[scheduled(cron=…|fixed_rate=…|fixed_delay=…, overlap=…, qualifier=…)]`) is a time-aware capability over the SAME `Spawner`, not a second executor: it emits a `ScheduledMethodDescriptor` into the `SCHEDULED` slice; the one reactive `SchedulerCore` (Role::Infrastructure) sleeps_until the earliest `Trigger::next_fire` and on wake spawns the body onto the Spawner, with `OverlapPolicy::SkipIfRunning` default enforced by a per-task `ConcurrencyGate` permit (limit-1). A missing facility is a HARD-FAIL refresh self-check (an async-first framework cannot run without it); a @Scheduled descriptor with no facility is the same anti-DCE expected-vs-found failure, never a silent no-fire.

**Resolved open questions:** @Async as chain advisor vs body-rewrite: CHAIN ADVISOR (uniform self-invocation across all advisors; ordering vs @Transactional governed by the one cmp_order sort so a tx never spans the async hop). The body-rewrite+chain-marker hybrid is DEFERRED to Phase 4 as an opt-in, never the default.; Cx across the spawn hop: only `Inherit` keys are captured by the `CxDecorator` at submit; the `TxResourceKey` (Isolate) is structurally excluded — the never-inheritable rule as a typed default, not a per-spawn discipline.; AsyncUncaughtFailureHandler granularity: single container-level sink routing into `ErrorKind::Integration{ContractId}` — composes with the unified error model, not a parallel handler per executor.; Executor shape: capability-split `Spawner`/`BlockingOffload`/`ConcurrencyGate` under one `ExecutionFacility` Role::Infrastructure bean, runtime-agnostic in core (leaf-tokio/leaf-smol impls), selected by primary/qualifier — consumed unchanged from async-context-model.; Scheduling: a Trigger SPI + one reactive SchedulerCore over the SAME Spawner (sleep_until+waker, never a spin); overlap via a shared ConcurrencyGate permit. Cron calendar dialect/DST and one-global-vs-per-scheduler timer-wheel are flagged remaining risks.; Bounds: uniform Send+'static on spawned work, steered by the `SpawnableWork` `#[diagnostic::on_unimplemented]`; no ?Send carve-out in core.; Detached lifecycle: DropPolicy::Detach for fire-and-forget; detached tasks are drained at shutdown via the TeardownLedger after stop_all(), not cancelled on caller-drop; documented that a task outliving the drain window is aborted.

### `exception-translation`
ONE `AdvisorDescriptor` (Role::Infrastructure, TRANSLATE_ORDER = INNERMOST so it catches the raw native error before a tx advisor re-wraps it). Pointcut = the repository-boundary stereotype marker (`#[repository]`/pluggable `#[dao]`) read as `AnnotationMetadata` presence — the natural pointcut, advising only boundary beans. The interceptor is a thin after-throwing mapper: `match next.proceed().await { Ok(v)=>Ok(v), Err(e)=>Err(self.translate(e)) }` — success passes through, only the Err path runs the chain (pure sync CPU, no extra await, no held resource → no cancel-safety burden). `make_interceptor` resolves the ordered chain `Box<[Arc<dyn DataAccessExceptionTranslator>]>` via collection-injection (`@Order`-sorted by cmp_order). `trait DataAccessExceptionTranslator: Send+Sync { fn translate(&self, e: &LeafError) -> Option<LeafError>; }` returning `None` = pass-to-next (Spring's null-is-pass), first `Some` wins, the original is ALWAYS chained as a `Cause` (never lost). The hierarchy is NOT a new error family: `DataAccessError` IS `ErrorKind::Integration { kind_id: ContractId }` carrying a structured `kind` discriminant (DuplicateKey/OptimisticLocking/DataIntegrity/Transient/NonTransient/Recoverable) in its data payload — a vendor crate (leaf-sqlx/leaf-diesel) contributes its taxonomy + translator by DATA, no `dyn DataAccessError` trait family, no core ABI bump. A translator that needs the typed native error downcasts over the LeafError chain (degrading to 'pass' on a wrong-type downcast, with a diagnostic, never swallowing). Opt-in-by-default via auto-config (`OnMissingBean` + `OnProperty` default-on + `OnBean` of a persistence integration); absent in the bare engine.

**Resolved open questions:** Ride the AOP chain vs codegen map_err: RIDE THE SHARED CHAIN (R1) — so translation composes/orders with tx on one proxy (the 'same proxy carries both' property) and is inspectable; the codegen-map_err (R2) is rejected as the default because it forks the weaving story and loses the tx compose, though `IntoDataAccess` impls remain a usable convenience for vendor crates.; Error matching: a translator receives the LeafError and downcasts over its typed `chain` to reach the concrete native error (the error-model carries the typed chain, so the seam is not a bare `dyn Error`); a wrong-type downcast passes with a diagnostic, never silently mis-translates.; Translator set: RUNTIME-PLUGGABLE via collection-injection of `dyn DataAccessExceptionTranslator` beans (Spring parity, any scope, @Order via cmp_order), covered by the mandatory anti-DCE expected-vs-found self-check so a never-linked vendor translator crate is a LOUD diagnostic — fixing the 'silently untranslated, looks like it worked' failure mode that is worse than a missing bean.; Silent-DCE surfacing: the SAME expected-vs-found self-check on the translators stream (an anti-DCE check, not a parallel system); `leaf doctor` lists 'discovered translators, N repository beans advised'.; Service-trait requirement: a @Repository bean MUST expose a service trait to be advised (the no-CGLIB consequence) — diagnosed at Tier-0 `#[diagnostic::on_unimplemented]`; a concrete-only DAO is injectable but un-advisable, a named divergence.; Order relative to tx: translation is INNERMOST (TRANSLATE_ORDER closest to the real method) so it maps the raw native error before any tx rollback re-wraps it — `order:i32` data, inspectable via advice_chain_of.; DataAccessError shape: `ErrorKind::Integration{ContractId}` extended BY DATA with a structured kind discriminant — open/extensible across crates without a core ABI bump and without a downcast-heavy dyn trait family; resolved jointly with the error-model.

### `transactional-events`
NOT an interceptor — the ONE member that rides the events listener-adapter layer, proving the multicaster is extensible. `#[transactional_event_listener(phase=…, fallback=…)]` is a thin meta-specialization of `#[event_listener]`: it emits the SAME `ListenerDescriptor` PLUS `defer: Option<TxDeferral { phase: TxPhase, fallback: bool }>` (TxPhase ∈ BeforeCommit|AfterCommit(default)|AfterRollback|AfterCompletion). It lands in the SAME EVENT_LISTENERS slice / typed-or-erased channel the events subsystem built — no new registry, no new bus. Deferral is realized at DISPATCH by the `CoreDispatch` step: for a listener whose descriptor has `defer=Some`, instead of awaiting inline it reads the ambient `Cx::current()?.get::<TxResourceKey>()` (the SAME tx-resource key the tx interceptor installs). (a) ACTIVE TX → it does NOT invoke; it registers a callback on the tx's `TxSyncRegistry` (transaction-management's synchronization protocol), capturing the `ErasedEvent` (Arc'd, Send+'static) + the listener adapter + the phase; when the tx interceptor reaches that phase boundary (BEFORE_COMMIT before its commit .await; AFTER_COMMIT only on success; AFTER_ROLLBACK only on rollback; AFTER_COMPLETION always) it fires the registered callbacks on the SAME logical task that owns the tx. (b) NO ACTIVE TX → `fallback=true` runs inline now; `fallback=false` SKIPS but emits a structured diagnostic-grade record ('transactional listener X skipped: no active transaction; set fallback=true to run regardless') — making Spring's §1897 silent no-op LOUD. Per-phase error contract: a BEFORE_COMMIT listener `Err` feeds the tx outcome (can abort the commit); AFTER_* errors are isolated (cannot un-commit) and route to the multicaster's `ErrorIsolation`/AsyncUncaughtFailureHandler. Defer-then-spawn: an AFTER_COMMIT listener marked async spawns at firing (the tx is already committed, so losing the tx Isolate context is correct; Inherit keys ride via CxDecorator).

**Resolved open questions:** Where the deferral binds: a synchronization callback registered on the active tx's `TxSyncRegistry` (R1), found via the ambient `TxResourceKey` Cx key — NOT a tx-owned buffer (R3), because the tx interceptor already exposes the synchronization protocol and this needs no change to the Transaction handle shape. The outbox-buffer (R3) is a flagged future variant.; Hard-coded branch vs DispatchPolicy SPI: a hard-coded `defer` branch in CoreDispatch (R1), NOT a pluggable DispatchPolicy SPI (R2) — the multicaster is ALREADY the proxy-substrate Interceptor chain (AsyncDispatch/ErrorIsolation/ContextPropagation entries), so async-per-listener is an existing chain entry and tx-deferral is one inline branch over the `defer` descriptor field; no second extensibility SPI, no hot-path policy consultation.; Loud no-tx: the skip path emits a structured LeafError-grade diagnostic record (distinct from a DCE-dropped descriptor: the events anti-DCE self-check catches 'descriptor missing', this catches 'descriptor present but no active tx'); a startup self-check WARNs 'transactional listeners exist but transaction-management is not wired'.; Per-phase error/cancellation: BEFORE_COMMIT Err is consumed before the manager's commit .await and can abort; AFTER_* errors are isolated and routed to the handler; a deferred listener whose tx is cancelled before its phase is reclaimed by dropping the captured event value (sync Drop, no async Drop). The ONE cancellation contract shared with tx/retry.; Condition + captured event timing: the event value is snapshotted at publish/enqueue time (Arc'd into the registration); the @EventListener condition is evaluated at publish time over the publish-time ambient Cx — documented so the deferred fire sees a consistent captured snapshot, not the (different) phase-boundary context.; Chaining from a deferred listener: DISABLED for AFTER_* phases (the original tx is finalized, so a chained event would have no active tx) — enforced as a macro hard-error / startup-pass contract, consistent with the events neighbour's 'chaining disabled on async listeners'.; Reactive-tx universality: Spring 6.1's special-case 'carry tx context in the event' is leaf's universal default — the deferred listener fires on the same logical task that owns the task-scoped tx (TxResourceKey), so no thread-bound synchronization manager is needed.; Persisted outbox: flagged as an ecosystem-integration concern layered on the same AFTER_COMMIT registration, NOT core v1.

## Cross-feature interactions
- **proxy-interception (upstream)** — ALL seven concerns are consumers of this substrate. Six are `Interceptor` chain entries placed by the frozen `ProxyPlan`; their order is the pinned `order:i32` const data sorted by the one `cmp_chain` (RoleTier-first composite over the unchanged `cmp_order`; SEAMS C6) at correctness grade. They depend on the two named substrate guarantees: `Next::proceed()` REPLAYABLE (retry) and SKIPPABLE (cache hit, validation short-circuit). Each emits a flat `AdvisorDescriptor` with a `make_interceptor` fn that resolves its service bean. Transactional-events does NOT add an interceptor — it rides the multicaster, which is ITSELF a proxy-substrate Interceptor chain (so async-per-listener is an existing chain entry). The service-trait-mandatory rule applies to every advised bean here (tx/cache/validation/translation targets).
- **execution-context (upstream)** — The single most-shared seam. The ONE `TxResourceKey: CxKey{POLICY=Isolate}` is installed by the transaction interceptor, read by nested data-access, and read by the transactional-events dispatch branch to find the active tx — three consumers, one key. @Async's CxDecorator captures only Inherit keys so the tx resource is structurally never inherited across a spawn. Retry's @ConcurrencyLimit and scheduling's overlap-policy share the ONE `ConcurrencyGate`/`Permit`. @Async and scheduling share the ONE `Spawner`/`ExecutionFacility` and the one reactive timer-wheel (also retry's backoff). All cancel-safety is sync RAII Drop + awaited TeardownLedger entries (tx finalization, scheduler quiesce, multicaster quiesce) — no async Drop anywhere.
- **events (upstream)** — Transactional-events is a `ListenerDescriptor` with an added `defer` field in the SAME EVENT_LISTENERS slice and channel — no new bus. Its dispatch reroutes through transaction-management's TxSyncRegistry. Caching's @CacheEvict is a canonical event consumer (cache eviction as a decoupled reaction). The multicaster's ContextPropagation/AsyncDispatch dispatch concerns are a SIBLING around-advice shape (the events `DispatchInterceptor`, which fans out over N listeners — structurally distinct from the advice `Interceptor` that wraps one Call, per C5), and its ordering uses the SAME `cmp_chain` + RoleTier grade (NOT the same Interceptor type) — so the events area and the advice area share one comparator and one ordering grade.
- **transaction-management <-> exception-translation** — Co-advised on the same @Repository proxy: translation sits INNERMOST (TRANSLATE_ORDER closest to the real method) so it maps the raw native error before the tx interceptor's rollback-rule matching re-wraps it. Both are chain entries; the relative order is data-driven and inspectable via advice_chain_of — the canonical 'one proxy, many concerns, invisible-if-misordered' case made visible.
- **transaction-management <-> caching <-> validation <-> retry <-> async** — The canonical chain order on a single bean (outermost→innermost): VALIDATE (reject bad args before any work) → RETRY (each attempt is a fresh tx) → @ASYNC (positioned so a tx never spans the spawn hop — but typically a method is either async-dispatched OR transactional at a boundary) → CACHE (a hit short-circuits BEFORE a tx is opened) → TX (demarcate) → TRANSLATE (raw native error). All positions are the pinned `leaf_core::order` `order:i32` constants (SEAMS C6: `CACHE_ORDER=400 < TX_ORDER=500`, `TRANSLATE_ORDER=600` innermost), sorted once into the frozen ProxyPlan by `cmp_chain`, surfaced by `Container::advice_chain_of` + `leaf doctor`. Caching OUTSIDE tx (so a hit avoids opening a tx) and translation innermost are the two correctness invariants Spring documents; here they are data (a leaf-core property test asserts `CACHE_ORDER < TX_ORDER`), not convention.
- **transaction-management <-> transactional-events** — Tightest pairing. The tx interceptor OWNS the TxSyncRegistry and the TxResourceKey; the deferred listener registers a phase callback on that registry and is fired by the tx interceptor at the phase boundary on the same logical task. BEFORE_COMMIT listener errors feed back into the tx interceptor's commit-or-rollback decision (consumed before the manager's commit .await); AFTER_* listener errors are isolated. The reactive-tx 'carry context in the event' fix is automatic because the tx is task-scoped via the Cx.
- **conditional-strategy + autoconfig (upstream)** — Every advisor's `condition: Option<&dyn Condition>` is a `CondExpr` over `ConditionId` evaluated at App<Resolve>; the opt-in-by-default gate is the standard `exclude > OnMissingBean back-off > OnProperty default-on > OnBean of an integration` ladder (each enable_*!() emits an auto-config descriptor). This is why each concern is absent in the bare engine and present-but-overridable under leaf-boot — one conditional/auto-config machinery, no feature-private activation.
- **error-model (upstream)** — Rollback-rule matching, retry classification, and translation all match over `ErrorKind` + downcast over the typed LeafError `chain` — one error vocabulary. `DataAccessError` and any vendor taxonomy ride the open `ErrorKind::Integration{ContractId}` arm (by-data, no ABI bump). Detached @Async / after-commit listener panics and the no-tx-skip diagnostic all become LeafError nodes; dangling tx-manager qualifiers and malformed cron aggregate into the Tier-2 AssemblyReport; mandatory-service-trait violations are Tier-0 diagnostics.

## Crate hints
- leaf-core: the ultra-stable ABI homes for the SHARED shapes only — the TransactionManager / Cache / CacheManager / DataAccessExceptionTranslator / Validate traits, TxResourceKey (CxKey), TxPhase / TxSyncRegistry, RetryPolicy / BackoffPolicy, the const attribute structs (TxAttribute / CacheOpMeta / AsyncMeta / TxDeferral), and the ADVISORS / SCHEDULED linkme slices. These are emitted-against workspace-wide (macros hard-code ::leaf_core paths), so freeze early and version conservatively. DataAccessKind rides ErrorKind::Integration (no separate error ABI).
- leaf-tx (feature crate): #[transactional] macro lowering, the TransactionInterceptor (begin/proceed/commit-or-rollback over the TxResourceKey + TxSyncRegistry), propagation logic, rollback-rule matching, and the tx AdvisorDescriptor. force-linked by enable_transaction_management!(). Concrete managers live in integration crates (leaf-sqlx-tx, etc.) as ordinary beans.
- leaf-cache: #[cacheable]/#[cache_put]/#[cache_evict] macros, the CacheInterceptor, the typed (MethodKey,CacheKey) store + single-flight in-flight map, and the in-memory default Cache. Caffeine/Redis backends are separate integration crates contributing Arc<dyn CacheManager> beans.
- leaf-validation: #[derive(Validate)], built-in constraint fns, the recursive ValidationContext driver + cycle guard, the MethodValidationInterceptor, and the binder-side adapter (face 3 reuses the shared ErrorSink). No central registry — impls travel with types.
- leaf-resilience: RetryTemplate/RetryPolicy/BackoffPolicy primitives, RetryInterceptor + ConcurrencyLimitInterceptor (over async-context-model's ConcurrencyGate), and enable_resilient_methods!() with the mandatory two-advisor self-check.
- leaf-tokio / leaf-smol (consumed, not authored here): provide the ExecutionFacility + AmbientStore + SchedulerCore + reactive timer-wheel the @Async / scheduling / retry-backoff paths run on. HARD-FAIL refresh self-check if no primary facility.
- leaf-events (consumed + one extension): the ListenerDescriptor gains the `defer` field and CoreDispatch gains the tx-deferral branch; #[transactional_event_listener] is a thin macro in leaf-tx that emits the events row. No new bus crate.
- leaf-boot: hosts the opt-in-by-default auto-config descriptors (the OnMissingBean+OnProperty gates) that install each concern's advisor, the CreatorPolicy admitting Role::Infrastructure advisors, and the leaf-doctor surfaces (advice_chain_of, discovered_translators, facility-present checks).
- leaf-macros: all attribute macros are THIN — each emits one const descriptor + (for cache/validation) typed closures + the make_interceptor fn-ptr via absolute ::leaf_core paths; cargo-expand-legible; generics hard-error with a register_proxy!(Concrete) / per-instantiation shim.

## Remaining risks (→ Phase 4)
- Cancellation-rollback soundness: the sync-Drop 'must-rollback' flag + explicit async rollback + connection-pool-timeout fallback is the agreed shape (no async Drop), but the exact guarantee when a future is dropped mid-commit (.await between flush and commit) — and how the pool-timeout reclaim window composes with the TeardownLedger drain deadline — needs measurement and a precise contract in Phase 4. Shared with context-close.
- ErasedArgs/ErasedRet pack-unpack ABI (inherited from proxy-substrate): cache-key-from-a-param and validation's typed-arg access both need concrete arg types at the seam without a per-arg box on the common path. The exact tuple-thunk vs small-vec-of-Box<dyn Any> representation is unmeasured; affects cache/validation hot-path cost.
- Cron calendar engine: dialect (6/7-field), timezone/DST, and missed-fire semantics (fire-all-missed vs once vs skip after a clock jump), plus whether the cron parse lives in the macro / build.rs / startup pass. The one CPU-bound bit on an otherwise reactive cold path. Feature-local to scheduling.
- One-global-vs-per-scheduler timer-wheel and the fixed_delay completion-feedback composition across the spawn boundary — measured, async-context-model flagged it.
- Single-flight (cache sync=true) async-cancellation: promotion of the next waiter when the computer is cancelled, and the ownership of the shared in-flight future, is a structured-concurrency design point shared with retry; the precise waker/handle protocol needs a tested implementation.
- @Async body-rewrite + chain-marker HYBRID (deferred opt-in): if offered, it must not reintroduce per-advisor self-invocation inconsistency; confirm the chain-marker keeps ordering vs @Transactional governed by the one sort.
- Validation groups / group-sequences are deliberately dropped in v1 (awkward in the compile-everything derive); reassess whether partial group support is worth threading a group selector through ValidationContext + tagging generated checks.
- Persisted transactional-outbox variant (R3's buffer persisted for at-least-once AFTER_COMMIT delivery): a rust-native opportunity aligned with AFTER_COMMIT's durable-side-effect intent, but pulls in serialization/durability — likely an ecosystem-integration concern layered on the same TxSyncRegistry registration, not core; decide ownership.
- Supertrait-emission policy for an advised bean exposing multiple service traits (which trait the chain dispatches through; one proxy implementing all vs one per trait) — couples to registry-substrate's TypeRow-emission sub-question; affects tx/cache/validation on multi-trait repository beans.
- Monomorphized single-advisor fast-path (proxy-substrate deferral): the common 'one advisor, known-concrete site' case skipping the boxed-future hop — measure whether the second codegen path is worth it for the hot cache-hit and method-validation paths.
