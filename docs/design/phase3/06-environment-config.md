# Environment & Externalized Configuration  `[environment-config]`

_The read/mutate-segregated Environment snapshotted at seal_environment, the ordered first-source-wins PropertySource stack with origin tracking, ${...} placeholder resolution, the deterministic single-pass config-data loader, and the source-producing adapters (SAJ, @PropertySource, cloud-platform, CPS view)._

This subsystem is the read/mutate-segregated Environment that is snapshotted by the toolkit's seal_environment() async fence at the App<Define>→App<Resolve> edge, plus everything that fills it: the ordered first-source-wins PropertySource stack with origin tracking, ${...} placeholder resolution, the deterministic single-pass config-data loader, and the source-producing adapters (SAJ, @PropertySource, cloud-platform detection, CPS view). The unification is that ALL of it lives on the value-phase side of a single freeze: the whole stack is mutated synchronously and single-threaded during App<Define>, sealed into one immutable Arc<EnvCore> at seal_environment().await, and read lock-free thereafter through the ownership-model's cheap-clone Env handle that ADR-05 already mandates as an always-present field on Context. There is exactly ONE Environment value, ONE freeze boundary (which IS the toolkit's seal_environment), ONE first-source-wins lookup primitive, ONE placeholder engine, and ONE precedence-rung ladder co-owned by config-data + profiles + auto-config. The read view (PropertyResolver) is what application beans get — handed out as the Env clone-handle; the mutate view (EnvBuilder) is a typestate-consumed builder that exists only inside App<Define>, so a post-seal edit is unrepresentable exactly as App<Resolve> consuming the RegistryBuilder makes a post-freeze register() unrepresentable. The config-data loader is the dominant EnvironmentPostProcessor-analogue: it is the ONLY place real async config IO happens, and it runs entirely inside seal_environment() (the 5f pre-materialization pipeline the compile-runtime-split ADR pins there), materializing every document into an in-memory PropertySource before the env seals. Externalized-config precedence becomes declarative data (a PrecedenceRung total order), the four invariants (first-source-wins, never-blend, null-never-overrides, name-as-identity) are type-level guarantees via Option<PropertyValue>, and provenance (Origin) rides the never-blend short-circuit untouched, surfaced through the one LeafError/Diagnostic path. SAJ, @PropertySource, cloud-platform, and the CPS adapter are all just producers/consumers of the one stack: SAJ and config-data file loaders are ConfigDataLoader impls placing rung-tagged contributions; @PropertySource rides the config-class Descriptor (no new registry) and applies at a fenced low rung; cloud-platform is a frozen-once detection fact feeding both config-data document activation and a CondExpr::OnCloudPlatform leaf; the CPS adapter is the binder-facing relaxed/tri-state view the downstream binder consumes.

## Shared machinery (how this rests on the toolkit)

Every feature rests on the SAME small set of toolkit primitives rather than minting its own:

1. ONE freeze boundary = the toolkit's seal_environment(). compile-runtime-split's ADR fixes that App<Define>::seal_environment(self).await is the 5f async-IO fence: ALL config IO (config-data files, SAJ blob acquisition, cloud-platform env snapshot, remote sources) completes here, the Env snapshots into EnvCore, and the entire App<Resolve>/seal()/App<Wired> chain that follows is synchronous and reads it lock-free. This subsystem does NOT invent a second seal — it IS the body of seal_environment. The EnvBuilder is the mutable env held in App<Define|... and is consumed by the seal_environment transition (the env analogue of App<Resolve> consuming RegistryBuilder); a post-seal source push is type-unrepresentable.

2. ONE handle currency for the read view. The everywhere-shared read handle is the ownership-model's Env = cheap-clone Arc<EnvCore> (ADR-05 names it verbatim as an always-present Context field). EnvCore is Send+Sync+'static so it crosses .await/threads as one Arc clone — riding the same safe-publication contract as ErasedBean, no env-specific lock. PropertyResolver is the least-privilege trait Env implements; beans receive Env by ordinary constructor injection (the aware-callbacks intent), never reflective magic.

3. ONE async/sync fence model from ADR-07. The hot read path (stack.get, placeholder walk, origin render) is synchronous, non-blocking, allocation-light — never .await, never a poll, satisfying ADR-07's reactive-hot rule. The ONLY async is the cold config-data load inside seal_environment, run on the bootstrap ExecutionFacility; a flaky-remote retry/backoff is the sanctioned cold-path reactive timer loop. No async Drop: config-data file handles/dir iterators close via explicit RAII guards inside the bounded cold pass (ADR-07 teardown discipline).

4. ONE diagnostic path. EnvError/ConfigDataError/PlaceholderError/BindOriginError are all ErrorKind arms folded into the one LeafError causal chain + Diagnostic renderer. Env-derived failures (missing required key, unresolved placeholder, missing non-optional config location, malformed JSON, unknown enforced cloud-platform) aggregate into the App<Wired> AssemblyReport tier where env work precedes wiring — the toolkit's four-tier timing ladder (compile / build-or-seal / App<Wired> aggregated / runtime) places malformed ${...} or CondExpr at Tier 0 and a missing required property at the seal/Wired tier.

5. ONE condition algebra for profiles + cloud-platform. Profile activation and cloud-platform document gating are CondExpr leaves (OnProfile, OnCloudPlatform) over the conditional-strategy ADR's runtime tier — the mandatory floor, always evaluated in App<Resolve> AFTER seal_environment, so they read the frozen Env lock-free. config-data's spring.config.activate.on-profile/on-cloud-platform reuse the IDENTICAL CondExpr evaluator config-class @Profile conditions use (one grammar, two phases: documents filtered during the load inside seal_environment; beans filtered in App<Resolve>).

6. ONE cross-crate discovery substrate for source producers. ConfigDataLoaders, SAJ transports, cloud-platform probes (open variant), and StringValueResolver hooks are discovered via the cross-crate-discovery ADR's linkme distributed slices of const descriptors with the triple anti-DCE defense (force-link + toolchain guardrails + ExpectedManifest self-check) — a configtree: prefix with no registered loader is a loud AntiDceError-class diagnostic, never a silent empty stack. Loader selection is by descriptor DATA (the handles predicate), never link order. Config VALUES are runtime data and never link-collected.

7. ONE precedence representation. The full externalized-config ladder is a declarative PrecedenceRung total order co-owned by config-data + profiles + auto-config (auto-config contributes the lowest defaultProperties rung; @PropertySource a bottom-but-one rung; SAJ a below-cmdline rung), applied in a single deterministic plan-then-apply fold inside seal_environment — not emergent add_* insertion order. This is the env analogue of the registry's freeze building all indices in one pass.

8. Origin rides the registry's never-blend short-circuit. PropertyValue { raw, origin } is the stack's value; first-source-wins returns the WHOLE winning value, so the winning source's Origin is the answer with zero extra walk — the same data-shape-makes-it-correct move the registry uses for coherence.

## Features

### `environment`
ONE concrete EnvCore behind the ownership-model Env handle (resolving the R1-vs-R2-vs-R3 open question onto the toolkit: R3's consumed-builder + structural-sharing, but the 'build' IS seal_environment). leaf-core: `#[derive(Clone)] pub struct Env(Arc<EnvCore>); struct EnvCore { stack: SealedStack, profiles: ActiveProfiles, cloud: CloudPlatform, origins: OriginStore, parent: Option<Env> }`. Env implements the least-privilege read trait `pub trait PropertyResolver { fn get(&self, key:&str)->Option<ResolvedValue>; fn get_required(&self,key:&str)->Result<ResolvedValue,EnvError>; fn contains(&self,key:&str)->bool; fn get_as<T:FromConfigValue>(&self,key:&str)->Result<Option<T>,EnvError>; fn resolve_placeholders(&self,t:&str)->Cow<str>; fn resolve_required_placeholders(&self,t:&str)->Result<String,EnvError>; fn accepts_profiles(&self,e:&ProfileExpr)->bool; fn active_profiles(&self)->&ActiveProfiles; fn cloud_platform(&self)->CloudPlatform; }`. The MUTATE view is `EnvBuilder` — a distinct owned type held by App<Define>; `App<Define>::seal_environment(self).await` runs config-data + SAJ + cloud-detect, then `EnvBuilder::seal() -> EnvCore` is consumed, producing the immutable Arc. Reachability resolved: beans inject `Env` by ordinary constructor DI (ADR-05 makes Env an always-present Context field; aware-callbacks = plain injection, not reflection). Hierarchy: a child Context's Env layers a new top over the parent's Arc<EnvCore> via `parent: Option<Env>` — child-wins + dedup is structural sharing (R3), one-directional (child holds parent, never reverse). typed get_as delegates to type-conversion at the read seam.

**Resolved open questions:** Reachability handle = the ownership-model cheap-clone Env(Arc<EnvCore>), NOT a fresh Arc<dyn PropertyResolver> — rides ADR-05's always-present Context field and the one Send+Sync+'static handle contract; Read/mutate seam = consumed builder (R3): EnvBuilder is consumed by seal_environment, so post-seal mutation is type-unrepresentable, the same typestate move App<Resolve> uses for RegistryBuilder; no token, no lock; EnvironmentPostProcessor-analogue discovery shares the cross-crate-discovery linkme substrate (loaders/transports), not a bespoke registry; Async-pre-step lives inside seal_environment (the 5f fence) on the bootstrap ExecutionFacility — async-fetch then in-memory snapshot then sync seal; Profile matching is accepts_profiles delegating to the profiles CondExpr evaluator over active_profiles() — profiles read as a property through the same stack, unified; Typed reads (get_as) live on the read surface AND at injection/binder points (one FromConfigValue seam), delegating to type-conversion; Provenance is carried per-value (Origin rides every PropertyValue) but the rich data is interned in OriginStore so the carrier stays cheap — see origin-tracking

### `property-sources`
R1 (flat ordered Vec + name index) for the MUTABLE phase, frozen at seal_environment into a read-optimized SealedStack. leaf-core: `pub trait PropertySource: Send+Sync { fn name(&self)->&SourceName; fn get(&self,key:&str)->Option<PropertyValue>; fn caps(&self)->SourceCaps; fn keys(&self)->Option<Box<dyn Iterator<Item=&str>+'_>>; }` where SourceCaps carries the enumerable bit (resolving the enumerable/non-enumerable dichotomy as a capability flag on the raw source, per the CPS-adapter and random.* open questions). `PropertyValue { raw: Cow<'static,str>, origin: OriginId }` (OriginId from origin-tracking keeps the carrier a u32). Mutable stack `PropertySourceStack { sources: Vec<Box<dyn PropertySource>>, index: IndexMap<SourceName,usize> }` lives on EnvBuilder with the literal Spring vocabulary add_first/add_last/add_before(name)/add_after(name)/replace/remove, each returning Err(NoSuchSource{name,present}) on a missing anchor. First-source-wins = linear Option short-circuit (never-blend = early-return whole value; null-never-overrides = None falls through, Some("") wins). At seal_environment, `freeze() -> SealedStack` interns SourceNames as Arc<str> and (resolving R1-vs-R3) builds an effective index `IndexMap<CanonicalKey,(PropertyValue,SourceName)>` over enumerable sources only, keeping non-enumerable sources (random.*, remote/WASM) as an ordered fallback chain with a precedence-cutoff rule so a high-precedence non-enumerable source correctly shadows a lower enumerable entry. Reads on SealedStack are lock-free (immutable, Arc-shared via EnvCore). Origin-agnostic: Box<dyn PropertySource> takes native/file/SAJ/configtree/future sources identically. Cross-crate source-contributors are ConfigDataLoaders/SajTransports discovered via the shared linkme substrate; position is descriptor DATA (PrecedenceRung), never link order.

**Resolved open questions:** Stack representation = mutable Vec+index pre-seal, frozen to an enumerable effective-index + non-enumerable fallback chain at seal (R1 mutation vocabulary + R3 read structure); hierarchy sharing is at the Env level (parent Arc), not the stack; Hot read path = O(1) effective index for enumerable sources + ordered fallback for non-enumerable; the precedence-cutoff rule keeps cross-path first-source-wins correct; Cross-crate source-contributor discovery = the one cross-crate-discovery linkme substrate with triple anti-DCE defense, shared with component scanning; zero-contribution = loud AntiDce-class diagnostic; Contributor ordering = PrecedenceRung descriptor DATA applied in a deterministic plan-then-apply pass inside seal_environment, never link/section order; Async pre-step = config-data materializes remote sources to in-memory snapshots inside seal_environment before they join the synchronous-read stack; Provenance attached to every value as a cheap OriginId; rich data in OriginStore (interned), so origin is hot-path-cheap and diagnostic-rich; Enumerable facet = SourceCaps capability bit + optional keys() on the raw source trait (not a downcast), so binder/relaxed-binding/CPS-adapter and the tri-state UNKNOWN flow from it uniformly

### `property-resolution`
R1 char-scanner is the engine, but borrowing the SEALED Env's stack (not a pre-seal borrow), with mode as a per-call argument (R2's fix). leaf-core: the PlaceholderEngine is a hand-rolled recursive scanner replicating PropertyPlaceholderHelper — scan for `${`, match the nesting `}`, recurse-on-key first (so `${${meta}}` resolves), `stack.get(canonical_key)`, recurse-on-value, split on first `:` for `${key:default}`, with a SmallVec<[&str;8]> visited set yielding CircularPlaceholder{chain}. Strict/lenient is the two-method split on PropertyResolver (resolve_placeholders lenient leaves literal; resolve_required_placeholders strict → UnresolvedPlaceholder{placeholder,surrounding,sources_searched}). DEFAULT framework-wide = strict-collect: a dedicated env-validation sub-step inside App<Wired>::validate() collects EVERY unresolved mandatory placeholder into the AssemblyReport (resolving the lenient-foot-gun-vs-charter-#9 tension toward the toolkit's aggregated startup pass). `@Value` reads do NOT stay lenient-at-the-call-site — under `[C2-config-locus]` they are dry-run-resolved STRICT (`resolve_into::<T>` invoked, the coerced value DISCARDED) by the SAME App<Wired>::validate() sub-pass, so a malformed `@Value` (`order.max-retries=abc → u16`) surfaces as a Tier-2 `Convert`/`UnresolvedValue` node in the ONE AssemblyReport rather than as a Tier-3 fault inside `Provider::provide` at refresh R5. Per-site `#[value("…", lenient)]`/`Option<T>` and the threaded `StartupValidation::Lenient` mode downgrade those two value-shape facets to `Severity::Warn` (the only lever-tunable facet); structural wiring stays HARD. Recursion robustness = an explicit depth cap (configurable knob, default 64) PLUS the cycle set. Synthetic sources (random.*) are NOT a StringValueResolver chain — they are non-enumerable PropertySources in the stack (R1/R2 self-contained walk wins over R3's chain machinery), so the resolver treats all sources uniformly via stack.get and stays cross-crate-inert. The ${...} engine owns ONLY ${...}; #{...} expression-language dispatch (if added) is a separate engine the value-injection split owns. Cow fast path keeps no-placeholder resolution allocation-free. Cold-path placeholder resolution of config-data import locations runs against the partial stack-so-far inside seal_environment.

**Resolved open questions:** Default mode = strict-collect as an App<Wired>::validate() sub-step (aggregates all unresolved placeholders AND all dry-run @Value coercion failures into the one AssemblyReport at Tier-2 per `[C2-config-locus]`; @Value is NOT lenient-at-the-call-site — it is strict-dry-run-then-discard at validate, downgraded to Warn only under per-site `lenient`/`Option<T>` or `StartupValidation::Lenient`) — leans on the toolkit's aggregated startup pass, not fail-on-first; Mode placement = per-call (two PropertyResolver methods: lenient resolve_placeholders / strict resolve_required_placeholders), so one Env serves both without two resolver instances; Parse-vs-fused = fused R1 char-scanner (one-shot config-data templates dominate); a #[value("...")] macro emits a thin call to the hand-written parser, malformed-${} caught at compile only for literal templates — stays a thin macro; Handler/discovery substrate = NOT shared with binder, NOT a StringValueResolver chain; random.* is a non-enumerable PropertySource so the resolver is self-contained and cross-crate-inert; ${...} vs #{...} = this engine owns only ${...}; the @Value dispatch/ordering between engines is owned by value-injection, not here; Placeholder-expanded value provenance = Origin::Derived chaining (see origin-tracking), bounded by the shared recursion depth cap; Recursion robustness = explicit depth cap (default 64) + cycle-detection visited set, cap is a configurable knob

### `origin-tracking`
R2 interned OriginStore is the framework choice (leanest carrier + natural env-dump surface), with R1's Derived chaining flattened into the table. leaf-core: `#[derive(Copy,Clone)] pub struct OriginId(NonZeroU32);` carried inline in PropertyValue (a u32, niche-optimized so Option<OriginId> is free). Rich data in `pub struct OriginStore { entries: Vec<OriginEntry>, resources: Vec<Arc<str>>, sources: Vec<SourceName> }` where `OriginEntry { source: SourceId, location: Option<TextLocation>, derived_from: Option<OriginId> }` — two altitudes (coarse Source always; fine TextResource{line,column,span} optional for text sources), Derived is a flat Option<OriginId> index link (no recursive Box, no unbounded heap). The OriginStore is built during config-data parsing inside seal_environment (loaders call store.intern_text(resource,line,col)/intern_source(name) and stamp the returned OriginId) and frozen into the immutable EnvCore at seal — one id authority per Env. Diagnostics resolve lazily: `OriginStore::resolve(id) -> ResolvedOrigin` walks id→entry→resource/source rendering 'application.yaml:12:5' / 'env var APP_MODE' / 'app.mode (via ${...} from application.yaml:12)'. Provenance rides the never-blend short-circuit untouched (winning value carries its OriginId, no merge). Compile-time injection-site origins (the second altitude with no Spring analogue) are captured via proc_macro::Span (stable 1.88) on #[value] and pre-interned as a code-origin OriginEntry, so a bind failure names BOTH 'injected at MyService::timeout (src/svc.rs:20)' AND 'value set in application.yaml:14'. Hierarchy id-space: a child Env's OriginStore references the parent store by an outer id-space tag (a generation field on OriginId resolution) so child-won values render parent-minted origins without re-interning; one-directional. Rendering routes through the one Diagnostic trait — Origin is the typed provenance field on LeafError, not a parallel system (error_generic_member_access stays rejected/nightly).

**Resolved open questions:** Representation = interned OriginId + frozen Arc<OriginStore> (R2): cheap Copy carrier, O(distinct origins) memory, natural /actuator/env-analogue surface; built inside seal_environment, frozen into EnvCore; Carry-vs-lookup = carry a cheap OriginId on every value (hot-path-present u32) but resolve rich data lazily on the diagnostic path — best of both; a release 'strip origins' build lever is deferred; Two altitudes unified = OriginEntry models runtime TextLocation (config-data parse) AND compile-time proc_macro::Span code-origin (value-injection); both stable 1.88, both feed the one Diagnostic renderer; Derived causal chain = flat derived_from: Option<OriginId> index link bounded by property-resolution's depth cap — no recursive Box; Cross-crate stamping = the env-owned OriginStore is HANDED to each ConfigDataLoader (a scoped capability, not an ambient global); reuses the shared source-contributor substrate, no bespoke origin registry; Hierarchy = child OriginStore references parent store by outer id-space tag (generation-checked), so parent-minted ids resolve in the child without re-interning, honoring one-directional visibility

### `config-data`
R2 plan-then-apply with an R1 worklist planner internals (the doc's named hybrid), running as the body of seal_environment. The async planner does ALL IO on the bootstrap ExecutionFacility and emits an ordered ConfigDataPlan of rung-tagged contributions; a pure sync applier folds it onto the EnvBuilder stack — eliminating the &mut-across-.await borrow because IO holds no stack borrow and the fold holds no .await. leaf-config: `pub trait ConfigDataLoader: Send+Sync { fn handles(&self,loc:&ConfigDataLocation)->bool; fn load<'a>(&'a self,loc:&'a ConfigDataLocation,ctx:&'a LoadCtx<'a>)->BoxFuture<'a,Result<Vec<LoadedDocument>,ConfigDataError>>; }` (BoxFuture per the ADR-07 dyn-seam standard). `pub struct ConfigDataPlan { contributions: Vec<Contribution> }`, `Contribution { rung: PrecedenceRung, source_name: SourceName, props: Vec<(KeyRef,PropertyValue)> }`, `pub enum PrecedenceRung { DefaultProperties, PropertySourceClass, ConfigDataFile{group:u16,profile_specific:bool,external:bool}, Random, SpringApplicationJson, OsEnv, CommandLine }` — the full ladder as a total-ordered enum CO-OWNED by config-data + profiles + auto-config (each contributes rungs), stable-sorted by (rung,group,profile_specific,external,discovery_index) so profile-specific-overrides + last-active-profile-wins + .properties-beats-.yaml are a comparator, not insertion order. Single-pass traversal: read early-control keys (config.name/location/additional-location) from the already-seeded high-precedence sources (never files); compute the location worklist; per location pick the winning loader; per LoadedDocument evaluate activate_on against the FROZEN ActiveProfiles (CondExpr OnProfile/OnCloudPlatform — profiles sealed up-front, the literal happens-before edge) and skip non-matches; push spring.config.import locations after the current item, deduped by canonical resource (idempotency); placeholders in import locations resolve against the provisional accumulated contributions. HARD RULE enforced inline: an activate.on-profile document setting profiles.active/include/group/default → loud ConfigDataError::IllegalActivationDocument. Loaders stamp fine OriginId into props as they parse. Loaders discovered via the shared linkme substrate (a configtree: with no loader = loud anti-DCE diagnostic). Explicit RAII closes file handles/dir iterators inside the cold pass (no async Drop).

**Resolved open questions:** Precedence ownership = a shared declarative PrecedenceRung total-order enum co-owned by config-data+profiles+auto-config, applied in one plan-then-apply fold inside seal_environment — one canonical order, not config-data-private insertion calls; Engine shape = R2 plan-then-apply (async planner / sync applier) with an R1 worklist planner internal; clean async/sync bisection, no &mut-across-.await, exhaustively unit-testable via an injected ConfigFs; Async fence = the whole pass is the body of App<Define>::seal_environment (compile-runtime-split's 5f fence) on the bootstrap ExecutionFacility; every document materialized to an in-memory PropertySource before the env seals; flaky-remote retry = the sanctioned cold-path reactive timer loop; refresh of a changed source is out of scope for v1 (Boot reloads only on restart); Format-loader discovery = the one cross-crate-discovery linkme substrate, shared with property-sources contributors and component scanning; selection by handles DATA, triple anti-DCE defense, configtree-with-no-loader fails loudly; Dynamic/WASM source contributing documents/subtrees = DROPPED (native only); ConfigFs/loader abstraction stays origin-agnostic but no dynamic second classpath; Two profile layers share one frozen ActiveProfiles across the phase boundary owned by the refresh template: profiles seal up-front → config-data filters documents (inside seal_environment) → env seals → App<Resolve> bean conditions evaluate; Rust immutability makes Boot 2.4's late-activation paradoxes structurally impossible; Import-location placeholders resolve against the provisional accumulated-contributions view inside the planner (synchronous, no IO), keeping the async/sync bisection intact; Deterministic single-pass preserved literally (R1 worklist in the R2 planner); no DAG/cycle-detection model — ConfigDataImportCycle is not a possible error, idempotency is the loaded-set

### `extra-9`

Flatten/array-index segment vocabulary is shared with the binder's tree-descent and relaxed-binding's index rules (one canonical key-segment primitive).

**Resolved open questions:** Precedence-rung emission = the loader DECLARES PrecedenceRung::SpringApplicationJson; config-data's plan-then-apply assembler owns all stack mutation — no scattered add_before; Transport extension = FIXED core-owned list (env-var + command-line); SajTransport stays origin-agnostic for future opt-in but no dynamic discovery in v1; JNDI/system-property transports dropped (no Rust analogue); Flatten/descent segment vocabulary UNIFIED with binder-descent and relaxed-binding index conventions into one canonical key-segment+[index] primitive (charter #11); JVM-transport mapping = JNDI/servlet/system-property rungs dropped; SAJ's relative claim (below cmdline, above env) survives, absolute JVM neighbours evaporate; Null/absence model = JSON null + missing key collapse to None (absent) at flatten time; empty-string is Some("") present-and-wins; the Option<PropertyValue> None-is-only-fall-through invariant is uniform across all sources; Command-line arg ownership = leaf-boot owns a minimal --key=value argv parser feeding both the SAJ cmdline transport and the CommandLine rung; documented to coexist with an app's clap/argh

## Cross-feature interactions
- **compile-runtime-split (toolkit)** — The whole subsystem is the body of App<Define>::seal_environment().await — the 5f async-IO fence the ADR pins between App<Define> and App<Resolve>. config-data/SAJ/cloud-detect run here on the bootstrap ExecutionFacility; the EnvCore snapshots and is frozen into the FrozenContainer-adjacent immutable Env before seal(). Strict-placeholder validation and missing-required-key/missing-config-location failures aggregate into the App<Wired> AssemblyReport. @PropertySource and bean-level profile/cloud conditions run in App<Resolve>, AFTER the env is sealed, reading it lock-free.
- **ownership-model (toolkit)** — The everywhere-read handle Env is the ownership-model's cheap-clone Arc<EnvCore>; Send+Sync+'static rides EnvCore exactly as it rides ErasedBean (one safe-publication contract, no env-specific lock). The atomic Arc clone is the happens-before edge into every async bean.
- **container-shape / registry-substrate (toolkit)** — Env is an always-present field on Context (ADR-05) injected into beans by ordinary DI; the EnvBuilder→seal_environment→sealed Env typestate mirrors RegistryBuilder→App<Resolve>→frozen Registry (consumed mutable access = post-seal edit unrepresentable). config-data sources are NOT registry beans — they live in EnvCore.stack — but loaders/transports are discovered via the same linkme/anti-DCE pipeline as components.
- **conditional-strategy (toolkit)** — profiles and cloud-platform are CondExpr leaves (OnProfile, OnCloudPlatform) over the mandatory runtime tier; config-data document activation (spring.config.activate.on-profile/on-cloud-platform) reuses the IDENTICAL CondExpr evaluator that gates @Profile/@ConfigurationCondition beans — one grammar, two phases (documents filtered inside seal_environment; beans filtered in App<Resolve>). The frozen ActiveProfiles + frozen CloudPlatform are the inputs both phases read.
- **auto-configuration (toolkit)** — auto-config co-owns the PrecedenceRung ladder (it contributes the lowest DefaultProperties rung) and its OnProperty/OnMissingBean back-off reads the FROZEN env — so config-data is a strict happens-before of auto-config condition evaluation (env sealed inside seal_environment, then App<Resolve> runs the auto-config deferred batch over it).
- **cross-crate-discovery (toolkit)** — ConfigDataLoaders, SAJ transports (opt-in), StringValueResolver hooks (if any), and the open cloud-platform variant ride the linkme distributed-slice substrate with the triple anti-DCE defense (force-link + toolchain guardrails + ExpectedManifest self-check). A configtree: prefix with no registered loader is a loud diagnostic, never a silent empty stack. Config VALUES are runtime data and never link-collected. cloud-platform v1 deliberately uses a closed enum + explicit probe install to avoid the DCE tax.
- **error-model (toolkit)** — EnvError/ConfigDataError/PlaceholderError/IllegalActivationDocument/UnknownCloudPlatform are ErrorKind arms in the one LeafError causal chain; Origin is the typed provenance field on the diagnostic, rendered through the one Diagnostic trait (error_generic_member_access stays rejected). Failures land at the correct timing tier: malformed ${...}/CondExpr at compile (Tier 0); ContractId/anti-DCE/missing-loader at seal (Tier 1); missing-required-key/unresolved-mandatory-placeholder/missing-non-optional-location aggregated at App<Wired> (Tier 2); a genuinely flaky runtime source fault at runtime (Tier 3).
- **async-context-model (toolkit)** — All config IO is cold-path async inside seal_environment on the bootstrap ExecutionFacility; the hot read path (stack.get, placeholder walk, origin render, CPS adapter, cloud_platform()) is synchronous/non-blocking per the reactive-hot rule. A flaky-remote retry/backoff is the sanctioned cold-path reactive timer loop. No async Drop: file handles/dir iterators close via explicit RAII in the bounded cold pass.
- **downstream binder / config-properties / value-injection / relaxed-binding / type-conversion (sibling subsystem)** — The CPS adapter is the ONLY surface the binder/relaxed walk touches; it carries the tri-state PRESENT/ABSENT/UNKNOWN, the collection-replacement mechanism, and unblended OriginId so bind/validation failures name the exact file:line. ## Remaining risks (→ Phase 4)
- Strict-collect placeholder validation as the default: running an App<Wired>::validate() sub-pass that resolves every mandatory placeholder AND dry-run-coerces every eager `@Value` (`resolve_into::<T>`, value discarded) doubles as a pre-flight over the whole env. RESOLVED in Phase 4 by the unified strictness lever (`[C2-config-locus]` + the SEAMS "Unified startup-validation strictness lever"): the opt-out is `BootstrapSettings.startup_validation: StartupValidation { Strict, Lenient, Skip }`, read ONCE at the head of `App<Wired>::validate()` and threaded down — `Strict` (default) collects all into the report, `Lenient` downgrades the two value-shape facets (unresolved mandatory placeholder, uncoercible `@Value`) to `Severity::Warn`, `Skip` (valid `cargo leaf prepare` plan) elides the cold walk only. The placeholder sub-pass and the `@Value` dry-run sub-pass read the SAME threaded value, so they cannot diverge. Residual: the cold-walk cost is measure-before-build; the AOT plan-hash must actually cover the bound config for `Skip` to be sound. NOTE: config-properties JSR validation (range/nested) is NOT one of the two downgradable facets — it has no Tier-3 fallback, so it stays HARD even under `Skip` (the cold cost is elided but the bind+validate still runs at validate-time materialization).
- Reserved 'too-late' bootstrap-key namespace ownership: @PropertySource too-late detection and config-data's early-control-key bootstrap need ONE canonical list of early-consumed key prefixes (leaf.main.*, logging.*, profile-activation). Phase 4 must pin who owns/extends that list so a new framework knob is covered without per-feature drift.
- Hierarchy OriginStore id-space: a child Env resolving a parent-minted OriginId via an outer id-space generation tag is correct but adds a generation-check on the diagnostic path; needs validation that the tag scheme stays sound across deep context hierarchies and never silently renders a wrong origin.
- CPS-adapter lazy memo cache coherence: the env is sealed/immutable so the read-through adapter is safe, but the append-only prefix→State memo's interaction with very large configtree binds (re-walk vs memoize) is a startup-cost-vs-memory call to measure — could fall back to an eager index for huge enumerable sources.
- PrecedenceRung total-order co-ownership: config-data + profiles + auto-config each contributing rungs into one enum is clean, but a fourth contributor (a test-support @TestPropertySource analogue) added later must extend the enum coherently. The discovery_index tie-break determinism under parallel loader IO (assign by deterministic worklist order, not IO-completion order) is a subtle constraint to enforce in the planner.
- K8s pair-heuristic degradation: cloud-platform's SERVICE_HOST/SERVICE_PORT enumeration over a non-enumerable source degrades to a coarse fingerprint — needs a clear diagnostic so a mis-detection-due-to-non-enumerable-source is visible, not a silent NONE.
- Format-loader sync facet for @PropertySource: the App<Resolve> contribution step needs a synchronous load entry point on config-data's (async) format loaders for the local-file case; the sync/async facet split must be designed so it does not bifurcate the loader trait (charter no-bimodality) — likely a sync local-file fast path with async reserved for genuinely remote loaders that never appear at this rung.
