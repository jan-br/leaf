# Conditions, Profiles & Auto-Configuration  `[conditions-autoconfig]`

_The one typed CondExpr algebra over a stable ConditionId with the per-kind tier-map (runtime as mandatory floor), profiles as a runtime preset, the condition report, configuration-class/@Bean and @Import composition, and the exclude>back-off>default auto-config ladder at @Fallback role._

This subsystem is the single gating-and-composition spine of leaf: ONE typed `CondExpr` algebra over a stable `ConditionId`, routed by the FIXED per-kind tier-map (`Cfg`/`ConstFold`/`Runtime`, runtime as the mandatory floor), evaluated in `App<Resolve>`'s Parse-then-Register sub-passes against the sealed `Env` and the growing `RegistryBuilder` definition set, with every outcome written into ONE `ConditionReport` frozen into the `FrozenContainer` at `seal()`. Eight Phase-1 features collapse onto this spine without re-fragmenting:\n\n- **conditions** is the `CondExpr` enum + `Condition` SPI + framework-inferred `phase()`/`tier()` = max-over-leaves (kills Spring's forgot-to-declare-REGISTER bug structurally).\n- **condition-family** is the catalog of `ConditionKind` leaves (OnProperty/OnBean/OnMissingBean/OnClass/OnProfile/...) — each a const tier/sub-phase declaration + a thin attr macro + a runtime `Condition` impl behind a `ConditionId`; OnClass is a `#[cfg]` leaf (compile-pruned, force-link-paired), not a runtime probe.\n- **condition-report** is the passive sink the one eval path writes, carrying the six cross-tier verdict classes (Positive/Negative/Exclusion/Unconditional/CompiledOutByCfg/BuildFoldedFalse) and consuming the substrate anti-DCE self-check — never reinventing it.\n- **auto-configuration** is the thin `leaf-boot` `run_autoconfig` orchestrator over the dedicated `AUTO_CONFIGS` linkme channel: the `exclude(ContractId) > user-bean back-off(OnMissingBean) > auto-config default` ladder in one cold pass, registering candidates incrementally so each `OnMissingBean`/`OnSingleCandidate` `DefinitionProbe` sees the growing set; auto-config beans register at `Role`-orthogonal `CandidateRole::Fallback` so a user bean transparently supersedes.\n- **auto-config-ordering** is the three-pass sort (alphabetical base → `order(i32)` → topological before/after with cycle detection) over `SequenceKey(epoch, ContractId)`, plus the three-source exclusion merge — definition order ONLY, structurally unreadable by the `App<Running>` instantiation scheduler.\n- **profiles** is the `ON_PROFILE` `CondExpr` preset (always Runtime/Parse), a thin `ProfileExpr` grammar (`!`/`&`/`|`, parens-mandatory-on-mixed, array=OR, negation=absence) parsed at macro time, evaluated against a sealed canonical `ActiveProfiles` value computed once at `seal_environment()` (active/include/group/default fallback resolved there, not in the matcher).\n- **configuration-classes** is lite-only: each `#[bean]` fn lowers to one `Descriptor` + `ProviderSeed` whose factory params are injection points; intra-config `@Bean`→`@Bean` self-calls are a `compile_error!`; the config struct is itself a bean.\n- **import-composition** keeps the four altitudes as thin metadata over the one descriptor stream: `#[import]` = a force-link-solving contributor edge resolved into the active set at lift; immediate selectors return a closed `ImportRef` set; deferred-group is exactly `run_autoconfig` (auto-config IS the headline DeferredImportSelector.Group); `Registrar` is the imperative escape hatch; `@Enable*` is a typed `EnableAttrs<A>` struct, not reflective metadata.\n\nThe whole subsystem is cold, synchronous, single-threaded, allocation-bounded, runs entirely before the `start().await` async fence, and adds ZERO hot-path cost (the frozen report is read lock-free at steady state). It owns only gating + composition semantics; it re-keys, re-stores, and re-emits NOTHING that registry-substrate, codegen-boundary, or environment-config already own.

## Shared machinery (how this rests on the toolkit)

The entire subsystem rests on the toolkit's named primitives, and the unification is that all eight features share them rather than minting their own:\n\n1. **ONE `CondExpr` algebra over `ConditionId`, ONE evaluator, ONE outcome stream.** conditions, condition-family, profiles, auto-config back-off, import-selector gating, component-scan filters, and `Registrar` conditional branches ALL emit one const `CondExpr` tree (`Leaf/All/Any/Not/Const`) and are interpreted by ONE recursive `evaluate(&CondExpr, &ConditionCtx) -> ConditionOutcome`. There is no second gating mechanism, no `AnyNestedCondition` subclass boilerplate (boolean composition is first-class syntax), and no author-declared phase (it is inferred `= max` over leaf tiers/sub-phases). This is the toolkit's fixed `conditional-strategy` ABI consumed verbatim.\n\n2. **ONE per-kind tier-map as the kernel ABI** (`ConditionKind { const ID, const TIER, const SUB }` in leaf-core). It is the SINGLE authoritative artifact condition-report reads to render verdict classes and auto-config reads to know which leaves are pre-decided. `Cfg` leaves (OnClass) are physically pruned at compile + force-link-paired; `ConstFold` leaves arrive as `CondExpr::Const(bool)`; `Runtime` is the mandatory floor. profiles' `ON_PROFILE` is a fixed `TIER=Runtime, SUB=Parse` entry in this same map — the asymmetry is made coherent by it being just one more row.\n\n3. **ONE `Descriptor` const row as the universal currency** (registry-substrate). A user `#[bean]`, an auto-config `#[bean]`, an imported config, and a `Registrar`-minted definition are all the SAME `Descriptor` + `Provider`; the only differences are `role` (orthogonal `Role::{Application,Support,Infrastructure}`), a new `CandidateRole::{Normal,Primary,Fallback}` field for the auto-config @Fallback default, and the `CondExpr` guard living in `meta`. There is no `AutoConfigDescriptor`/`ComponentDescriptor`/`RawBeanDefinition` second seed type — those Phase-1 placeholders all collapse into `Descriptor`. The dedicated `AUTO_CONFIGS` linkme slice carries `Descriptor`s flagged auto-config (a second discovery channel, same row shape, same anti-DCE contract).\n\n4. **ONE `seal()` boundary and ONE phase placement** (compile-runtime-split). All of this subsystem's runtime work is the body of `App<Resolve>`: `seal_environment().await` ends the 5f async fence and produces the sealed `Env` + `ActiveProfiles`; then `route_conditions()` (Parse sub-pass over leaves with no OnBean) and `run_autoconfig()` (Register sub-pass, incremental) run synchronously; then `seal()` freezes the dense `Registry` + the `ConditionReport` into `FrozenContainer`. A post-seal condition or auto-config edit is type-unrepresentable (the typestate consumes the builder).\n\n5. **ONE stable cross-build identity `ContractId(u64)`** for everything durable: auto-config exclusion keys, before/after ordering edges, name-based `leaf.autoconfigure.exclude` property strings (resolved against the candidate name index), and the report's element key for the silent-now/loud-later join. `TypeId` is the in-process fast key only, never serialized. `SequenceKey(epoch, ContractId)` is the deterministic batch tie-break below `cmp_order`.\n\n6. **ONE `DefinitionProbe` = the no-instantiation candidate-resolver** (injection-mechanics, shared with `App<Wired>::validate`). OnBean/OnMissingBean/OnSingleCandidate are `probe.would_resolve_unique(ty) -> Resolvability` over the frozen-candidate-set-so-far, running the SAME primary/fallback/qualifier policy as injection — never a bean-construction, never a naive count. There is one definition of \"unambiguous,\" used by conditions AND startup validation.\n\n7. **ONE anti-DCE triple defense** (cross-crate-discovery: force-link + section-retention guardrails + `ExpectedManifest` self-check at the `App<Define>→App<Resolve>` edge). It guards the `COMPONENTS` and `AUTO_CONFIGS` channels AND the `CONDITIONS` `ConditionId→&dyn Condition` registry uniformly. condition-report CONSUMES its result to distinguish the four absence states; it never reinvents it. \"enabled-starter-zero-contribution\" and \"family crate not force-linked so OnProperty always passes\" are the SAME `AntiDceError::SourceVanished`.\n\n8. **ONE error/diagnostic path with four timing tiers** (error-model). Malformed `CondExpr`/`ProfileExpr`/unknown cfg feature/generic guarded element = compile error (Tier 0); `ContractId` collision / ordering cycle / exclusion-typo / post-seal edit = build-or-seal `LeafError` (Tier 1); condition-not-met = a silent recorded `ConditionReport` Negative (never an error); the later `NoSuchBean` enriches FROM the frozen report at `App<Wired>` (Tier 2). condition-not-met and wiring-broken stay crisply time-separated.\n\n9. **ONE sealed `Env`/`ActiveProfiles` read view** (environment-config). All Parse-tier leaves (OnProperty/OnProfile/OnResource/runtime-OnCloudPlatform) read the lock-free `Env` snapshot synchronously; profiles' `resolve_active` runs inside `seal_environment`, so `ActiveProfiles` is fully materialized before the first condition evaluates.

## Features

### `conditions`
The `conditions` SPI is the toolkit's fixed `conditional-strategy` ABI, made concrete. leaf-core hosts: `pub enum CondExpr { Leaf(ConditionId, AttrSlice), All(&'static [CondExpr]), Any(&'static [CondExpr]), Not(&'static CondExpr), Const(bool) }`; `pub struct ConditionId(pub u32)`; `pub enum EarliestTier { Cfg, ConstFold, Runtime }`; `pub enum SubPhase { Parse, Register }`; `pub trait ConditionKind { const ID: ConditionId; const TIER: EarliestTier; const SUB: SubPhase; }`. `impl CondExpr { pub const fn tier(&self) -> EarliestTier; pub const fn phase(&self) -> SubPhase; }` compute max-over-leaves at const-eval time (so the framework infers the earliest-sound phase — an OnBean nested anywhere auto-defers the whole guard to Register). The runtime SPI is `pub trait Condition: Send + Sync { fn matches(&self, ctx: &ConditionCtx<'_>, attrs: &AttrSlice) -> ConditionOutcome; }` — synchronous, side-effect-free, no `.await`, no IO. `ConditionCtx<'a> { env: &'a Env, defs: &'a DefinitionView<'a>, probe: &'a DefinitionProbe<'a>, caps: &'a CapabilitySnapshot, report: &'a ReportSink<'a> }`. `ConditionOutcome { matched: bool, reason: ReasonMsg }`. ConditionId→impl resolution rides a `#[distributed_slice(::leaf_core::CONDITIONS)] static CONDITIONS: [CondImplRow]` (`CondImplRow { id: ConditionId, imp: &'static dyn Condition }`) lifted at `App<Define>` and self-checked by the same anti-DCE pass (an unresolved ConditionId is a Tier-1 `AssemblyError`, never a silent pass-all). The thin macro `#[conditional(all(on_property("x"), any(on_bean(Foo), not(on_class("redis")))))]` emits exactly one `const CondExpr` tree onto `Descriptor.meta` via absolute `::leaf_core` paths; cargo expand shows one legible const. Evaluation is `evaluate(&CondExpr, &ConditionCtx) -> ConditionOutcome` driven by `App<Resolve>::route_conditions()` (Parse sub-pass: trees whose `phase()==Parse`) then inside `run_autoconfig` (Register sub-pass: trees with any OnBean leaf, seeing the growing set). A miss is a graceful recorded Negative, never an error.

**Resolved open questions:** Conditional strategy blend: FIXED by the toolkit — R3 typed CondExpr algebra is the authoring surface, R2 cfg-gate is OnClass, R1 runtime evaluator is the mandatory floor. Phase 3 adopts verbatim; no re-litigation.; OnClass realization: a `Cfg`-tier leaf emitted under `#[cfg(feature=...)]` (physically removed when the capability feature is off) PAIRED with cross-crate-discovery force-link; never a runtime registry probe (a linked type is always present).; Two-phase ordering at runtime: established as DATA via the per-kind `SUB` const + `CondExpr.phase()=max`, resolved by the Parse-then-Register sub-pass sequencing in App<Resolve>, NEVER link/section order.; Phase author-declared vs inferred: framework-INFERRED (max over leaves) — kills Spring's #1 forgot-REGISTER bug; the inferred tier is surfaced in the ConditionReport to avoid the surprise-deferral trap.; ConditionContext ownership/lifetime: a borrowed read-only snapshot `ConditionCtx<'a>` over the sealed `Env` (cheap-clone `Arc<EnvCore>`) + the borrowed growing `DefinitionView` + the `DefinitionProbe` — no Arc-per-condition, no global lock.; One shared predicate primitive (unification #11): YES — conditions, programmatic-registration guards, component-scan filters all emit/evaluate the identical `CondExpr` over the identical `ConditionCtx`.; Silent-now/loud-later: condition-not-met writes a Negative into the ReportSink; never a LeafError. The App<Wired> NoSuchBean enriches by querying the frozen report by ContractId.; Runtime-overlaid state / WASM dropped: the Runtime tier is mandatory regardless (OnProperty/OnProfile read runtime env); WASM is dropped, so no dynamic-source carve-out is needed.; AttrSlice schema (the open Phase-3 question): adopt a shared compile-time attr-parsing helper in leaf-macros that validates known keys per ConditionKind at macro-expansion (typo like `having_valeu=` is a Tier-0 compile error with a span), lowering to a uniform `AttrSlice = &'static [Attr]` (`Attr::{Str,Bool,Int,Type}`) read by the impl — typed-validation at the macro boundary, uniform stringly carriage in the const data.; CondExpr combinator freeze: KEEP `All/Any/Not/Leaf/Const` only for v1; `count(n, [..])`/`xor` are macro sugar lowered to `All/Any/Not` (e.g. xor(a,b) = Any([All([a,Not(b)]),All([Not(a),b])])) so the kernel enum stays frozen and minimal — adding a variant is a kernel-ABI break, deferred.

### `condition-family`
The catalog is a set of `ConditionKind` impls in the `leaf-conditions` crate, each: (1) a stable `ConditionId` const + `const TIER`/`const SUB`; (2) a thin per-member attr macro emitting `Leaf(ID, attrs)`; (3) a hand-written `Condition` impl registered into `CONDITIONS`. Concrete members and their fixed tier-map entries: `on_property`/`on_boolean_property`/`on_expression`/`on_resource` = (Runtime, Parse); `on_profile`/runtime `on_cloud_platform` = (Runtime, Parse); `on_bean`/`on_missing_bean`/`on_single_candidate` = (Runtime, Register); `on_class`/`on_capability` = (Cfg, n/a — compile-pruned + force-linked); `on_rust_version`/target-`on_cloud_platform`/`on_threading`/`on_war_deployment` = (ConstFold, decided at build → `Const(bool)`). The OnBean family delegates its verdict to `ctx.probe.would_resolve_unique(ty) -> Resolvability::{None, Unique(BeanId), Ambiguous(u16)}`: `on_missing_bean(T)` matches iff `!= Unique`, `on_bean(T)` iff `== Unique`, `on_single_candidate(T)` iff `== Unique` (which, by running the candidate-resolver's primary/fallback policy, correctly counts a @Primary-among-several as unique and a lone @Fallback as resolving). `SearchStrategy{Current,Ancestors,All}` is an attr the OnBean impl reads to scope its `DefinitionView` query through `Context::names_including_ancestors`-style hierarchy walk. OnProperty multi-name semantics: all listed names must pass; `match_if_missing` is a per-condition attr (default false); the property keys read through the `Env`'s relaxed-binding view. Booleans are NOT catalog members — `all/any/not` are first-class `CondExpr` nodes (no `AnyOf`/`AllOf` ConditionIds). Each impl authors a kind-specific `ReasonMsg` (`OnProperty x.enabled: found "false", expected "true"`; `OnMissingBean(Foo): found user bean bar`) so the report populates for free. JVM-ecosystem reframing is baked into the tier-map: OnWebApplication → an `on_capability("web")` Cfg leaf or a runtime web-context flag (no servlet model); OnThreading → execution-facility capability; OnJava → `on_rust_version`; OnWarDeployment/OnJndi/OnCheckpointRestore are DROPPED (Rust-irrelevant).

**Resolved open questions:** Per-kind tier-map (the load-bearing artifact): MINTED and FROZEN as the `ConditionKind` consts in leaf-core (conditional-strategy ABI). condition-family populates rows; it does not own the map shape.; OnClass = Cargo capability feature (Cfg tier) + mandatory force-link pairing; OnMissingClass = `not(on_class(...))` evaluated at compile (the cfg makes the descriptor absent, so the complement is the cfg-false branch). Additive-only discipline: backend XOR stays a runtime OnProperty/OnProfile decision.; Attribute schemas: typed-per-member validation at macro-expansion (early span-rich errors) lowering to the uniform stringly `AttrSlice` the impl reads — same shared helper as conditions.; Phase author-declared vs inferred: inferred via the per-kind `SUB` const + CondExpr.phase()=max — no author phase declaration anywhere in the family.; JVM members reframing: OnWebApplication→capability/runtime-flag; OnThreading→execution-facility capability; OnJava→OnRustVersion; OnWarDeployment/OnJndi/OnCheckpointRestore DROPPED. Resolved per-member in the tier-map.; OnSingleCandidate delegation: reuses EXACTLY injection-mechanics' `DefinitionProbe.would_resolve_unique` (one definition of unambiguous), running the candidate-resolver over the DEFINITION view with no instantiation (verified shared with App<Wired> validation).; Cross-tier accountable account: each member writes its verdict+reason to the ReportSink; cfg-pruned members contribute via the report's CompiledOutByCfg class (from the CapabilitySnapshot), not a separate breadcrumb channel.; WASM dropped: the runtime tier is mandatory anyway; no WASM-sourced OnClass degradation needed.; ConditionContext ownership + hierarchy-aware views: borrowed `ConditionCtx<'a>` exposing a `DefinitionView` with SearchStrategy-scoped (Current/Ancestors/All) queries.; Boolean grammar richness: first-class `all/any/not` syntax over the frozen CondExpr enum; xor/count are macro sugar, not new kernel variants (ABI frozen).

### `condition-report`
condition-report is the passive accounting sink — NOT an evaluator and NOT a reason-author. leaf-core hosts `pub enum ConditionReportClass { Positive, Negative(ReasonMsg), Exclusion(ContractId), Unconditional, CompiledOutByCfg(SourceTag), BuildFoldedFalse(ConditionId) }` and `pub struct ConditionRecord { element: ContractId, self_type: Option<TypeId>, class: ConditionReportClass, leaves: Box<[LeafOutcome]>, provenance: CodeSpan }` (provenance baked at macro-expansion via `proc_macro::Span`, carried as `&'static str` + line on `Descriptor.meta`). During `App<Resolve>` the one eval path writes through `trait ReportSink { fn record(&self, rec: ConditionRecord); }` — the default sink is a single cold-path `&mut`-owned `RefReport` rooted on the refresh stack (no Arc, no lock, amortized Vec append). At `seal()` it freezes into `Arc<ConditionReport>` stored in `FrozenContainer.report` (the field compile-runtime-split already reserves), keyed by ContractId for O(1) lookup. Cross-tier reconciliation is done by FUSING three inputs at the report-finalize step, NOT a second manifest: (1) the runtime Negative/Positive rows; (2) the `CapabilitySnapshot` (resolved feature set, from the binary's force-link/ExpectedManifest emission) to label an absent element CompiledOutByCfg; (3) the anti-DCE `self_check` result so a manifest-present-but-zero-rows-and-no-cfg-label crate is the substrate's `AntiDceError::SourceVanished` (LOUD) — kept DISTINCT from a CompiledOutByCfg (feature off) and a Negative (genuine back-off). The report does NOT double as the self-check; it CONSUMES the single substrate self-check's result (resolving the coherence-note duplication). Delivery is two pure cold readers over the frozen report: `render_startup(&ConditionReport) -> String` (the `--debug`/`LEAF_DEBUG` analogue, gated by `leaf.condition-report.enabled` read once from Env) and `serialize(&ConditionReport) -> ReportJson` (serde, the `/actuator/conditions` analogue an optional introspection crate exposes). The silent-now/loud-later bridge: `App<Wired>::validate`'s `NoSuchBean` enrichment does an O(1) `report.lookup(contract)` and appends `backed off because <reason>` / `feature redis not enabled (enable it in Cargo.toml)` (the actionable Action: spirit, available because CompiledOutByCfg knows the gate).

**Resolved open questions:** Own sink vs unified subscriber: a dedicated cold-path `RefReport` sink frozen to `Arc<ConditionReport>` (R1), NOT a hot-path observation bus (R2). The frozen report is the single artifact; an optional tracing/actuator crate reads `serialize()` — extensibility without a runtime pub/sub.; Cross-tier reconciliation: a runtime sink that FUSES runtime rows + CapabilitySnapshot + anti-DCE result at finalize (the toolkit makes the six-class taxonomy fixed) — no separate compile-time GuardManifest second channel (R3 rejected as a second DCE surface); the CapabilitySnapshot already carries the cfg facts.; ElementId keying: `ContractId` (stable cross-build) is the canonical key, matching what auto-config exclusions, ordering edges, and the App<Wired> NoSuchBean enrichment all use — the four overlapping Phase-1 identity schemes unify on ContractId (the coherence-note's highest-risk gap, closed). `self_type: Option<TypeId>` is the in-process fast secondary key.; Resolved feature/cfg set: obtained from the single authoritative `CapabilitySnapshot` emitted by the binary's build.rs/force-link step (cross-crate-discovery owns it), not probed ad hoc — trustworthy across -p/--workspace because it reads the FINAL resolved set.; Report doubles as DCE self-check: NO — it CONSUMES the single substrate-level self-check (resolving the coherence note: one check, the report a consumer). 'bean backed off' vs 'registration vanished' stay distinct because they come from different inputs (runtime row vs AntiDceError).; Delivery channels: env-gated text dump + serde structured value; no-op default; tracing bridge as an optional crate reading serialize(). No mandated logging backend.; Report ownership/lifetime: `&mut`-owned RefReport on the refresh stack (cold), frozen to `Arc<ConditionReport>` in FrozenContainer (read possibly-hot during lazy-resolution NoSuchBean enrichment, lock-free).; WASM dropped: the runtime report tier is the floor regardless; no WASM-sourced-record provenance gap.; Verdict taxonomy: the six classes are FROZEN in leaf-core ABI (conditional-strategy's ConditionReportClass) — all producers/consumers agree; CompiledOutByCfg/BuildFoldedFalse are the leaf-specific additions.; Machine-parseable reasons: condition-report imposes a `ReasonMsg{kind, expected, found, gate}` shape that condition-family members fill (consistent, serde-able for CI config-drift checks) while still rendering human-readable.

### `auto-configuration`
The engine is the codegen-boundary R1 runtime-deferred-batch, made concrete as `leaf-boot::run_autoconfig(ctx: &mut AssemblyCtx, candidates: &[&Descriptor]) -> Result<(), AssemblyError>` (taking the candidate set as a PARAMETER so test-slice mode can pass an explicit subset). An auto-config is `#[leaf::auto_config]` on a lite-only config (the macro LINTS `#[bean]` self-calls as a `compile_error!`), emitting `Descriptor`s into `#[distributed_slice(::leaf_core::AUTO_CONFIGS)]` — a SECOND channel beside `COMPONENTS`, inheriting the full anti-DCE contract (force-link + ExpectedManifest + per-crate self-check), ContractId keying, and SequenceKey ordering. The engine runs ONE cold synchronous pass in `App<Resolve>`'s Register sub-pass, AFTER `seal_environment` + all user defs + the eager user-config batch: `if env.get_as::<bool>("leaf.enable-autoconfiguration") == Some(false) { return Ok(()) }` (kill-switch); `let ordered = auto_config_ordering::order_batch(candidates, ctx.exclusions)?`; then for each `c` in SequenceKey order: `if ctx.exclusions.contains(c.contract) { report.record(Exclusion(c.contract)); continue; }` (excluded mints NO BeanId, never enters back-off); else `match evaluate(c.meta.guard, &cond_ctx) { matched => ctx.defs.register_lite(c), no_match(reason) => report.record(Negative(c.contract, reason)) }`. `register_lite` appends the candidate's `Descriptor`+`Provider` into the growing `RegistryBuilder` so later candidates' `OnMissingBean` `DefinitionProbe` see it — Spring's definition-order-for-correctness as a lifecycle sub-pass. Auto-config beans register with `CandidateRole::Fallback` on their `Descriptor` (the new orthogonal field): the candidate-resolver treats a Fallback candidate as losing to any non-fallback of the same contract, so a user bean transparently supersedes even when BOTH are present — a softer override than all-or-nothing OnMissingBean. The starter mechanism is the FIXED autoconfig-starters-bom topology: a `dep:`-hidden umbrella feature pulls `leaf-starter-*` (an aggregator crate, Cargo.toml only) whose integration crates the umbrella's `#[leaf::main]`/build.rs auto-force-links + adds to ExpectedManifest. Test-slice mode `slice!{ WebStack, DataLayer }` calls the SAME `run_autoconfig` over an explicit `&[&Descriptor]` subset, optional-if-absent entries (whose crate isn't force-linked) skipped. The engine outputs DEFINITION order/contents ONLY — discarded before `App<Running>`'s dependency-driven WiringPlan exists (doc-1234 structurally enforced).

**Resolved open questions:** Dependency-based auto-config (global manifest vs capability-scoped): BOTH, layered as the toolkit fixes — participation = enabled capability feature ∪ scan! list (force-link-scoped, the Cfg-tier OnClass physically removes absent-capability candidates at compile), WIRING = the runtime CondExpr guard match over the AUTO_CONFIGS slice. The candidate set IS a function of linked capabilities, gated by runtime back-off.; Build-time vs runtime pipeline: RUNTIME R1 deferred batch is the DEFAULT (back-off needs the final runtime user-bean set); the build-time `cargo leaf prepare` plan is the OPT-IN accelerator whose residual lands in the SAME App<Resolve> slot. Phase 3 ships R1; the build plan is deferred (a Phase-4 maintenance-surface measure).; Deferral-for-correctness moment: the Register sub-pass of App<Resolve], AFTER all user defs, with incremental register_lite so each OnMissingBean DefinitionProbe sees a growing-then-final set. Survives the partly-cfg strategy because cfg leaves are already pruned/decided before the runtime OnBean leaves evaluate.; Two-phase fast-path: the Cfg-tier compile pruning (absent-capability candidates physically removed) IS the rust-native spring-autoconfigure-metadata pre-filter; the runtime set is small.; Starter mechanism: FIXED by autoconfig-starters-bom — dep:-hidden umbrella feature → leaf-starter-* aggregator → integration crates auto-force-linked + ExpectedManifest; capability starter vs stack starter; resolver='3' + [workspace.dependencies] single-copy invariant.; Anti-DCE self-check distinguishing missing-default/back-off/typo: the single substrate self-check (DCE-vanished=loud AntiDceError) + Exclusion(ContractId)=no-BeanId + Negative(reason)=silent + CompiledOutByCfg=feature-off — four crisply distinct report classes, one leaf-doctor account. An enabled-starter-zero-contribution is a loud self-check warning.; condition-not-met vs wiring-broken: kept distinct via the report (silent Negative now; the App<Wired> NoSuchBean enriches from it later). One accountable account across tiers.; Two-source WASM duality: DROPPED — native-only; the engine is origin-blind regardless but no dynamic candidate lane is built.; Test-slice mode: `slice!{...}` runs the SAME engine over an explicit `&[&Descriptor]` candidate list (the engine takes candidates as a parameter); optional-if-absent entries skipped when their crate isn't force-linked; same exclusion/back-off ladder.; Engine read-mostly world view + incremental mutate: `AssemblyCtx { env: &Env, defs: &mut DefinitionView, probe: &DefinitionProbe, exclusions: &ExclusionSet }` over the borrowed growing RegistryBuilder — single-threaded synchronous loop, no global lock (the builder is owned by App<Resolve>, not shared).; @Fallback vs OnMissingBean interaction (the open Phase-3 question): auto-config beans register as `CandidateRole::Fallback` by DEFAULT (the soft override for the common 'user overrides default' case); an explicit `#[on_missing_bean]` guard is the belt-and-suspenders all-or-nothing back-off reserved for when the candidate must not even register (e.g. it would conflict at definition level). Default = Fallback role; OnMissingBean is opt-in on top.; Kill-switch granularity: `leaf.enable-autoconfiguration=false` is the global off-switch read first (short-circuits the whole batch before exclusions); per-starter disable is expressed as `exclude=`/`leaf.autoconfigure.exclude` by ContractId (not a separate per-starter property in v1).

### `auto-config-ordering`
A pure deterministic function in leaf-boot: `order_batch(candidates: &[&Descriptor], excl: &ExclusionSet) -> Result<Vec<&Descriptor>, OrderingError>`, run inside `run_autoconfig`'s cold pass. Ordering data rides `Descriptor` (a new `OrderHint` in meta): `OrderHint { order: i32 /*DEFAULT 0*/, before: &'static [ContractId], after: &'static [ContractId] }` plus inline `before_name`/`after_name` resolved against the candidate name index. Three passes, in order: (1) STABLE alphabetical base by the candidate's canonical name (the deterministic anchor making results reproducible regardless of link/section order — never rely on linkme collection order); (2) STABLE sort by `order: i32` via `leaf_core::order::cmp_order` over `OrderKey{value, source}` (the ONE comparator, NOT collection-injection's @Order — auto-config priority is a distinct `OrderSource`); (3) STABLE topological sort honoring before/after edges with EXPLICIT cycle detection emitting `OrderingError::Cycle{path: Vec<ContractId>}` naming the contributors. `SequenceKey(epoch, ContractId)` is the deterministic tie-break BELOW cmp_order (cross-crate-discovery owns it). Exclusions: `ExclusionSet::merge` over THREE sources — typed `exclude=[ContractId]` (compile-known via `#[leaf::main(exclude=[...])]`), `exclude_name` strings, and the `leaf.autoconfigure.exclude` Env list property (relaxed-bound list) — all keyed/resolved to `ContractId`. ### `profiles`
Profiles are the `ON_PROFILE` `CondExpr` preset — NOT a parallel engine. leaf-core hosts `pub const ON_PROFILE: ConditionId` with the fixed tier-map entry `(TIER=Runtime, SUB=Parse)`, `pub enum ProfileExpr { Name(&'static str), Not(&'static ProfileExpr), And(&'static [ProfileExpr]), Or(&'static [ProfileExpr]) }`, and the canonical `pub struct ActiveProfiles { set: HashSet<Arc<str>>, ordered_provenance: Box<[(Arc<str>, ActivationReason)]> }` (Send+Sync, lock-free read). The thin `#[profile("prod & (eu | us)")]` / `#[profile("p1","p2")]` macro parses the grammar at MACRO time into a `const ProfileExpr` and emits `Leaf(ON_PROFILE, attrs)` onto `Descriptor.meta`; mixed `&`/`|` without parens, empty names, illegal chars are `compile_error!` with a span (the fail-fast-on-ambiguity intent as a Tier-0 error). Array form → `Or([...])`; negation matches absence (`Not(Name)` = `!set.contains(name)`, so `!prod` registers under default). The activation ALGEBRA is owned by profiles but RUNS inside `seal_environment()` (the env analogue of seal): a pure `resolve_active(levers: ProfileLevers, validate: bool) -> Result<ActiveProfiles, ProfileError>` encoding every buried rule as named branches: empty active∧empty include → `{default}`; any explicit active∨include → `default` dropped entirely (`default_suppressed_by_explicit`); include prepended before active; group fan-out applied TRANSITIVELY with cycle detection (`ProfileError::GroupCycle{path}`); gated name validation. `ProfileLevers{active, include, groups, default}` is HARVESTED by environment-config/config-data from the property stack (transport), not by profiles — profiles owns only the algebra. The `OnProfile` `Condition` impl reads the const `ProfileExpr` and evaluates `profiles::matches(&expr, env.active_profiles())` — a pure synchronous read over the sealed `ActiveProfiles`, lock-free. config-data's `spring.config.activate.on-profile` reuses the IDENTICAL `ProfileExpr` grammar+evaluator (one grammar, two phases: documents filtered during the load inside seal_environment; beans filtered in App<Resolve>). The same parser is exposed as a runtime `accepts_profiles("<str>") -> Result<bool, ProfileParseError>` escape hatch for dynamic expressions.

**Resolved open questions:** Conditional strategy side: profiles are ALWAYS Runtime/Parse tier (a fixed ON_PROFILE row in the per-kind tier-map) even when sibling conditions are static — the coherence constraint, made coherent because ON_PROFILE is just one more leaf kind. NOT a cfg/feature (features can't be un-set; activation is ops-replaceable, mutually-exclusive-capable).; Where ActiveProfiles is produced / transport-vs-algebra boundary: profiles owns the pure `resolve_active(ProfileLevers)` ALGEBRA; environment-config/config-data own HARVESTING ProfileLevers from the property stack (the relaxed-bound active/include/group/default keys). resolve_active runs once inside seal_environment.; include 'per-property-source not list-merged': lives UPSTREAM in environment-config's harvesting (ProfileLevers carries the already-per-source-resolved include list); profiles' resolve_active takes a flat list — the buried rule is a property-stack concern, keeping the algebra pure.; Matching as method vs free fn: a free fn `profiles::matches(&ProfileExpr, &ActiveProfiles)` that `Env::accepts_profiles` delegates to — one seam shared by conditions, environment, and config-data (resolves the cross-design open question once).; Late/additive activation against immutable-after-seal: a pre-seal lever merge or a child-derived ActiveProfiles (for hierarchical contexts), NEVER in-place mutation — honors the seal_environment builder-consumed contract.; Group fan-out one-level vs transitive: TRANSITIVE with explicit cycle detection (ProfileError::GroupCycle) — a defensible robustness improvement over Spring's flat model, surfacing a cycle as a loud seal-time error.; Named profile-GROUP scope (R2) worth it: NO — profiles stay a per-element CondExpr preset; group-as-introspection is a condition-report VIEW, not a registration scope (avoids the near-parallel-engine debt and the 'group'-scope vs 'group'-lever name collision). ### `configuration-classes`
Lite-only canonical (Phase-1 R1): full-mode CGLIB illusion is DESIGNED OUT, not emulated — Rust cannot intercept an intra-impl method call, and lite is Spring's own best practice + the auto-config mandate. `#[configuration] impl AppConfig { #[bean] async fn pool(&self, cfg: Ref<DbConfig>) -> Result<Pool, ResolveError> {..} #[bean] fn repo(&self, pool: Ref<Pool>) -> Repo {..} }`. The thin macro reads each `#[bean]` fn and emits exactly ONE `Descriptor` + one `ProviderSeed` per method (the toolkit's registry-substrate currency), whose `Provider::provide` resolves each fn PARAMETER as an injection point through the `ResolveCtx` (origin-blind: native/test-double identically). The config struct is itself a registered `Descriptor` bean (so methods read `&self` shared injected state). Bean name = method name (bean-naming overlay); return type = the produced TypeId (declare the most concrete type for candidate matching). `#[bean(scope=.., init=.., destroy=.., primary, fallback, lazy, order=..)]` and `#[conditional(...)]`/`#[profile(...)]` map straight onto `Descriptor` fields + the `meta.guard` CondExpr. CRUCIAL lint: the macro inspects bodies and emits `compile_error!` with the rewrite hint (`take repo: Ref<Repo> as a parameter instead`) on an intra-config `#[bean]`→`#[bean]` self-call — turning Spring's silent lite-mode footgun into a loud diagnostic (the macro CAN see the body Spring's enhancer treats opaquely). Singleton-correctness is structural: a param resolves once through the candidate-resolver to the managed singleton, so the 'second unmanaged instance' bug is impossible. Async `#[bean]` fns are first-class (boxed-future at the dyn Provider edge — the toolkit's fixed async-across-dyn standard). Destroy is EXPLICIT by default (`#[bean(destroy=..)]` names it; no silent `close()`/`shutdown()` inference — leaf has no async Drop, teardown via the `TeardownLedger` from bean-lifecycle). A `#[configuration]` in any crate is an ordinary contributor to the `COMPONENTS` stream; `#[leaf::auto_config]` is the same lowering flagged into `AUTO_CONFIGS` with `CandidateRole::Fallback`.

**Resolved open questions:** Full-mode guarantee at all / how: NO full mode — lite-only (R1). ### `import-composition`
The four altitudes survive as thin metadata over the ONE descriptor stream (Phase-1 R1), with altitudes 2-3's deferred-group being EXACTLY the auto-config engine (auto-config IS the headline DeferredImportSelector.Group). Altitude 1: `#[configuration] #[import(DbConfig, MetricsConfig)] struct AppConfig;` — the macro emits an `ImportEdge{ from: ContractId, to: &'static [ContractId] }` const onto `Descriptor.meta`; at `App<Define>` lift, the builder walks edges transitively, de-dups by ContractId (diamond-safe), and includes every imported config's Descriptors in the active set. Because the edge names a real TYPE, the importer path-references it, solving Layer-0 DCE for that edge for free (the import graph is largely DCE-immune; the anti-DCE self-check only covers entry-point configs + selector/registrar discovery). Altitude 2 (immediate selector): `trait ImportSelector: Send + Sync { fn select(&self, meta: &ImportMetadata, cx: &SelectCtx<'_>) -> Vec<ImportRef>; }` where `ImportRef` is a CLOSED enum over `ContractId`/registered markers (no classloader string resolution); runs synchronously in the Parse sub-pass. Altitude 3 (deferred + group): collapses into `run_autoconfig` — a deferred selector registers candidates into the AUTO_CONFIGS batch, merged/de-duped/topo-sorted by auto-config-ordering's three-pass over SequenceKey, gated by the CondExpr guard in the Register sub-pass (the deferral-for-correctness moment). Altitude 4 (registrar): `trait Registrar: Send + Sync { fn register(&self, cx: &mut RegistrarCtx<'_>) -> Result<(), RegisterError>; }` (the discovery-codegen SPI, reused verbatim) — the imperative escape hatch minting Descriptors, with a macro lint forbidding it from registering a definition-phase post-processor (Spring's BDRPP rule as a loud diagnostic). `@Enable*`/ImportAware: a typed `#[derive(EnableAttrs)] struct EnableScheduling { pool_size: usize }` — the importer's attributes are a TYPED struct threaded to the importee as `EnableAttrs<EnableScheduling>` (an injectable read-only value), replacing Spring's stringly `setImportMetadata` reflection. Multi-importer attribute resolution: per-import-edge attribute INSTANCE (each edge carries its own EnableAttrs value, keyed by edge) — NOT a single merged value (resolving the orphaned injection-kind coherence note: import-composition owns it, per-edge-instance rule). Selectors/registrars are synchronous, IO-free, cold-path planning logic (the read-mostly ConditionCtx discipline). v7 @Import-on-interfaces is honored (an edge can name a config-implementing trait).

**Resolved open questions:** Which altitudes survive: ALL FOUR as thin metadata over one stream (R1), but altitude 3 (deferred-group) IS the auto-config engine (no separate Group subsystem) and altitude 4 IS the discovery-codegen Registrar SPI — maximal sharing, not four bespoke subsystems.; selectImports arbitrary class-NAME dynamism: NOT supported — ImportRef is a CLOSED set of linked ContractIds/markers (no classloader). ## Cross-feature interactions
- **conditions** — The substrate every other feature in this subsystem rests on: condition-family supplies the leaf catalog, profiles the ON_PROFILE preset, auto-configuration the OnMissingBean back-off, condition-report the sink, import-composition the selector/registrar gating, configuration-classes the per-#[bean] guard. ONE CondExpr enum, ONE evaluator, ONE outcome stream — no feature mints a second gating mechanism.
- **condition-family** — condition-family populates the per-kind tier-map (the conditional-strategy ABI) with concrete ConditionKind rows; conditions owns the map shape. The OnBean family delegates to injection-mechanics' DefinitionProbe; the OnClass member couples to cross-crate-discovery force-link (cfg pairs with link); profiles' ON_PROFILE is one more row in the same map.
- **condition-report** — The passive sink the one eval path writes during App<Resolve], frozen into FrozenContainer.report at seal(). It CONSUMES the single substrate anti-DCE self-check result (not a second manifest) and the CapabilitySnapshot to render the six verdict classes. The App<Wired> NoSuchBean enriches FROM it by ContractId — the silent-now/loud-later bridge.
- **auto-configuration** — The thin App<Resolve> orchestrator that DELEGATES gating to conditions (CondExpr/DefinitionProbe), ordering to auto-config-ordering (three-pass over SequenceKey), exclusion to cross-crate-discovery (ExclusionSet), and registration to the RegistryBuilder. It is the headline consumer of import-composition's deferred-group (it IS the Group) and registers lite-only configuration-classes #[bean]s at CandidateRole::Fallback.
- **auto-config-ordering** — Fixes DEFINITION order ONLY (doc-1234), feeding run_autoconfig's incremental register so each OnMissingBean DefinitionProbe sees deterministic predecessors. The ordered Vec is an App<Resolve> artifact structurally unreadable by App<Running>'s dependency-driven WiringPlan (compile-runtime-split typestate boundary).
- **profiles** — The ON_PROFILE CondExpr preset (always Runtime/Parse). Its resolve_active algebra runs inside seal_environment producing the canonical ActiveProfiles before any condition evaluates; config-data's spring.config.activate.on-profile reuses the IDENTICAL ProfileExpr grammar+evaluator (one grammar, two phases). A profile-gated auto-config is still ordered in the batch but backs off when inactive.
- **configuration-classes** — Lite-only configs whose #[bean]s are the Descriptors auto-configuration registers and import-composition arranges; #[bean] guards are CondExpr trees, #[profile]s are ON_PROFILE leaves. The config struct is itself a Descriptor bean. Auto-configs are #[leaf::auto_config] = the same lowering, flagged into AUTO_CONFIGS with Fallback role.
- **import-composition** — Altitudes 1-4 over the one descriptor stream: altitude-1 #[import] edges resolved at App<Define> lift (DCE-safe, by ContractId); altitudes 2-3 (deferred-group) ARE the auto-config engine; altitude-4 Registrar IS the discovery-codegen SPI. @Enable* is a typed EnableAttrs<A> threaded per-import-edge (the owned injection kind).
- **registry-core** — Builds on it directly: one Descriptor row (with new orthogonal CandidateRole + meta.guard CondExpr + OrderHint + ImportEdge), the dedicated AUTO_CONFIGS linkme slice beside COMPONENTS, ContractId identity, the RegistryBuilder.register/freeze seal boundary, and the DefinitionProbe via the candidate-resolver. Re-keys/re-stores NOTHING; auto-config beans are ordinary Providers the Engine cannot tell apart (origin-blind, CandidateRole is a candidate-resolution input only).
- **environment-config** — seal_environment() ends the 5f async fence and produces the sealed Env + canonical ActiveProfiles (profiles' resolve_active runs there) BEFORE the Parse sub-pass. All Parse-tier leaves (OnProperty/OnProfile/OnResource) read the lock-free Env. The kill-switch + leaf.autoconfigure.exclude property are read from the sealed Env in App<Resolve>. ProfileLevers harvesting (transport) is environment-config's job; profiles owns only the resolve_active algebra.
- **discovery-codegen** — Reuses the linkme/anti-DCE pipeline (force-link + section-retention + ExpectedManifest self-check) for the AUTO_CONFIGS and CONDITIONS channels; the Registrar SPI is discovery-codegen's, reused for import altitude-4; the thin #[conditional]/#[profile]/#[bean]/#[auto_config] macros emit const data via ::leaf_core paths (compile-time merge, no runtime overlay).
- **bean-lifecycle** — #[bean] destroy hooks register Destroyers in the TeardownLedger (no async Drop, explicit-by-default — no silent close() inference); auto-config/lite-config factories are Providers driven by Engine::create through the single publish pipeline; a condition-gated-out bean simply never registers a definition (no lifecycle participation).
- **injection-mechanics** — OnBean/OnMissingBean/OnSingleCandidate ARE injection-mechanics' DefinitionProbe.would_resolve_unique (the no-instantiation candidate-resolver running primary/fallback/qualifier policy over the definition view) — the SAME primitive App<Wired> whole-graph validation uses. CandidateRole::Fallback is a candidate-resolver tie-break input (a Fallback loses to any non-fallback of the same contract).
- **compile-runtime-split** — All runtime work of this subsystem IS the body of App<Resolve> (route_conditions + run_autoconfig) between seal_environment and seal(); the ConditionReport freezes into FrozenContainer at seal(); the App<Wired> NoSuchBean enriches from it. condition-not-met (silent) stays distinct from the App<Wired> AssemblyReport (loud, aggregated) — the four-tier timing ladder.

## Remaining risks (→ Phase 4)
- @Fallback role vs OnMissingBean overlap: Phase 3 makes CandidateRole::Fallback the default soft-override and reserves explicit OnMissingBean for the hard all-or-nothing case. The precise candidate-resolver arithmetic when a contract has {one Fallback auto-config, one user Normal, plus a @Primary among them} must be co-verified with injection-mechanics' candidate-resolver and the collection-injection expected-count self-check (which subtracts excluded/backed-off candidates) — a Phase-4 cross-ADR test target.
- CapabilitySnapshot trustworthiness: the CompiledOutByCfg report label and the OnClass cfg verdict depend on a single authoritative resolved-feature snapshot from the binary's build step. The resolved set differs between -p and --workspace and can be flipped by a sibling via feature unification; resolver='3' mitigates but Phase 4 must validate the snapshot is computed from the FINAL resolved set and that a crate whose ONLY rows were all cfg-gated-out is not misclassified as DCE-vanished (shared VanishReason inference with cross-crate-discovery).
- DefinitionProbe definition-only mode: OnBean/OnMissingBean/OnSingleCandidate and App<Wired> validation both require the candidate-resolver to run primary/fallback/qualifier policy over the DEFINITION view with zero instantiation. This must be the same code path (one definition of 'unambiguous'); a divergence would make back-off and startup validation disagree. Verify with injection-mechanics that the candidate-resolver exposes a true no-instantiation mode.
- cmp_order ownership across the subsystem: auto-config-ordering's order(i32) pass uses leaf_core::order::cmp_order with a dedicated OrderSource distinct from collection-injection's @Order; the SequenceKey(epoch, ContractId) tie-break sits below it. Phase 4 must confirm auto-config priority and injection @Order do not alias through the same OrderKey and silently reorder one another.
- ProfileLevers transport/algebra boundary: profiles owns resolve_active but environment-config owns harvesting (including the per-property-source-not-list-merged include rule). If ProfileLevers is over-flattened, the include rule is lost; if it carries per-source structure, transport leaks into the algebra. The exact ProfileLevers shape is a co-decided seam with environment-config to finalize.
- Per-import-edge EnableAttrs injection kind: import-composition now owns this new injection kind (a config imported by two importers receives two edge-keyed EnableAttrs instances). annotation-model must reserve the descriptor shape for it before it freezes; the resolution rule (per-edge instance, not merged) must be honored by autowiring so it doesn't get treated as an ordinary ambiguous by-type injection.
- Opt-in cargo leaf prepare build plan deferral: shipping only the R1 runtime batch in v1 means the build-frozen/runtime-residual seam legibility is untested until the plan ships. If startup cold-cost over hundreds of candidates proves too high on hostile targets, the plan becomes load-bearing and its manifest convention + --check + OUT_DIR hygiene must be designed — measure cold-assembly cost in Phase 4 before committing.
