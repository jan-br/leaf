# Instantiation, Scopes & Bean Lifecycle  `[bean-lifecycle]`

_Single-phase Engine::create (construct->populate->init->publish) through the Provider/Published seam: scope as three data axes, OnceCell singleton publication/concurrency contract, lazy/eager waves, init/destroy/aware/smart-init callbacks, background-bootstrap, the post-processor SPI, and @DependsOn._

This subsystem owns the value-phase of leaf: turning a frozen `Descriptor`+`Provider` row into a live, published bean, and tearing it back down. It is `Engine::create` — the single concrete creation driver named in the container-shape ADR — plus the three data axes of scope, the `OnceCell` singleton publication contract, the lazy/eager WiringPlan-driven waves, the per-bean init/destroy/aware/smart-init callback chains, background-bootstrap, the post-processor SPI, and @DependsOn forced ordering. Phase 2 already pre-decided the hard architectural calls this subsystem's Phase-1 features each independently raised: construction is SINGLE-PHASE (no two-phase construct-then-populate, no early-exposure middle cache, no `&mut self` populate hook); the pluggable-creation-strategy SPI (bean-instantiation R3) and the swappable Scope SPI as `Box<dyn>` are REJECTED in favor of one concrete driver and scope-as-data; cycles are deferral-edge-only and constructor cycles are fatal at the `App<Wired>` validation pass; there is no global lock (publication is the `OnceCell` per-slot data shape, the atomic Arc clone is the happens-before edge). Phase 3's job is therefore not to choose between R1/R2/R3 — it is to express every feature as a concrete consumer of the already-fixed primitives, resolving each feature's openSubQuestions against them. The result is radically unified: one `Provider::provide -> BoxFuture<Published>` seam, one `Engine::create` driver, one `LifecyclePlan` const per bean (init/aware/destroy steps as fn-pointers), one `TeardownLedger` for all async cleanup, one `cmp_order` comparator for every ordering need, and one `Cx`/`AmbientStore` for every context-scope store. The whole subsystem is the body of the `start().await` async fence and the `shutdown().await` drain that bracket `App<Running>`.

## Shared machinery (how this rests on the toolkit)

Every feature here rests on the SAME fixed Phase-2 primitives; the unification is that they share them rather than each minting a mechanism:

1. ONE creation seam + ONE driver. `Provider::provide(&ResolveCtx) -> BoxFuture<Result<Published,ResolveError>>` is the only construction primitive, and `Engine::create(id, cx)` is the only driver. Native beans, FactoryBean products, the aware-installing infrastructure beans, background-marked beans, and prototype factories are all just `Provider`s the driver cannot tell apart (origin-blind: `Origin` is diagnostic-only). The Phase-1 "pluggable creation strategy" (bean-instantiation R3) and "dyn Scope SPI" (scopes R1) are NOT minted — populate/init/publish/guard are each one concrete step inside `Engine::create`.

2. ONE publication contract via `Published` + the `OnceCell` store. `{Once,PerContextKey} -> Published::Shared(ErasedBean)`, `PerResolution -> Published::Owned(Box)`. The singleton store IS `registry.singletons: Box<[OnceCell<ErasedBean>]>` indexed by `BeanId.0`; `get_or_init` is the creation guard (at-most-once by data shape, lock-free ready reads — no global mutex, no strict/lenient lock dance). The atomic Arc clone is the safe-publication happens-before edge — this IS the concurrency-contract, carried by the `Send+Sync+'static` bound on `ErasedBean`/`Bean` in one leaf-core location.

3. SCOPE = three data axes `ScopeDef{Multiplicity, StoreSource, TeardownPolicy}`, never a handle type and never a `Box<dyn Scope>`. `StoreSource::ContainerStore` -> the `OnceCell` array; `StoreSource::AmbientStore(ScopeKind)` -> a per-`Cx`-key `InstanceStore` reached through the async-context-model's `Cx` bundle (never a ThreadLocal). Custom scopes contribute a const `ScopeDef` triple + a `ScopeKind` registration; they pick where instances live, never a new mechanism.

4. ONE const lifecycle metamodel. The macro emits a `LifecyclePlan` (init steps, aware-want bitset, destroy steps, smart-init flag, depends_on contract list) as a flat const on/beside the `Descriptor` — the same thin-macro/`::leaf_core`-paths discipline as the `Descriptor` itself. Callbacks are origin-agnostic `fn(&dyn Any, &Cx) -> BoxFuture<Result<(),CallbackError>>` fn-pointers (boxed-future-on-dyn is the fixed async-across-dyn standard). Participation is DATA (a flag/bitset), never runtime instanceof.

5. ONE teardown path. The `TeardownLedger` (ownership-model) drained LIFO by `Context::shutdown().await` after `stop_all()`. The singleton destroyer, a context-scope's per-key destroyers, a half-built bean's compensating partial-destroy, and the cancelRefresh unwind are ALL ledger entries — no async Drop anywhere; sync-only cleanup rides RAII `Drop`. The Arc strong-count governs the real free, so in-flight scoped work past a boundary stays sound.

6. ONE plan + ONE comparator + ONE error path. The eager waves come from `App<Wired>::validate()`'s `WiringPlan` (dependency-ordered, lazy beans excluded from the eager bitset, @DependsOn folded as forced construction edges). All ordering (init/destroy phase order, lifecycle start/stop phases, smart-init participant order) calls `leaf_core::order::cmp_order`. All failures are `ResolveError`/`CallbackError` -> `LeafError`/`Diagnostic`, aggregated at the validation pass or surfaced as a typed runtime fault during the `start().await` eager pass.

7. The post-processor SPI is dogfooded infrastructure, not a kernel SPI surface beyond the one decoration seam. The instance-decoration "after_init" hook IS the proxy-substrate auto-proxy installation point (origin-agnostic `ErasedBean -> ErasedBean` swap); the definition-tier (BFPP) work is `App<Resolve>`-phase `Registrar` emission already owned by the autoconfig/codegen pipeline. Aware-provision and smart-init are themselves expressed as the framework's own `Role::Infrastructure` participants, not a parallel system.

## Features

### `bean-instantiation`
ONE concrete driver, single-phase, no strategy SPI. `Engine::create(id, cx) -> Result<Published, ResolveError>` is: `let p = &self.registry.providers[id.0]; let scope = self.registry.rows[id.0].scope;` then branch on `scope.multiplicity`: (Once) `let cell = &self.registry.singletons[id.0]; cell.get_or_init_async(|| publish_pipeline(p, cx)).await` returning `Published::Shared(cell.clone())`; (PerContextKey) resolve the ambient `InstanceStore` via `cx.scope_store(scope.store)` and call its `get_or_init` keyed by `BeanId`; (PerResolution) run `publish_pipeline` once, no store write, return `Published::Owned`. `publish_pipeline` = `let raw = p.provide(cx).await?;` (the macro-emitted factory whose typed params ARE the injection points — populate is FUSED into provide, there is NO separate populate step and NO early-exposure middle cache, per the Phase-2 single-phase decision) `-> run_aware(&raw, cx).await? -> run_before_init_decorators(&raw).await? -> run_init(&raw, cx).await? -> raw = run_after_init_decorators(raw)?` (the proxy swap point) `-> register_destroyer_if_shared(id, &raw, cx); Ok(into_published(raw, scope))`. The OnceCell `get_or_init` IS the creation guard — there is no per-bean keyed Mutex, no strict/lenient policy; concurrent first-creators of the same singleton race into `get_or_init` and exactly one wins (lock-free ready read after). The eager pass is `App<Wired>::start().await` walking `WiringPlan.waves`: each wave's BeanIds whose `eager` bit is set are constructed (wave-parallel via the `Spawner`, see background-bootstrap); a constructor body fault is the ONLY failure mode here (all wiring already validated at `App<Wired>`), aborts start, and triggers cancelRefresh ledger unwind. Prototype hand-off-and-forget is structural: `Published::Owned`, no `singletons` slot, no destroyer registered.

**Resolved open questions:** Singleton-handle type: fixed by ownership-model as ErasedBean=Arc<dyn Any+Send+Sync>; arena/BeanRef is the deferred opt-in fast lane only, NOT used here.; Creation-guard substrate: the OnceCell-per-slot store IS the guard (no keyed Mutex, no strict/lenient lock); the Phase-1 hardest-won lenient-locking code is dissolved by data shape, since planned waves prove intra-wave independence so distinct beans never contend and same-bean contention is the OnceCell's at-most-once.; Two vs three cache levels: TWO (ready=OnceCell, no early-exposure middle level) — single-phase construction means there is never a half-built instance to expose, so the middle cache is eliminated; circular references are deferral-edge-only.; async-across-dyn boxing: BoxFuture is the fixed standard; the monomorphized known-concrete fast-path is the deferred Phase-3 escape (measure-first), not used by the default driver.; Validation-to-build-time: wiring validation is the App<Wired> whole-graph pass; construction in App<Running> can only fail for genuine constructor/IO faults.

### `scopes`
Scope is the const `ScopeDef{multiplicity, store, teardown}` triple on the `Descriptor`, consumed by `Engine::create`'s branch (above). NO `Box<dyn Scope>` trait. Built-in `SINGLETON`/`PROTOTYPE`/`REQUEST` are the consts already in ownership-model. The `InstanceStore` trait (for `StoreSource::AmbientStore(ScopeKind)`) is reached through the async-context-model `Cx`: `cx.scope_store(ScopeKind) -> Option<&dyn InstanceStore>` reads a context-scope binding from the ambient `Cx` bundle (an `Isolate`-or-`Inherit` `CxKey` the web/request layer installs at the scope boundary via `Holder::scope(store, fut)`). `InstanceStore::get_or_init(BeanId, provider, cx) -> BoxFuture<ErasedBean>` mirrors the singleton OnceCell but is keyed inside the per-context map and DROPPED at scope-end (refcount->0 unless in-flight work holds a clone — the Arc is what makes referenced-past-the-boundary SOUND). Custom scopes register a `ScopeKind` + supply an `InstanceStore` impl bound into a `CxKey`; they NEVER define a handle type. The singleton-holds-shorter-scope gotcha is resolved by the injection-resolution subsystem's deferral family (`Lookup`/`LazyRef`) or a scoped-proxy `TargetSource` (proxy-substrate) — not re-solved here. Scope-mismatch (a singleton constructor-injecting a request-scoped bean by value) is an `App<Wired>` validation error (ScopeMismatch class). Custom-scope teardown-on-scope-end drains that store's ledger entries; nested scopes (request-in-session) are nested `Cx` derivations — the inner store's ledger drains first on the inner boundary, the Arc strong-count bounds in-flight stragglers to the deadline.

**Resolved open questions:** dyn Scope SPI rejected: scope is data (ScopeDef triple) + a concrete InstanceStore impl reached via Cx, not a five-method async trait the engine calls per resolution (no boxed-future-per-get tax for the common singleton path, which hits OnceCell directly).; Who owns the singleton map vs the Singleton scope object: the registry owns it (singletons:Box<[OnceCell]>); there is no Singleton scope object — Multiplicity::Once is a branch in Engine::create, not a dyn dispatch.; Web-scope across .await: solved by the async-context-model Cx re-installed per poll (never ThreadLocal); the InstanceStore binding rides an Isolate/Inherit CxKey.; Nested-scope teardown ordering: inner scope (request) ledger drains at the inner Cx boundary before the outer (session); straggler Arc clones bound by the per-bean shutdown deadline.; Custom-scope handle ownership: always the uniform ErasedBean (Published::Shared); a custom scope picks StoreSource+TeardownPolicy, never an owned/borrowed divergence.

### `lazy-initialization`
Lazy is a GATE on the `WiringPlan.eager` bitset, not a construction path. A `#[lazy]`/`lazy=true` descriptor still registers and freezes; `App<Wired>::validate()` still validates its definition and its injection points, but `validate()` CLEARS that BeanId's bit in the eager bitset so `start().await` skips it. First `Context::get::<T>()`/first injection resolution then drives `Engine::create` on-demand (the same driver, same OnceCell guard — so first-access build is at-most-once and lock-free thereafter). The global flag is an `App<Resolve>`-phase default applied to every descriptor that did not explicitly set its lazy flag, with the THREE structural exclusions enforced as hard rules at that phase: `Role::Infrastructure` beans stay eager; smart-init participants (the `smart_init` flag) stay eager (else their once-after-all-singletons hook never fires); and an `exclude` predicate SPI (a const-fn or a `Registrar`-supplied filter run in `App<Resolve>`) can re-include beans. `lazy(false)` is the explicit per-bean override. The central trap (direct injection of a lazy bean by an eager bean silently forces eager build) is handled by the WiringPlan: if any eager bean has a non-deferral construction edge to a lazy bean, the lazy bean's eager bit is FORCED back on (it must be built before its eager dependent) and a diagnostic notes the laziness was defeated — laziness only holds when the edge is a `Deferral`/`Lazy` kind (injection-resolution's `LazyRef<T>`, which removes the construction edge). A lazy bean carrying lifecycle-event/listener registration that is never built is a `App<Wired>` warning (the @EventListener-silently-lost footgun made loud).

**Resolved open questions:** Deferred-handle/proxy realization: reused from injection-resolution (LazyRef<T> = OnceCell-cached for singleton targets; the deferral edge is removed from the construction graph) — not re-invented; the singleton slot the lazy bean eventually fills is the same OnceCell.; Global-flag locus: App<Resolve> phase (runtime tier), applied to the growing definition set before seal(); explicit per-bean flags win over the global default.; Exclude-filters: an App<Resolve> predicate (const-fn or Registrar-supplied), resolved before seal() so the eager bitset is final at App<Wired>.; Lazy + ready-hook (smart-init) contradiction: force-include — a smart-init participant is excluded from global-lazy (its eager bit stays set), matching Spring SPR-7014; a per-bean lazy(true) on a smart-init bean is an App<Wired> error.; Lazy + auto-start lifecycle bean: still started (runtime-lifecycle reads the participant set, not the eager bitset).

### `lifecycle-callbacks`
Two mirrored chains as DATA on a const `LifecyclePlan` (R2's single-table shape adopted for the mirror-can't-drift property, emitted as flat const per R1's thin-macro discipline): `LifecyclePlan { init: &'static [LifecycleStep], destroy: &'static [LifecycleStep], aware_wants: AwareFlags, smart_init: bool, depends_on: &'static [ContractId] }` where `LifecycleStep { phase: LifecyclePhase, call: LifecycleFn, id: StepId }` and `LifecycleFn = fn(&(dyn Any+Send+Sync), &Cx) -> BoxFuture<'_, Result<(), CallbackError>>`. Init phases (canonical discriminant order): `PostConstruct < AfterPropertiesSet < InitMethod`; destroy phases: `PreDestroy < DisposableDestroy < DestroyMethod`, run in REVERSE. The macro emits one fn-pointer per annotated `#[post_construct]`/`#[pre_destroy]` async method or trait impl (annotation-over-interface DX: a plain annotated method stays a POJO). `run_init` is invoked by `Engine::create` as the init step, AFTER aware + before-init decorators, BEFORE after-init decorators/publish — and crucially HOLDS NO GUARD (the OnceCell `get_or_init` future is the only thing in flight; user init `.await`s never sit under a held lock — Spring's @PostConstruct-under-lock deadlock dissolved by execution context). Publication gate: `into_published` runs only after `run_init` returns Ok. DEDUP: `StepId` (a macro-assigned stable id per source method) collapses a method serving two mechanisms to one step (earlier init phase / later destroy phase). DESTROY-METHOD INFERENCE: a runtime trait-object probe `if Some via downcast to dyn Closeable` in leaf-core (token-only macro cannot resolve types), emitting a synthetic `DestroyMethod` step UNLESS `#[no_destroy_inference]`; a loud diagnostic ('leaf will call X.close() at shutdown; annotate #[no_destroy_inference] to suppress') guards the silent-teardown hazard. SINGLETON: `register_destroyer` pushes a `Destroyer{bean_id, run: closure over the destroy chain + ErasedBean}` onto the `TeardownLedger`. PROTOTYPE: pushes NOTHING (never-destroyed structural). PARTIAL-DESTROY on init failure: the runner derives the mirror of init phases < N (the failing phase) and runs those destroy steps on the unwind, via the ledger.

**Resolved open questions:** Discovery: annotation-metadata via thin macro emitting fn-pointers (POJO stays leaf-trait-free) is the primary; the trait form (InitializingBean/DisposableBean analogue) is the typed escape hatch — both feed the same const LifecyclePlan table.; Callback seam shape: erased fn(&dyn Any, &Cx) -> BoxFuture (origin-agnostic, boxed-on-dyn per the fixed async standard); monomorphized typed escape is deferred.; Destroy-method inference placement: runtime dyn-Closeable probe in leaf-core at registry-build (cold), not macro-time; opt-out + loud diagnostic mandatory.; Partial-construction cleanup: derived mirror-of-init-steps-that-ran, run via the same TeardownLedger on the cancelRefresh/init-failure unwind; no async Drop.; Pluggable LifecycleContributor SPI (R3) rejected: it widens the ABI and overlaps post-processor-spi; the fixed init/destroy/aware/inference sources are emitted by the thin macro into one const table.; Dedup identity: a macro-assigned StepId (not fn-pointer equality, which has codegen-unit caveats).; FactoryBean product teardown: the product's destroyer rides the factory's registered destroyer (disciplined, no orphan leak) — co-decided with factory-bean.

### `aware-callbacks`
Awares are a GROUP fired as the first sub-step of `run_init` inside `Engine::create`, gated by the `AwareFlags` bitset on the `LifecyclePlan` (cheap bitmask skip; no speculative downcast). The Phase-1 R2 insight (construction-seam delivery is idiomatic Rust) is adopted as the PRIMARY door, with R1's post-population callback as the secondary door for facts only knowable after registration: most infrastructure (Environment, EventPublisher, ResourceLoader, MessageSource, the Context) is delivered by ORDINARY constructor injection — these are `Role::Support` beans resolvable by type through the injection-resolution Selector (the `resolvable_dep` layer / a `HashMap<TypeId,ErasedBean>` of always-present infra handles), so an infra bean simply takes `Ref<Environment>` as a constructor param and there is NO setter-into-a-shared-bean, NO OnceCell-per-aware. For the residual Spring-exact timing (a bean needing its OWN registered name, or the Context handle post-build), a single optional grouped hook `async fn on_context_ready(&self, infra: &ContextInfra) -> Result<(),CallbackError>` fires at the aware slot; `ContextInfra` is a thin borrowed bundle of always-present `Context` handles (`bean_name()`, `env() -> &Env`, `events()`, `resources()`, `messages()`, `context()`). The factory-tier (bean_name/factory) is available on a bare `Engine`; the context-tier handles are present only when the `Context` façade installed them (so context awares are inert on a bare engine — structural, exactly Spring's plain-BeanFactory parity). The callback-vs-by-type duality is single-sourced: the SAME infra `ErasedBean`s in the resolvable-dependency map back both `on_context_ready`'s bundle accessors and a by-type `get::<Environment>()`. Infrastructure-only steering: a `#[diagnostic]` warns if `on_context_ready` is impl'd on a non-`Role::Infrastructure`/`Support` bean. BeanClassLoaderAware is DROPPED (no Rust classloader). Generics with awares hard-error (no 'static symbol).

**Resolved open questions:** Delivery shape: construction-seam injection (the dominant door, dissolves interior mutability) + one optional post-population on_context_ready hook for registration-fact-dependent needs (e.g. own bean name).; Injected-handle ownership: owned Arc-backed cheap-clone handles (Env=Arc<EnvCore>, Ref<T> for the rest) a bean MAY retain — the uniform shared-handle currency, no borrow-leaking lifetimes into get<T>.; Taxonomy granularity: collapsed to constructor injection + one grouped ContextInfra hook (not ten traits), with named-by-type resolution preserving the fine-grained 'I need exactly Environment' signal.; How 'does this bean want aware X': the macro-emitted AwareFlags bitset on the descriptor (thin const), not trait-upcast probing or per-trait Any downcasts.; Async-across-dyn: on_context_ready is the boxed-future-on-dyn standard; runs off any held guard (no executor-blocking).; Callback-vs-by-type single source: one resolvable-dependency map backs both — Spring's ignoreDependencyInterface/registerResolvableDependency split collapses to one source.; Capability-set openness: CLOSED core set (Env/events/resources/messages/context) delivered as Role::Support beans; a new infra capability is a new bean by type, not a kernel aware-trait — no WantsContext/CapabilityProvider SPI (R3 rejected as ABI-widening).

### `smart-initializing`
R1 adopted (literal lock-free second loop), implemented as a `start().await` sub-step. A bean opts in via the `smart_init: bool` flag on its `LifecyclePlan` (the macro sets it when the type implements the `AfterSingletonsReady` trait, or carries an `#[after_singletons_ready]` method). After the final WiringPlan wave completes AND all background-bootstrap join handles have resolved (the barrier), `Engine` walks the eager-singleton publication set in `cmp_order` order and, for each whose `smart_init` flag is set, awaits `bean.after_singletons_ready(&ready_cx)` — OUTSIDE any guard (no OnceCell future in flight; an `.await` here cannot block a guarded worker). `ReadyCx` is a read-only view exposing the complete-graph registry queries (`all_of::<T>()` / `get::<T>()` via the injection-resolution Selector) — SAFE because every eager singleton already exists, so a query cannot trigger a half-built peer (it may still lazily build a genuinely-`@Lazy` bean on demand on this task). It is NOT unified with lifecycle-events' ContextRefreshed milestone (R2 rejected to avoid coupling a core bring-up guarantee to the events subsystem and to preserve the inside-finishBeanFactoryInit vs after-finishRefresh timing distinction): smart-init fires inside the eager pass, BEFORE runtime-lifecycle auto-start; ContextRefreshed fires after. The @EventListener-discovery dogfooding is vestigial in Rust (listeners are macro-discovered at link time, not runtime-swept), so smart-init serves genuine cross-bean AGGREGATION, not method discovery. Failure aborts start all-or-nothing with a rich error; a hook that acquired async resources before failing has its cleanup run via the TeardownLedger on the cancelRefresh unwind. Fired exactly once; never for lazy/prototype/scoped beans (they are not in the eager publication set).

**Resolved open questions:** Bespoke vs unified-milestone vs work-ledger: bespoke AfterSingletonsReady trait + flag (R1) — keeps the core bring-up guarantee independent of the events subsystem; the work-ledger (R3) and event-milestone (R2) rejected.; Distinction from ContextRefreshed preserved: smart-init inside the eager pass (graph complete, components not yet running); ContextRefreshed after runtime-lifecycle auto-start.; Invocation over erased handle: boxed-future-on-dyn (AFIT not dyn-safe), cold and once-per-refresh so negligible.; ReadyCx query contract: read-only complete-graph view over the frozen Registry + Selector; a query of a genuinely-@Lazy bean builds it on demand on the same task.; @Lazy + ready-hook: force-include (handled in lazy-initialization).; Firing barrier vs background-bootstrap: after the final wave AND all background join handles resolved; a background-built participant is itself in the set.; Sequential vs concurrent participants: sequential in cmp_order for deterministic diagnostics (the graph is read-only so concurrent is a deferred optimization).; Multiple work items per bean: held to one hook per bean (predictable mental model; R3's multi-item rejected).

### `background-bootstrap`
Adopted as R1's marker-spliced-into-the-eager-pass, but the wave structure comes from the WiringPlan (so R2's plan-proven-independence advantage is inherited for free — the planner already partitioned beans into dependency waves, so intra-wave beans ARE independent). A `bootstrap: Background` flag on the descriptor (the macro emits it like `lazy`/`primary`). During `start().await`'s wave walk, an UNMARKED eager bean is constructed inline on the bootstrap task; a MARKED bean's `Engine::create` future is `Spawner::spawn`ed (the async-context-model `ExecutionFacility`, resolved as the `Role::Infrastructure` bootstrap executor) and its `SpawnHandle` pushed into a per-refresh `Vec<SpawnHandle>`. There is NO strict/lenient creation guard — the OnceCell `get_or_init` is the only guard and the wave plan proves intra-wave independence (Spring's hardest-won lenient-locking code is eliminated, not ported). Dependency-edge modeling falls out of wave ORDERING: a non-deferral dependent is in a LATER wave, so it simply isn't started until the dependency's wave (including its spawned background future) completes — realized by awaiting the relevant `SpawnHandle`s at the wave boundary; a deferral (`LazyRef`) dependent shares a wave (no edge), the parallelism enabler; @DependsOn forces an earlier-wave edge built before the marked bean is spawned. JOIN-BEFORE-REFRESH: after the final wave, all `SpawnHandle`s are awaited (try_join) before smart-init fires; any background fault aborts start all-or-nothing. EXECUTOR-REQUIRED is LOUD (charter §2.9 divergence from Spring's silent-ignore): if any bean is background-marked but no `ExecutionFacility`/bootstrap executor resolves, the `App<Wired>` self-check fails naming the marked beans (the missing-executor is already a HARD-FAIL at refresh per async-context-model, so this composes). Background futures must be `Send+'static` (they run on the multi-thread executor) — a non-Send factory on a marked bean is a compile/validate error via the `SpawnableWork` diagnostic. Cancellation cancels the spawned futures (SpawnHandle drop = abort); partial builds run their compensating partial-destroy via the ledger.

**Resolved open questions:** Marker vs computed-eligibility: marker (R1) decides WHAT runs background; the WiringPlan waves provide the independence proof (R2's dividend) so no per-bean lenient guard is needed.; Executor handle: the async-context-model ExecutionFacility (Spawner) resolved as the Role::Infrastructure bootstrap executor by qualifier; missing = loud HARD-FAIL (not silent-ignore).; Creation-guard substrate for background: OnceCell get_or_init only; no strict/lenient policy, no thread-name-prefix heuristic.; Join/structured-concurrency primitive: collect SpawnHandles, try_join at the wave/refresh barrier; first error aborts all-or-nothing.; Dependency-graph discovery: the WiringPlan (App<Wired>) already computed it; non-deferral edge = later wave (await handle), deferral edge = same wave, @DependsOn = forced earlier wave on the bootstrap task.; Send/'static constraint: enforced via SpawnableWork #[diagnostic::on_unimplemented]; surfaced at compile or validate.; Deadlock (logical await cycle): impossible — a construction cycle is a fatal App<Wired> error before any build; deferral edges (the only cycle break) don't create wave-ordering constraints.

### `post-processor-spi`
Split across the two phases the toolkit already owns, NOT minted as a five-trait runtime SPI. (A) DEFINITION-TIER (BFPP/BDRPP): there is no `dyn DefinitionPostProcessor` the engine drives — definition rewriting/registration is the `App<Resolve>` phase's `Registrar` emission + the condition tier-router + the auto-config back-off ladder, all already owned by compile-runtime-split/autoconfig. A user who needs to add/edit definitions contributes a `Registrar` (the `App<Define>::with(plugin)` escape hatch or an `#[auto_config]` module) that runs in `App<Resolve>` before `seal()`; the freeze boundary IS the 'no more definition edits' line (the typestate makes a post-seal edit unrepresentable). Three-band Priority>Ordered>unordered ordering is `cmp_order` over the emitted-registrar order keys. (B) INSTANCE-TIER (BPP): the only surviving runtime decoration seam is `after_init` (and `before_init`), and it is OWNED BY proxy-substrate as the auto-proxy installation point: `run_after_init_decorators(raw: ErasedBean) -> ErasedBean` does an O(1) `ProxyPlan` lookup (frozen at seal) and swaps in the transparent newtype proxy where advised. This is origin-agnostic (`ErasedBean -> ErasedBean`) and is where AOP/tx/cache/validation/retry/@Async attach as `Interceptor`-chain entries — leaf's own features dogfood the SAME seam. The MergedBeanDefinitionPostProcessor (injection-metadata caching) is dissolved: injection metadata is the const `InjectionPlan` emitted by the macro, computed at compile time, not discovered at a runtime BPP. The SmartInstantiationAware.getEarlyBeanReference (early-proxy-into-cycle) is dissolved: single-phase construction has no early-exposure, and cycles are deferral-edge-only. Role discovery is from the `Descriptor.role` field (data), never runtime instanceof. A bare `Engine` runs zero decorators; the `Context` installs the standard infrastructure (the inert->live switch via `refresh()` auto-detecting `Role::Infrastructure`).

**Resolved open questions:** Two tiers unify into the existing phases: definition-tier = App<Resolve> Registrar/condition/auto-config work (no new SPI); instance-tier = the proxy-substrate after_init seam (one origin-agnostic ErasedBean->ErasedBean swap).; Proxy/early-reference mechanism: getEarlyBeanReference eliminated (single-phase, no early exposure, deferral-only cycles); after_init is the proxy attach point.; async-across-dyn for BPP: before_init/after_init are the boxed-future-on-dyn standard; merged-def metadata is the compile-time const InjectionPlan (no per-bean runtime metadata BPP).; Tiers-unify-into-one-hook (R3): not needed — there are structurally only two seams (App<Resolve> definition work, after_init instance decoration), each already owned.; Role discovery: Descriptor.role data, never instanceof; the App<Wired>/refresh self-check asserts expected infrastructure processors by role (anti-DCE).; Co-decided with refresh: the fixed-order refresh template (auto-detect Infrastructure -> install -> eager-instantiate -> smart-init) is the concrete sequence; BPPs and the beans they reference are eager and not auto-proxied (Spring's documented cost preserved).

### `concurrency-contract`
Realized as a TYPE BOUND, not a runtime promise (R1). The contract has no code of its own — it is anchored on the `Send+Sync+'static` bound that rides `ErasedBean`/`Bean` in one leaf-core location (ownership-model). (a) SAFE PUBLICATION: `Engine::create` builds a singleton on the bootstrap task and publishes by `OnceCell::set` of `Arc<dyn Any+Send+Sync>`; the atomic Arc clone handed to every later observer IS the happens-before edge (the publish-side write is sequenced before the consumer's atomic load) — Spring's final-like visibility WITHOUT a singleton mutex, because move-then-share is the publication. Config-only fields need no synchronization, enforced: an immutable field of a `Sync` type is provably safe to share. (b) NOT METHOD-THREAD-SAFE, by the type system: a singleton is shared as `&self` behind `Arc`, so the borrow checker FORBIDS mutating a plain field through it — to mutate shared state the developer MUST reach for `Mutex`/atomics, exactly Spring's pointer to volatile/locks, except the compiler refuses to let you forget. (c) SCOPE IS THE PRIMARY LEVER: a non-`Send+Sync` bean is legal at a NARROWER scope (prototype `Published::Owned` needs only `Send` to cross a spawn, or nothing if used inline) — the diagnostic steers there. Deliverables: the documented doctrine + a doctrine-citing `#[diagnostic::on_unimplemented]` on the `Bean`/`ShareableBean`/`SpawnableWork` bounds ('shared-scope bean must be Send+Sync; if it holds mutable per-interaction state, make it prototype/request-scoped'), NOT a raw trait-solver error. NO global lock anywhere (the only synchronization is the per-slot OnceCell atomic and the single-controller serialization of lifecycle transitions). This is the unification anchor: the same bound satisfies the executor's spawn requirement, the Cx-propagated-value requirement, and the shared-bean requirement, stated once.

**Resolved open questions:** Contract as type bound vs runtime promise: type bound (R1) — Send+Sync IS the safe-publication-and-shared-access proof; no runtime machinery.; Send/Sync bound placement: on ErasedBean/Bean in one leaf-core location (ownership-model), propagating to every cross-crate contribution automatically.; ?Send single-threaded carve-out: deferred behind the pluggable-runtime seam (no carve-out in core; steered by the doctrine diagnostic), per async-context-model.; How the happens-before edge is earned: the atomic Arc clone (OnceCell publish), replacing Spring's singleton-mutex JMM edge — no global lock.; Diagnostic shape: #[diagnostic::on_unimplemented] citing scope-as-concurrency-lever, not a cryptic bound error.

### `extra-0`
@DependsOn folds into ONE construction graph alongside injection edges (R1). The macro emits `depends_on: &'static [ContractId]` on the `LifecyclePlan`/`Descriptor` (ContractId is the stable cross-build key; resolved to BeanIds at freeze). `App<Wired>::validate()` adds a forced construction edge from the dependent to each depends_on target (in addition to the injection-point edges from the InjectionPlan), so the unified edge set drives BOTH the eager-init WiringPlan wave ordering AND the context-close reverse-teardown order (the TeardownLedger's LIFO publication order already honors this, since a depends_on target is constructed and published before its dependent). A @DependsOn cycle (with the injection edges) is a fatal `App<Wired>` error with the full path and the 'convert injection point X to LazyRef<T>' fix hint — the same Cycle classification injection-resolution uses (only `Deferral`/`Lazy` edges are removed from the graph; @DependsOn edges are always mandatory and never breakable). Cross-crate: a depends_on ContractId that resolves to no registered bean is a loud `App<Wired>` NoSuchBean naming the missing target (the dropped-target anti-DCE self-check). Interaction with background-bootstrap: a @DependsOn target of a background-marked bean is forced to construct on the MAIN bootstrap task in an earlier wave BEFORE the marked bean is spawned (never parallelized by the dependent's marker).

**Resolved open questions:** Graph substrate: the WiringPlan/construction-edge graph already built by App<Wired>::validate() (injection-mechanics) — @DependsOn edges are just additional mandatory edges, no separate graph.; Cross-crate target resolution: ContractId resolved at seal()/freeze; a dropped target is a loud App<Wired> NoSuchBean (anti-DCE self-check).; Cycle handling: fatal at App<Wired> (depends_on edges are non-breakable; only deferral edges break cycles); full path + LazyRef fix hint.; Bidirectional startup/teardown: one edge set drives both eager waves (forward) and TeardownLedger LIFO (reverse), so depends-on ordering holds at both ends.; Background interaction: @DependsOn target forced to main-thread earlier wave before the marked bean is spawned.

## Public API sketch

// ── leaf-core: the creation driver lives on Engine (container-shape); these are the lifecycle additions ──

// Const lifecycle metamodel — thin-macro-emitted per bean, via ::leaf_core paths
pub struct LifecyclePlan {
    pub init:        &'static [LifecycleStep],      // canonical phase order, run forward
    pub destroy:     &'static [LifecycleStep],      // run reverse
    pub aware_wants: AwareFlags,                     // bitset; cheap skip
    pub smart_init:  bool,                           // participates in after_singletons_ready
    pub depends_on:  &'static [ContractId],          // forced construction edges (extra-0)
}
pub struct LifecycleStep { pub phase: LifecyclePhase, pub call: LifecycleFn, pub id: StepId }
pub enum LifecyclePhase { PostConstruct, AfterPropertiesSet, InitMethod, PreDestroy, DisposableDestroy, DestroyMethod }
pub type LifecycleFn = fn(&(dyn std::any::Any + Send + Sync), &Cx)
    -> futures::future::BoxFuture<'_, Result<(), CallbackError>>;
#[derive(Clone, Copy)] pub struct AwareFlags(u16);
#[derive(Clone, Copy, PartialEq)] pub struct StepId(u32);

// Optional typed escape-hatch traits (the bean stays a POJO if it uses the annotation form instead)
pub trait InitializingBean: Send + Sync { fn after_properties_set(&self) -> BoxFuture<'_, Result<(), CallbackError>>; }
pub trait DisposableBean:   Send + Sync { fn destroy(&self) -> BoxFuture<'_, Result<(), CallbackError>>; }
pub trait Closeable:        Send + Sync { fn close(&self) -> BoxFuture<'_, Result<(), CallbackError>>; } // destroy-inference target

// Aware: construction-seam injection is the primary door; this is the post-population grouped hook
pub trait AwareReady: Send + Sync {
    fn on_context_ready<'a>(&'a self, infra: &'a ContextInfra<'a>) -> BoxFuture<'a, Result<(), CallbackError>>;
}
pub struct ContextInfra<'a> { /* bean_name(), env()->&Env, events(), resources(), messages(), context() */ }

// Smart-init: lock-free, complete-graph, eager-singletons-only, once per refresh
pub trait AfterSingletonsReady: Send + Sync {
    fn after_singletons_ready<'a>(&'a self, cx: &'a ReadyCx) -> BoxFuture<'a, Result<(), CallbackError>>;
}
pub struct ReadyCx<'a> { /* read-only complete-graph view: get::<T>(), all_of::<T>() via the Selector */ }

// Scope: data axes + a concrete store for ambient scopes (NO Box<dyn Scope>)
pub trait InstanceStore: Send + Sync {  // request/session/custom; reached via Cx, dropped at scope-end
    fn get_or_init<'a>(&'a self, id: BeanId, p: &'a dyn Provider, cx: &'a ResolveCtx<'a>)
        -> BoxFuture<'a, Result<ErasedBean, ResolveError>>;
    fn ledger(&self) -> &TeardownLedger;
}
pub fn register_scope(kind: ScopeKind, def: ScopeDef);  // custom scope = a const triple + a ScopeKind; built-ins reserved

// Background-bootstrap: a descriptor flag, not an API surface
pub enum Bootstrap { Default, Background }   // descriptor field; Background => spawn + join-before-refresh

// Teardown unit pushed by the singleton publish step (prototype pushes nothing)
pub struct Destroyer { pub bean_id: BeanId, pub run: Box<dyn FnOnce() -> BoxFuture<'static, Result<(), CallbackError>> + Send> }

// Errors (-> LeafError/Diagnostic)
pub enum CallbackError { Init { bean: BeanName, phase: LifecyclePhase, step: StepId, cause: LeafError },
                         Destroy { bean: BeanName, phase: LifecyclePhase, cause: LeafError },
                         Aware { bean: BeanName, cause: LeafError } }

// Concurrency-contract: no API — a doctrine carried by the bound, with a steering diagnostic
#[diagnostic::on_unimplemented(message = "shared-scope bean `{Self}` must be Send + Sync (shared across executor threads); \
   if it holds mutable per-interaction state, make it prototype- or request-scoped (scope is the concurrency lever)")]
pub trait ShareableBean: Send + Sync + 'static {}

## Cross-feature interactions
- **injection-resolution** — Engine::create's provide() factory params ARE the InjectionPlan injection points (single-phase, populate fused into provide). The App<Wired> validation pass resolves every eager single-valued point via the Selector, classifies the construction graph for cycles (with @DependsOn edges folded in by extra-0), and emits the WiringPlan waves bean-instantiation/background-bootstrap consume. Lazy/deferral edges (LazyRef/Lookup) are removed from the construction graph — the only cycle break and the laziness-holds enabler. Scope-mismatch (singleton injecting request-scoped by value) is a Selector/validation error.
- **registry-core** — Consumes the frozen Registry verbatim: singletons:Box<[OnceCell<ErasedBean>]> IS the creation guard + singleton store; providers:Box<[Arc<dyn Provider>]> is the create() target; Descriptor.scope (ScopeDef), .role (Role), and the macro-emitted LifecyclePlan/depends_on are read here. reopen_epoch carries forward initialized OnceCell cells for retained BeanIds (the settled singleton-carry-forward seam) so the Once contract holds across dynamic epochs.
- **execution-context (async-context-model)** — Cx is the substrate for context-scope InstanceStores (request/session reached via a CxKey, never ThreadLocal; re-installed per poll). The ExecutionFacility/Spawner runs background-bootstrap waves and is a HARD-FAIL-if-missing Role::Infrastructure bean. The TeardownLedger (drained LIFO by shutdown after stop_all) carries every async destroyer, partial-destroy, and cancelRefresh unwind — no async Drop. cmp_order orders init/destroy phases, smart-init participants, and lifecycle start/stop phases. The Send+Sync bound (concurrency-contract) is the same one the executor's spawn and Cx-value requirements use.
- **proxy-substrate** — The after_init decoration step in Engine::create IS the auto-proxy install point (post-processor-spi instance-tier): an O(1) ProxyPlan lookup swaps in the transparent newtype where advised. An advised bean injected by concrete type is an App<Wired> error (settled seam); an advised prototype gets a per-resolution proxy whose TargetSource constructs-and-owns a fresh Published::Owned instance (settled Owned-lane seam) so the no-refcount prototype invariant holds. tx/cache/validation/retry/@Async dogfood this seam as Interceptor entries.
- **refresh-lifecycle / runtime-lifecycle / context-close (lifecycle orchestration)** — Context::refresh()'s fixed-order template drives this subsystem: auto-detect Role::Infrastructure -> install decorators -> eager-instantiate WiringPlan waves (with background spawn+join) -> smart-init -> runtime-lifecycle auto-start -> publish ContextRefreshed. shutdown() drains the TeardownLedger LIFO after runtime-lifecycle stop_all(). cancelRefresh (failed bring-up) drains the same ledger but skips stop_all and ContextRefreshed (Failed not Closed). The init chain runs under NO guard (dissolving @PostConstruct-under-lock).
- **factory-bean / conditions / autoconfig** — A FactoryBean is one Provider (FactoryBeanProvider) the driver treats identically; the factory is a full bean with its own LifecyclePlan, and its product's destroyer rides the factory's registered destroyer (no orphan leak). Conditions/auto-config run in App<Resolve> before seal() (the definition-tier post-processor work); a bean's eager bit and scope are final by App<Wired>. The lazy global-flag default and the smart-init/Infrastructure exclusions are applied in App<Resolve>.

## Remaining risks (→ Phase 4)
- WiringPlan wave-parallelism contract (open in compile-runtime-split): do declared-independent eager beans OVERLAP across the Spawner within a wave, or strictly sequence? Overlap minimizes the not-ready window but risks an under-declared-edge data race; this is exactly where background-bootstrap's join-before-refresh and the §5 async model meet. Phase 4 must pin the structured-concurrency scope semantics for a wave and the cancel-cascade when one node in a wave faults.
- Destroy-method inference via a runtime dyn-Closeable probe assumes every bean's ErasedBean can be downcast to dyn Closeable — this requires the bean's stored handle to expose that upcast (a TypeRow upcast emitted by the macro for Closeable impls) or a per-bean flag the macro sets; if the macro must detect the Closeable impl it borders on resolving types it cannot see. Phase 4 should confirm a marker-trait/blanket-impl trick or a macro-emitted has_close flag closes this without violating thin-macro discipline.
- Context-scope (request/session) InstanceStore teardown-on-cancellation ordering when a straggler Arc clone outlives the scope boundary: the Arc strong-count governs the real free, but the per-key destroyer drain must still run within the deadline and the nested-scope (request-in-session) drain order must be precise. Shared open question with async-context-model; needs an end-to-end stress test in Phase 4.
- The aware delivery split (constructor injection primary + on_context_ready secondary) leaves a thin seam: a bean needing its OWN registered name at construction cannot get it from a constructor param (the name is a registration fact). Phase 4 must confirm the set of facts that genuinely require the post-population hook is small and that on_context_ready firing off-guard composes cleanly with the single-phase create() (it runs inside run_aware, before init).
- Background-bootstrap Send+'static enforcement: a marked bean whose provide() future captures a non-Send collaborator should fail at compile (SpawnableWork diagnostic) or at App<Wired>; the exact surfacing (compile vs validate) depends on whether the macro can express the Send bound on the emitted factory. Measure-first per the toolkit's monomorphized-fast-path note.
- @DependsOn + background interaction precision: forcing a depends_on target onto the main bootstrap task in an earlier wave before the dependent is spawned must be reconciled with the WiringPlan's wave numbering when the target is ALSO independently background-marked — Phase 4 should verify the forced-main-thread rule wins over the target's own Background flag without creating an ordering contradiction.
