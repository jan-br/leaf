# Metadata, Codegen & Cross-Crate Discovery  `[discovery-codegen]`

_The thin-macro -> const Descriptor + ProviderSeed -> linkme-slice -> binary-crate force-link pipeline, the annotation/stereotype vocabulary, scanning-as-participation, programmatic and mid-bootstrap registration, plus the triple anti-DCE defense._

This subsystem is the link-time-collection pipeline that fills the frozen registry: thin proc-macros emit one const `Descriptor` row + one const `ProviderSeed` per bean via absolute `::leaf_core` paths; `linkme` distributed slices collect them at link time with zero life-before-main; the binary crate (via `#[leaf::main]` + a generated `build.rs`) force-links every participating crate, emits an `ExpectedManifest`, and the cold `App<Define→Resolve>` assembly pass lifts the rows, runs the triple anti-DCE defense, applies exclusions, runs auto-config, and `seal()`s the dense `BeanId` registry. It consolidates six features that all reduce to "contribute or read const `Descriptor`/metadata rows on the one substrate": (1) annotation-model — the flat const `AnnotationMetadata` merge engine (compile-time flattening, default), the `MarkerId` vocabulary, and `@AliasFor` forwarding; (2) component-stereotypes — `@component`/`@service`/etc. as macros that emit the same `Descriptor` differing only in `Role` + meta markers, with a `stereotype!` minting helper; (3) component-scanning — discovery-is-registration: a synchronous `ScanPlan` exclude-then-include filter fold over the link-collected `&[Descriptor]` scoped to the force-linked participating set, with the silent-empty self-check as the headline defense; (4) programmatic-registration — `RegistryBuilder::register` exposed as the one hand-writable `BeanRegistry` SPI, with a typed `BeanSpec<T>` + explicit-pull async `supplier`, lowering to the identical `Descriptor`+`Provider` rows; (5) extra-3 (resolvableDependencies) — infrastructure handles published into the registry's `HashMap<TypeId, ErasedBean>` consulted as injection-mechanics' lowest-precedence `ResolvableDependency` layer; (6) extra-5 (mid-bootstrap registration) — the freeze-semantics contract: a single hard `seal()` after a pre-instantiation assembly fixpoint, with late registration after seal a loud `RegistrationAfterFreeze` error and `reopen_epoch()` the only sanctioned re-entry. The unification is that all six are producers of, or readers from, ONE `&'static [Descriptor]` shape on ONE `linkme` substrate, assembled in ONE cold synchronous pass before any `.await`.

## Shared machinery (how this rests on the toolkit)

Every feature rests on the toolkit's fixed pipeline primitives rather than minting its own:

1. ONE const row shape `Descriptor` is the universal currency. annotation-model OWNS the `meta: &'static AnnotationMetadata` field; component-stereotypes emit a full `Descriptor` that differs from a plain `@component` ONLY in `role` + `meta.markers`; component-scanning FILTERS `&[Descriptor]` and never invents a `CandidateDescriptor`; programmatic-registration LOWERS each `register_bean` to one `Descriptor`+`Provider` indistinguishable from a macro-emitted one; extra-5 governs WHEN the `Descriptor` set stops growing. There is no second seed type — Phase-1's `ComponentDescriptor`/`CandidateDescriptor`/`RawBeanDefinition`/`StereotypeDef` placeholders all collapse into `Descriptor`.

2. ONE `linkme` substrate, many typed slices. `COMPONENTS` carries beans (stereotypes + scanned + `@bean`); `AUTO_CONFIGS` is the second channel (autoconfig-starters-bom); `SOURCES` carries the per-crate `provides_descriptors!` anchors for the self-check; `STEREOTYPES` carries `StereotypeDef` meta-edge rows ONLY for the optional runtime-closure fallback. Annotation merge happens at MACRO time (compile-time flatten), not in a slice — the slice carries already-merged const data.

3. ONE assembly pass on the `App<Define→Resolve>` phase machine, owned by `RegistryBuilder`. `from_slices` lifts; `self_check(&EXPECTED)` runs anti-DCE; `apply_exclusions(&ex)` (ContractId-keyed) and `run_autoconfig` run the ladder; the `ScanPlan` filter fold runs here; registrars' `register()` run here to a fixpoint; `freeze()` is the single `seal()`. extra-5's "freeze" is exactly this one `seal()` boundary — the dense `BeanId` materialization point.

4. ONE stable cross-build identity `ContractId(u64)` for everything durable: stereotype/marker identity, exclusion keys, scan-conflict detection, and resolvableDependency provenance. `MarkerId` is itself a `ContractId` (a marker is just an author-stable string hashed). `TypeId` is the in-process fast key only.

5. ONE anti-DCE triple defense (force-link + section-retention guardrails + `ExpectedManifest` self-check) guards ALL discovery: stereotyped components, scanned candidates, auto-configs, registrars, AND resolvableDependency-providing infra crates. component-scanning's "silent-empty" footgun and programmatic-registration's "DCE'd registrar = silent missing beans" are the SAME `AntiDceError::SourceVanished`.

6. ONE origin-agnostic `Provider`/`Published` creation seam. A stereotyped bean's factory, a registrar's `supplier` closure, a resolvableDependency's pre-built handle, and a `FactoryBean` product are all `Arc<dyn Provider>` the `Engine::create` driver cannot tell apart. A registrar's `supplier` erases to `Provider::provide → BoxFuture<Published>` exactly like a native bean.

7. ONE error/diagnostic path: malformed `@AliasFor`/generic bean/self-referential stereotype = compile error (Tier 0); `ContractId` collision/anti-DCE/exclusion-typo/scan-conflict/post-seal registration = build-or-seal-time `LeafError` (Tier 1); the rest aggregate into the `App<Wired>` `AssemblyReport` (Tier 2).

## Features

### `annotation-model`
ADOPT Phase-1 R1 (compile-time-merged const descriptor) as the DEFAULT, with R3's runtime-closure kept ONLY as a narrow opt-in fallback, and R3's typed-merge reserved to nothing (the trait-upcast axis belongs to registry-substrate's TypeRow, not here). Concrete types in leaf-core (ultra-stable ABI):

  pub struct AnnotationMetadata { pub markers: &'static [MarkerId], pub attrs: &'static [AttributeEntry], pub qualifiers: &'static [QualifierReq], pub depends_on: &'static [ContractId] }
  #[derive(Clone,Copy,PartialEq,Eq,Hash)] pub struct MarkerId(pub ContractId);  // a marker IS a stable contract id
  pub struct AttributeEntry { pub key: &'static str, pub value: AttrValue }   // key is kebab-canonical
  pub enum AttrValue { Bool(bool), Int(i64), Str(&'static str), Type(TypeId), List(&'static [AttrValue]) }
  impl AnnotationMetadata { pub const fn has(&self, m: MarkerId) -> bool; pub const fn attr(&self, k: &str) -> Option<&AttrValue>; }
  pub const COMPONENT: MarkerId = MarkerId(contract_hash("leaf::Component"));

The merge (meta-presence flatten + distance-ordered nearest-wins + @AliasFor forwarding) runs ENTIRELY in leaf-codegen (a normal testable library the thin macro calls) at expansion: the macro emits one already-flattened `&'static AnnotationMetadata`. `@AliasFor` is a `forward = ::leaf::Component::name` path arg validated at expansion against the imported base-marker const schema (reciprocity, same-type, identical-default) → `compile_error!` on misconfig (strictly earlier than Spring). The runtime sees O(1)/O(n) const reads only; `depends_on` materializes @DependsOn as ContractId edges the WiringPlan reads. `meta` drops verbatim into `Descriptor.meta` — annotation-model OWNS this one field. Cross-crate alias CONFLICT (two crates' markers disagreeing on one element) is the ONLY runtime residual: detected at `freeze()` over the assembled rows as a `LeafError` Tier-1 diagnostic.

**Resolved open questions:** Where does merge happen — compile-time flatten is the default (zero overhead + earliest diagnostics); a runtime overlay is NOT offered for native beans (WASM dropped removes the only late-overlay justification); String-keyed records vs types — string-keyed const `AnnotationMetadata` is the vocabulary; the trait-upcast/TypeId axis is reserved to registry-substrate's TypeRow 'injectable as supertype', never the whole annotation model; Cross-crate aggregate set + conflict detection — assembled at `freeze()` over link-collected rows where the full set is visible; alias conflict is a Tier-1 build/seal error; @AliasFor misconfiguration timing — compile-time `compile_error!` at expansion against the imported base-marker schema; Do conditions need runtime-overlaid attributes — no; all condition-relevant attributes are const, so the model is fully precomputed (CondExpr leaves read const `AttrValue`s); present-vs-find duality — only 'present/get' (element + meta-annotations) survives; Rust composition favours traits over inheritance, no supertype-walk needed; One shared metadata primitive — yes: `AnnotationMetadata` is THE shape bean-definition (Descriptor.meta), config-metadata, and stereotypes all read

### `component-stereotypes`
ADOPT R1 (codegen-flattened): a stereotype is a thin attribute macro that expands to the SAME `Descriptor` a plain `@component` emits, differing ONLY in `role: Role` and `meta.markers`. leaf ships `@component` + `@service`/`@repository`/`@controller`/`@configuration` as one-liners over a leaf-codegen `descriptor!` helper. Container behaviour reads ONLY the flat `Descriptor`, so a stereotype is STRUCTURALLY (not conventionally) behaviourally identical to `@component`. Concrete:

  // leaf-macros: #[service] ≡ leaf_codegen::descriptor!(self, role=Application, markers=[COMPONENT, SERVICE], name_attr=name)
  // user stereotype minting (the Spring 'annotate an annotation' DX analogue):
  #[macro_export] macro_rules! stereotype { ($name:ident => $($marker:path),+ ; role=$r:expr) => { /* emits a thin attr macro forwarding into ::leaf::descriptor! with the flattened transitive marker set */ } }
  // leaf::stereotype!{ domain_service => COMPONENT, SERVICE, DOMAIN_SERVICE ; role = Role::Application }

The transitive marker closure (@RestController = @Controller = @Component) is flattened by leaf-codegen at expansion and frozen into `meta.markers`, so the @Indexed-follows-@Component edge is just 'this Descriptor is in COMPONENTS'. `Role{Application,Support,Infrastructure}` is the orthogonal axis carried in `Descriptor.role` (the SAME field Context::refresh() auto-detects `Role::Infrastructure` on — NOT the autoconfig `CandidateRole`). @Repository carries ZERO behaviour here: it is `markers=[..,REPOSITORY]` data the exception-translation advisor queries (`meta.has(REPOSITORY)`); the pluggable repositoryAnnotationType analogue is 'point the advisor at any MarkerId'. The R3 runtime `StereotypeDef`-closure is kept ONLY as a deferred escape for a future need to mint stereotypes whose meta-edges are not knowable at expansion — not built in v1. Generic stereotyped beans hard-error with a `register_component!(Concrete)` hint.

**Resolved open questions:** Where meta-presence is resolved — compile-time flatten (R1), co-resolved identically with annotation-model; Vocabulary as data vs types — DATA (MarkerId-keyed `meta.markers`); the trait axis is reserved to registry-substrate's TypeRow only; Stereotype-minting DX + extensibility cost — a `stereotype!` macro-generator (one declaration per user stereotype); matches annotation-model's compile-time-flatten realization; @Repository-as-pointcut + repositoryAnnotationType pluggability — a `MarkerId` on `meta.markers` queried by the exception-translation advisor; pluggable = configurable target MarkerId, no recompile of leaf-core; Misconfiguration timing — compile-time `compile_error!` (alias/self-reference); cross-crate name/duplicate conflict at seal; Cross-crate aggregate + conflict detection + force-link — shares the anti-DCE triple defense; conflicts detected at `freeze()`; One shared seed — yes: the stereotype `Descriptor` IS bean-definition's seed IS scanning's filter input; one shape

### `component-scanning`
ADOPT R1 (macro-flattened descriptors + runtime filter fold) as THE model: discovery = filtering the link-collected `&[Descriptor]`, NOT scanning a classpath. R2 (build-time codegen sweep) is folded into the OPTIONAL `cargo leaf prepare` accelerator (deferred, measured before built); R3's runtime ScanFilter SPI is NOT a separate path — its late-binding need (profile-driven excludes) is served by the CondExpr runtime tier, not a second filter chain. Concrete in leaf-boot (runs in `App<Resolve>`, cold sync):

  pub struct ScanPlan { pub scope: ParticipatingSet, pub filters: Vec<ScanFilter> }
  pub enum ScanFilter { MetaPresent(MarkerId), AssignableTo(TypeId), ModulePathRegex(&'static str), Custom(fn(&Descriptor)->bool) }
  impl RegistryBuilder { pub fn apply_scan(&mut self, plan: &ScanPlan) -> Result<(), AssemblyError>; }

The default include filter is exactly `[MetaPresent(COMPONENT)]` — every stereotype caught transitively because the flattened `meta.markers` already contains COMPONENT. EXCLUDES evaluated first + short-circuit, then ≥1 include must match. 'Read without loading' is structural: a Custom filter sees only the const `Descriptor`, never an instance. SCOPE re-expresses Spring's basePackages as the force-linked participating set (binary crate + enabled capability features ∪ `scan!` list) — honouring 'scope discovery, don't make it global'; there is no resource enumeration stage. The CONCRETE+independent gate is the macro hard-erroring on generic/abstract targets. The silent-empty footgun (the deprecated-index cautionary tale) is defeated by the SAME `RegistryBuilder::self_check(&EXPECTED)` — a scoped crate contributing zero rows is a loud `AntiDceError::SourceVanished`, surfaced in `leaf doctor` with per-SourceTag row counts. @Conditional gating runs AFTER filtering (filters decide what-could-be a candidate over const metadata; CondExpr runtime tier decides what-DOES-register). The AutoConfigurationExcludeFilter boundary is structural: auto-configs live in the SEPARATE `AUTO_CONFIGS` slice, so component scanning over `COMPONENTS` never picks them up.

**Resolved open questions:** Where the filter pipeline runs — at runtime in `App<Resolve>` over flattened const descriptors (R1); build-time sweep deferred to the opt-in `cargo leaf prepare`; no separate runtime ScanFilter SPI (profile excludes ride CondExpr); Default classpath substrate — linkme distributed slices (`COMPONENTS`); build-codegen is the opt-in portable fallback only; Base-package scoping re-expression — the force-linked participating set: binary crate + enabled capability features ∪ `scan!` list ('radiate downward' = 'the app + enabled integration crates'); Silent-empty guard — `RegistryBuilder::self_check(&ExpectedManifest)` at the Define→Resolve edge + `leaf doctor` per-crate row counts; the non-optional headline defense; One shared seed — yes: scanned candidate = stereotype Descriptor = bean-definition seed (Descriptor); @Conditional vs filter composition — filters (const metadata, no construction) decide candidacy; CondExpr runtime tier decides registration; the documented split; AutoConfigurationExcludeFilter / multiple channels — auto-configs in the separate `AUTO_CONFIGS` slice are never scanned from `COMPONENTS`; channels distinct at the slice level, unified at `freeze()`

### `programmatic-registration`
ADOPT a synthesis of R1+R3: `RegistryBuilder::register(d, p)` IS the irreducible hand-writable SPI (the codegen-boundary escape hatch — what every macro emits by hand), and a registrar is ANY `fn(&mut RegistrarCtx)` run in the cold definition phase. A typed `BeanSpec<T>` fluent builder + explicit-pull async supplier sit over it; each `register_bean` lowers to ONE `Descriptor`+`Arc<dyn Provider>` indistinguishable from a macro-emitted bean. Concrete in leaf-core/leaf-boot:

  pub trait Registrar: Send + Sync { fn register(&self, cx: &mut RegistrarCtx<'_>) -> Result<(), RegisterError>; }
  pub struct RegistrarCtx<'a> { builder: &'a mut RegistryBuilder, env: &'a Env }
  impl RegistrarCtx<'_> { pub fn env(&self) -> &Env; pub fn register_bean<T:'static>(&mut self, name: Option<&str>, f: impl FnOnce(BeanSpec<T>)->BeanSpec<T>) -> Result<BeanId,RegisterError>; pub fn register_alias(&mut self, alias:&str, target:&str)->Result<(),RegisterError>; }
  pub struct BeanSpec<T> { /* scope, lazy, role, candidate_role(Primary/Fallback), order, description, supplier */ }
  impl<T:'static> BeanSpec<T> { pub fn prototype(self)->Self; pub fn lazy(self)->Self; pub fn primary(self)->Self; pub fn fallback(self)->Self; pub fn order(self,i:i32)->Self; pub fn supplier<F,Fut>(self, f:F)->Self where F:Fn(&ResolveCtx)->Fut+Send+Sync+'static, Fut:Future<Output=Result<T,ResolveError>>+Send; }

The supplier is an async closure type-erased into the SAME origin-agnostic boxed-future `Provider` the engine stores — a `SupplierProvider<T>` wrapping the boxed `make`. Suppliers PULL (`cx.bean::<Foo>().await` via the engine get-path); there is no reflective autowiring, so this is the natural shape, not a concession. Reachability: `App::with_registrar(r)` (Architecture D) OR a `Registrar` descriptor in a `REGISTRARS` linkme slice via `#[import(MyRegistrar)]`. Registrars run STRICTLY pre-seal (extra-5), so `&mut RegistryBuilder` is sound — no mid-instantiation registration. A generic registrar/target hard-errors with a `register_concrete!` shim. A registrar attempting to register a definition-phase post-processor or registering after seal is a LOUD diagnostic (Spring's silent 'too late' becomes a Tier-1 error). The R2 `.wired()`/opaque-supplier split is NOT adopted in v1 (it pre-judges a build-time substrate); every registrar bean uses the boxed supplier path, with the monomorphized `.wired()` accelerator deferred behind `cargo leaf prepare`.

**Resolved open questions:** Thin façade over the same descriptor stream vs runtime-re-invoke island — thin façade: every `register_bean` lowers to one `Descriptor`+`Provider` identical to `@bean`; no AOT re-invoke island in v1 (WASM dropped removes the dynamic-source driver); when-can-a-registrar-run — strictly in the pre-seal cold definition phase (extra-5's assembly fixpoint); never mid-instantiation; supplier opaqueness / build-time freeze — the boxed-future supplier is the uniform path; the `.wired()` monomorphized accelerator is deferred to the opt-in `cargo leaf prepare`; &mut registry vs immutable snapshot — sound because registrars run pre-`seal()`; the typestate makes post-seal `&mut` unrepresentable; planning context width — narrow `RegistrarCtx{env}` only (no live container, no resolved bean set during planning); conditional-registration diagnostics blind spot — accepted; a registrar's imperative `if env.matches_profiles()` is legible-by-code; declarative gating on the `#[import]` is the condition-report-visible route; async teardown — no async Drop; `BeanSpec` carries a destroy hook driven by the container's reverse-order `shutdown().await` via the TeardownLedger; override/collision policy — registrar-minted defs go through the SAME `RegistryBuilder::register` collision policy as macro-minted ones (the #36648 cautionary tale)

### `extra-3`
ADOPT Phase-1 realization 1 (typed extra-provider table on the engine), NOT the synthetic-bean realization. The registry owns `resolvables: HashMap<TypeId, ErasedBean>` (injection-mechanics already fixes this as the terminal `ResolvableDependency` layer of the fixed selection ladder, consulted at LOWEST precedence — after Primary/Name/Qualifier/Priority/DefaultCandidate). Concrete:

  impl Engine { pub fn register_resolvable<T: ?Sized + 'static>(&mut self, handle: Ref<T>); pub fn register_resolvable_erased(&mut self, ty: TypeId, handle: ErasedBean); }
  // injection-mechanics Layer 'ResolvableDependency': consulted iff the keyed candidate set is empty

Context::refresh() calls `register_resolvable` for each infrastructure handle it exposes (the Context handle, EventPublisher, ResourceLoader, Env, MessageSource) right after building the Engine, before eager instantiation, so they are frozen-then-lock-free-read. Lowest precedence falls out of the ladder ORDER (the layer is last + abstains when any real definition candidate exists) — no synthetic priority constant, no `@Fallback` coupling. The `ignoreDependencyInterface` HALF COLLAPSES ENTIRELY: Rust has no reflective setter scan, and Aware-style delivery (if any) is a syntactically distinct trait impl, so there is nothing to subtract; the only residual is the predictability policy (below). Handle type = the SAME `ErasedBean` the singleton store holds; trait views match via the same TypeRow upcast registry-substrate emits, so `register_resolvable::<dyn EventPublisher>` works. The container-self-reference Arc cycle is broken by storing a `Weak`-backed handle for the Context's own handle (a `Ref` over an `Arc` whose strong owner is the run pipeline, the table holds a derived handle). Resolvables are SINGLE-VALUED ONLY: excluded from collection injection (a `List<T>` never includes a resolvable) — encoded as the ResolvableDependency layer not contributing to the candidate SET, only to single-valued fall-through. The table is FROZEN at refresh (no post-bring-up mutation; WASM dropped removes the dynamic-register-after-bring-up need), so reads are lock-free against the immutable snapshot.

**Resolved open questions:** Does ignoreDependencyInterface survive — NO; it collapses entirely (no reflective setters); the feature is the registration SPI + lowest-precedence lookup only; Second resolution source vs synthetic bean — a registry-owned `HashMap<TypeId,ErasedBean>` consulted as injection-mechanics' terminal ResolvableDependency layer (realization 1); NOT a synthetic @Fallback bean (avoids dragging cross-crate count semantics into a container-local concern); Container self-reference Arc cycle — a Weak-backed handle: the table holds a derived `Ref` whose strong owner is the run pipeline, never an Arc cycle; Canonical handle shape + trait-upcast match — `ErasedBean`; trait views match via registry-substrate's TypeRow upcast; concrete-struct handles match exact TypeId only; Lowest precedence realization — query ORDER (the terminal layer abstains when any real candidate exists), not a synthetic priority or @Fallback flag; Predictability of accidental shadowing — a user bean of an infra handle's type wins (real candidate beats the resolvable); documented, and the resolvable's presence is shown in `leaf doctor`; Collection injection participation — NEVER; resolvables are single-valued only, encoded by not contributing to the candidate SET; Dynamic post-bring-up registration — NO; the table is frozen at refresh (WASM dropped); lock-free reads against the immutable snapshot

### `extra-5`
ADOPT R3 (closed two-phase model: a single hard `seal()` after a pre-instantiation assembly fixpoint) — the WASM drop and the typestate phase machine make this the correct and simplest fit; R1's generational append and R2's base+overlay are rejected as machinery for a dynamism leaf no longer needs. extra-5 OWNS the freeze-semantics contract that bean-registry, programmatic-registration, and background-bootstrap deferred. Concrete: the freeze IS the toolkit's single `seal()` boundary (`RegistryBuilder::freeze() -> Registry`, the `App<Resolve> → App<Wired>` transition). Three Spring boundaries are pinned distinctly:

  (a) the soft by-type-query cache hint (Spring freezeConfiguration) — NOT a separate thing in leaf; the frozen `Registry`'s prebuilt `by_type` index IS it.
  (b) the per-bean markBeanAsCreated edit-freeze — folded into the single `seal()`; there is no separate per-bean blueprint-edit window after seal.
  (c) the actual immutable-store seal — `seal()` itself, materializing the dense BeanId space.

The assembly phase (App<Define→Resolve>) runs ALL definition contribution to a FIXPOINT before `seal()`: lift slices → self_check → evaluate_conditions → apply_exclusions → run_autoconfig (incremental) → run registrars → re-run until no new `Descriptor` appears (a bounded fixpoint, cap N=16 default, exceeding it is a loud `RegistrationStorm` diagnostic naming the last registrar). Then ONE irreversible `seal()`. Register-after-seal is a LOUD `RegisterError::RegistrationAfterFreeze { name, phase }` with the remediation hint 'move this to a Registrar so it runs in the assembly phase'. The eager driver then executes a genuinely-sealed snapshot (plan-then-execute holds absolutely; lock-free reads for the whole instantiation phase). A SmartInitializingSingleton-analogue is a pure read/aggregate callback after all eager singletons exist — it may NOT register (post-seal). `reopen_epoch()` is the ONLY sanctioned later mutation: it consumes `App<Running>`, returns a fresh `App<Define>` with a COW-seeded builder (NOT an in-place edit), carrying forward the prior epoch's initialized OnceCell cells for retained BeanIds (the COHERENCE.md singleton-carry-forward fix). Determinism: late-drive order keys on registration generation, never link order.

**Resolved open questions:** Append-mostly (R1) vs base+overlay (R2) vs hard-seal (R3) — R3: a single hard `seal()` after a pre-instantiation assembly fixpoint; bean-registry's freeze-early/plan-then-execute holds absolutely; Preserve register-after-eager-pass freedom — NO; all contribution is redirected into the pre-seal assembly fixpoint (charter §2.2 'proven unnecessary'); the WASM drop removes the only runtime-discovery driver; What exactly is 'the freeze' — the single `seal()`/`freeze()` boundary; the by-type-cache-hint is the frozen index, the per-bean edit-freeze is folded in, the store-seal is `seal()` itself; Re-entrancy + cross-task concurrency without a global lock — moot under R3: no mid-pass mutation; the assembly fixpoint is single-threaded cold-path; only `RegistryBuilder::register` (append-only) runs, never concurrent with the eager driver; Where conditions/profiles evaluate vs freeze — at runtime over collected descriptors BEFORE seal (CondExpr runtime tier in App<Resolve>), consistent across conditions/profiles/configuration-classes/auto-config; Dynamic source participation — only via `reopen_epoch()` (a fresh epoch), never mid-bootstrap; WASM dropped, so v1 has no dynamic source at all; Fixpoint cap + deterministic order — cap N=16, exceeding = loud `RegistrationStorm`; late-drive order keys on registration generation (not link order); SmartInitializingSingleton registering — NO; post-seal registration is always a loud `RegistrationAfterFreeze`; the callback is read/aggregate only

## Cross-feature interactions
- **annotation-model** — is the engine beneath component-stereotypes (meta-presence flatten + @AliasFor name-forwarding) and component-scanning (the MetaPresent(COMPONENT) include filter). Stereotypes and scanning MUST consume the SAME compile-time-flatten realization so the merge never diverges across code paths — co-resolved here.
- **component-stereotypes** — produces the `Descriptor` whose `meta.markers` is component-scanning's filter input and whose `role` is the orthogonal axis Context::refresh() auto-detects (Role::Infrastructure). @Repository is a `MarkerId` data point the exception-translation advisor (out-of-subsystem) queries — the pointcut is data here, the behaviour lives elsewhere.
- **registry-substrate (upstream)** — every feature contributes/reads its `Descriptor` row; annotation-model OWNS `Descriptor.meta`; the TypeRow upcast rows (not this subsystem's annotation model) serve 'injectable as supertype'; extra-3's resolvables live in the registry's `HashMap<TypeId,ErasedBean>`; extra-5's `seal()` IS registry-substrate's `freeze()`.
- **injection-mechanics (upstream)** — extra-3's resolvableDependency table is the terminal `ResolvableDependency` layer of the fixed selection ladder, consulted at lowest precedence; programmatic-registration's `BeanSpec` qualifiers/order feed the same Selector; the QualifierReq/MarkerId vocabulary is shared with annotation-model.
- **cross-crate-discovery / codegen-boundary (upstream)** — all six features ride the one `linkme` substrate and the one anti-DCE triple defense; component-scanning's silent-empty self-check and programmatic-registration's DCE'd-registrar check are the SAME `AntiDceError::SourceVanished`; exclusions key on ContractId; the `ScanPlan` and registrar fixpoint run in the `RegistryBuilder` assembly pass.
- **autoconfig-starters-bom (upstream)** — auto-configs use the SEPARATE `AUTO_CONFIGS` slice so component-scanning over `COMPONENTS` never picks them up (the AutoConfigurationExcludeFilter boundary, made structural); auto-config beans register at `CandidateRole::Fallback` (distinct from the stereotype `Role` axis); extra-5's fixpoint registers auto-config candidates incrementally so each OnMissingBean sees the growing set.
- **compile-runtime-split (upstream)** — extra-5's hard-seal contract IS the `App<Define→Resolve→Wired→Running>` phase machine's single `seal()`; the assembly fixpoint runs in App<Define→Resolve>; register-after-seal is unrepresentable by the typestate; `reopen_epoch()` re-enters App<Define> with a COW builder (the COHERENCE.md typestate-vs-reopen_epoch clarification).
- **error-model (upstream)** — four timing tiers: malformed @AliasFor/generic bean/self-referential stereotype = compile error (Tier 0); ContractId collision / anti-DCE / exclusion-typo / scan name-conflict / RegistrationAfterFreeze / RegistrationStorm = build-or-seal-time LeafError (Tier 1); residual unsatisfiable injection = aggregated AssemblyReport (Tier 2). condition-not-met stays a silent recorded report entry, distinct from the loud states.

## Crate hints
- leaf-macros (proc-macro=true, host target, exports only macros): #[component]/#[service]/#[repository]/#[controller]/#[configuration]/#[bean] expanding to one const Descriptor + ProviderSeed via absolute ::leaf_core paths; the stereotype! minting macro; #[import(Registrar)]; #[leaf::main]. ## Remaining risks (→ Phase 4)
- The macro supertrait-emission policy for TypeRow upcasts (all-declared vs only-used) is shared with registry-substrate and still open; component-stereotypes' Descriptor carries these rows, so row-count bloat must be measured before fixing 'all declared injectable supertraits' (the cross-crate-safe default).
- The exact contract_hash inputs (package + module-path + ident + declared-name?) are now a triple semver/ops surface (MarkerId identity, exclusion keys, scan-conflict detection). A crate rename silently changes MarkerId/ContractId and breaks a stereotype meta-edge or an exclusion. Must be documented as a public contract and the freeze-time collision guard must hard-error (not salt). Co-decided with codegen-boundary + cross-crate-discovery — verify the inputs are identical across all three readers.
- Whether `cargo leaf prepare` (the build-time scan sweep + monomorphized `.wired()` registrar accelerator + auto-config plan) ships in v1 or is deferred — measure whether the default R1 runtime ScanPlan fold + boxed-supplier path is fast enough that the build step is only worth it for determinism-sensitive deployments.
- The registrar fixpoint cap (N=16) and the RegistrationStorm diagnostic are a guess; needs measurement against real configuration-classes/import-composition cases to confirm legitimate deep-import chains never hit it.
- extra-3's Context-self-handle Weak-backed lifetime needs validation against the actual ownership model — confirming the run pipeline holds the strong Arc and the resolvables table holds only derived/Weak-upgradeable handles so shutdown ordering (TeardownLedger drain) does not strand a live resolvable handle.
- The boxed-future supplier cost (one heap alloc per construction) is acceptable for singletons but a real §2.5 cost for prototype/scoped beans minted in a registrar loop; the deferred `.wired()` monomorphized path is the mitigation but is not in v1 — measure prototype-registrar workloads before committing.
- With WASM dropped, the entire dynamic-source / second-epoch precedence question (can a dynamic row shadow a static @Primary; in-flight reconciliation across the swap) is moot for v1, but `reopen_epoch()` still exists as the sanctioned re-entry; its singleton-carry-forward (COHERENCE.md fix) must be implemented and tested even though no v1 feature drives it, to avoid the Once-multiplicity violation if any future feature uses it.
