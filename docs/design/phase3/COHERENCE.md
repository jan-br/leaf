# Phase 3 — Composition Review

**Verdict:** COMPOSES — with bounded, fixable seams. The 14 Phase-3 subsystem redesigns genuinely re-unify onto the fixed Phase-2 toolkit rather than re-fragmenting it: every subsystem's "sharedMachinery" rests on the SAME named primitives (ErasedBean/Ref/Published; one Provider seam + one Engine::create driver; the dense-BeanId frozen Registry + stable ContractId; the Descriptor/linkme/triple-anti-DCE pipeline with one COMPONENTS plus typed sibling slices; the App<Define→Resolve→Wired→Running> machine with one seal() and one aggregated AssemblyReport; one CondExpr tier-map + DefinitionProbe; the Interceptor-chain + transparent delegating-newtype proxy + TargetSource; the Cx ambient + per-key Propagation + TeardownLedger; one cmp_order; one LeafError/Diagnostic/FailureAnalyzer). I checked the toolkit's four "settled cross-ADR seams" (singleton carry-forward, typestate-vs-reopen_epoch, advised-concrete-injection rejection, advised-prototype Owned lane) and every subsystem that touches them states them consistently with COHERENCE.md and the owning ADR — no subsystem reopens a fixed decision. Cross-feature interactions are largely bidirectionally consistent: where subsystem A names B's primitive, B's own design exposes exactly that primitive (e.g. injection-resolution's DefinitionProbe ⇄ conditions-autoconfig's OnBean family ⇄ ADR-10; events' multicaster-as-Interceptor-chain ⇄ proxy-interception's substrate ⇄ ADR-08; environment-config's seal_environment ⇄ ADR-04's 5f fence; declarative-advice's TxResourceKey Isolate ⇄ ADR-07's per-key Propagation). The crate topology is acyclic and strictly inward (leaf-macros→leaf-core; everything else→leaf-core only; leaf-boot→core+conditions+config+codegen; starters→integration crates; umbrella = unique DAG sink), the single-copy-of-leaf-core invariant is correctly load-bearing, and every subsystem maps to a crate or is explicitly fused into leaf-boot. The defects are NOT contradictions: they are (1) one literally-empty subsystem record (container-lifecycle = "x"), (2) one open question that is genuinely SHARED by 4+ subsystems and now blocks convergence (TypeRow/proxy multi-supertrait-emission), (3) one identity-derivation contract (ContractId inputs) that has quietly become a triple semver/ops/exclusion surface read by 3+ subsystems and must be pinned ONCE, and (4) a handful of count-arithmetic / ordering / wave-parallelism seams where two subsystems must agree on a shared invariant that no single one currently owns. These are exactly the integrate-and-iterate work Phase 4 exists for.

## Gaps & fixes

- **container-lifecycle subsystem vs the rest (its record is a stub: sharedMachinery="x", one "x" interaction, one "x" risk)** — The Refresh/Runtime-Lifecycle/Teardown subsystem — the single most cross-cutting orchestration concern — has NO actual Phase-3 design. It is the fixed-order Context::refresh() template (ADR-05) + the WiringPlan eager-instantiation waves (ADR-04) + the TeardownLedger LIFO drain + stop_all() (ADR-01/07) + RunState watch + ContextRefreshed/Closed event firing (events subsystem). Six other subsystems already point AT it as if it were specified: bean-lifecycle ('Context::refresh()'s fixed-order template drives this subsystem'), events ('refresh fires Refreshed + drains the EarlyEventBuffer'), bootstrap-diagnostics ('refresh_context() delegates wholly to Context::refresh()'), execution-context (scheduler arms only after the SmartInitializing barrier), declarative-advice (ProxyPlan installs at after_init during refresh), environment-config (seal_environment is the body of App<Define>). The fusion point exists but its CONTRACT (exact phase ordering, the wave-parallelism semantics, the cancel-cascade, the unified RunState/RunDown/exit-code state machine) is unwritten, so the consumers cannot be validated against it.
  - **Fix:** Phase 4 must author the container-lifecycle subsystem as the leaf-boot refresh+teardown template that COHERENCE fuses, pinning: (a) the exact fixed-order sequence (defns-frozen → BFPP-cold → auto-detect Role::Infrastructure → install ProxyPlan decorators → init context services [messageSource/multicaster/ResourceLoader/Env] → eager-instantiate WiringPlan waves with background spawn+join → SmartInitializing barrier → runtime-lifecycle auto-start → publish ContextRefreshed); (b) the WiringPlan wave-parallelism contract (ADR-04 open Q — overlap vs sequence, cancel-cascade on a faulting node); (c) ONE unified application state machine reconciling RunState (runtime-lifecycle), RunDown (context-close), and exit-code-shutdown so keep-alive's watch_run_state() and shutdown coordination don't redundantly model the same transitions (bootstrap-diagnostics flags this); (d) cancelRefresh unwind = drain ledger but skip stop_all + ContextRefreshed (Failed not Closed).
- **registry-core + discovery-codegen + proxy-interception + declarative-advice + binding-conversion (all five carry the SAME open TypeRow supertrait-emission question)** — The macro's supertrait-emission policy (emit a TypeRow upcast row for ALL declared injectable supertraits vs only-used) is left open in ADR-02 AND ADR-08, and FIVE subsystems independently flag it as a remaining risk that couples to their design: registry-core (row-count bloat unmeasured), discovery-codegen (component-stereotypes' Descriptor carries these rows), proxy-interception (one proxy implementing ALL declared supertraits vs one-per-trait, and which trait the chain dispatches through), declarative-advice (tx/cache/validation on a multi-trait repository bean), and binding-conversion (collection/Map injection over a supertype). This is a single decision whose answer changes the proxy codegen shape, the injectable-as-supertype surface, and collection-injection-over-supertype arithmetic simultaneously — it cannot be resolved per-subsystem or the five will diverge.
  - **Fix:** Phase 4 co-decides ONE policy in a single cross-ADR note owned by registry-substrate (storage) + proxy-substrate (proxy shape): default to 'all declared injectable supertraits' for cross-crate safety (an undeclared supertrait is not injectable; the macro can't see cross-crate use sites), measured for row-count bloat on a realistic large graph; and pin that an advised multi-trait bean generates ONE proxy implementing all declared injectable supertraits (upcasting to each dyn view) with the chain dispatching per-method, so collection/Map injection over a supertype sees the proxy under every view consistently. This single answer unblocks proxy codegen, the injectable-as-supertype contract, and the collection-over-supertype count.
- **registry-core + discovery-codegen + conditions-autoconfig + environment-config (all read/derive ContractId; cross-crate-discovery keys exclusions on it)** — ContractId derivation inputs (Cargo package + module-path + ident + declared-name?) and its collision policy (hard-error vs salt) are open in ADR-02 and ADR-09, but ContractId has quietly become a TRIPLE durable surface read by multiple subsystems with different stakes: registry-core (the in-process collision guard at freeze), discovery-codegen (MarkerId IS a ContractId; stereotype meta-edges, scan-conflict detection), conditions-autoconfig (auto-config exclusion keys + before/after ordering edges + the leaf.autoconfigure.exclude config-file string), and environment-config indirectly via auto-config exclusion. A re-export or crate rename silently changes the ContractId and breaks an exclusion / a stereotype meta-edge / a config-file exclude key — a latent semver footgun across three readers that currently each assume their own derivation.
  - **Fix:** Phase 4 must pin ContractId derivation as ONE documented public contract verified identical across all three readers (registry-substrate, codegen-boundary, cross-crate-discovery), make the freeze-time collision guard HARD-ERROR (never salt — salting would make a config exclude string non-reproducible), and document the rename/re-export semver hazard explicitly. The exclusion-typo guard (ExclusionError::NoSuchTarget) and the MarkerId identity must demonstrably derive from the same inputs.
- **conditions-autoconfig (@Fallback CandidateRole) + injection-resolution (Selector candidate arithmetic) + autoconfig-starters-bom** — Auto-config beans register at CandidateRole::Fallback so a user bean transparently supersedes (ADR-11), AND OnMissingBean back-off still exists (ADR-10). Three subsystems flag the same unresolved arithmetic: the candidate-resolver's exact behavior when a contract has {one Fallback auto-config, one user Normal, plus a @Primary among them}, whether @Fallback REPLACES OnMissingBean or is belt-and-suspenders, and how an excluded/backed-off candidate is subtracted from the collection-injection expected-count self-check. injection-resolution owns the Selector but the @Fallback-demote layer and the count-subtraction rule are described in conditions-autoconfig and autoconfig-starters-bom without a single owner pinning the arithmetic.
  - **Fix:** Phase 4 co-verifies in injection-mechanics' Selector: (a) the PrimaryPromote/FallbackDemote layer ordering for the {Fallback + Normal + @Primary} case (a Fallback loses to any non-fallback of the same contract; @Primary among non-fallbacks wins); (b) that @Fallback role is the default soft-override and OnMissingBean is reserved for the hard all-or-nothing case (not both by default); (c) that excluded/conditioned-out candidates are subtracted from the collection-injection expected-count BEFORE the self-check fires, so a backed-off candidate never reads as a count mismatch. One shared test target across the three subsystems.
- **events (multicaster Interceptor pipeline ordering) + proxy-interception + declarative-advice (one cmp_order, RoleTier-first)** — Events makes the multicaster an Interceptor chain (AsyncDispatch/ErrorIsolation/ContextPropagation/Metrics entries sorted by cmp_order) reusing the proxy-substrate primitive verbatim, and declarative-advice pins a canonical chain order (VALIDATE→RETRY→@ASYNC→TX→CACHE→TRANSLATE). Both rely on cmp_order being RoleTier-first at correctness grade. But events flags that AsyncDispatch MUST sit OUTSIDE ErrorIsolation for sync-listener isolation to hold, and declarative-advice flags translation-innermost / cache-outside-tx as correctness invariants — these are RoleTier/order assignments that no single subsystem authoritatively fixes, and a wrong relative position is invisible in object identity (the exact misordering hazard ADR-08 names).
  - **Fix:** Phase 4 fixes the RoleTier assignment table ONCE (in proxy-substrate, the comparator's correctness-grade owner): assign each infrastructure concern (tx/cache/validation/retry/@Async/translate) AND each multicaster interceptor (AsyncDispatch/ErrorIsolation/ContextPropagation/Metrics) a concrete RoleTier + order:i32 so the canonical chain order and the multicaster pipeline order are both derivable and inspectable via advice_chain_of/leaf doctor. Add a property test that the assembled order matches the documented correctness invariants (translation innermost, cache outside tx, AsyncDispatch outside ErrorIsolation).
- **environment-config (CPS adapter tri-state, strict-collect placeholder pass) + binding-conversion (ConfigurationPropertySource consumer, strict @Value default) + compile-runtime-split (always-on App<Wired> validation)** — Three subsystems independently raise the SAME always-on-vs-opt-out lever from different angles: compile-runtime-split (whether App<Wired> whole-graph wiring validation is always-on or opt-out for AOT-checked startup), environment-config (strict-collect placeholder validation as a default App<Wired> sub-pass — 'same always-on-vs-opt-out question the toolkit flags... the two should share the opt-out lever'), and binding-conversion (strict-by-default @Value diverging from Spring's lenient, needing a migration toggle). These are three facets of one startup-validation-cost policy that must share a single opt-out mechanism or users get three inconsistent knobs.
  - **Fix:** Phase 4 designs ONE startup-validation strictness lever shared by wiring validation, strict-placeholder collection, and strict @Value, with a single opt-out path (intended for cargo-leaf-prepare-checked deployments) — measured for cold-walk cost — rather than three independent toggles. Pin strict-by-default with a documented FailureAnalyzer for the @Value migration surprise.
- **expr-i18n-resources + binding-conversion (both depend on an OPTIONAL ExpressionEvaluator for #{...})** — expr-i18n-resources drops the runtime SpEL interpreter (parse(str).getValue(ctx)) as a Spring-parity divergence justified by WASM being gone, and binding-conversion routes every #{...} @Value segment to the expr subsystem's ValueExpr<T> closure treated as 'optional Option<&dyn>'. If no evaluator is linked, binding-conversion silently yields a Tier-2 error for any #{...} @Value. The two subsystems must agree on whether the evaluator is always-present, opt-in, or whether a surviving feature (a config-loaded condition string or admin-supplied cache-key) silently needs the interpreter that was dropped.
  - **Fix:** Phase 4 confirms across both subsystems that NO surviving feature needs a runtime-authored expression (audit cache-key, @ConditionalOnExpression, config-loaded condition strings); if one does, it returns as an explicit opt-in leaf-expr crate behind the SAME EvalCx shape, never on the event hot path. Add a build-time leaf doctor check: 'this @Value uses #{} but no expression engine is linked' so the missing-evaluator case is loud at build, not a surprising Tier-2 startup error.

## Phase 4 agenda (integrate & converge)
- AUTHOR the missing container-lifecycle subsystem (the leaf-boot refresh+teardown template COHERENCE fuses): pin the fixed-order refresh sequence, the WiringPlan wave-parallelism contract + cancel-cascade, and ONE unified application state machine reconciling RunState / RunDown / exit-code-shutdown. This is the highest-leverage gap — six subsystems already depend on a contract that does not yet exist, and the wave-parallelism + cancel-cascade question is the single open item that touches background-bootstrap, structured concurrency, scope teardown, and the async fence simultaneously.
- CO-DECIDE the TypeRow / proxy multi-supertrait-emission policy in ONE cross-ADR note (default all-declared-injectable-supertraits, measured for bloat; one proxy implementing all declared views, per-method dispatch). It is the single open question shared by five subsystems (registry-core, discovery-codegen, proxy-interception, declarative-advice, binding-conversion) and blocks proxy codegen, the injectable-as-supertype surface, and collection-over-supertype arithmetic at once.
- PIN the ContractId derivation inputs + hard-error collision policy as ONE documented public contract, verified byte-identical across its three readers (registry-substrate, codegen-boundary, cross-crate-discovery), and document the rename/re-export semver hazard. It is now a triple semver/ops/exclusion surface (collision guard, MarkerId/stereotype identity, config-file exclude keys) that silently breaks if the readers diverge.
- RESOLVE the @Fallback-role vs OnMissingBean candidate arithmetic in injection-mechanics' Selector as a single cross-ADR test target: the {Fallback + user Normal + @Primary} resolution, @Fallback-as-default-soft-override vs OnMissingBean-for-hard-cases, and excluded/backed-off subtraction from the collection-injection expected-count. Co-verified across conditions-autoconfig, autoconfig-starters-bom, and injection-resolution.
- FIX the RoleTier + order:i32 assignment table ONCE in proxy-substrate for every infrastructure concern AND every multicaster interceptor, with a property test asserting the assembled chain matches the documented correctness invariants (translation innermost, cache outside tx, AsyncDispatch outside ErrorIsolation). Misordering is invisible in object identity, so this needs a single authoritative owner and an inspectable diagnostic (advice_chain_of / leaf doctor).
- UNIFY the startup-validation strictness lever: ONE shared opt-out covering App<Wired> wiring validation + strict-placeholder collection + strict @Value (measured cold-walk cost), instead of three independent knobs across compile-runtime-split, environment-config, and binding-conversion. Ship strict-by-default with a FailureAnalyzer for the Spring-migrant @Value surprise.
- CONFIRM the dropped-runtime-interpreter decision across expr-i18n-resources + binding-conversion: audit that no surviving feature needs runtime-authored expressions; gate any survivor behind an opt-in leaf-expr crate on the same EvalCx shape; add a build-time 'uses #{} but no expression engine linked' doctor check so the optional-evaluator gap is loud at build, not a Tier-2 startup error.
- MEASURE-BEFORE-BUILD the deferred performance escapes as one batched decision (they recur across nearly every subsystem and share infrastructure): the monomorphized single-advisor proxy fast-path + ErasedArgs/ErasedRet pack-unpack ABI, the singleton-only BeanRef arena, skip-Cx-install-when-unchanged, the per-resolution narrowing-Trace gating, and whether cargo leaf prepare's build-time plan ships in v1. Decide the ABI-affecting ones (ErasedArgs/ErasedRet, BeanRef key) before leaf-core freezes; defer the pure codegen fast-paths.
