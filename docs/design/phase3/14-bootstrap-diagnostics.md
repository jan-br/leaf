# Bootstrap Entry Point & Diagnostics  `[bootstrap-diagnostics]`

_The opinionated run() pipeline over App<...> (app-type deduction, args, context-initializers, the fixed named run-event sequence, runners readiness-gate window, banner, transient bootstrap-context, exit-code/shutdown coordination, spring.main.* self-binding) and the FailureAnalyzer rendering policy over the one LeafError/Diagnostic spine._

The opinionated `App::run()` pipeline and the FailureAnalyzer rendering policy, re-unified onto the fixed Phase-2 toolkit. The Phase-1 critic flagged this subsystem as the place where JVM-shaped scaffolding (a standalone early multicaster, `bindToApplication` mutating a live builder, a `withHook` thread-local read across `.await`, two parallel exit-code SPIs, `FailureAnalysis` duplicating condition-report machinery) is most at risk of being ported blindly. Phase 3 resolves all thirteen features by making them THIN orchestration over primitives the toolkit already fixed, not new machinery.

The unifying call (confirmed verbatim by container-shape ADR-05 line 47): `App::run()` is a THIRD orchestration layer ATOP `Context`, never a third container type. It is one concrete `async fn` that walks the FIXED `App<Define→Resolve→Wired→Running>` typestate the toolkit already mandates — there is NO separate `RunPhase` enum and NO re-orderable RunStage DAG (R2/R3 rejected: they would re-open the compile-runtime-split keystone and weaken the "you cannot accidentally reorder bootstrap" guarantee). The run-event sequence (`Starting → EnvironmentPrepared → ContextInitialized → Prepared → Refreshed → Started → Liveness=Correct → [runners] → Ready → Readiness=AcceptingTraffic | Failed`) maps 1:1 onto the typestate transitions, so the fixed named order is a structural property of the typestate, not a hand-maintained list.

The phase boundaries land on the toolkit's seams exactly: the FIRST four events fire on the upstream `EarlyEventBuffer` (the events subsystem's pre-multicaster primitive) — NOT a bootstrap-local "standalone early multicaster"; `EnvironmentPrepared` IS the body of `App<Define>::seal_environment().await` (the 5f async fence where config-data/SAJ/cloud-detect run); `Refreshed`/`Started`/`Ready` fire on the live `Context`'s `EventPublisher` after `Context::refresh()`. Availability rides the upstream `AvailabilityHandle` watch cells. FailureAnalysis is pure rendering policy over the ONE `LeafError`/`Diagnostic` spine: analyzers are `FAILURE_ANALYZERS` linkme descriptors (already in the toolkit's slice family), exit codes are `ErrorKind::exit_code()` (no separate ExitCodeExceptionMapper), and a boot `RunFailure` simply carries a `LeafError`.

## Shared machinery (how this rests on the toolkit)

Every feature rests on the SAME small set of toolkit primitives rather than minting its own. The eight unifications:

1. ONE pipeline shape = the typestate. `App::run()` is the third orchestration layer over `Context` (ADR-05). It does NOT carry a parallel `RunPhase` enum; the FIXED phase machine `App<Define>::seal_environment().await → App<Resolve> (route_conditions + run_autoconfig) → seal() → App<Wired>::validate() → Context::refresh().await → App<Running>` IS the run sequence. Each named run-event is a side effect fired at a typestate transition, so "the canonical order holds / fires once / cannot be skipped" is enforced by the consuming typestate (a sealed builder is unrepresentable to re-seal), the same guarantee R3-of-events asked for, now free because compile-runtime-split fixed the typestate.

2. ONE discovery substrate for ALL run-participants. ContextInitializers, Runners, RunListeners/early-listeners, FailureAnalyzers, AppTypeDeducers, ContextFactories, FlavorSeeders, PlatformProbes, ExitCodeContributors, BootstrapRegistryInitializers are EACH a typed `linkme` distributed slice of const descriptors joining `COMPONENTS`/`CONDITIONS`/`AUTO_CONFIGS`/`FAILURE_ANALYZERS`/`EVENT_LISTENERS` (cross-crate-discovery ADR-09 line 117 names this verbatim: "ALL run-participant streams share THIS one substrate ... No bespoke per-stream registry"). All are force-linked, `ContractId`-keyed, ordered by the ONE `cmp_order`, and covered by the SAME triple anti-DCE defense (force-link + section-retention + `ExpectedManifest` self-check). The acute asymmetric hazards bootstrap flags — a DCE'd early RunListener silently absent, a DCE'd migration Runner = data corruption, a DCE'd analyzer = a quieter message, a DCE'd deducer = silently-wrong AppType — are ALL one `AntiDceError::SourceVanished` surfaced at the `App<Define>→App<Resolve>` self-check edge.

3. ONE ordering comparator. Every ordered participant stream (initializers, the single merged Runner stream, analyzers, exit-code contributors, deducer chain) sorts by `leaf_core::order::cmp_order` over `OrderKey{value,source}` with `SequenceKey(epoch, ContractId)` as the deterministic tie-break below it. Never link/section order (ADR-09 line 34). The Phase-1 "flat @Order vs declared edges" question is resolved to flat `cmp_order` + ContractId tie-break — the framework-wide single comparator, no bus-/runner-local mechanism.

4. ONE async + ambient + teardown spine (ADR-07). `run()` is one awaited `async fn` on the bootstrap `ExecutionFacility`. The ONLY async before `seal()` is the cold config-data/SAJ/cloud IO inside `seal_environment` (5f fence); the whole `App<Resolve>/seal()/App<Wired>` chain is synchronous. The `withHook` single-run RunListener + AbandonedRun signal are EXPLICIT values threaded through a `RunOverlay` argument to `run()` (NOT an ambient thread-local read across `.await` — the §2.3 hazard is dissolved by passing data, the same move the toolkit uses for `Cx`). Shutdown is the explicit awaited `Context::shutdown()` draining the `TeardownLedger` LIFO after `stop_all()` — no async Drop; the shutdown hook is a cold `tokio::signal` task in leaf-tokio that triggers it once via a CAS.

5. ONE reactive-state primitive for availability + keep-alive. Liveness/Readiness are the upstream `AvailabilityHandle` (the events subsystem's `watch<RunState>`-shaped cells); the run flips them and a single projector re-publishes `AvailabilityChanged` on the bus. `spring.main.keep-alive` (the Phase-1 unowned gap) is realized as `run()` awaiting `watch_run_state()` reaching `Terminated` after Ready when keep-alive is set and no other foreground task holds the process — a sanctioned reactive subscribe, never an is_running poll-loop.

6. ONE diagnostic spine = `LeafError`/`Diagnostic`/`FailureAnalyzer` (ADR-12). FailureAnalysis is NOT a parallel error system: it is the rendering-policy layer. A boot `RunFailure` carries a `LeafError` whose `chain` records the failing phase; analyzers are `FAILURE_ANALYZERS` const descriptors `@Order`-sorted, first-non-None wins, matching on `ErrorKind`/downcast over the chain (not Spring's reflective generic match). Exit codes are `ErrorKind::exit_code()` + one `App`-level override hook (no ExitCodeExceptionMapper SPI). The renderer consults the frozen `ConditionReport` + `ExpectedManifest` to enrich a `NoSuchBean` ("backed off because…" / "feature X not enabled" / "crate expected but produced zero rows"). The banner is the ONE deliberate exception to fail-fast (cosmetic → degrade-and-warn via a `Severity::Warn` LeafError, never aborts).

7. ONE env mutate→seal boundary for self-binding + flavor + cloud. `spring.main.*` self-binding, `EnvironmentConverter` flavor reseed, and cloud-platform detection ALL happen inside `seal_environment` over the upstream `EnvBuilder`, before it is consumed into `Env`. `bindToApplication`'s "config rebinds the live builder" becomes: the builder records programmatic `spring.main.*` calls as low-precedence rungs, then `seal_environment` binds the `spring.main` subtree into a frozen `BootstrapSettings` via the upstream binder (config = last write wins) — NO live-object mutation, a single owned resolution on the cold path. A post-seal `spring.main.*` push via `@PropertySource` is type-unrepresentable (the builder is consumed) and the silent no-op becomes a loud `LeafError`.

8. ONE condition algebra for app-type + cloud. `OnWebApplication`/`OnAppType` and `OnCloudPlatform` are `CondExpr` leaves over the runtime tier (ADR-10), evaluated in `App<Resolve>` AFTER `seal_environment`, reading the frozen `Env`/`AppType`/`CloudPlatform` lock-free. AppType deduction is a pure cold-path fold over `AppTypeDeducer` descriptors against a `CapabilitySet` (the resolved force-linked capability markers — the rust-native "classpath"); the deduced value is final before the first condition evaluates.

## Features

### `spring-application-run`
`App::run()` is the concrete async orchestrator, the third layer over `Context` (ADR-05 line 47). Signature: `impl App<Define> { pub async fn run(self, args: Args, overlay: RunOverlay) -> Result<RunningApp, RunFailure>; }` where `#[leaf::main]` emits the hand-writable `App::new(PrimaryConfig).run(Args::from_env(), RunOverlay::none()).await` plus the force-link prelude + `ExpectedManifest`. The body walks the FIXED toolkit typestate, firing the named events at each transition (there is NO separate RunPhase enum — the typestate IS the state machine, resolving the R1-vs-R2-vs-R3 open question to R1-on-the-toolkit): (1) start the startup timer (`StartupRecorder` from ADR-12, the run is the root step); (2) arm the shutdown hook if `BootstrapSettings.register_shutdown_hook` (a leaf-tokio signal task, CAS-once → `Context::shutdown()`); (3) create the transient `BootstrapContext` + run `BootstrapRegistryInitializer`s; (4) buffer-fire `Starting` on the upstream `EarlyEventBuffer`; (5) parse `Args`; (6) `let app: App<Resolve> = app.seal_environment().await?` — this IS `EnvironmentPrepared` (config-data/SAJ/cloud/profiles/self-bind/flavor all run inside it, the config-data feature already owns the body); (7) print banner from frozen `BootstrapSettings`; (8) deduce `AppType` + pick `ContextFactory`; (9) apply ContextInitializers (`cmp_order`-sorted, merged user+library+generated) then buffer-fire `ContextInitialized`, promote BootstrapContext singletons, register the args+banner singletons, fire `Prepared`; (10) `route_conditions()` + `run_autoconfig()` → `seal()` → `App<Wired>::validate()` — the aggregated `AssemblyReport` pass, which reads `BootstrapSettings.startup_validation` ONCE at its head and (per `[C2-config-locus]`) PRE-MATERIALIZES the pure-projection config-properties beans (binder-backed `Provider::provide` + JSR validation, bound `Arc` stored into the slot OnceCell) and DRY-RUNS every eager `@Value` coercion (`resolve_into::<T>`, value discarded) — so config bind / JSR / @Value coercion failures aggregate HERE at Tier-2, BEFORE refresh; (11) `Context::refresh().await` (R0..R8: auto-detects Role::Infrastructure incl. the Multicaster, eager-instantiates — its R5 eager bitset EXCLUDES the config-properties beans validate() already pre-bound, publishing their existing Arc) — `Refreshed`/`ServerInitialized` fire DURING via the now-live `EventPublisher`; the `EarlyEventBuffer` is drained at the refresh phase that installs the multicaster; (12) fire `Started` + `AvailabilityHandle.set_liveness(Correct)`; (13) `call_runners()`; (14) if running, fire `Ready` + `set_readiness(AcceptingTraffic)`. ANY error routes to `handle_run_failure(LeafError)`: if `overlay.abandoned` → rethrow (no analysis/close); else fire `Failed`, `analyze_first()` over `FAILURE_ANALYZERS`, render via `Diagnostic`, drive `Context::shutdown()` (post-refresh) or drop the unsealed builder (pre-refresh — nothing constructed), return `RunFailure{ phase, error: LeafError, analysis: Option<FailureAnalysis> }`.

**Resolved open questions:** Run-orchestration shape: R1 (fixed async fn) wins, but realized AS the toolkit typestate walk — no bespoke RunPhase enum, no re-orderable RunStage DAG (R2/R3 rejected: would re-open compile-runtime-split and weaken the no-accidental-reorder guarantee); Cross-crate substrate: linkme distributed slices per participant stream, force-linked + ExpectedManifest self-checked (ADR-09); inventory/build-codegen rejected as defaults; Run events REUSE the general substrate: first four on the upstream EarlyEventBuffer, later on the live Context EventPublisher; no bootstrap-local standalone multicaster; Unified error model: FailureAnalysis is rendering policy over the one LeafError/Diagnostic (ADR-12), analyzer-match by ErrorKind/downcast; bindToApplication modeled as a frozen BootstrapSettings bound inside seal_environment (config last-write-wins), not live-object mutation; withHook/Abandoned carried as an explicit RunOverlay value argument, never an ambient thread-local across .await

### `application-type-deduction`
`AppType` is an OPEN interned value (`AppType(ContractId)` over a stable &'static str like "web.servlet"), core knows only `AppType::NONE`; integrations contribute the rest as data (no web knowledge in core, charter §1). Deduction is R1-on-the-toolkit: a `#[linkme::distributed_slice] static APP_TYPE_DEDUCERS: [DeducerDescriptor]` where `DeducerDescriptor { app_type: AppType, order: OrderKey, applies: fn(&CapabilitySet)->bool }`. `deduce(&CapabilitySet) -> AppType` sorts by `cmp_order`, first `applies()`==true wins, else `NONE`. The `CapabilitySet` is the resolved set of force-linked capability markers (the rust-native classpath, populated by the SAME force-link discipline + self-check), shared with `OnWebApplication`/`OnFeature` so condition and deduction cannot disagree (R2's unification dividend taken WITHOUT R2's separate machinery — `OnWebApplication(t)` is a `CondExpr` leaf reading the final `AppType`). Three override edges resolved by precedence: programmatic `app.set_app_type(t)` < config `spring.main.web-application-type` (bound inside seal_environment, last write wins). A post-deduction self-check: a deduced/overridden `AppType` with no registered `ContextFactory`/`FlavorSeeder` is a loud `LeafError{kind: AntiDce}` naming the type + triggering capability + "add AND force-link leaf-web". Deduction is a pure synchronous cold-path fold run at the head of seal_environment (provisional) and re-checked at its tail after config (final), final before any condition evaluates. Build-time freezing (R3) is NOT v1: the runtime fold is cheap and avoids a mandatory build.rs; left as a measured Phase-4 accelerator.

**Resolved open questions:** R1 deducer chain on the toolkit; R3 build-time freeze deferred (measured, not v1); AppType is an open interned ContractId-keyed value, core knows only NONE; CapabilitySet unified with OnWebApplication/OnFeature (R2 intent) but deduction stays a simple fold — no separate ranked-rule engine; Override precedence config>programmatic>deduced applied inside seal_environment via the frozen BootstrapSettings; Same-order ambiguity (two deducers claiming incompatible types) is a detected LeafError, not silent first-match; Silent-DCE mis-deduction caught by the shared anti-DCE self-check + the no-provider self-check

### `application-arguments`
ONE owner (collapsing the Bootstrap+extra-6 double-design): `leaf_core::env::ApplicationArguments`, an `Arc`-shared value object parsed ONCE by a pure sync `fn parse(argv: Vec<OsString>) -> Result<ApplicationArguments, LeafError>` (R1 fixed `--key[=value]` grammar in core, the Layer-0-safe always-present choice; R2 parser-SPI rejected to avoid two divergent argv interpretations + a DCE silent-wrong-parse hazard). It exposes the three Spring questions (`source_args()`, `option_names()`/`option_values(name)` multi-valued, `non_option_args()` incl. the `--` tail). The SAME parse is wrapped by a thin `CommandLineArgsSource: PropertySource` (the upstream environment-config trait) added at the highest operational `PrecedenceRung::CommandLine` inside `seal_environment`, so `--server.port=8080` resolves through first-source-wins and a `--leaf.profiles.active`/`--leaf.main.*` override works schema-independently. The one `Arc<ApplicationArguments>` backs all three consumers (the `PropertySource`, the injectable singleton via `register_resolvable`, and the Runner argument) — no triple copy (the ownership-model handle currency). Multi-value→property adaptation: BOTH CSV (`foo=a,b`, Spring default) and indexed (`foo[0]`) emitted, reconciled with relaxed-binding's list conventions; `OsString` decode is per-token (option keys strict-UTF8 → `LeafError` on the offending token+position; positional args lossy). Parse errors render through the FailureAnalyzer chain. Lenient by default (unknown args accepted, Spring-faithful) so ad-hoc overrides work; a typed schema layer (R3) is deferred.

**Resolved open questions:** R1 fixed core grammar wins (Layer-0-safe, no DCE hazard); R2 parser-SPI / R3 typed-schema deferred; Multi-value adaptation: emit BOTH CSV and indexed keys to satisfy relaxed-binding + binder; OsString decode per-token: option keys strict, positionals lossy; Lenient (accept unknown) by default so schema-independent --leaf.* overrides work; Shared handle = Arc<ApplicationArguments> over the ownership-model currency, backs all three consumers no-copy; Single owner resolves the Bootstrap-vs-extra-6 duplication: leaf_core::env::ApplicationArguments

### `context-initializers`
A dedicated `ContextInitializer` trait (R1 — its own crisp 'customize the pre-refresh context' surface; R2 fold-into-post-processor-spi NOT adopted here since that pre-judges post-processor-spi's keystone and blurs Spring's deliberate initializer-vs-BFPP distinction): `trait ContextInitializer: Send + Sync { fn initialize(&self, cx: &mut PreRefreshCtx<'_>) -> Result<(), LeafError>; }` collected via `#[linkme::distributed_slice] static CONTEXT_INITIALIZERS: [InitializerDescriptor]` (force-linked, self-checked, same substrate). Three origins merge into ONE `cmp_order` stream: USER (programmatic `App::add_initializer` + the `RunOverlay` single-run for tests), LIBRARY (the slice), and the GENERATED-AOT initializer which gets a structural 'runs first' rule (not a collidable reserved Order band) — it is prepended before the sort. The `&mut`-across-`.await` clash is resolved by EXECUTION CONTEXT (R1/R2 path, the toolkit's `App<Resolve>` is synchronous single-threaded cold-path — `initialize` is a sync fn, never awaits, sidestepping aliasing without a lock). `PreRefreshCtx` is a thin capability projection over seams other features own (no second way to do anything): `env_mut()` → the `EnvBuilder` mutate seam, `register_post_processor()` → post-processor-spi, `engine_policy()` → allow-override/lazy-init/allow-circular knobs on the builder. The 'register a pre-refresh singleton' affordance (Spring's springApplicationArguments/banner) is dropped in favor of the run pipeline's existing singleton registration + BootstrapContext promotion (avoids two promotion paths). An initializer failure is a `LeafError{kind: ConstructionFailed}` aggregated like NoSuchBean.

**Resolved open questions:** R1 dedicated trait wins (not R2 fold-into-post-processor-spi, which would pre-judge that keystone; not R3 frozen plan); Same linkme substrate as all run-participants, force-linked + self-checked; &mut-across-await resolved by execution context (App<Resolve> is sync single-threaded), not structurally; Capability scope = thin projection over EnvBuilder/post-processor-spi/engine-policy seams, no new way to do the same thing; Generated-AOT initializer 'runs first' is a structural prepend rule, not a collidable Order band; withHook threaded via RunOverlay not thread-local; Pre-refresh singleton-registration affordance designed OUT — reuse run-pipeline registration + BootstrapContext promotion

### `run-listeners-events`
The standalone early multicaster is DESIGNED OUT (charter §2.2 — it existed in Spring only because the real multicaster is a refresh-created bean). leaf's `EventPublisher` is an always-present structural Context field (events subsystem), and the FIRST four facts (Starting/EnvironmentPrepared/ContextInitialized/Prepared) fire before the Multicaster is installed, so they go to the upstream `EarlyEventBuffer` (the events subsystem's pre-multicaster buffering primitive, shared with startup-instrumentation), drained at the refresh phase that installs the Multicaster. The named run-event sequence is the set of built-in lifecycle facts: `Starting, EnvironmentPrepared, ContextInitialized, Prepared` (early-buffered) then `Refreshed, Started, AvailabilityChanged(Liveness=Correct), Ready, AvailabilityChanged(Readiness=AcceptingTraffic) | Failed` (on the live publisher) — each a typed event fired at a typestate transition, so 'fired once, in order, never skipped' is structural. The `RunListener` seven-callback contract collapses into ordinary `@EventListener`-style subscriptions on these typed facts (R2 intent) PLUS pre-context early-listeners that cannot be beans: those are `#[linkme::distributed_slice] static EARLY_LISTENERS: [EarlyListenerDescriptor]` (force-linked, self-checked — the acute asymmetric silent-DCE hazard handled by the same mandatory self-check). The NON-UNIFORM error contract uses the events subsystem's `DispatchErrorMode`: early/ordinary listener failures `IsolateEach`, milestone facts (Started/Ready) `ObserveAndFailStartup` → route a `LeafError` into `handle_run_failure`. The `withHook` single-run RunListener is threaded as an explicit value in `RunOverlay`, not an ambient thread-local. The run-event-name taxonomy is a published versioned const vocabulary in leaf-boot, kept coherent with startup-instrumentation/lifecycle-events as ONE shared milestone vocabulary.

**Resolved open questions:** Standalone early multicaster designed OUT; always-present EventPublisher + upstream EarlyEventBuffer replace it; Early buffering = the events subsystem's EarlyEventBuffer drained at multicaster-install (R1 buffer realized on the shared primitive); EARLY_LISTENERS linkme slice for pre-context listeners; acute asymmetric silent-DCE caught by the mandatory self-check; Non-uniform error contract = the events DispatchErrorMode (IsolateEach for ordinary, ObserveAndFailStartup for milestones); Sequence = typed lifecycle facts at typestate transitions (R3 verified-transition guarantee, free from the toolkit typestate); withHook threaded via RunOverlay; seven-callback RunListener collapses into typed-fact subscriptions; BootstrapContext-carrying first callbacks kept minimal; run-event taxonomy unified with startup-instrumentation/lifecycle-events vocab

### `runners`
ONE `Runner` async trait (collapsing Spring's ApplicationRunner/CommandLineRunner two-trait lineage as JVM scaffolding, §2.2): `trait Runner: Send + Sync { fn run<'a>(&'a self, args: &'a ApplicationArguments) -> BoxFuture<'a, Result<(), LeafError>>; }` (the toolkit BoxFuture standard at the dyn edge; raw-args is just `args.source_args()`). Runners are ordinary beans resolved as a typed collection from the frozen `Context`, merged into ONE `cmp_order`-sorted stream (flat Order + ContractId tie-break, NOT declared edges — the framework-wide single comparator), awaited STRICTLY SEQUENTIALLY by default (R1/R2 safe Spring-faithful; declared-independent concurrency waves R3 deferred behind a measured opt-in to avoid the data-race footgun). `call_runners()` runs in the precise window: after `Context::refresh()`+`Started`+Liveness=Correct, BEFORE `Ready`+Readiness=AcceptingTraffic (the K8s readiness gate — runners finish loading/migrating, THEN traffic flips). A runner returning Err aborts startup → `RunFailure{ runner: ContractId, order, error: LeafError }` → handle_run_failure (post-refresh, so `Context::shutdown()` not cancel). A blocking runner must use the `BlockingOffload` capability (documented). The uniquely-dangerous silent-DCE-migration-never-runs failure is correctness-critical here, caught by the mandatory anti-DCE self-check + force-link discipline; a panicking runner is `catch_unwind`→`LeafError`.

**Resolved open questions:** ONE Runner trait (CommandLineRunner two-trait lineage designed OUT, §2.2; raw args = source_args()); Flat cmp_order + ContractId tie-break (the framework-wide single comparator), not declared edges; STRICTLY SEQUENTIAL by default; declared-independence concurrency waves (R3) deferred behind a measured opt-in; Runtime collection from the frozen Context (R1/R2); build-time RunnerPlan (R3) deferred; Same linkme/force-link substrate; silent-DCE-migration is correctness-critical, the mandatory self-check is non-optional; RunnerError = LeafError feeding FailureAnalysis; panicking runner catch_unwind→LeafError; ApplicationArguments shared with the args feature, near-empty for non-CLI apps (no ceremony)

### `banner`
R2-on-the-toolkit: the slimmest realization. The banner is a compile-time-lowered const template (`include_str!`/default figlet) with placeholders split at macro time; `${...}` placeholders (`application.version`, `leaf.version`, `application.title`) resolve against the frozen `Env` via the upstream `resolve_placeholders` walk, mostly from Cargo metadata (`env!("CARGO_PKG_VERSION")`) known at compile time. The MODE tri-state (`Console/Log/Off`) is read from the frozen `BootstrapSettings.banner_mode` (config-overridable via spring.main.banner-mode, bound inside seal_environment). `print_banner()` runs after `seal_environment`, before context creation. It is the ONE deliberate exception to charter §1.7 fail-fast: a banner failure NEVER aborts a boot — it degrades to no-banner + a `LeafError{ mode: Severity::Warn }` (this 'cosmetic/optional steps degrade' is a recognized category in the error model via Severity::Warn, not a banner-specific carve-out). The runtime Banner-SPI + source-discovery chain (R1) is DROPPED (no integration crate ships a banner; the discovery substrate is not dragged into a cosmetic feature). The rendered-banner-as-injectable-singleton (springBootBanner) ceremony is DROPPED (§2.2 — no leaf consumer injects it). External-file override (spring.banner.location) is a rare runtime-load lane via the resource-loading neighbour, behind the embedded default. ANSI styling is a small self-contained `auto/always/never` with NO_COLOR/tty detection.

**Resolved open questions:** R2 slim compile-time-lowered const + thin mode-match; R1 SPI-chain and R3 frozen-PartialBanner dropped/deferred; Banner EXEMPTED from cross-crate auto-discovery — app-explicit only; Rendered-banner-as-singleton ceremony designed OUT (§2.2); Embedded include_str!/Cargo-metadata is the dominant path; external-file override a rare resource-loading lane; Banner is the ONE fail-fast exception: degrade-and-warn via Severity::Warn (a recognized error-model category, not a carve-out); ANSI = small self-contained auto/always/never + NO_COLOR; Log-mode reuses the diagnostics/tracing channel

### `bootstrap-context`
A DISTINCT minimal transient registry (R1 — intent-faithful, covers shared objects beyond env; R2 fold-into-env-window rejected since it pre-decides environment's seam and scopes too narrowly; R3 resource-DAG rejected as over-engineering a deliberately minimal concept). Type-erasure reuses the toolkit's currency: `BootstrapContext` stores `ErasedBean = Arc<dyn Any+Send+Sync>` keyed by `TypeId` (the same handle the registry uses — NOT a bespoke map). Surface: write `register(instance)`/`register_lazy(supplier)`/`add_close_listener`, read `get<T>()/get_or_else`/`is_registered`. The lazy supplier resolves async cold IO (the motivating expensive remote-config/vault client) on the bootstrap `ExecutionFacility`, with the async→snapshot→sync-handle fence happening INSIDE seal_environment (the framework-wide async-pre-materialization owner is seal_environment itself, resolving the cross-feature 'who owns the pre-step' question): `get<T>()` is SYNC over already-resolved instances; all async resolution is forced into the seal_environment pre-step. Prototype scope is DROPPED (§2.2 — a share-one-expensive-object phase has no need for re-create-per-get). Promote-at-close happens during prepare_context (Spring timing, after the context exists): selected entries become real container singletons via `register_resolvable`, the same warm client kept alive; a `BootstrapContextClosed` fact fires on the `EarlyEventBuffer`. Lifetime is structural: the `BootstrapContext` is owned by the run frame and dropped after prepare_context (no reaching in post-refresh). A bootstrap-resource failure is a `LeafError{kind: ConfigIo}` through the FailureAnalyzer chain. No async Drop — close listeners are awaited explicitly.

**Resolved open questions:** R1 distinct minimal registry; R2 env-fold and R3 resource-DAG rejected; Type-erasure = ErasedBean/TypeId (the toolkit registry currency), not a bespoke bootstrap map; Async pre-step owner = seal_environment (the framework-wide async-pre-materialization fence); get() is sync over resolved instances; Lazy supplier IS async-resolving but resolution is confined to the seal_environment cold pre-step; Prototype scope designed OUT (§2.2); promote-at-close at prepare_context via register_resolvable; BootstrapContextClosed dispatched on the upstream EarlyEventBuffer; close listeners awaited (no async Drop); Resource-dependency DAG (R3) not adopted — flat imperative registry per Spring + §5 don't-over-build

### `exit-code-shutdown`
R1-leaning + R3 exit-code-fold: three thin roles + a coordinator delegating to `Context::shutdown()`, with exit-code folded into the error model (NO separate ExitCodeExceptionMapper SPI). (1) Shutdown hook: a runtime-agnostic core seam `trait ShutdownTrigger` whose leaf-tokio impl is a cold `tokio::signal` (SIGTERM/ctrl-c) task, armed early when `BootstrapSettings.register_shutdown_hook`, CAS-once → `Context::shutdown().await`. (2) Exit code: `ErrorKind::exit_code() -> ExitCode` is the failure-path source (ADR-12, one story: 'port-in-use → analysis + action + conventional code'), plus a success-path `#[linkme::distributed_slice] static EXIT_CODE_CONTRIBUTORS: [ContributorDescriptor]` (beans yielding i32); aggregation = Spring's highest-magnitude-wins (max-of-positives or min-of-negatives, else 0). One `App`-level exit-code override hook. (3) Coordinator (`App::exit()` analogue): trigger `Context::shutdown()` (which drains the `TeardownLedger` LIFO after `stop_all()`, per-bean timeouts, `ShutdownReport`), THEN compute the code, publish `ExitCodeEvent`. ## Remaining risks (→ Phase 4)
- WiringPlan wave-parallelism (ADR-04 open Q): whether Context::refresh() overlaps declared-independent eager beans is measured-before-built and affects the not-ready window before Started; the run pipeline awaits refresh all-or-nothing regardless, so this is a downstream perf risk not a correctness one for bootstrap.
- Runner concurrency waves (R3) deferred behind a measured opt-in: minimizing the readiness window collides with a data-race footgun absent from Spring's always-sequential model — needs honestly-declared independence before it can be safe; default stays sequential.
- One unified application-lifecycle state machine: runtime-lifecycle (RunState), context-close (RunDown), and exit-code-shutdown each touch a watch-based machine — Phase 4 must reconcile whether these are ONE machine spanning bring-up→running→draining→terminated or a composed stack, so keep-alive's watch_run_state() await and shutdown coordination do not redundantly model the same transitions.
- RenderStyle::StructuredJson schema as a stable tooling contract (leaf doctor / CI gates) vs human-only in v1 — the FailureAnalysis/AssemblyReport/ConditionReport JSON schema is a versioning commitment; measure tooling demand (shared with ADR-12 open Q).
- cargo leaf prepare build-time plan (RunPlan/AppType freeze, R3 across features) NOT shipped in v1 — the runtime fold is the default; if startup cost or wasm/embedded/native-image targets demand it later, the RunPlan/DEDUCED_APP_TYPE const + manifest format become an ABI commitment that must version cleanly. Measured before built.
- The run-event-name taxonomy + run-participant descriptor traits are a versioned cross-crate ABI with no classpath to anchor them — must stay coherent as ONE shared milestone vocabulary with startup-instrumentation and lifecycle-events (three overlapping name vocabularies risk); ownership/versioning is a topology decision to finalize.
- ExitCode space: std::process::Termination's portable 0-255 vs Spring's arbitrary-i32 highest-magnitude aggregation — the public ExitCode type and whether the OBSERVED code (ExitCodeEvent) matches the COMPUTED one (Termination) needs a final call shaping the #[leaf::main] signature.
