# Execution & Ambient Context  `[execution-context]`

_The capability-split ExecutionFacility (Spawner/BlockingOffload/ConcurrencyGate), the TaskScheduler/@Scheduled cold-path timer-wheel, and the Cx per-poll re-install with per-key Propagation{Inherit,Isolate} that carries locale/request/tx context across thread hops._

This subsystem is the async spine of leaf, made concrete. It consolidates four Phase-1 features that the Phase-2 ADR-07 (async-context-model) already fused at the decision level — task-execution, scheduling, context-propagation, locale-context — and gives each an implementable design that shares ONE set of primitives rather than minting its own. The unification is total and was pre-settled by ADR-07's five sub-decisions (5a-5f): there is ONE capability-split executor (`ExecutionFacility` = `Spawner` + `BlockingOffload` + `ConcurrencyGate`, a `Role::Infrastructure` bean, runtime named only in `leaf-tokio`/`leaf-smol`); scheduling is NOT a second executor but a time-aware layer over the SAME `Spawner` (a `Trigger` SPI + one reactive `SchedulerCore` timer-wheel); context-propagation is NOT a ThreadLocal nor a runtime accessor-registry but ONE cheap-clone `Cx` bundle re-installed per poll over a leaf-owned `AmbientStore` task-local, with per-key `Propagation{Inherit,Isolate}`; and locale-context is NOT a separate holder mechanism but simply a set of `CxKey` declarations over that same `Cx` substrate, sugared by a thin `#[holder]` macro. Every async dyn seam returns `BoxFuture` (forced by the upstream `Provider::provide` shape, AFIT/RPITIT not dyn-compatible); teardown is the explicit container-driven `TeardownLedger` drain (no async Drop); reactive primitives on hot paths, the one shared timer-wheel on the sanctioned cold paths (retry/backoff/reconnect/@Scheduled). The Phase-1 candidate sets collapse cleanly: task-execution R2 (capability split) won; scheduling R2 (Trigger SPI + SchedulerCore over the executor) won, absorbing R3's optional config-bound trigger specs as a value-injection feature; context-propagation R2 (poll-wrapping combinator over one bundle) won as the primary mechanism with R1 demoted to a `CxBridge` seam-adapter and R3 retained as the always-present explicit `Cx` value escape hatch; locale-context realization-1 (task-scoped RAII holders) won, reframed as CxKey declarations.

## Shared machinery (how this rests on the toolkit)

The whole subsystem rests on five toolkit primitives, and the unification is that the four features share them rather than each re-deriving an ambient/spawn/timer mechanism.

1. ONE async seam currency: `BoxFuture<'a, T> = Pin<Box<dyn Future + Send + 'a>>` at every origin-agnostic dyn boundary, identical to the upstream `Provider::provide -> BoxFuture<Published>`. `Spawner::spawn`, `BlockingOffload::run_blocking`, `ConcurrencyGate::acquire`, `Lifecycle::start/stop_graceful`, the `@Async`/`@Scheduled` advisor seam, and `Shutdown::shutdown` all box their future. The monomorphized fast path stays available at known-concrete sites (Phase-3 escape, measure-first). Authored via `trait_variant::make(:Send)` + dynosaur, or a thin-macro-emitted `Pin<Box<dyn Future + Send>>` — no nightly, no RTN.

2. ONE ambient substrate: the cheap-clone `Cx(Arc<CxNode>)` bundle (Send+Sync+'static) over ONE leaf-owned `AmbientStore` task-local seam (backed by `tokio::task_local!` in leaf-tokio, never named in core). `Scoped<F>` re-installs the bundle on EVERY poll (the ThreadContextElement analogue, correct across work-stealing). This is the ONE place "the current X" lives — locale-context's holders, the request-scope binding the scoped-proxy `TargetSource` reads, the tx resource binding, and any tracing bridge ALL read this one bundle. Per-key `Propagation::{Inherit,Isolate}` on each `CxKey` carries locale/request/trace (Inherit) while pinning the tx/connection resource (Isolate, never auto-captured across a hop). This is the registry-substrate's by-data discipline applied to context: a `CxKey` is `{NAME, Value type, POLICY}` — data, not a registered accessor object, so there is no DCE-prone empty-registry bug.

3. ONE creation/lifecycle seam: the `ExecutionFacility`, `SchedulerCore`, and every accessor/bridge are ordinary beans resolved through the upstream `Provider`/`Engine::create` driver and selected by primary/qualifier. The facility and scheduler are `Role::Infrastructure`, so `Context::refresh()` auto-detects and installs them in the fixed-order template before any application bean is built (container-shape ADR). A missing execution facility is a HARD-FAIL at the refresh self-check (an async-first framework cannot run without it); a missing `AmbientStore` backing is degraded-not-fatal (context stops moving — a loud WARN, surfaced by the same anti-DCE expected-vs-found pass).

4. ONE teardown path: the upstream `TeardownLedger` drained LIFO by `Context::shutdown().await` after `stop_all()` — no async Drop anywhere. The facility's graceful drain, the scheduler quiesce, and any async tx finalization are awaited ledger entries; the Arc strong-count governs the real free so in-flight scoped work past a boundary stays sound. Sync-only cleanup (Cx restore on poll-exit, `Permit` release, holder scope restore) rides RAII `Drop` because it is cancel-safe.

5. ONE ordering + error spine: `@Scheduled` overlap ordering, advisor placement of `@Async`, and decorator order all use the single `cmp_order` comparator (RoleTier-first, lower-value-wins, ContractId tiebreak). All failures are nodes in the one `LeafError` causal chain rendered by `Diagnostic`: a malformed cron / dangling qualifier is a Tier-2 `AssemblyError` aggregated into the `AssemblyReport`; a failed scheduled fire or detached-task panic routes to the container-level `AsyncUncaughtFailureHandler` and enters the chain via the open `Integration{ContractId}` arm; the "@Scheduled descriptor discovered but no facility" and "accessor crate produced zero rows" checks are the same anti-DCE expected-vs-found self-check, not a parallel system.

## Features

### `task-execution`
REALIZATION: ADR-07 5c capability-split (Phase-1 R2) is the committed shape; R1's fat `TaskExecutor` and R3's config-profile dispatcher are rejected. THREE object-safe traits in leaf-core, each boxing its future at the dyn seam:
```rust
pub trait Spawner: Send + Sync { fn spawn(&self, fut: BoxFuture<'static,()>) -> SpawnHandle; }
pub trait BlockingOffload: Send + Sync { fn run_blocking(&self, f: Box<dyn FnOnce()+Send+'static>) -> BlockingHandle; }
pub trait ConcurrencyGate: Send + Sync { fn acquire(&self) -> BoxFuture<'static, Permit>; }
```
bundled under ONE `Role::Infrastructure` bean `ExecutionFacility` that impls all three over one runtime (the `applicationTaskExecutor` identity preserved as a single named bean). Consumers depend on EXACTLY the capability they need (`Ref<dyn Spawner>` / `Ref<dyn BlockingOffload>` / `Ref<dyn ConcurrencyGate>`), so 'this code may block' is a type-level property (charter 2.3). `SpawnHandle` awaits to `Result<T,JoinError>`; DROP=abort (cancel at next await), with `DropPolicy::Detach` for fire-and-forget (`@Async`, async-event, scheduled fire). `Permit` is RAII: `Drop` releases even on cancel (no leak), limit-1 = declarative instance lock — this is the ONE bounded-concurrency primitive shared by retry-resilience and scheduling overlap. Selection is pure DI primary/qualifier; the `execution.mode=force` lever is an `Environment`-bound `EnginePolicy`-adjacent flag that rebinds the qualifier resolution to the primary facility. SELECTED IMPLS: `leaf-tokio` registers a primary `TokioExecutionFacility` (spawn / spawn_blocking / Semaphore); `leaf-smol` an alternative; both are force-linked by `leaf-boot` and HARD-FAIL the refresh self-check if absent.
RESOLVED OPEN QUESTIONS: (a) shape = capability split (R2), settled by ADR-07 5c. (b) Send/'static-vs-swap-by-config: uniform `Send+'static` on spawned work and `Send+Sync+'static` on shared beans is MANDATED (ADR-07 5e), rides the upstream `Bean`/`ErasedBean` bound, enforced by a doctrine `#[diagnostic::on_unimplemented]` (`SpawnableWork: Send+'static`) — the single-threaded `?Send` lane is deferred behind the pluggable-runtime seam, demand-measured. (c) cancellation = SpawnHandle drop aborts, fire-and-forget detaches and is drained at shutdown then aborted past the deadline — framework-wide contract. (d) uncaught failure = ONE container-level `AsyncUncaughtFailureHandler` bean, composing with the error model via the `Integration{ContractId}` arm. (e) mandatory-runtime DCE = force-link + hard-fail self-check (worse-than-ordinary, so non-negotiable). (f) ConcurrencyGate owner = THIS feature (the shared lower primitive), consumed by retry/scheduling. (g) TaskScheduler is NOT separate (see scheduling). Construction is cold-path async, runtime-dependent, so the facility is among the earliest `Role::Infrastructure` beans `refresh()` installs.

**Resolved open questions:** Executor shape: capability split (Spawner/BlockingOffload/ConcurrencyGate) under one ExecutionFacility role bean — R1 fat trait and R3 config-dispatcher rejected (ADR-07 5c); Send/'static vs swap-by-config: uniform Send+'static / Send+Sync+'static mandated framework-wide, doctrine #[diagnostic::on_unimplemented], single-threaded ?Send lane deferred to pluggable-runtime; Cancellation: SpawnHandle drop=abort, fire-and-forget=detach-drained-then-aborted-past-deadline; cooperative, no thread interrupt; Uncaught task failure: one container-level AsyncUncaughtFailureHandler routing into the LeafError Integration arm; Mandatory-runtime Layer-0 DCE: force-link default runtime in leaf-boot + hard-fail expected-vs-found self-check at refresh; ConcurrencyGate ownership: belongs to this feature as the one bounded-permit primitive, consumed by retry-resilience and scheduling overlap

### `scheduling`
REALIZATION: ADR-07 5c + Phase-1 R2 — scheduling is a TIME-AWARE CAPABILITY over the SAME `Spawner`, NOT a second executor abstraction (no injectable `TaskScheduler` bean). Three pieces:
(1) A SYNC `Trigger` SPI (kept sync to avoid boxing the hot timing path): `trait Trigger: Send+Sync { fn next_fire(&self, ctx: TriggerContext) -> Option<Instant>; }` with built-in `CronTrigger`, `FixedRateTrigger`, `FixedDelayTrigger` and user-implementable custom cadences. `TriggerContext` carries `last_scheduled / last_completion / last_actual_fire` (Spring's Trigger.nextExecution shape) so fixedDelay's completion-feedback and cron's calendar both compute from one struct.
(2) ONE container-owned `SchedulerCore` (`Role::Infrastructure`, internal — not user-injectable): a single reactive timer-wheel min-heap of `(next_fire, task_id)` that `sleep_until`s the global earliest on the shared reactive timer (ADR-07 5f — never an interval spin), and on wake SPAWNS the due body onto the injected `Spawner` (the body NEVER runs on the driver — this is the structural fix for Spring's single-thread serialization gotcha, line 2045). The timer-wheel is the ONE shared cold-path timer also used by retry/backoff/reconnect.
(3) `#[scheduled(cron="..." | fixed_rate=... | fixed_delay=..., initial_delay=..., qualifier="...")]` is a THIN macro emitting one const `ScheduledMethodDescriptor { bean_contract, method_key, trigger_spec, qualifier }` into a `linkme` `SCHEDULED` slice via `::leaf_core` paths (data only, charter 2.10). An instance-tier post-processor at `after_init` binds each descriptor to the live bean `Ref` and registers `(Trigger, body)` into `SchedulerCore`; arming waits for the SmartInitializing all-singletons barrier in `refresh()` so a fire never hits a half-built graph.
Firing as a `Role::Infrastructure` advisor: a `@Scheduled` method that is also `@Async`/`@Transactional` composes through the ONE proxy chain (proxy-substrate), ordered by `cmp_order`.
RESOLVED OPEN QUESTIONS: (a) separate-vs-capability = capability over the executor (R2). (b) overlap policy DEFAULT = skip-if-still-running (async has no implicit ceiling — the inverted gotcha), with queue / allow-concurrent as per-trigger opt-ins, bounded by the shared `ConcurrencyGate`. (c) fixedDelay completion-feedback: `SchedulerCore` awaits the body's `SpawnHandle` before computing `next_fire`, threading completion into `TriggerContext.last_completion` (getting this wrong silently turns fixedDelay into fixedRate — a documented correctness contract). (d) compile-vs-runtime: literal trigger specs validated at a Tier-2 startup pass (parse all crons, dedupe ids, resolve `${...}` placeholders via value-injection, check qualifiers) aggregated into the `AssemblyReport` — earliest legible, NOT first-fire; config-bound specs (the R3 idea) are absorbed as ordinary value-injection placeholders, not a separate schedule-table substrate. (e) cron engine: own a 6/7-field calendar engine (a `leaf-cron` helper crate) parsed at the startup pass (the rare CPU-bound bit on a cold path); missed-fire policy = skip-to-next after a clock-jump by default, with fire-once opt-in (documented divergence from any single Rust crate). (f) cancellation/drain: `SchedulerCore` participates in `shutdown().await` — stop arming, drain in-flight via the facility's drain, abort past deadline. (g) failed fire routes to the SHARED `AsyncUncaughtFailureHandler` and the schedule CONTINUES (one bad fire is not a dead task). (h) timer driver = ONE global wheel shared with retry. (i) scheduling-specific anti-DCE: the expected-vs-found self-check enumerates discovered-vs-expected `SCHEDULED` descriptors — a dropped descriptor is a task that silently never fires, so this is non-optional. (j) context into a fire: the body is spawned with `.scoped(Cx::current())` so Inherit keys (locale/trace) follow; the Isolate tx key does not (see context-propagation).

**Resolved open questions:** Separate scheduler vs time-aware capability: capability — a Trigger SPI + one SchedulerCore timer-wheel delegating all running to the injected Spawner; no separate TaskScheduler bean (ADR-07 5c); Overlap/concurrency default: skip-if-still-running, bounded by the shared ConcurrencyGate; queue/allow-concurrent are per-trigger opt-ins; fixedDelay completion-feedback: SchedulerCore awaits the body SpawnHandle before next_fire, threaded via TriggerContext.last_completion; Compile-vs-runtime: literal specs validated at a Tier-2 startup pass into AssemblyReport; config-bound specs absorbed as value-injection ${...} placeholders, not a separate table; Cron engine + missed-fire: own a 6/7-field calendar engine (leaf-cron), parse at startup pass, default skip-to-next on clock-jump with fire-once opt-in; Cancellation/drain + failed fire: participates in shutdown().await drain; failed fire routes to shared AsyncUncaughtFailureHandler and schedule continues; Timer driver: one global reactive timer-wheel shared with retry-resilience backoff and periodic-refresh cold paths; Scheduling anti-DCE: discovered-vs-expected SCHEDULED-descriptor enumeration in the expected-vs-found self-check (a dropped descriptor never fires)

### `context-propagation`
REALIZATION: ADR-07 5a — the poll-wrapping combinator over ONE bundle (Phase-1 R2) is the PRIMARY mechanism; R1's runtime accessor-registry is DEMOTED to a `CxBridge` seam-adapter (tracing scope-stacks, WASM marshalling); R3's explicit value is RETAINED as the always-present escape hatch (`Cx` is itself the explicit value). Core types (leaf-core, runtime-naming-free):
```rust
#[derive(Clone)] pub struct Cx(Arc<CxNode>);                 // Send+Sync+'static, structural-sharing
pub trait CxKey: 'static { type Value: Send+Sync+'static; const NAME: &'static str; const POLICY: Propagation; }
pub enum Propagation { Inherit, Isolate }
impl Cx { fn get<K:CxKey>(&self)->Option<&K::Value>; fn with<K:CxKey>(&self,v:K::Value)->Cx; fn current()->Option<Cx>; fn enter<R>(&self, f: impl FnOnce()->R)->R; }
pub trait AmbientStore: Send+Sync { fn with_installed<R>(&self,cx:&Cx,f:&mut dyn FnMut()->R)->R; fn current(&self)->Option<Cx>; }
pub struct Scoped<F>{/* cx+inner; poll = store.with_installed(&cx, || inner.poll()) EVERY poll */}
pub trait CxFutureExt: Future+Sized { fn scoped(self, cx: Cx) -> Scoped<Self>; }
pub trait CxBridge: Send+Sync { fn project(&self, cx:&Cx) -> BridgeGuard; }   // sync Drop = pop
pub trait CxDecorator: Send+Sync { fn capture(&self) -> Cx; }                  // opt-in propagation at submit
```
The `AmbientStore` trait is the ONE storage seam; `leaf-tokio` backs it with `tokio::task_local!`. `Scoped<F>` re-installs the bundle on EVERY poll (the only model correct across work-stealing migration). Propagation across a hop = `child.scoped(Cx::current())` = ONE Arc clone + a poll-wrapper — far cheaper than R1's per-hop box-dance on hot fan-out. ISOLATE by default; INHERIT keys auto-captured by the `ExecutionFacility`'s `CxDecorator` ONLY when propagation is enabled (the `task.execution.propagate-context` analogue, an `Environment`-bound per-facility/per-profile toggle). The tx/connection resource is an `Isolate` `CxKey` — read ambiently within one logical task, NEVER auto-captured across the `@Async`/scheduled spawn hop (the never-inheritable rule as a TYPED default, ADR-07's keystone for transaction-management).
RESOLVED OPEN QUESTIONS: (a) THE KEYSTONE = poll-re-install over one bundle (R2). (b) task-local primitive = the leaf-owned `AmbientStore` task-local, runtime-backed (core names no runtime); skip-install-when-unchanged (Arc-ptr compare) is a deferred Phase-3 measured optimization. (c) Send bounds = `Send+Sync+'static` riding `ErasedBean`/`Bean` in one place; a non-Send/Sync ambient value cannot ride a hop, surfaced by the doctrine diagnostic. (d) one bundle vs N accessors = ONE bundle (cheaper, one inspectable schema, kills the DCE empty-registry bug). (e) scope-based concerns (tracing/observation open-scope, NullObservation masking) do NOT map to a value bundle and are served by `CxBridge` adapters that own their push/pop inside the bridge — an explicit bounded non-uniformity. (f) explicit-vs-ambient line: ambient is primary (AOP/event/schedule seams cannot thread a param and read `Cx::current()`); the explicit `Cx` value is the user-call-graph escape hatch — no second model to desync because both ARE the same bundle. (g) accessor discovery: `CxKey` declarations are data (one bundle schema), inspectable at startup via `leaf doctor`; the bundle schema + active executor/decorator set are listed by the anti-DCE self-check (degraded-not-fatal WARN if backing absent). (h) per-key policy = the typed `Propagation` const on each `CxKey` (Inherit/Isolate), the never-inheritable tx case as the typed default. (i) cancellation: sync restore rides `Scoped` poll-exit `Drop` cleanly (cancel can't skip it, the slot is pristine between polls); any async context cleanup is an awaited `TeardownLedger` entry, never Drop. (j) WASM marshalling: deferred (WASM dropped from active design).

**Resolved open questions:** The ambient-across-.await keystone: poll-re-installing combinator (Scoped<F>) over ONE Cx bundle, re-installed every poll — R1 accessor-registry demoted to CxBridge, R3 explicit value kept as escape hatch (ADR-07 5a); Task-local primitive: leaf-owned AmbientStore task-local seam, runtime-backed (core names no runtime); skip-install-when-unchanged deferred as a measured Phase-3 optimization; Send/Sync bounds: Send+Sync+'static riding ErasedBean/Bean in one place; non-Send/Sync ambient values cannot propagate, surfaced by doctrine diagnostic; One bundle vs N accessors: ONE cheap-clone bundle with one inspectable schema — kills the silent-DCE empty-registry bug; Scope-based contexts (tracing/observation masking): served by CxBridge seam-adapters owning their scope push/pop, not the core value bundle; Explicit-vs-ambient line: ambient primary (seams read Cx::current()), explicit Cx value as the escape hatch — same bundle, no desync model; Per-key propagation policy: typed Propagation const per CxKey (Inherit locale/trace, Isolate tx/connection) — never-inheritable as a typed default; Cancellation/restore: sync restore rides Scoped poll-exit Drop (slot pristine between polls); async context cleanup is an awaited TeardownLedger entry, never Drop

### `locale-context`
REALIZATION: Phase-1 realization-1 (task-scoped RAII holders), reframed onto the toolkit so locale-context introduces NO new ambient mechanism — it is simply a set of `CxKey` declarations over the ONE `Cx` substrate that context-propagation owns. 'The current X' (locale, request attributes, tx-sync) are `CxKey` types declared in the crate that owns the concept (locale in the i18n crate, request attrs in a web integration crate, tx-sync in the tx crate — NEVER hardcoded in core):
```rust
pub struct LocaleKey; impl CxKey for LocaleKey { type Value = Locale; const NAME: &'static str = "locale"; const POLICY: Propagation = Inherit; }
pub struct TxResourceKey; impl CxKey for TxResourceKey { type Value = TxBinding; const NAME: &'static str = "tx.resource"; const POLICY: Propagation = Isolate; }
```
A thin `#[holder]` attribute macro is PURE SUGAR over a `CxKey` impl + a typed `Holder<K>` accessor handle (`HOLDER.scope(v, fut).await` = `fut.scoped(Cx::current().unwrap_or_default().with::<K>(v))`; `HOLDER.get()` = `Cx::current().and_then(|c| c.get::<K>().cloned())`; `HOLDER.with(|v| ...)` for borrow). Reads are pure task-local lookups (no global lock, no Arc churn for Copy values) — hot-path safe. The 'name' field is repurposed from Spring's NamedThreadLocal into the `NAME` const used in 'no current X bound — were you inside HOLDER.scope(...)?' diagnostics (rich-diagnostics, charter 2.9) and in the startup-inspectable bundle schema. Dual-visibility (Spring's inheritable/non-inheritable dual-field) IS the `Propagation` const: Inherit holders are auto-captured into spawned children by the facility's `CxDecorator`; Isolate holders (tx) are not. The `LocaleKey` is the canonical input to messages-i18n: `MessageSource::resolve(code, args)` reads `LocaleKey` from the ambient `Cx` when no explicit locale is passed (Spring's LocaleContextHolder feeding MessageSource).
RESOLVED OPEN QUESTIONS: (a) canonical task-local primitive = the shared `AmbientStore`/`Cx` (no separate holder storage); leaf takes a soft default-runtime dependency for the backing via leaf-tokio. (b) propagate-by-default vs isolate = ISOLATE by default (ADR-07 5a, data-bleed defense), Inherit per-key opt-in. (c) ambient-everywhere vs explicit-&Ctx = ambient `Cx` is the DX default with the explicit `Cx` value always available — the hybrid is the same bundle, so realization-3's multi-backing complexity collapses (the only sync-bridge case is a `CxBridge` projecting into a `spawn_blocking` island, which leaf-tokio's `BlockingOffload` wraps). (d) holder registry substrate = `CxKey` declarations are data; uniqueness is enforced like bean-naming (declare-once or a loud `AssemblyError` at the freeze collision guard, keyed by the `NAME`/`ContractId`). (e) holder-scope vs bean-scope lifetime: a request/session scope opens at a `Cx::with`/scope boundary and its instance-store ledger drains at scope-end via the explicit async teardown — values are `Send+Sync+'static` (clone-out or `with(|&v|...)` borrow). (f) Send/Sync discipline: Inherit holders MUST be `Send+Sync+'static` (they cross a hop); the bound rides the handle. (g) unified Scope primitive: ONE `Scoped`/`Cx::with` serves holder scoping, propagation restore, and the bridge — under cancellation it rides sync Drop (no async Drop). std::thread_local survives ONLY as a bounded sync-bridge inside thread-pinned `spawn_blocking` islands via a `CxBridge`, never as the async hot-path mechanism.

**Resolved open questions:** Canonical task-local primitive: the shared AmbientStore/Cx substrate (no separate holder storage); soft default-runtime dependency via leaf-tokio; Propagate-by-default vs isolate: ISOLATE by default (data-bleed defense), Inherit is a per-key typed opt-in; Ambient-everywhere vs explicit-&Ctx: ambient Cx is the DX default with the explicit Cx value always available — same bundle, so the multi-backing hybrid collapses; Holder registry substrate: CxKey declarations are data; declare-once enforced like bean-naming (loud collision error at freeze, keyed by NAME/ContractId); Holder-scope vs bean-scope lifetime: scope opens at a Cx::with boundary, instance-store ledger drains at scope-end via explicit async teardown; values Send+Sync+'static (clone-out or with(|&v|)); Send/Sync discipline: Inherit holders must be Send+Sync+'static (they cross a hop); bound rides the handle; Unified Scope primitive: one Scoped/Cx::with serves holder scoping + propagation restore + sync-bridge; rides sync Drop under cancellation; thread_local survives only as a bounded spawn_blocking CxBridge

## Cross-feature interactions
- **task-execution ↔ scheduling** — Scheduling owns WHEN (the Trigger + SchedulerCore timer-wheel), task-execution owns WHERE (the injected Spawner). The scheduled body is SPAWNED onto the facility, so virtual-thread support, the ConcurrencyGate overlap ceiling, the CxDecorator, and graceful-drain lifecycle are all inherited, not duplicated. The single-thread serialization gotcha is structurally impossible because the driver never runs the body inline.
- **context-propagation ↔ task-execution** — task-execution exposes the per-facility/per-profile propagate-context toggle and the CxDecorator submit-time capture; context-propagation defines what is captured (the Cx bundle) and the Scoped poll-wrapper that re-installs it in the spawned task. The hop is child.scoped(Cx::current()). ISOLATE-by-default means an undecorated spawn carries no context (data-bleed defense).
- **context-propagation ↔ locale-context** — locale-context is the storage half (CxKey declarations = the holders), context-propagation is the movement half (the Scoped combinator + Propagation policy). They share ONE bundle and ONE Scope abstraction — the prime unification: locale-context introduces no second mechanism, it is data over the same Cx. The critical shared constraint (holder storage must be task-scoped, not thread_local) is satisfied by construction because both ride AmbientStore.
- **context-propagation ↔ transaction-management (downstream)** — The tx/connection resource binding is an Isolate CxKey read ambiently within one logical task and NEVER auto-captured across an @Async/scheduled hop — the never-inheritable rule realized as a typed Propagation::Isolate default. Tx finalization under cancellation has no async Drop: it is explicit awaited cleanup via the interceptor's around-advice / the TeardownLedger.
- **this subsystem ↔ proxy-substrate (downstream)** — @Async is a Role::Infrastructure dispatch advisor in the ONE shared proxy chain (not a body-rewrite by default), so its order vs @Transactional/@Cacheable is governed by the single cmp_order chain sort (a tx must not span the async hop). A @Scheduled @Async @Transactional method composes through one chain. Interception seams read the ambient Cx (they cannot thread a param). proceed() crossing a hop participates in propagation; cancel-safety is sync RAII Drop + the teardown ledger.
- **task-execution/scheduling ↔ container-shape refresh-lifecycle** — ExecutionFacility and SchedulerCore are Role::Infrastructure beans auto-detected by Context::refresh() in the fixed-order template and installed before any application bean. A missing facility HARD-FAILS the expected-vs-found self-check (an async-first framework cannot run without it); a missing AmbientStore backing is degraded-not-fatal (WARN). The scheduler arms only after the SmartInitializing all-singletons barrier.
- **task-execution/scheduling ↔ retry-resilience (downstream)** — The ConcurrencyGate bounded-permit is the ONE shared primitive (task-execution owns it, retry's @ConcurrencyLimit and scheduling's overlap ceiling consume it; limit-1 = instance lock). Retry/backoff, reconnect, periodic refresh, and @Scheduled triggers all await the ONE shared reactive timer-wheel — the only sanctioned cold-path 'poll' loops; the permit releases on cancel via RAII even across backoff sleeps.
- **this subsystem ↔ error-model** — Malformed cron / dangling qualifier / duplicate scheduled-id and duplicate-holder-name are Tier-2 AssemblyError nodes aggregated into the AssemblyReport at the startup/freeze pass. A failed scheduled fire and a detached-task panic route to the one container-level AsyncUncaughtFailureHandler and enter the LeafError chain via the open Integration{ContractId} arm. The 'scheduled descriptor discovered but no facility' and 'accessor crate produced zero rows' checks are the same anti-DCE expected-vs-found self-check, rendered by the one Diagnostic.
- **locale-context ↔ messages-i18n / events (downstream)** — LocaleKey is the canonical ambient input to MessageSource::resolve when no explicit locale is passed. Synchronous event dispatch runs on the caller's task and preserves Cx naturally; async dispatch carries context only when the CxDecorator is enabled (ISOLATE-by-default). Validation message resolution defers to format-time so it reads the ambient LocaleKey then, avoiding touching the holder on the validate path.

## Remaining risks (→ Phase 4)
- skip-Cx-install-when-unchanged optimization (Arc-ptr compare to elide the per-poll with_installed swap) is unmeasured — the per-poll cost on deep-await hot paths must be benchmarked before deciding it is worth the branch (ADR-07 defers it as a Phase-3 measured escape).
- The single global reactive timer-wheel (one sleep/one waker for the whole app, maximally reactive) is a single point of failure for every scheduled task; whether to shard per-scheduler and the exact fixedDelay completion-feedback composition across the spawn boundary needs measurement under load.
- Cron calendar semantics divergence: owning leaf-cron means picking DST/timezone and missed-fire behavior that may differ from Spring's exact dialect AND from any single Rust crate — a parity-vs-pragmatism risk that needs an explicit documented contract and conformance tests.
- AsyncUncaughtFailureHandler granularity (single container-level vs per-executor vs per-@Async-method) and its exact composition with the LeafError Integration arm is left open by ADR-07; the single-level default may be too coarse for per-task error routing.
- Single-threaded ?Send carve-out is deferred behind the pluggable-runtime seam (demand-measured); if a current-thread runtime backing is later demanded, reconciling a relaxed bound without re-introducing sync/async bimodality (charter 2.3) is unresolved.
- CxBridge scope-based concerns (tracing/observation NullObservation masking) are an explicit bounded non-uniformity outside the value bundle; full scope-masking parity lives in the bridge layer and its cancel/teardown interaction with the per-poll model needs validation when an observability integration is built.
- Detached fire-and-forget tasks can outlive the shutdown drain window and are then aborted; the exact deadline policy and whether a straggler holding an Isolate tx binding can corrupt state past the boundary needs a concrete cancel-safety audit alongside transaction-management.
