# Binding, Type Conversion & Typed Config  `[binding-conversion]`

_The central ConversionService, the relaxed-binding canonical key identity, the Binder tree-descent engine and its BindResult/BindHandler, @ConfigurationProperties typed targets and emitted config-metadata, @Value injection, and the ${}-vs-#{} dispatch + placeholder escaping at the value-string boundary._

This subsystem is the value-shape half of leaf: it turns frozen string-keyed configuration into typed Rust values and lands them either at scattered injection points (@Value) or onto cohesive typed config objects (@ConfigurationProperties). It sits ENTIRELY in the post-seal, synchronous, lock-free read world the upstream environment-config subsystem hands it: every feature reads the immutable EnvCore/SealedStack (the cheap-clone Env handle) and never re-opens IO. It consolidates nine features into one chain — Canonical key identity (relaxed-binding) → first-source-wins value (the upstream PropertySource stack via the binder-facing ConfigurationPropertySource adapter) → ${}-vs-#{} dispatch + escaping at the value-string boundary (extra-12/extra-13) → trait-native coercion (type-conversion) → tree-descent binding (binder) → typed targets and their emitted metadata (config-properties, config-metadata). The central unification is that there is NO second registry, NO second error type, NO second async fence, and NO second comparator: conversion is compile-time-dispatched (FromConfigValue) at statically-known target types with a thin erased ConversionService only where the binder genuinely loses the type; @ConfigurationProperties types ARE ordinary beans whose Provider is a binder-backed factory (so they ride the registry-substrate, the App<Wired> validation pass, and the LeafError/AssemblyReport spine like every other bean); @Value is an ordinary PointKind::Value injection point on the existing InjectionPlan (step-2 on the same selection spine as by-type autowiring); and every diagnostic — UnresolvedValue, Convert, Bind, Validation — is a LeafError ErrorKind node aggregated into the one AssemblyReport. The four Phase-1 keystone questions (trait-vs-registry conversion, lowered-vs-runtime @Value, the ${}/#{} dispatch owner, and the bind-target self-description) all resolve toward the rust-native, compile-time, thin-macro, zero-extra-substrate answer.

## Shared machinery (how this rests on the toolkit)

The whole subsystem mints almost nothing; it rests on the fixed toolkit primitives and the unification is that all nine features share them.

1. ONE freeze boundary, consumed not re-created. Every feature reads the post-seal world. The upstream seal_environment() async fence (compile-runtime-split 5f) is the ONLY IO; after it, the cheap-clone Env = Arc<EnvCore> (ownership-model Context field) is read lock-free. Relaxed-name indexes, the placeholder PlaceholderSyntax grammar config, the binder-facing ConfigurationPropertySource (CPS) adapter, and the random.* source are all materialized at/before seal and frozen — no value-phase mutation is representable (the EnvBuilder is consumed). @ConfigurationProperties binding+JSR-validation runs at exactly ONE site — `App<Wired>::validate()` (the one place validate() materializes beans) over this frozen Env, per `[C2-config-locus]`: validate() PRE-MATERIALIZES every config-properties bean (calls its binder-backed `Provider::provide` early, aggregating BindError/Convert/Validation into the one Tier-2 AssemblyReport) and STORES the bound `Arc` into the singleton slot's OnceCell; refresh R5's eager bitset then EXCLUDES config-properties beans (they are eager-EXCLUDED-because-PREBOUND) so `Provider::provide` is NEVER re-run at R5 and a malformed value can NEVER route into the R5 cancel-cascade. config-metadata emits at compile/build and is never read by the running app.

2. ONE creation/identity seam. A @ConfigurationProperties target is NOT a parallel pipeline — it is an ordinary bean: a const Descriptor + a ProviderSeed (codegen-boundary) whose Provider::provide builds the value by calling the binder against Env (never the bean container — pure-projection enforced structurally because the factory receives only &Env + a BindHandler, no ResolveCtx bean access). It lands in the dense BeanId registry, is resolved by the one Selector spine, and is injectable as Ref<AppProperties> like any singleton. TIMING (the `[C2-config-locus]` carve-out): because its Provider is pure-projection (&Env+BindHandler only, no bean collaborators), `App<Wired>::validate()` can construct it EARLY without any wiring being live — this is the ONE bean class validate() materializes. validate() calls `Provider::provide` for each config-properties BeanId, aggregates failures at Tier-2, and writes the resulting `Published::Shared(Arc)` into that slot's OnceCell; refresh R5 sees the slot already-initialized (eager-EXCLUDED-because-PREBOUND) and publishes the existing Arc without re-running the factory. So binding runs exactly once, at validate-time, never at R5. @Value is a PointKind::Value row on the same const InjectionPlan (injection-mechanics) the macro already emits per bean — step-2 on the identical fixed-order layer-fold, no magic side channel, no second BeanPostProcessor. A `@Value`'s `resolve_into::<T>` coercion is likewise DRY-RUN at validate() (the value computed and discarded) so a malformed coercion is a Tier-2 node, not a Tier-3 `Provider::provide` fault.

3. ONE async/sync fence model (async-context-model ADR-07). The entire chain — relaxed lookup, placeholder walk, escape de-escaping, ${}/#{} dispatch, conversion, tree-descent, validation — is SYNCHRONOUS, non-blocking, allocation-light on the hot/common path (Cow-borrowing fast paths). It NEVER .awaits and NEVER polls (reactive-hot rule): any remote/async-backed source was snapshotted into EnvCore before seal. The only async-shaped seam is that binding runs inside the cold App<Resolve>→App<Wired> pass and a @ConfigurationProperties Provider::provide is a ready-future. No async Drop. The one exception surface (a #{} expression segment) delegates to the expr subsystem's sync-pure ValueExpr<T> closure (expr-i18n-resources), which is contractually sync — so even dispatch stays off the await path.

4. ONE diagnostic spine (error-model ADR-12). UnresolvedValue, Convert, Bind, Validation, ConfigIo are existing closed ErrorKind arms — never new top-level enums. The ONE ErrorSink accumulator collects ALL bind/convert/validation failures across a target (collect-all, not fail-on-first) and folds them into the App<Wired> AssemblyReport at Tier 2; a malformed ${...}/#{...} grammar or a field type lacking FromConfigValue is a Tier-0 compile error at the attribute span; a LazyValue.get()/random.* grammar miss is necessarily Tier-3 runtime. Every node carries Origin = File{path,line} (or the OriginId interned by the upstream OriginStore) so a coercion failure reads 'cannot convert "abc" to u16 for order.max-retries (application.yaml:12, matched env ORDER_MAX_RETRIES)'. FailureAnalyzers ship for BindError. The strict-vs-lenient @Value default (the error-model open question) resolves to STRICT-by-default surfaced as a Tier-2 Convert/UnresolvedValue from the `[C2-config-locus]` validate-time dry-run, with lenient either a per-site `#[value("…", lenient)]`/`Option<T>` opt-in OR the global `BootstrapSettings.startup_validation = Lenient` lever (Spring-migrant escape) — both downgrade the @Value/placeholder value-shape facets to `Severity::Warn`. Config-properties bind+JSR-validation is the OTHER Tier-2 facet and rides the SAME aggregated report at the SAME validate-time site; its range/nested JSR checks are NOT lever-downgradable (no Tier-3 fallback).

5. ONE candidate/selection + ordering reuse. @Value rides the existing Selector::resolve_one as PointKind::Value; the value layer short-circuits the by-type candidate search exactly as Spring's doResolveDependency step-2. Config-properties beans order via the one cmp_order. No new comparator, no new resolver.

6. ONE cross-crate discovery substrate (codegen-boundary + cross-crate-discovery). config-properties contributes a const Descriptor into COMPONENTS (it is just a bean) — so its scan route IS the existing component-discovery substrate with the triple anti-DCE defense; a force-linked-but-zero-contribution config crate is a loud AntiDceError (high stakes: a missing config bean = app runs on defaults). config-metadata uniquely RELAXES this: it emits into a sibling CONFIG_METADATA slice whose DCE miss is BENIGN (tooling-only), so it needs no force-link self-check — only a leaf metadata --check CI gate. type-conversion's FromConfigValue needs NO discovery at all (coherent trait impls resolved at the call site, zero DCE exposure); only the optional ConversionService custom-converter registry would ride CONVERTERS, and it is opt-in.

7. ONE key-identity model. relaxed-binding's CanonicalName (uniform-form equivalence) is THE single owner of key identity consumed by the placeholder walk (canonicalize-before-lookup, preserving Spring's asymmetric-placeholder contract), the binder's tree-descent, OnProperty conditions (conditional-strategy already routes its property reads through uniform-form), and config-metadata's advertised form. The CPS adapter's tri-state (Present/Absent/Unknown) is the binder-facing view over the SealedStack.

## Features

### `type-conversion`
ADOPT trait-native FromConfigValue as the default, monomorphized at the statically-known target type, with a thin erased ConversionService used ONLY where the binder loses the type. `pub trait FromConfigValue: Sized { fn from_config_value(v: &ConfigValue, cx: &ConvertCtx) -> Result<Self, LeafError>; }` where ConfigValue is the upstream origin-carrying value (scalar Cow<str> + OriginId, or a list/map cursor the binder supplies). leaf-core ships blanket impls: `impl<T: FromStr> FromConfigValue for T` (number/bool), a `#[derive(LeafEnum)]` thin macro emitting a case-insensitive match, and leaf-owned grammar newtypes Duration/DataSize/Period implementing the Spring suffix grammar; collections via `impl<T: FromConfigValue> FromConfigValue for Vec<T>/HashSet<T>` (comma-split or indexed children) and `HashMap<K,V>` (child keys). ConvertCtx carries the strict/lenient policy, the unit-default hint (from a #[config(unit="s")] attribute the binder/value macro forwards — NOT an ambient thread-local, honoring ADR-07), and the locale pulled from the ambient Cx Holder<LocaleKey> (shared with messages-i18n) only when a locale-sensitive converter is invoked. Convert failures are ErrorKind::Convert LeafError nodes carrying the raw value, the concrete target type name (free, statically known), and Origin. The erased fallback: `pub struct ConversionService` is a frozen-after-seal HashMap<(TypeId,TypeId), Arc<dyn Converter>> consulted by the binder's erased dyn path and by a custom `#[converter]` bean contributed via the CONVERTERS linkme slice; it is OPT-IN and built once at seal (lock-free reads after). Orphan-rule escape: a leaf-owned `Conv<T>` newtype bridge for foreign target types.

**Resolved open questions:** Trait vs runtime registry: trait-native FromConfigValue is the default (zero registry, zero TypeId lookup, compile-error on a missing impl); the runtime ConversionService is a thin opt-in fallback for the genuinely erased binder path and Spring-parity custom-converter registration — NOT the primary mechanism.; serde adoption: NOT adopted into leaf-core's mandatory surface; a feature-gated serde-bridge converter is offered but FromConfigValue is the canonical trait so diagnostics keep Origin/provenance richness.; ConvertCtx threading without an .await-unsafe thread-local: strict/lenient + unit-default are passed explicitly via ConvertCtx (forwarded by the value/bind macro); locale is read from the ambient Cx Holder<LocaleKey> only at the locale-sensitive call, never captured eagerly.; Cross-crate converter discovery: rides the existing CONVERTERS linkme slice with the triple anti-DCE defense, but only for the opt-in registry path; the trait path is DCE-inert.; Fail-fast timing: a missing FromConfigValue impl is Tier-0 (compile); a coercion failure is Tier-2 (App<Wired> AssemblyReport via the ErrorSink) for both @Value and @ConfigurationProperties — consistent across both surfaces.; Orphan-rule owner: a leaf-owned Conv<T> newtype + #[derive(LeafEnum)] cover the common cases; the registry exists regardless for the dynamic-target case but is not forced on the default path.; Grammar types Duration/DataSize/Period: leaf-owned newtypes in leaf-core for coherent downstream impls + controllable suffix grammar.

### `relaxed-binding`
ADOPT R1 — a CanonicalName value type as the single owner of key identity — fused with the upstream CPS adapter and a per-enumerable-source uniform-form index built at seal. `pub struct CanonicalName { segments: Box<[Segment]> }` with `enum Segment { Named(Box<str>), Indexed(u32), MapKey(Box<str>) }`; `CanonicalName::parse(&str) -> Result<Self, NameSyntaxError>` validates kebab canonical form; the equivalence is `uniform()` — a const-fn char fold (lowercase, drop non-alnum separators except inside a bracket-escaped segment). Two reconciled lookup paths under one rule: an ENUMERABLE source (parsed file/JSON/env snapshot) is wrapped once at seal into a RelaxedIndex (HashMap<UniformKey, IndexedEntry{ raw_key, OriginId }>) so a relaxed get is one normalize + one hashmap hit (the upstream ConfigurationPropertySource::get/iter is implemented over this); a NON-enumerable source (raw env, random.*) is probed by the bounded env four-candidate generator (dash-removed + legacy-underscore × upper/lower, capped at four WHOLE-name forms — never a per-segment cartesian product). Both paths share one normalization fn so the asymmetric-placeholder contract can't diverge. The env-only namespacing prefix is a field on the system-env source's mapper, applied to env only. RelaxedCollision (first-name AND firstName in one source) is detected at index-build and surfaced as a Tier-2 LeafError listing every collision at once (resolving toward catch-early). The placeholder walk canonicalizes a placeholder's KEY through CanonicalName::parse before lookup (preserving Spring's documented asymmetry as intent). config-properties derives each Rust field's canonical kebab name from its snake_case ident via this one rule.

**Resolved open questions:** Relaxed collision policy: detected at seal-time index build, surfaced as an aggregated Tier-2 LeafError (one report listing all collisions) rather than silent last-wins — catch-early per the diagnostics spine, while not hard-failing a config that merely repeats one form.; Where canonicalization lives: R1 CanonicalName value type is the single identity owner; per-enumerable-source RelaxedIndex (R3) is the storage acceleration UNDER it; non-enumerable sources use bounded candidate probing (R1/R2) — one normalization rule shared across both, so it is R1-owns-identity with R3-as-implementation, not three competing models.; Placeholder key always canonicalized: YES — the walk canonicalizes the key before lookup, preserving Spring's asymmetry (${demo.itemPrice} does NOT match demo.item-price) as deliberate intent; this is owned here and consumed by property-resolution.; Bracket-escape owner: relaxed-binding owns the escape semantics (a MapKey segment preserves its contents verbatim, disabling normalization for that segment); the binder consumes the parsed Segment and never re-strips brackets; the dispatcher passes ${[..]} keys through unchanged.; Custom mapper cross-crate substrate: the two stock mappers (dotted-kebab + system-env) ship in leaf-core always-present; an optional third-party mapper rides the existing component-discovery substrate keyed by source-kind DATA — shared, not a parallel slice.; Env candidate fan-out bound: capped at four whole-name forms (matching Spring); the same bounded set is computed by the index builder and the probe path so the two never diverge.; Mutable/refreshable index: out of v1 scope (config is sealed); a future config-refresh rebuilds the RelaxedIndex into a fresh EnvCore epoch and arc_swaps it (the registry-substrate reopen_epoch lane), never a hot-path lock.

### `binder`
ADOPT R1 — a derived BindTarget const schema + a hand-written recursive walker over the CPS adapter, with the BindHandler unified onto property-resolution's ResolveHandler shape. `pub trait BindTarget: Sized { const SCHEMA: &'static NodeSchema; fn bind(c: &mut BindCursor<'_>) -> BindOutcome<Self>; }`; `enum NodeSchema { Scalar, Object{ method: BindMethod, fields: &'static [Field] }, List(&'static NodeSchema), Map(&'static NodeSchema), Set(&'static NodeSchema) }`; recursive const-self-reference (Vec<Self>) handled by a fn-pointer indirection `&'static NodeSchema` resolved lazily. `pub struct Bindable<T>{ existing: Option<T>, supplied: Option<Box<dyn FnOnce()->T>>, method_override: Option<BindMethod>, .. }`; `pub enum BindResult<T>{ Bound(T), Unbound, Failed(LeafError) }` (three-state monad — Unbound is NOT an error, distinct from Failed). `pub struct Binder<'s>{ cps: &'s dyn ConfigurationPropertySource, conv: &'s ConversionService, handler: &'s dyn BindHandler }` with `bind<T:BindTarget>(&self, prefix:&CanonicalName, b:Bindable<T>) -> BindResult<T>` and `bind_or_create`. The walker selects BindMethod (override > existing-forces-JavaBean > derived-from-shape) and descends one segment per object level building child CanonicalNames. Collection REPLACEMENT (one source owns the whole List/Map) is answered via the CPS adapter's contains_descendant_of(name) tri-state: the single highest-precedence source whose state is Present for any descendant owns the collection; never assembled across sources (preserves never-blend). Rust JavaBean convention: a #[derive(BindTarget)] struct with Default + settable fields = JavaBean; a type with a single non-injected constructor (records) = ValueObject; existing-value-forces-JavaBean falls back to field-wise set (an immutable type bound onto an existing instance is a Tier-2 error, not silent). BindHandler is the SAME observer trait property-resolution uses: `trait BindHandler { fn on_start/on_success/on_failure/on_finish(..); }`; the stock ValidationBindHandler calls Validate::validate after each Object node (the shared validation engine from declarative-advice). All bind failures push into the one ErrorSink (collect-all) → Tier-2 BindError nodes carrying prefix + field path + raw value + matched raw key + OriginId + candidates.

**Resolved open questions:** Target self-description: a leaf-owned #[derive(BindTarget)] emitting a const NodeSchema + a cursor-calling bind fn (R1) — reflection-free, lets config-metadata co-emit from the same token input, keeps macros thin; serde-bridge is an opt-in alternate, not the canonical path.; Shared handler/discovery substrate: BindHandler IS property-resolution's ResolveHandler shape (one observer family, charter unification); custom handlers, when discovered cross-crate, ride the existing component-discovery substrate — but the core binder binds a named concrete T at a known call site, so it is DCE-inert.; Bind-method model: both JAVA_BEAN (Default+settable-fields convention leaf defines) and VALUE_OBJECT (single non-injected constructor) supported; existing-value-forces-JavaBean honored, and an immutable type bound onto an existing value is a loud Tier-2 error rather than silent mutation.; Aggregated vs fail-first diagnostics: AGGREGATED — every bad field/coercion/constraint across a bind is collected into the one ErrorSink and reported together in the App<Wired> AssemblyReport (Tier 2), framework-wide.; Bracket-escape owner: relaxed-binding owns it; the binder consumes the parsed MapKey Segment.; Collection replacement across enumerable+non-enumerable: computed via the CPS adapter's contains_descendant_of tri-state (Present/Absent/Unknown) — enumerable sources answer Present/Absent cheaply from the RelaxedIndex; non-enumerable sources answer Unknown and fall to candidate probing; the highest-precedence Present source owns the whole collection.; Config-refresh atomic swap: out of v1; a re-bind produces a fresh owned value swapped via the registry epoch lane, not a hot-path lock.; Async pre-materialization: lives entirely in the upstream seal_environment fence; the binder walk is sync over already-materialized values.

### `config-properties`
ADOPT R3 — a @ConfigurationProperties type IS an ordinary bean whose Provider is a binder-backed factory — unifying it into the registry-substrate rather than a parallel CONFIG_PROPS pipeline. `#[config_properties(prefix="my.app", ignore_unknown=true, ignore_invalid=false)]` on a #[derive(BindTarget)] struct emits exactly one const Descriptor + one ProviderSeed (codegen-boundary), where Provider::provide is the pure-projection factory: it calls leaf_core::bind::<T>(env, &handler) against the cheap-clone Env and publishes Published::Shared(Arc::new(value)). The factory receives ONLY &Env + a BindHandler — NEVER a ResolveCtx — so the pure-projection 'discourage injecting other beans' rule is a STRUCTURAL guarantee (the type physically cannot autowire a collaborator). The const PREFIX/POLICY/BIND_METHOD live in a tiny ConfigProperties trait impl as const data. THREE enabling routes map onto the toolkit: (a) explicit — register_config_properties::<T>() / #[leaf::enable_config_properties(MyProps, ThirdPartyProps)] on the app root (the only route for third-party types leaf can't annotate and for generic instantiations; the universal escape hatch); (b) scan — the macro-emitted Descriptor in COMPONENTS IS the scan route, force-linked per crate with the triple anti-DCE defense (a force-linked-but-zero-config crate is a loud AntiDceError — high stakes since a missing config bean means default/empty config); (c) stereotype — a #[component] config type is registered JavaBean-only (the macro refuses ValueObject for this route, transcribing Spring's restriction, because the container instantiates it leaving no binder seam for constructor args). Binding+JSR-validation run at exactly ONE site — `App<Wired>::validate()` — per `[C2-config-locus]`: validate() PRE-MATERIALIZES each config-properties bean (invokes its pure-projection `Provider::provide` = `bind::<T>(&env,&handler)` then the stock `ValidationBindHandler` running `Validate::validate` after each Object node), all-or-nothing, aggregating every BindError/Convert/Violation across every config-properties target into the one AssemblyReport (Tier-2 BindError/Validation naming prefix+key+origin), and STORES the bound `Published::Shared(Arc)` into the bean's singleton OnceCell. refresh R5 then EXCLUDES these beans from the eager wave (eager-EXCLUDED-because-PREBOUND) and publishes the already-bound Arc — `Provider::provide` is NEVER re-run at R5, so `DB_POOL_SIZE=0` failing `range(min=1)` is a Tier-2 AssemblyReport node, NEVER an R5 cancel-cascade (RunState→Failed) fault. This resolves the cross-doc LOCUS contradiction (binding ran "at App<Resolve>→App<Wired>" vs config-beans-are-ordinary-beans-built-at-R5): the ONE site is validate(), and R5 is pre-bound-Arc-publication only. The two leniency flags map onto BindHandler decisions (ignore_unknown = on_start skip-unknown; ignore_invalid = on_failure recover-vs-rethrow), with config-properties owning the default policy + per-type override; the range/nested JSR checks are NOT downgradable by the global `StartupValidation` lever (they have no Tier-3 fallback — see the lever interaction below). Generic config types hard-error on the scan route (generics-can't-register), directing to route (a).

**Resolved open questions:** Opt-in + prefix-carry mechanism: R3 — the attribute macro emits a normal bean Descriptor + a binder-backed ProviderSeed, so config-properties unifies with the bean model (one creation seam, one validation pass, one error spine) rather than a parallel registry.; Three routes onto no-classpath Rust: explicit enable-list (route a, DCE-safe, third-party + generics), the COMPONENTS scan descriptor (route b, the existing discovery substrate with the high-stakes anti-DCE self-check), and JavaBean-only stereotype (route c, refusing ValueObject); constructor-binding restricted to (a)/(b) exactly as Spring.; Pure-projection without bean injection: STRUCTURAL — the binder-backed Provider receives only &Env + BindHandler, never a ResolveCtx, so a config-properties type cannot inject a sibling bean.; Leniency flag policy: config-properties owns the defaults (ignore_unknown=true, ignore_invalid=false) and the per-type override surface, lowered onto binder BindHandler on_start/on_failure decisions.; Bind-method choice surface: surfaced via type shape + an optional #[config(bind=value_object|java_bean)] override; execution delegated to the binder.; Derived bean name: <prefix>-<fully::qualified::Type> deterministic, ByName-resolvable in the name overlay.; Conditional gating: a #[conditional(..)]/#[config_properties_if] guard is a CondExpr on the same Descriptor evaluated in App<Resolve>, composing with the scan slice without double-registration.

### `config-metadata`
ADOPT R1 — proc-macro co-emission of a const metadata row alongside the binding artifacts — plus an R2 binary-crate rollup, exploiting the UNIQUE benign-DCE property. The same #[config_properties]/#[derive(BindTarget)] expansion that emits the bean Descriptor ALSO emits a const ConfigGroup into a sibling `#[linkme::distributed_slice] static CONFIG_METADATA: [ConfigGroup]` (a new sibling to COMPONENTS/CONDITIONS, joining the codegen-boundary substrate). `pub struct ConfigGroup{ prefix, type_name, description: Option<&'static str>, properties: &'static [Property] }`; `pub struct Property{ name: &'static str /*canonical kebab via relaxed-binding's rule*/, ty: &'static str, description: Option<&'static str> /*from /// → #[doc]*/, default: Option<&'static str>, deprecation: Option<Deprecation>, hints: &'static [Hint], origin: CodeSpan }`. Descriptions are read from #[doc] attributes the proc-macro already has (Javadoc analogue, stable); declaration provenance uses proc_macro::Span file/line (stable 1.88) interned via the upstream OriginStore::intern_code. A `leaf metadata` CLI / binary build.rs walks the CONFIG_METADATA slice at the FINAL BINARY (the only whole-graph vantage), merges a hand-authored leaf-config-metadata.toml override/ignore supplement (for un-inferable collection/map defaults, foreign-key descriptions, ignored keys), and serializes a canonical config-metadata.json with a `leaf metadata --check` CI gate. The cross-module case is served because each crate emits its own ConfigGroup locally. UNIQUE relaxation: a DCE-dropped ConfigGroup degrades only IDE/tooling, never app behavior — INVERTING config-properties' high-stakes DCE — so config-metadata needs NO force-link self-check (only the --check CI warning). The rust-type→metadata-type-string rendering and enum-variants→value-hints vocabulary is owned here and kept consistent with type-conversion's accepted grammar.

**Resolved open questions:** Metadata/codegen boundary: R1 proc-macro co-emission (single source of truth, atomic with binding, cleanest /// capture, zero runtime cost) + an R2 binary-crate rollup over the CONFIG_METADATA slice for the whole-app artifact — macro emits per-crate const data, binary aggregates.; Cross-module/v4 case: each crate emits its own ConfigGroup const locally; the binary rollup picks them all up from the linkme slice without seeing the source — truly foreign types use the hand-authored supplement.; Runtime-queryable vs compile-time-only: compile-time build artifact by default (Spring's actual intent, zero runtime cost); an optional cold-path runtime catalog read is explicitly out of v1 scope.; Artifact location + --check: a canonical config-metadata.json at the workspace root with a leaf metadata --check CI gate (sqlx-prepare analogue); per-type files are a future option.; Override/ignore channel: a checked-in leaf-config-metadata.toml at the app root, MERGED OVER the generated tree at the binary rollup, supplying collection/map defaults + foreign-key descriptions + an ignored list.; Rust-type→type-string + hint vocabulary: owned by config-metadata, kept consistent with type-conversion's accepted suffix/enum grammar (enum variants → value hints, Duration/DataSize → suffix hints).; Shared substrate: config-metadata rides a sibling linkme slice but, because its DCE miss is BENIGN (tooling-only), it RELAXES the force-link + zero-contribution self-check the other discovery features mandate — a cheaper substrate justified by the benign-DCE asymmetry.

### `value-injection`
ADOPT R1 — macro-lowered typed injection as a PointKind::Value row on the existing InjectionPlan — with R3's typed handle family as opt-in escape hatches. #[value("${server.port:8080}")] on a constructor param/field lowers (in leaf-macros) to a const ValueSpec of pre-split segments + a monomorphized call the bean's Provider::provide makes: `let port: u16 = ::leaf_core::value::resolve_into::<u16>(&env, &PLAN.points[i], &ctx)?;`. VALIDATE-TIME DRY-RUN (`[C2-config-locus]`): so a malformed coercion does not first surface at refresh R5 inside `Provider::provide` (Tier-3), `App<Wired>::validate()` runs a THROWAWAY @Value dry-run sub-pass — for every eager bean's `InjectionPlan`, for each `PointKind::Value` row, it calls the SAME monomorphized `resolve_into::<T>(&env, point, &ctx)` and DISCARDS the value, collecting any `Convert`/`UnresolvedValue` into the one Tier-2 AssemblyReport. The dry-run is a pure-projection over the sealed `Env` (the same `&Env`+`ConvertCtx` the real call uses, threaded with the resolved `StartupValidation` leniency), reads no bean container, and is monomorphized at the SAME static target type `T` the bean will use — so the dry-run and the real R5 coercion are provably the same code path (no second parser, no drift). The grammar SPLIT (Literal | Placeholder{key,default} | Expr{body}) happens once at macro time (so runtime only resolves, never re-parses); malformed ${...}/#{...} is a Tier-0 compile error at the attribute span; a field type lacking FromConfigValue is a Tier-0 compile error. The injection point is a PointKind::Value row on the SAME const InjectionPlan (injection-mechanics) — so @Value is literally step-2 on the one Selector spine, short-circuiting the by-type candidate search; no second BeanPostProcessor, no magic field write (constructor/initializer-shaped per the rust-native nudge). resolve_into orchestrates: canonicalize+walk the ${} via property-resolution (relaxed-binding canonicalizes the key), dispatch ${}-then-#{} via the extra-12 interpreter, then FromConfigValue::from_config_value at the statically-known target. Strict-vs-lenient is a per-site knob (#[value("...", lenient)]) over a STRICT framework default (resolving the error-model open question toward strict — an unresolved placeholder is a Tier-2 UnresolvedValue node, never a silently-injected literal). The R3 handle family is opt-in: Option<T> (absent=None), RequiredValue<T>/bare-T (absent=Tier-2 error), and LazyValue<T> (the value-level analogue of the deferral family LazyRef<T>, OnceCell-cached, resolves on first .get() — contractually sync-non-blocking, so a #{} expression's async inputs must be pre-materialized into Env before first touch). A bare-T field's strict/lenient default wins over the type encoding only when no per-site knob is set. **Resolved open questions:** Default mode: STRICT by default — an unresolved ${} is a Tier-2 UnresolvedValue node collected into the AssemblyReport (every unresolved @Value reported at once), with lenient an explicit per-site opt-in; ### `extra-8`

### `extra-12`
ADOPT R3 — a const segment-AST emitted by the macro + one shared interpret() ordering function — the hybrid that eliminates dual-implementation drift. The dispatcher owns ONE grammar in leaf-core: `enum Segment{ Literal(Cow<'static,str>), Placeholder{ key: Cow<'static,str>, default: Option<&'static [Segment]> }, Expr(Cow<'static,str>) }` and one `interpret(template: &[Segment], env: &dyn PropertyResolver, expr: Option<&dyn ExpressionEvaluator>, policy) -> Result<String, LeafError>` that encodes the two-phase law in exactly one place: PHASE 1 expands every ${} (and its default recursively) via property-resolution over the SealedStack into a scratch buffer (relaxed-binding canonicalizes each key; placeholder strict/lenient is property-resolution's), then PHASE 2 — only if template.has_expr — hands the intermediate to the expr subsystem's evaluator. Once phase 2 begins its output is NOT re-scanned for ${} (non-recursion across the seam, stated precisely). The #[value] macro parses the literal once at compile time, validates the grammar, and emits a const-promoted &'static [Segment] + the interpret call + the inline FromConfigValue convert (target type known); for non-literal runtime strings the SAME parser produces a Vec<Segment> fed to the SAME interpret (one parser, one interpreter — no drift). The expr engine is treated as a fully opaque, OPTIONAL, origin-agnostic Option<&dyn ExpressionEvaluator> (the expr-i18n-resources subsystem's sync-pure ValueExpr); when absent, a template whose has_expr is true yields ONE canonical Tier-2 error AND the macro emits a #[diagnostic::on_unimplemented] guard for the earliest-legible compile-time catch. The dispatcher has ZERO discovery surface (each @Value site emits its own self-contained pipeline → DCE-immune by construction); only the expr PROVIDER, if pluggable, rides the registry. config-properties never invokes phase 2 (inert data — sharpening the @Value-is-code/config-is-data line). Origin threads through: a phase-2 failure names that its ${host} input came from application.yaml:12 (never a blended origin).

**Resolved open questions:** Whether a #{} layer exists: YES, via the expr-i18n-resources subsystem's ValueExpr<T>/ExpressionEvaluator; the dispatcher treats it as an OPTIONAL Option<&dyn ExpressionEvaluator> so an absent engine is a well-explained error, not a silent literal.; Where the split lives: R3 const-AST + shared-interpreter — compile-time grammar validation + const AST for the literal case, one runtime ordering interpreter and one parser shared with the dynamic-string path, eliminating the dual-implementation drift of a pure compile-time (option A) or pure runtime (option B) design.; Strict default at the seam: the absent-#{}-engine case is a HARD Tier-2 error (with a Tier-0 compile guard where the literal is visible), diverging from Spring's silent pass-through, reconciled with the framework-wide strict @Value default.; Unified micro-string grammar: the Segment grammar is the ONE embedded-value grammar, shared with @PropertySource locations and banner text; cron strings/event conditions reuse the expr subsystem's evaluator, not this placeholder grammar.; ExpressionEvaluator unification: the dispatcher is ONE caller of the expr-i18n-resources shared evaluator and does not over-specify the EvalCx; @ConditionalOnExpression/cache-key/event-condition share that evaluator.; Async interaction: interpret() phase 1 is sync over the SealedStack; the expr closure is contractually sync-pure (the expr subsystem hard-errors on an async body), so dispatch never .awaits and re-entrant ${} inside a #{} result is bounded by a depth/visited guard shared with property-resolution.; ${a:#{b}} legality: a #{} body inside a ${} default is legal — phase-1 expands the default's literal text including the #{} substring untouched, and phase 2 then evaluates it; stated explicitly in the one interpret() contract.

### `extra-13`
ADOPT the inline escape-aware char-scanner driven by a borrowed, sealed PlaceholderSyntax — parameterizing property-resolution's existing scanner rather than adding a subsystem. `pub struct PlaceholderSyntax{ prefix: Cow<'static,str> /*"${"*/, suffix: Cow<'static,str> /*"}"*/, value_separator: Option<Cow<'static,str>> /*Some(":")*/, escape: Option<char> /*Some('\\')*/ }`. The Env holds ONE PlaceholderSyntax built once at seal (read from the stack itself, e.g. leaf.placeholder.escape, then frozen — never a runtime-mutable global; the mutate seam is on EnvBuilder, the read seam only ever sees the frozen syntax). The scanner walks left-to-right; at the escape char it checks whether the next chars are the prefix OR (inside a body) the separator OR the escape itself — if so it consumes the escape and emits the sequence LITERALLY (\${ → literal ${, not a placeholder; sub\://host → key sub://host, the first : not a separator); an escape before any other char is emitted verbatim (the escape only 'bites' the special tokens). Output is Cow<str> — Cow::Borrowed when no placeholder and no escape fired (the zero-alloc low-overhead fast path), allocating only on actual expansion/de-escaping. The separator split inside a body is 'first UNESCAPED separator'. De-escaping happens BEFORE the placeholder-open decision but is NOT re-applied to a resolved value's own bytes except via the normal recursion contract (a resolved value IS re-scanned with the same syntax — stated explicitly). Multi-char delimiters (%{...}) work via substring compare with no grammar recompile. This shares the extra-12 Segment grammar: extra-12's parser consumes the de-escaped, syntax-parameterized output (one grammar, the escape rules live in the scanner; the ${}-vs-#{} split lives in extra-12; both parameterized by the one PlaceholderSyntax). Strict-mode adds two diagnostic inputs — an escape immediately before end-of-input, and (optionally) an unrecognized escape — surfaced as Tier-2 LeafError nodes carrying the original text + offending column; lenient mode passes them through verbatim (Spring parity). The post-file-parse contract (config-data loaders may consume one backslash first) is a documented shared invariant with the upstream config-data.

**Resolved open questions:** Where escape/delimiter config lives + how de-escaping threads: a borrowed value-type PlaceholderSyntax passed into the existing hand-written char-scanner, de-escape fused into the one scan pass returning Cow (zero-alloc fast path) — NOT a separate tokenizer (rejected: forfeits the Cow path) and NOT a separate compiled-matcher subsystem.; Per-resolver grammar config without a global mutable knob: PlaceholderSyntax is built once from the stack at seal and frozen into EnvCore; the mutate seam is EnvBuilder-only (consumed at seal), the read seam sees only the immutable syntax — no runtime-mutable global.; Disable/reconfigure: escape=None disables escaping; value_separator=None disables default-splitting; prefix/suffix reconfigurable to %{...} via substring compare — all as PlaceholderSyntax config variants mirroring Spring's spring.placeholder.escapeCharacter.default disable + no-default mode.; De-escape boundary vs recursion: de-escape the literal \${ BEFORE the placeholder-open decision; a resolved value is re-scanned with the same syntax (escapes in resolved values are honored, Spring parity) — the recursion contract states this explicitly.; Source-level escaping interaction: documented shared invariant — the placeholder escape operates on the POST-file-parse string (config-data's doubled-backslash unescape runs first); a cross-crate documented contract, not a registered artifact.; Cross-crate: no registry/DCE surface — PlaceholderSyntax is a plain kernel value type a downstream crate constructs and installs on EnvBuilder via the public builder; fully hand-writable.; Strict-mode escape diagnostics: a trailing escape (and optionally an unrecognized escape) surface as Tier-2 LeafError nodes with the offending column; lenient passes through verbatim.

## Cross-feature interactions
- **environment-config (upstream)** — This subsystem is the consumer side of seal_environment: it reads the cheap-clone Env=Arc<EnvCore> and the SealedStack lock-free, builds relaxed RelaxedIndexes and the PlaceholderSyntax at/before seal, and implements the binder-facing ConfigurationPropertySource (CPS) adapter (Present/Absent/Unknown tri-state) over the SealedStack. random.* is a PropertySource placed by config-data at PrecedenceRung::Random. It never re-opens the IO fence; Env::get_as delegates to type-conversion's FromConfigValue.
- **expr-i18n-resources (upstream)** — The extra-12 dispatcher routes every #{} segment to the expr subsystem's sync-pure ValueExpr<T>/ExpressionEvaluator (treated as opaque, optional Option<&dyn>); the evaluate-then-convert chain hands the expr result to FromConfigValue. type-conversion's locale-sensitive converters and ConvertCtx read the same ambient Cx Holder<LocaleKey> that messages-i18n reads. @ConditionalOnExpression and cache-key SpEL share that one evaluator; this subsystem provides the dispatch+convert, expr owns evaluation.
- **declarative-advice (upstream)** — The binder's stock ValidationBindHandler calls declarative-advice's shared Validate::validate engine after each Object node (face 3 of validation), so config-binding validation and method-validation are the SAME engine, SAME Violation model, SAME message resolution. Bind+validation failures push into the one ErrorSink and surface as Tier-2 LeafError nodes.
- **injection-mechanics (toolkit)** — @Value is a PointKind::Value row on the existing const InjectionPlan resolved by the one Selector spine (step-2, short-circuiting by-type candidate search); the R3 LazyValue<T> handle is the value-level analogue of the LazyRef<T> deferral family routed through the one resolve(Strictness,Cardinality) driver. No new resolver, no second injection surface.
- **registry-substrate + codegen-boundary (toolkit)** — config-properties targets are ordinary beans (Descriptor+ProviderSeed in COMPONENTS, dense BeanId, ByName-resolvable derived name); config-metadata emits const ConfigGroup rows into a sibling CONFIG_METADATA linkme slice; the optional custom-converter registry rides CONVERTERS. config-properties' scan route inherits the high-stakes triple anti-DCE defense; config-metadata RELAXES it (benign DCE).
- **conditional-strategy (toolkit)** — OnProperty/OnBooleanProperty leaves read the sealed Env through relaxed-binding's uniform-form equivalence and type-conversion's bool coercion; config-properties conditional gating (#[conditional]) is a CondExpr on the same Descriptor evaluated in App<Resolve>. - **error-model (toolkit)** — UnresolvedValue/Convert/Bind/Validation/ConfigIo are existing closed ErrorKind arms; the one ErrorSink aggregates all bind/convert/validation failures (collect-all) into the App<Wired> AssemblyReport at Tier 2, with Origin=File{path,line} via the upstream OriginStore; malformed grammar + missing-converter are Tier-0; LazyValue.get()/random.* grammar miss are Tier-3. A BindError FailureAnalyzer ships.
- **relaxed-binding ↔ binder ↔ value-injection ↔ extra-12/13 (intra-subsystem)** — CanonicalName is the single key-identity owner consumed by the placeholder walk (canonicalize-before-lookup), the binder tree-descent, and config-metadata's advertised form; the bracket-escape is owned by relaxed-binding and merely consumed downstream. ## Remaining risks (→ Phase 4)
- The CPS adapter's contains_descendant_of Unknown state for non-enumerable sources means collection-REPLACEMENT (one source owns the whole List/Map) falls to candidate-probing for env/random; the exact bound and correctness across enumerable+non-enumerable for deeply-nested indexed collections (servers[0].host) needs property-test validation in Phase 4 to ensure the index path and probe path agree.
- The const NodeSchema self-reference for recursive config types (Vec<Self>) requires a fn-pointer/lazy indirection; the exact const-promotion shape on stable (const fn returning &'static NodeSchema) must be prototyped — risk it forces a non-thin macro emission or a runtime lazy-init wrinkle.
- Strict-by-default for @Value diverges from Spring's lenient default; RESOLVED by the unified strictness lever (`[C2-config-locus]` + SEAMS "Unified startup-validation strictness lever"): the migration-friendly per-app global toggle IS `leaf.main.startup-validation=lenient` → `BootstrapSettings.startup_validation = StartupValidation::Lenient`, read once at the head of `App<Wired>::validate()` and threaded into the @Value dry-run's `ConvertCtx` (`Lenient → Leniency::Lenient`) and the placeholder sub-pass; a per-site `#[value("…", lenient)]`/`Option<T>` still wins over the global default. A `StartupValidationLenientAnalyzer` (FAILURE_ANALYZERS) renders the Spring-migrant prose when a @Value/placeholder would have aborted under Strict. Residual: the cold-walk cost is measure-before-build; `Skip` requires a hash-matching `cargo leaf prepare` plan.
- The ConvertCtx unit-hint forwarding (#[config(unit="s")]) vs the FromConfigValue blanket FromStr impl can collide (a Duration field whose source omits a unit); the precedence between a type's own suffix grammar and an attribute unit-default must be pinned to avoid double-interpretation.
- RandomValueSource's interior-mutable RNG behind Cell/atomic must be verified Send+Sync and lock-free under concurrent placeholder resolution during wave-parallel eager instantiation; a contended atomic-PCG could serialize, though the cold path makes this low-severity.
- config-metadata's rust-type→type-string + enum-variant→hint vocabulary must stay byte-consistent with type-conversion's accepted grammar; a drift makes IDE autocompletion suggest values the converter rejects — needs a shared const vocabulary table or a generation-time cross-check.
- The expr subsystem's ExpressionEvaluator being optional means a workspace that links no evaluator gets a Tier-2 error for any #{} @Value; the leaf metadata/doctor tooling should report 'this @Value uses #{} but no expression engine is configured' at build to avoid a surprising startup failure.
- The leaf-config-metadata.toml merge precedence over generated metadata (override vs supplement vs ignore) and its per-crate-vs-app-root ownership is under-specified; a wrong precedence could silently mask a generated property or fail to document a foreign key.
