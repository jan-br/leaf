# ADR 10 — Conditional Strategy + Profiles + Auto-Config Back-off  `[conditional-strategy]`

> **Note (2026-06-14):** the optional *dynamic-WASM source / WASM-"classpath"* mentioned below is **DROPPED** — leaf autodiscovery is native link-time registration only. Ignore WASM/dynamic-module references here; they are struck from the active design.


## Decision
ADOPT a SINGLE typed condition algebra — `CondExpr` (a const-constructible `All/Any/Not/Leaf` tree) over a stable `ConditionId` kernel ABI — as the ONE gating primitive every feature (conditions SPI, condition-family, profiles, auto-config back-off, programmatic-registration guards, component-scan filters) emits and the container evaluates. The "blend" the charter foresees is realized as a PER-KIND TIER-MAP keyed on `ConditionKind` (frozen in `leaf-core`), the load-bearing kernel ABI that decides, per condition kind, the EARLIEST sound tier — exactly the three tiers `compile-runtime-split` already named. THREE concrete pieces this ADR owns and the two upstream ADRs deferred to it:

(1) THE TIER-MAP (the kernel ABI). Every `ConditionKind` declares a `const TIER: EarliestTier` in `leaf-core`. The mapping is FIXED, not per-author:
  • `Cfg` tier (pruned at COMPILE): `OnClass`/`OnMissingClass` (= cargo capability feature presence), `OnCapability`. The macro emits the guarded `Descriptor` inside `#[cfg(feature="…")]` so an absent optional dependency is physically removed — the rust-native "read without loading". This MUST pair with force-link (cross-crate-discovery's Defense 0): a cfg-true leaf both compiles AND auto-force-links its crate, else the gated-in row silently vanishes.
  • `ConstFold` tier (decided at BUILD → arrives as `CondExpr::Const(bool)`): `OnRustVersion` (OnJava analogue), target-cfg-decidable `OnCloudPlatform`/`OnThreading` (platform-by-target), `OnWarDeployment`. `leaf-codegen`/`build.rs` folds these to a `const MATCHED: bool` leaf.
  • `Runtime` tier (evaluated in `App<Resolve>`, the MANDATORY FLOOR): `OnBean`/`OnMissingBean`/`OnSingleCandidate` (REGISTER sub-phase), `OnProperty`/`OnBooleanProperty`/`OnExpression`/`OnResource`/`OnProfile`/runtime-`OnCloudPlatform` (PARSE sub-phase), AND every dynamic/WASM-contributed definition's guard (no cfg, no Rust type → gateable only here). cfg/const are accelerator fast-paths over an always-present runtime evaluator.

(2) THE `@ConditionalOnMissingBean` ANALOGUE = a `DefinitionProbe` over the FROZEN-CANDIDATE-SET-SO-FAR, NOT a bean-construction. Back-off is a definition-level no-instantiation resolve: `OnMissingBean(ty)` matches iff `candidates(ty)` would NOT resolve to a unique winner (delegating to autowiring's no-instantiation candidate-resolver per area-coherence). It runs in the REGISTER sub-pass of `App<Resolve>` (compile-runtime-split's slot), AFTER all user defs + AFTER the auto-config batch's earlier candidates have incrementally registered, so each auto-config sees a growing-then-final definition set exactly as Spring's definition-order semantics require. Precedence is cross-crate-discovery's ladder, enforced in ONE ordered cold pass: `exclude(ContractId)` > `user-bean back-off (OnMissingBean)` > `auto-config default`. An excluded `ContractId` mints no BeanId and NEVER enters the back-off batch.

(3) PROFILES = the most-ergonomically-named `CondExpr` preset, ALWAYS Runtime-tier. `@Profile("…")` lowers to `CondExpr::Leaf(ON_PROFILE, expr_attrs)` reading `env.active_profiles()`; it is NOT a parallel engine. The `!`/`&`/`|` grammar (parens-mandatory on mixed `&`/`|`, implicit-OR arrays, negation-matches-absence) and the three levers (`active`/`include`/`group`) + reserved `default` fallback live in a thin `profiles` preset crate that emits a canned `CondExpr`; the SPI evaluates it. Profiles-always-runtime is the stated coherence constraint (a profile can be flipped by env/dynamic source, never const).

Authoring is ONE uniform surface regardless of tier: `#[conditional(all(on_class("redis"), any(on_property("x"), on_missing_bean(Foo))))]` emits a const `CondExpr` tree; the framework infers `phase()`/`tier()` = max over leaves (an `OnBean` nested anywhere auto-defers the whole guard to the REGISTER sub-pass — killing Spring's #1 forgot-to-declare-REGISTER bug). Boolean composition is first-class syntax (`all/any/not`), not subclassed helper conditions. A miss is a GRACEFUL recorded no-bean (never an error); the silent-now/loud-later bridge is realized by the condition report (frozen into the FrozenContainer at seal()) which a later `App<Wired>` NoSuchBean enriches from ("backed off because…" / "feature X not enabled").

## Rationale
- The blend is FORCED, not chosen: compile-runtime-split already established that the Runtime tier is the mandatory floor (owner-decision #1 maximal-magic discovery + the dynamic-WASM lane contribute definitions with NO cfg and NO Rust type, gateable only at runtime), with cfg/const as accelerators. This ADR honors that verbatim and supplies the missing concrete artifact it deferred: the per-kind tier-map as a frozen leaf-core ABI. condition-family's Phase-1 design explicitly named this tier-map 'a load-bearing kernel ABI' and the area-coherence note flagged that three features assume they can read tier-per-element but none mints it — this ADR is where it is minted and frozen.
- A SINGLE CondExpr algebra (R3 of the conditions SPI) is the unification answer (§2.11): conditions, condition-family, profiles, programmatic-registration guards, and component-scan filters all emit ONE const tree the container interprets with ONE eval path producing ONE ConditionOutcome stream. Boolean composition becomes first-class syntax (strictly better DX than Spring's AnyNestedCondition/AllNestedConditions subclass boilerplate, §2.1), and framework-inferred phase (max over leaves) structurally eliminates Spring's most common authoring bug — a §2.9-spirit 'prevent the error' win over 'diagnose it late'. Phase 1 explicitly recommended exactly this hybrid: R3's algebra as the surface, R2's cfg-gate for OnClass, R1's runtime planner as the mandatory floor.
- Back-off correctness (owner-decision #2) is intrinsically a RUNTIME definition-level fact: OnMissingBean must observe the FINAL user-bean set, which only exists after user-def registration in App<Resolve>. Realizing it as a DefinitionProbe (no-instantiation resolve over the candidate set, delegating to autowiring's candidate-resolver) — not a bean-construction and not a naive 'count beans' — is what makes OnSingleCandidate ('would by-type autowire resolve unambiguously, incl. @Primary') correct, and is exactly the no-instantiation definition-level resolve mode compile-runtime-split FORCES on autowiring. This is Spring's DeferralForCorrectness intent realized as a lifecycle sub-pass, not a perf trick.
- OnClass-as-cfg is the only honest rust-native 'read without loading': a linked Rust type is ALWAYS present, so the real signal is capability-feature presence, which physically removes the absent optional dependency from compilation — a cleaner guarantee than Spring's ASM trick (naming a missing capability can never build-error). This is condition-family's R2 tier-CFG and the cross-crate doc's 'enabling a feature = adding a starter'. It MUST pair with cross-crate-discovery's force-link (Defense 0), because a feature gates COMPILATION not LINKAGE — this ADR binds the cfg-true verdict to the auto-force-link the discovery ADR owns, keeping the two consistent so a 'condition true' default can never silently vanish.
- Profiles-as-a-CondExpr-preset (not a parallel engine) is Spring's own conceptual economy (profiles predate the SPI; Spring retrofitted them onto @Conditional) and the Phase-1 profiles design's load-bearing intent. Keeping profiles ALWAYS runtime-tier — even when sibling conditions are static — is the coherence constraint compile-runtime-split stated: a profile is env/dynamic-overlaid and can never be a const fact. The asymmetry (most conditions can be cfg/const, profiles never) is made coherent precisely because profiles are just one Leaf kind whose TIER const is fixed to Runtime in the same tier-map.
- The exclusion>back-off>default precedence and ContractId identity are honored from cross-crate-discovery unchanged: exclusions key on the stable ContractId (TypeId is not cross-build/cross-crate stable, and a config-file leaf.autoconfigure.exclude string + a foreign-toolchain WASM module agree only on ContractId), and an excluded ContractId mints no BeanId so it never participates in back-off. This ADR adds only the SEQUENCING (exclusion check before the back-off batch in the single App<Resolve> cold pass) — it does not re-key or re-store anything the discovery ADR owns.
- Thin macros (§2.10) and one accountable account (§6) are preserved by owning ONLY the gating semantics: the macro emits one const CondExpr tree + ConditionId leaves (hand-writable, cargo-expand-legible); ALL predicate logic lives in normal testable crates (leaf-conditions for the family, leaf-boot for the engine); the single eval path threads ConditionOutcome into the ONE condition report compile-runtime-split freezes into FrozenContainer at seal(). condition-not-met (silent, recorded) stays crisply distinct from the App<Wired> AssemblyReport's wiring-broken (loud, aggregated) — the silent-now/loud-later time-separation Spring depends on and §6 demands.

## Options considered & rejected
- **Opaque Fn/trait-object Condition (Spring-faithful R1: each condition is a boxed `dyn Condition` with an author-declared Phase, evaluated as a flat AND-fold)** — rejected: Reproduces Spring's #1 authoring footgun (forgetting to declare REGISTER phase silently evaluates a bean-introspecting condition against a half-built world) and forecloses framework phase-inference, because an opaque closure's phase cannot be derived. It also makes boolean composition require subclassed AnyNestedCondition/AllNestedConditions helpers (the exact boilerplate §2.1 should beat), and an opaque Fn cannot const-fold its cheap leaves — so the R1↔R2 static/runtime seam (the whole point of the blend) is impossible. Kept ONLY as the leaf evaluator behind a ConditionId (a typed leaf resolves to a runtime predicate), never as the authoring/composition surface.
- **Push the whole conditional strategy to cfg/build-time (R2 alone): every condition is a feature/cfg gate or a build const-fold, no runtime evaluator** — rejected: Intrinsically PARTIAL and contradicts compile-runtime-split's mandatory-runtime-floor: OnBean/OnMissingBean need the live definition set (a runtime fact), profiles are env-overlaid (always runtime), and owner-decision #1's maximal-magic + the dynamic-WASM lane contribute definitions with no cfg and no Rust type that can ONLY be gated at runtime. A build.rs also sees only its own crate, so it cannot answer 'is bean X defined in some other crate'. Amputating the order-sensitive half of Spring's two-phase model would break back-off entirely. cfg/const are retained strictly as accelerator fast-paths over the always-present runtime evaluator.
- **Build-time-frozen condition+autoconfig plan as the DEFAULT (R3 of auto-configuration): leaf-codegen aggregates manifests via cargo metadata, evaluates all build-decidable conditions, emits a frozen ordered plan; runtime only does OnMissingBean residual** — rejected: compile-runtime-split already settled this: the build-frozen plan is the OPT-IN R3 accelerator (cargo leaf prepare) for determinism/wasm/native-image, NOT the default. It cannot be the default because (a) OnMissingBean back-off needs the final RUNTIME user-bean set, (b) the cross-crate single-crate wall means the binary's build can't enumerate foreign rows without an unenforced manifest convention, and (c) a dynamic-WASM-contributed auto-config has no build presence. So a runtime residual MUST exist regardless; making build-time the default merely adds a build-frozen/runtime-residual confusion seam ('why didn't my plan change when I set the exclude property?'). Honored as the opt-in accelerator whose residual lands in the SAME App<Resolve> slot.
- **Realize @ConditionalOnMissingBean by eager-instantiating and checking the live singleton cache (Spring's literal 'consult the bean factory') ** — rejected: Construction-during-condition-evaluation violates the seal() boundary (no async/IO/construction before freeze, compile-runtime-split) and would make a pure gating decision depend on constructor side-effects and ordering. OnMissingBean is a DEFINITION question ('would a unique provider win?'), not a VALUE question ('is the singleton built?'). A DefinitionProbe over the frozen-candidate-set-so-far (no instantiation, delegating to the candidate-resolver) is strictly earlier, side-effect-free, and is exactly what OnSingleCandidate needs (it counts resolvable winners incl. @Primary, not built instances). It also keeps back-off origin-agnostic (a WASM-contributed candidate participates without being constructed).
- **Author-declared condition Phase (Spring-faithful) instead of framework-inferred phase()=max(leaf tiers)** — rejected: Author-declared phase is the documented #1 Spring authoring mistake (a bean-introspecting condition that forgets REGISTER silently answers against a half-built world — a silent correctness bug). Because leaf models conditions as typed CondExpr data (not opaque closures), the framework can INFER the earliest-sound phase from the leaves' fixed tier-map entries, making the bug structurally impossible. The only cost — an author who nests an OnBean inside a 'cheap parse prune' silently makes the whole guard late — is mitigated by surfacing the inferred tier in the condition report (turning a wrong-answer bug into a visible, explainable deferral).
- **Model OnClass/OnMissingClass as a runtime registry probe over the descriptor table (keep it uniform with OnBean at the Runtime tier)** — rejected: Dishonest in Rust: a linked type is always 'present', so a runtime probe could only test a capability/cfg fact already known at compile time — paying a cold-path cost to re-discover a build fact, and unable to make 'name an absent optional dependency' safe (the type either compiles or it doesn't). The cfg tier physically removes the absent type, the cleanest realization of the intent, and it doubles as the rust-native starter mechanism (capability feature = the dependency is present). The residual runtime sub-case (a WASM-contributed def with no cfg) is gated at the Runtime tier as an OnCapability-string check, not via a phantom OnClass.

## Consequences
- FORCES a stable, minimal `leaf-core` kernel ABI: `ConditionId`, `ConditionKind`, the `const TIER: EarliestTier` per-kind tier-map, the `CondExpr` enum (All/Any/Not/Leaf/Const), `ConditionOutcome`, and the `Condition` SPI trait. Emitted const trees hard-code absolute paths to these, so the CondExpr enum is a load-bearing ABI that must be ultra-stable and minimal — evolving it (e.g. adding an xor/count combinator) is a kernel-ABI change. The tier-map is the SINGLE authoritative artifact condition-report and auto-configuration both read per element (resolving the area-coherence 'three features assume a tier-map but none mints it' gap).
- FORCES autowiring/candidate-resolution to expose a NO-INSTANTIATION, definition-level resolve mode: `DefinitionProbe` powering OnBean/OnMissingBean/OnSingleCandidate must run the candidate-resolver's primary/fallback/qualifier policy over the FROZEN candidate-set-so-far without constructing, and return 'would-resolve-unique?'. This is the same primitive compile-runtime-split's App<Wired> whole-graph validation needs and the area-coherence note's OnSingleCandidate delegation requires — one definition-only candidate-resolver, not two.
- FORCES the runtime condition tier into compile-runtime-split's App<Resolve> as two ordered sub-passes: (1) a PARSE sub-pass evaluating order-insensitive leaves (OnProperty/OnProfile/OnResource/runtime-OnCloudPlatform) over the sealed Environment, then (2) a REGISTER sub-pass — after ALL user defs are collected and run INSIDE the auto-config batch incrementally — evaluating OnBean-family leaves so each candidate sees the growing set. The Environment MUST be sealed (compile-runtime-split's 5f async fence ends at seal_environment) BEFORE the PARSE sub-pass, fixing the environment/binder sealing order.
- FORCES the cfg tier to bind to cross-crate-discovery's force-link: a `Cfg`-tier leaf (OnClass/OnCapability) emits the guarded Descriptor under `#[cfg(feature=…)]` AND that feature must auto-add its crate to the force-link participating set + the ExpectedManifest. The condition-report's 'feature X not enabled' label requires the resolved-feature/cfg snapshot (register §6) to trustworthily distinguish 'compiled out by cfg' from 'DCE-vanished' — so a cfg-gated-out element is a distinct report class (CompiledOutByCfg) from a DCE'd one (AntiDceError) and from a runtime no-match (Negative).
- FORCES the condition report to carry THREE+ verdict classes reconciled across tiers into ONE account frozen into FrozenContainer at seal(): Positive / Negative(runtime no-match, with reason) / Exclusion(ContractId) / Unconditional / CompiledOutByCfg(feature absent) / BuildFoldedFalse(const-fold). The App<Wired> AssemblyReport's NoSuchBean enriches by querying the frozen report by the missing ContractId/BeanId ('candidate backed off because OnMissingBean found user Foo' / 'feature redis not enabled'). condition-not-met (silent, recorded) and wiring-broken (loud, aggregated) stay time-separated (§6).
- FORCES the auto-config engine to be the thin App<Resolve> orchestrator compile-runtime-split placed: it applies the ordered precedence ladder (exclude > OnMissingBean back-off > default) in ONE cold synchronous pass, delegating ordering to cross-crate-discovery's SequenceKey + auto-config-ordering's topo-sort and gating to this ADR's CondExpr eval. The ordered batch fixes DEFINITION order ONLY and is a discarded App<Resolve> artifact; the App<Running> instantiation scheduler (dependency-driven WiringPlan waves) cannot read it (doc-1234 boundary structurally enforced by compile-runtime-split).
- FORCES profiles to own only the expression grammar + activation levers as a thin preset emitting `CondExpr::Leaf(ON_PROFILE, …)`, ALWAYS Runtime-tier, reading env.active_profiles() at PARSE-sub-pass time. The `!`/`&`/`|` grammar (parens-mandatory on mixed, implicit-OR arrays, negation-matches-absence) and active/include/group + reserved `default` fallback are profile-local; ActiveProfiles must be sealed into the Environment before the PARSE sub-pass. No bespoke profile registration path exists — profiles ARE conditions.
- ACCEPTS a bounded one-shot COLD cost in App<Resolve>: evaluate residual runtime CondExpr trees (cfg leaves already pruned, const leaves already decided) + run the incremental OnMissingBean back-off batch over potentially hundreds of auto-config candidates. The cfg-tier compile pruning is the rust-native analogue of Boot's build-time spring-autoconfigure-metadata pre-filter — most absent-capability candidates are gone at compile time, so the runtime set is small. The opt-in cargo leaf prepare plan can pre-decide the build-decidable half for hostile targets. Zero hot-path cost (the FrozenContainer is read lock-free at steady state).
- ACCEPTS that 'compile-time conditional errors' are deliberately SCOPED to locally-legible cases: a malformed CondExpr, an unknown cfg feature, a statically-contradictory cfg(all(a,not(a))), and a generic guarded element (no 'static symbol) are compile errors. A backed-off auto-config, a runtime no-match, and a cross-bean dependency a dropped condition would have supplied are NOT compile errors — they are the runtime report (silent) and the App<Wired> startup pass (loud). leaf must document that the conditional strategy delivers FAST+EARLY+RICH gating diagnostics, not a fully compile-checked condition graph (impossible under maximal-magic + dynamic sources + runtime profiles/properties).

## API / type sketch

// ═══ leaf-core: the ONE gating ABI (consumed by conditions, condition-family, profiles,
//     auto-config back-off, programmatic-registration guards, component-scan filters) ═══
//     Built on compile-runtime-split's tier-router + cross-crate-discovery's ContractId/SequenceKey.

// ── THE TIER-MAP: the load-bearing kernel ABI (frozen here, read by report + auto-config) ──
#[derive(Clone, Copy, PartialEq, Eq)]
pub enum EarliestTier { Cfg, ConstFold, Runtime }     // Runtime is the MANDATORY floor
#[derive(Clone, Copy, PartialEq, Eq)]
pub enum SubPhase     { Parse, Register }              // Register = order-sensitive (OnBean family)
pub struct ConditionId(pub u32);                       // stable id; resolves to a Condition OR a const fact
pub trait ConditionKind {                              // each family member implements this as data
    const ID:   ConditionId;
    const TIER: EarliestTier;                          // OnClass=Cfg, OnRustVersion=ConstFold, OnBean=Runtime
    const SUB:  SubPhase;                              // OnBean/OnMissingBean=Register, else Parse
}

// ── THE CONDITION ALGEBRA: one const-constructible tree the macro emits (thin-macro, §2.10) ──
pub enum CondExpr {
    Leaf(ConditionId, AttrSlice),                      // parameterized: reads the triggering annotation's attrs
    All(&'static [CondExpr]),                           // stacked conditions = All (the AND default)
    Any(&'static [CondExpr]),  Not(&'static CondExpr),  // first-class boolean syntax (no subclass helpers)
    Const(bool),                                        // a ConstFold-tier leaf already decided at build
}
impl CondExpr {
    pub const fn tier(&self)  -> EarliestTier;          // = max over leaves; Cfg leaves are ABSENT (pruned)
    pub const fn phase(&self) -> SubPhase;              // = max over leaves; any OnBean ⇒ whole guard Register
}

// ── THE RUNTIME SPI (the mandatory floor; cfg/const are accelerators over it) ──
pub trait Condition: Send + Sync {                     // synchronous, side-effect-free read (no .await, no IO)
    fn matches(&self, ctx: &ConditionCtx<'_>, attrs: &AttrSlice) -> ConditionOutcome;
}
pub struct ConditionCtx<'a> {                           // read-mostly snapshot, NO construction
    pub env:    &'a SealedEnv,                          // sealed BEFORE the Parse sub-pass (5f fence done)
    pub defs:   &'a DefinitionView<'a>,                 // frozen-candidate-set-so-far (for the Probe)
    pub probe:  &'a DefinitionProbe<'a>,                // the OnMissingBean engine (below)
    pub caps:   &'a CapabilitySnapshot,                 // resolved cfg/feature set ⇒ trustworthy "feature X off"
}
pub struct ConditionOutcome { pub matched: bool, pub reason: ReasonMsg } // reason IS the report content

// ── THE @ConditionalOnMissingBean ANALOGUE: a no-instantiation DEFINITION probe ──
pub struct DefinitionProbe<'a>(&'a DefinitionView<'a>);
impl<'a> DefinitionProbe<'a> {
    pub fn would_resolve_unique(&self, ty: TypeId) -> Resolvability; // delegates to candidate-resolver, NO build
}
pub enum Resolvability { None, Unique(BeanId), Ambiguous(u16) }      // OnMissingBean ⇐ !Unique; OnSingleCandidate ⇐ Unique
// Built-ins (leaf-conditions crate; each is a ConditionKind + a thin attr macro):
//   on_missing_bean(T) = matches iff probe.would_resolve_unique(T) != Unique  (REGISTER)
//   on_bean(T)         = matches iff probe.would_resolve_unique(T) == Unique  (REGISTER)
//   on_single_candidate(T)                                                    (REGISTER)
//   on_property/on_boolean_property/on_resource/on_expression                 (PARSE,  Runtime tier)
//   on_class(feat)/on_capability(feat)  ⇒ #[cfg(feature=feat)] + force-link   (Cfg  tier, compile-pruned)
//   on_rust_version(range)/on_cloud_platform(target)/on_threading(target)     (ConstFold tier)

// ── PROFILES: the ergonomic preset (ALWAYS Runtime tier) — NOT a parallel engine ──
pub const ON_PROFILE: ConditionId = ConditionId(/*…*/);
// #[profile("prod & (eu | us)")] ⇒ CondExpr::Leaf(ON_PROFILE, parse_profile_expr!("prod & (eu | us)"))
// grammar: ! & | ; mixed &/| REQUIRE parens; ["a","b"] = implicit OR; !p matches ABSENCE.
// levers: active (ops) / include (per-source, not list-merged) / group (fan-out); reserved `default` fallback.

// ── THE ENGINE PLACEMENT (compile-runtime-split's App<Resolve> slot; ONE ordered cold pass) ──
//   route_conditions(): Parse sub-pass over sealed env  →  evaluate residual runtime trees
//   run_autoconfig():   apply EXCLUSIONS (ContractId, cross-crate-discovery) →
//                       for each candidate in SequenceKey order:        // exclude > back-off > default
//                         if exclude.contains(c.contract) { continue }  // mints NO BeanId
//                         if eval(&c.guard, ctx).matched { defs.register(c); } // Register sub-pass sees growing set
//                         else { report.record(Negative, c, outcome.reason); }
//   ⇒ condition report frozen INTO FrozenContainer at seal(); App<Wired> NoSuchBean enriches from it.

pub enum ConditionReportClass {                         // reconciled across tiers into ONE account (§6)
    Positive, Negative(ReasonMsg), Exclusion(ContractId),
    Unconditional, CompiledOutByCfg(SourceTag), BuildFoldedFalse(ConditionId),
}

## Forces on other concepts
- **Compile-vs-Runtime Resolution Split (upstream compile-runtime-split ADR)**: Honored and completed: this ADR supplies the concrete CondExpr enum + ConditionId + per-kind tier-map that that ADR's condition tier-router executes, the DefinitionProbe its OnMissingBean back-off needs, and the report classes its App<Wired> NoSuchBean enriches from. The runtime tier evaluates in its App<Resolve> Parse-then-Register sub-passes; the auto-config back-off batch runs in its slot AFTER user defs / BEFORE seal(); phase()/tier()=max-over-leaves is its framework-inferred CondExpr.phase(). Adds NO new lifecycle phase — only the gating semantics inside the existing timeline.
- **Cross-Crate Discovery + Anti-DCE + Exclusions (upstream cross-crate-discovery ADR)**: Honored unchanged: exclusions key on its ContractId (never TypeId); the precedence ladder exclude>back-off>default is its ladder, sequenced by this ADR; the Cfg-tier OnClass/OnCapability leaf MUST pair with its force-link (Defense 0) and feed its ExpectedManifest, because a feature gates compilation not linkage; the CapabilitySnapshot for 'feature X not enabled' reuses its resolved-feature set; condition ordering/tie-break reads its SequenceKey (epoch,ContractId), never link order. A cfg-gated-out element is a distinct report class from its AntiDceError (SourceVanished).
- **Autowiring / Candidate Resolution & Collection Injection**: MUST expose a no-instantiation, definition-level resolve mode (DefinitionProbe.would_resolve_unique) running the primary/fallback/qualifier candidate-resolver policy over the frozen candidate-set WITHOUT constructing — powering OnBean/OnMissingBean/OnSingleCandidate AND (per area-coherence) the App<Wired> whole-graph validation. OnSingleCandidate is 'would by-type autowire resolve unambiguously incl. @Primary', NOT 'exactly one bean'. Excluded ContractIds are absent from candidate sets; collection-injection expected-count self-checks subtract excluded/conditioned-out candidates.
- **Condition Evaluation Report (condition-report feature)**: The report is a passive sink the single eval path writes ConditionOutcome rows into, frozen INTO compile-runtime-split's FrozenContainer at seal(). It MUST carry the cross-tier verdict classes (Positive/Negative/Exclusion/Unconditional/CompiledOutByCfg/BuildFoldedFalse) reconciling compile-gated-out + build-folded + runtime-no-match into ONE account — a problem Spring never has. It CONSUMES the substrate anti-DCE self-check (does not reinvent it) and feeds the §6 silent-now/loud-later bridge: a later NoSuchBean queries the frozen report by ContractId/BeanId.
- **Profiles (profiles feature)**: Profiles are a CondExpr preset (ON_PROFILE Leaf), ALWAYS Runtime-tier — never cfg/const, even when sibling conditions are static (the coherence constraint). profiles owns only the !/&/| grammar (parens-mandatory on mixed, implicit-OR arrays, negation-matches-absence) and the active/include/group levers + reserved `default` fallback. ActiveProfiles MUST be sealed into the Environment before the Parse sub-pass; include is per-source (not relaxed-binding list-merged); group fans out. No bespoke profile registration path.
- **Environment / Binder / Property Sources**: The SealedEnv MUST be fully materialized (compile-runtime-split's 5f async pre-materialization fence completed at seal_environment) BEFORE the Parse sub-pass, since OnProperty/OnProfile/OnExpression read it synchronously. The kill-switch (leaf.enable-autoconfiguration=false) and the runtime exclusion source (leaf.autoconfigure.exclude, ContractId-string, relaxed-binding + list) are read from the sealed env in App<Resolve>. Conditions are synchronous side-effect-free reads — no condition may .await or do IO (any such work belongs to a bean Factory post-fence).
- **Component Scanning Filters & Programmatic Registration Guards**: Both consume the SAME CondExpr primitive (unification §2.11): a component-scan include/exclude filter and a programmatic Registrar's conditional branch emit/evaluate the identical CondExpr over the identical ConditionCtx — there is no separate gating mechanism. A Registrar's conditional outcomes are accountable to the same condition report (§6). cfg-tier scan filters pair with force-link identically to OnClass.
- **WASM / Dynamic Second-Source Layer**: A dynamic/WASM-contributed definition has no cfg and no Rust type, so its guard is gateable ONLY at the Runtime tier (the floor) — evaluated in the reopen_epoch's App<Define>'→re-seal() Parse/Register sub-passes against that epoch's frozen-candidate-set. OnClass/OnCapability for a dynamic def degrades to a runtime OnCapability-string check (no phantom compile-time OnClass). Dynamic candidates back off and order indistinguishably from native (origin-agnostic), keyed on ContractId, sorted after static rows via SequenceKey epoch.
- **Unified Error / Diagnostics Model (register §6)**: MUST keep condition-not-met (silent no-bean now, a recorded report Negative) and wiring-broken (loud App<Wired> AssemblyReport NoSuchBean later) DISTINCT and time-separated, while the latter enriches from the former. Errors split by timing: compile (malformed CondExpr / unknown feature / contradictory cfg / generic guarded element) → build-or-seal (cargo leaf prepare --check staleness, ContractId exclusion typo via cross-crate-discovery's ExclusionError::NoSuchTarget) → App<Resolve> runtime no-match (silent, recorded) → App<Wired> startup wiring (loud, aggregated).

## Open sub-questions (→ Phase 3)
