# ADR 11 — Auto-Configuration Model + Starters + BOM + Distribution  `[autoconfig-starters-bom]`

> **Note (2026-06-14):** the optional *dynamic-WASM source / WASM-"classpath"* mentioned below is **DROPPED** — leaf autodiscovery is native link-time registration only. Ignore WASM/dynamic-module references here; they are struck from the active design.


## Decision
ADOPT a THREE-LAYER distribution-and-activation model that turns the FIXED owner decisions (Boot-style opt-in auto-config #2; starters + BOM #3) into concrete, named Rust artifacts on top of the two upstream substrate ADRs (cross-crate-discovery and conditional-strategy), owning FOUR things those ADRs deferred:

(1) THE AUTO-CONFIG ENGINE = the codegen-boundary ADR's R1 runtime deferred-batch orchestrator, made concrete as a thin `leaf-boot` function `run_autoconfig(&AssemblyCtx) -> Result<(), AssemblyError>`. An auto-config module is `#[leaf::auto_config]` (the LITE-ONLY config marker; macro LINTS self-calls between `#[bean]` methods) emitting an `AutoConfigDescriptor` row into a DEDICATED `#[distributed_slice(::leaf_core::AUTO_CONFIGS)]` slice — a SECOND discovery channel beside `COMPONENTS`, force-linked + self-checked + SequenceKey-ordered + ContractId-keyed by the IDENTICAL cross-crate-discovery machinery (no bespoke registry). The engine runs in ONE cold synchronous pass in `App<Resolve>` AFTER all user defs + AFTER the user-config eager batch, in the precedence ladder `exclude(ContractId) > user-bean back-off(OnMissingBean) > auto-config default`, registering each surviving candidate's seeds INCREMENTALLY so later candidates' `OnMissingBean`/`OnSingleCandidate` see the growing-then-final definition set (Spring's definition-order-for-correctness, realized via conditional-strategy's DefinitionProbe). Back-off, ordering, and exclusion are NOT reinvented — they are conditional-strategy's `CondExpr`/`DefinitionProbe`, auto-config-ordering's topo-sort, and cross-crate-discovery's `ExclusionSet`, respectively. The engine OWNS only: the AUTO_CONFIGS channel ABI, the LITE-ONLY mandate + self-call lint, the `@Fallback`-role default registration (an auto-config bean is registered as a FALLBACK candidate so a user bean transparently supersedes even when both are present — softer than all-or-nothing back-off), and the kill-switch `leaf.enable-autoconfiguration=false`.

(2) OPT-IN ACTIVATION = TWO orthogonal gates, NEVER an arbitrary transitive dep (the owner-decision-#2 / cross-crate-discovery predictability posture). A `leaf-redis::RedisAutoConfig` PARTICIPATES (its descriptor is compiled, force-linked, self-checked, and enters the candidate batch) iff its crate is in the force-link participating set = `dep:`-hidden capability feature ENABLED ∪ explicit `scan!` list. It WIRES iff, additionally, its runtime `CondExpr` guard matches (back-off + OnProperty + OnProfile). So "add the starter feature" = compile + force-link + candidate-in-batch; "the default actually appears" = guard matched AND not excluded AND not overridden. A plain transitive dependency that is NOT an enabled capability feature and NOT in `scan!` is NEVER force-linked and so NEVER injects beans — the implicit-global concern is structurally answered.

(3) STARTERS = `leaf-starter-*` AGGREGATOR CRATES (the Spring-Boot-starter analogue). A starter crate is an (almost) code-free crate whose Cargo.toml depends on one-or-more integration crates + their transitively-needed peers, re-exports nothing of substance, and exists to be named ONCE by the app. It composes with the umbrella via a `dep:`-hidden umbrella feature: `[features] redis = ["dep:leaf-starter-redis"]` on the `leaf` umbrella. Enabling the `redis` umbrella feature pulls `leaf-starter-redis` (which pulls `leaf-redis` + e.g. `leaf-tokio`), AND the umbrella's generated force-link shim adds every starter's leaf-integration crates to the participating set + ExpectedManifest. Two starter SHAPES: a CAPABILITY starter (one integration crate, e.g. `leaf-starter-redis`) and a STACK starter (a curated bundle, e.g. `leaf-starter-web` = router + tokio + json + validation), the latter being the Rust analogue of `spring-boot-starter-web`.

(4) BOM-EQUIVALENT VERSION COORDINATION = a deliberately DUAL mechanism, since Cargo has no native BOM: (a) INTERNAL — `[workspace.dependencies]` at the leaf workspace root pins ONE version of `leaf-core`, `leaf-macros`, `linkme`, and every integration crate, with all members using `key.workspace = true`; this is what prevents the load-bearing "two copies of the type/auto-config registry" hazard (a duplicated `leaf-core` = two `AUTO_CONFIGS` slices = silently-split batch). (b) DOWNSTREAM — the version-pinned `leaf` umbrella/facade crate IS the BOM surrogate: a user depends on exactly ONE crate `leaf = { version = "x.y", features = ["redis","web"] }`, and because `leaf x.y` pins exact `=`-or-caret internal versions of every starter/integration crate it re-exports, picking `leaf`'s version transitively pins the entire aligned set — the umbrella is the single version-coordination point a Maven BOM would otherwise provide. Integration/starter crates depend on `leaf-core` (NEVER the umbrella) to preserve the strict inward DAG. A `leaf-bom` "virtual BOM" doc-table is shipped for users who DON'T use the umbrella (direct `leaf-core` + hand-picked integration crates), listing the mutually-tested version tuple, but the umbrella is the blessed path.

COMPOSITION WITHOUT SURPRISE: the three "magic" surfaces compose along ONE legible axis surfaced by `leaf doctor`/the startup report: maximal-magic discovery answers WHAT-could-contribute (the force-linked participating set, per-crate row counts from the self-check); auto-config back-off + the condition report answer WHAT-actually-wired-and-WHY (Positive/Negative(reason)/Exclusion/CompiledOutByCfg/BuildFoldedFalse); the FOUR distinct "absent" states (DCE-vanished=loud AntiDceError; excluded=no BeanId; condition-not-met=silent recorded Negative; compiled-out-by-cfg=feature off) stay crisply separate. The rule documented loudly: PARTICIPATION = enabled capability feature ∪ scan! list (never an arbitrary transitive dep); WIRING = participation ∧ guard-match ∧ not-excluded ∧ (fallback-loses-to-user-bean).

## Rationale
- The Boot-style auto-config engine is the FIXED owner decision #2 and the codegen-boundary ADR already settled its SHAPE (R1 runtime deferred-batch, with R3 build-time plan as the opt-in cargo-leaf-prepare accelerator); this ADR honors that verbatim and only supplies the concrete artifacts it deferred — the dedicated AUTO_CONFIGS discovery channel, the LITE-ONLY marker + self-call lint, and the @Fallback-role default registration. Phase 1 (03-conditions-auto-configuration §499, §503-515) grounds every piece: per-crate manifest as a thin const descriptor on the shared substrate, the deferred-for-correctness moment, incremental in-batch registration so OnMissingBean sees the growing set, and the kill-switch.
- Back-off MUST be the DefinitionProbe over the frozen-candidate-set-so-far, not a bean-construction or a naive count — this is conditional-strategy's decided realization (OnMissingBean ⇐ would_resolve_unique != Unique), and reusing it (not minting a parallel back-off engine) is charter §2.11 unification. The precedence ladder exclude > user-bean back-off > auto-config default and the ContractId keying are cross-crate-discovery's decisions; this ADR only SEQUENCES the engine into App<Resolve> after user defs (the deferral moment Phase 1 §545 and Spring doc 1166/1204 require for correctness) and adds NO new keying or storage.
- Opt-in activation as (force-link participating set ∪ scan! list) ∧ runtime guard is the EXACT predictability posture cross-crate-discovery fixed for the maximal-magic owner decision (#1): participation is an explicit, scoped, predictable decision = enabled dep:-hidden feature UNIONED with scan!, never an arbitrary transitive dep, mitigating the implicit-global concern WITHOUT per-component declaration. The cross-crate research (§201-208) proves a cargo feature gates COMPILATION not LINKAGE, so a starter feature MUST do BOTH — gate compilation AND auto-add to force-link — which is why activation is two gates, not one; this is forced by T1, not chosen.
- leaf-starter-* aggregator crates are the literal Spring-Boot-starter intent (owner decision #3) realized in idiomatic Cargo: the cross-crate research (§196-206, §219) endorses dep:-hidden capability features over integration crates and additive-only capabilities, and a starter crate is the natural unit a user names once. Two shapes (capability vs stack starter) mirror Spring's own split (spring-boot-starter-data-redis vs spring-boot-starter-web), giving intent parity (charter §2.2) while staying DX-first (charter §2.1): one feature flag = one curated, version-aligned, force-linked bundle.
- The BOM-equivalent is FORCED to be non-native and dual because Cargo has no BOM (owner decision #3 explicitly says design it): the cross-crate research (§266) nails that [workspace.dependencies] inheritance gives the single-copy version unification that prevents the two-registries hazard INTERNALLY, while the umbrella facade (§257, §265) is the one-dependency downstream surface whose own version transitively pins the aligned set — exactly the coordination a Maven BOM provides, achieved by re-export + exact internal version pins. The leaf-bom doc-table is the honest escape hatch for non-umbrella users (charter §2.10 escape-hatch-at-every-level), since a pure [workspace.dependencies] table is not consumable by an external crate the way a Maven BOM is.
- The strict inward DAG (leaf-macros -> leaf-core -> integration crates -> leaf-starter-* -> leaf umbrella) is mandated by charter §6 and the cross-crate research (§243-264): no dependency cycles is a hard Cargo constraint, integration/starter crates depend on leaf-core NEVER the umbrella, and the umbrella owns the force-link + ExpectedManifest emission because the binary/umbrella is the only point seeing the whole graph (§244). This ADR's starter+BOM topology slots into that DAG without inventing new layering.
- Composition-without-surprise is realized by keeping the FOUR absence states distinct (the cross-crate-discovery + conditional-strategy decision) and routing them through ONE leaf-doctor view: discovery answers what-could-contribute (per-crate row counts), the condition report answers what-wired-and-why, and a DCE-vanished starter is a LOUD AntiDceError naming the crate — turning Phase 1's worst failure mode (§552, a silently-bare object graph indistinguishable from genuine back-off) into an early rich diagnostic (charter §2.9). The @Fallback-role default (Phase 1 §570) further softens override so a user bean wins transparently even when the auto-config default is also present, beating all-or-nothing back-off on DX.
- Everything stays thin-macro and hand-writable (charter §2.10): #[leaf::auto_config] emits one const AutoConfigDescriptor row (hand-writable against AUTO_CONFIGS::register), the starter crate is plain Cargo.toml (zero macro magic), and the umbrella's force-link shim is the same use crate as _; + const ExpectedManifest cross-crate-discovery already specified. The engine is a normal testable leaf-boot function over collected descriptors + a DefinitionView — unit-testable without a container, a link step, or a runtime (Phase 1 §506), which is the highest-confidence validation path.

## Options considered & rejected
- **Capability-keyed dependency-based engine (Phase 1 R2): make an enabled capability feature the PRIMARY gate, cfg-gating the whole #[auto_config] descriptor so the candidate set is scoped by COMPILATION, with OnMissingBean as a thin runtime residual** — rejected: Most intuitive 'add a dependency = add a starter' DX and lowest runtime cost, but it PRE-LEANS on making cfg the primary gate, which collides with the upstream decisions: conditional-strategy fixed the Runtime tier as the MANDATORY floor (dynamic/WASM-contributed auto-configs have no cfg and no Rust type; OnMissingBean needs the live def set), and cross-crate-discovery fixed that participation is force-link UNIONED with scan!, not a pure cfg fact. Cfg-scoping ALSO weakens the condition report's 'we considered X and it didn't match' accountability for the capability dimension (a cfg-gated-out candidate is simply absent, not considered-then-rejected). We KEEP its honest kernel — a Cfg-tier OnClass/OnCapability leaf physically removes an absent optional dep — but as the conditional-strategy tier-map's accelerator over the always-present runtime engine, not as the primary engine shape.
- **Build-time frozen auto-config plan as the DEFAULT (Phase 1 R3): leaf-codegen aggregates per-crate manifests via cargo metadata, runs union+dedup+exclusion-merge+build-prune+sort at build time, emits an inspectable autoconfig_plan.rs; runtime evaluates only OnMissingBean residual** — rejected: Earliest diagnostics and near-zero startup, but the codegen-boundary ADR already settled the build-frozen plan as the OPT-IN cargo-leaf-prepare accelerator for hostile targets, NOT the default — because (a) OnMissingBean back-off needs the FINAL runtime user-bean set, (b) a build.rs sees only its own crate (the directionality wall, cross-crate research §232), so the binary's codegen cannot introspect foreign #[auto_config] annotations without an unenforced manifest convention, and (c) a dynamic/WASM auto-config has no build presence. A runtime residual MUST exist regardless, so making build-time the default merely adds the build-frozen/runtime-residual confusion seam ('why didn't my plan change when I set the exclude property?'). Honored as the opt-in accelerator whose residual lands in the SAME App<Resolve> slot.
- **Re-using the single COMPONENTS distributed slice for auto-config descriptors (no dedicated AUTO_CONFIGS channel), distinguishing auto-configs by a marker field on the Descriptor** — rejected: Saves one slice, but conflates two channels with different lifecycle timing: auto-config candidates must be assembled in a SEPARATE deferred batch AFTER user defs (deferral-for-correctness), while normal components register in the main pass. A marker-field filter over one slice forces every consumer to re-partition the slice and risks an auto-config accidentally entering via the normal component path (Spring's AutoConfigurationExcludeFilter exists precisely to prevent this double-registration, doc 1180, Phase 1 §566). A dedicated AUTO_CONFIGS slice (same substrate, same self-check, same SequenceKey) makes the channel separation structural and lets the engine iterate exactly the deferred set — cheaper and less error-prone than a filtered scan.
- **A real importable BOM crate that re-exports [workspace.dependencies] as an external dependency table (mimicking Maven's <dependencyManagement> import scope)** — rejected: Cargo has no <dependencyManagement> import-scope: [workspace.dependencies] is consumable ONLY by workspace members, and a published crate cannot export a 'version constraint table' that downstream crates inherit without depending on it. Attempting it would require every downstream user to vendor leaf into their workspace or hand-copy the table — defeating the one-dependency DX. The umbrella facade IS the only Cargo-native version-coordination point (its own version transitively pins the re-exported set), so we make the umbrella the blessed BOM surrogate and ship leaf-bom as a documented version-tuple TABLE (not an importable constraint mechanism) for the non-umbrella escape hatch.
- **Per-integration-crate auto-activation: any integration crate, when present as a transitive dependency, auto-registers its auto-config (no feature/scan! opt-in) — pure maximal-magic for auto-config too** — rejected: Directly contradicts FIXED owner decision #2 ('activated by adding a starter-like crate and/or a cargo feature — NEVER by an arbitrary transitive dep') and the cross-crate-discovery predictability posture. Maximal magic applies to COMPONENT discovery within force-linked crates; auto-config activation is deliberately the opt-in gate that mitigates the implicit-global concern. A transitive dep that force-links would inject opinionated defaults a user never asked for — the exact surprise the owner chose to prevent via opt-in activation + exclusions. Participation requires the enabled capability feature ∪ scan!, so a transitive dep alone (not force-linked) never participates.
- **Mutually-exclusive backend features for starters (e.g. starter-redis-cluster XOR starter-redis-standalone as cargo features)** — rejected: Cargo features are additive-only and unify ON across the graph (cross-crate research §190-194); mutually-exclusive features are officially unsupported and will be unioned-on-together, producing a silent mis-configuration or a compile_error! at best. A backend CHOICE must be a runtime/profile decision (OnProperty/OnProfile over a runtime CondExpr), not a feature toggle. Starters are therefore strictly ADDITIVE capabilities; any 'A xor B' selection is pushed to the conditional-strategy runtime tier, which this ADR documents as a per-starter discipline the macro/lint enforces.

## Consequences
- FORCES a SECOND cross-crate discovery channel — `#[distributed_slice(::leaf_core::AUTO_CONFIGS)]` beside COMPONENTS — that inherits the FULL cross-crate-discovery contract: force-link (Defense 0), section-retention (Defense B), the expected-vs-found self-check (Defense MANIFEST with per-crate provides_descriptors anchor), ContractId keying, and SequenceKey ordering. An auto-config in a crate the binary never force-links is a LOUD AntiDceError::SourceVanished naming the crate, NOT a silently-bare graph. leaf-core's stable ABI grows by exactly one slice + the AutoConfigDescriptor row shape (lite-config seeds + CondExpr guard ref + @Fallback role flag).
- FORCES the engine into the conditional-strategy + codegen-boundary lifecycle slot unchanged: it runs in App<Resolve>'s REGISTER sub-pass, AFTER seal_environment, AFTER all user defs and the eager user-config batch, applying exclude(ContractId) BEFORE the back-off batch (an excluded ContractId mints no BeanId and never enters back-off) and registering surviving candidates INCREMENTALLY in SequenceKey order so each later candidate's DefinitionProbe sees the growing set. It outputs DEFINITION order/contents ONLY and NEVER touches the async instantiation scheduler (the doc-1234 definition-order-vs-creation-order boundary, structurally enforced).
- FORCES the umbrella `leaf` crate + #[leaf::main]/build.rs to own starter force-link: enabling a `leaf` umbrella feature (e.g. `redis`) AUTO-ADDS every leaf-integration crate the corresponding leaf-starter-* pulls to (a) the force-link participating set (`use leaf_redis as _;`) AND (b) the const ExpectedManifest sources. The user states the dependency ONCE (the feature flag); force-link + expected-manifest are automatic. A starter feature that is enabled but contributes ZERO descriptors is a loud self-check failure (the 'enabled-but-empty' warning Phase 1 §508/§552 mandates).
- FORCES the strict inward crate DAG with starters as a new tier: leaf-macros (proc-macro) -> leaf-core (COMPONENTS + AUTO_CONFIGS slices + RegistryBuilder + condition ABI) -> integration crates (leaf-redis, leaf-tokio; depend on leaf-core ONLY) -> leaf-starter-* aggregators (depend on integration crates; NEVER the umbrella) -> leaf umbrella facade (re-exports + force-links enabled starters). No starter or integration crate may depend on the umbrella (cycle/hard-error). leaf-boot hosts the engine; leaf-core hosts the ABI (charter §6 topology).
- FORCES the BOM-equivalent onto [workspace.dependencies] (one copy of leaf-core/leaf-macros/linkme/every integration crate, key.workspace=true on all members) + the version-pinned umbrella as the downstream coordination point. Duplicating leaf-core (two versions in the graph) = two AUTO_CONFIGS slices = a silently-split auto-config batch — so the single-copy invariant is LOAD-BEARING, and resolver='3' must be set explicitly at the workspace root (cross-crate research §186) so dev/build/target features don't leak into the production candidate set. Picking the umbrella's version transitively pins the aligned starter/integration set; a leaf-bom doc-table documents the tested tuple for non-umbrella users.
- FORCES the auto-config bean to register with a @Fallback role (not just OnMissingBean back-off) so a user bean transparently supersedes a present auto-config default — co-binding this ADR with the @Primary/@Fallback ADR: an auto-config default is a fallback candidate, a user bean is non-fallback, and the candidate-resolver picks the user bean. This is a SOFTER override than all-or-nothing OnMissingBean back-off and must be the documented default registration role for auto-config beans (Phase 1 §570).
- FORCES FOUR crisply-distinct 'absent' states the diagnostics model and leaf doctor must keep separate and route to ONE account: (a) DCE-vanished starter = LOUD AntiDceError (crate in ExpectedManifest, zero rows); (b) excluded ContractId = no BeanId minted (silent-by-design, in the exclusion report); (c) condition-not-met / backed-off = silent recorded condition-report Negative('OnMissingBean found user Foo' / 'OnProperty leaf.redis.url absent'); (d) feature-off = CompiledOutByCfg(SourceTag). Conflating any two reintroduces the silent-vanish hazard or makes a deliberate exclusion or a genuine back-off look like a bug (Phase 1 §578-579).
- FORCES a curated TEST-SLICE mode `slice!{ WebStack, DataLayer }` reusing the SAME engine over an EXPLICIT candidate list (not the full contributed set), with optional-if-absent entries skipped when their crate isn't force-linked — the leaf analogue of @WebMvcTest/@DataJpaTest (Phase 1 §504/§581). This forces the engine to accept its candidate set as a parameter (full AUTO_CONFIGS slice vs an explicit subset) rather than hard-coding the global slice, keeping it a pure function over (candidates, user_defs, env).
- ACCEPTS that distribution is NOT a Maven BOM and must be documented honestly: there is no importable cross-workspace version-constraint mechanism in Cargo; the umbrella's version IS the coordination point, and a user who bypasses the umbrella (direct leaf-core + hand-picked crates) owns their own version alignment against the leaf-bom doc-table. leaf MUST document that 'add a transitive dep and it injects opinionated defaults' is NOT how activation works (participation = enabled capability feature ∪ scan!), which is also the predictability posture owner decisions #1/#2 endorse.
- ACCEPTS a bounded one-shot COLD cost in App<Resolve> (dedup + exclusion-subtract + topo-sort + per-candidate runtime CondExpr eval + incremental DefinitionProbe back-off over potentially hundreds of candidates), mitigated by the conditional-strategy Cfg-tier compile pruning (absent-capability candidates are physically removed at compile time, the rust-native analogue of Boot's spring-autoconfigure-metadata fast-path) and optionally by the codegen-boundary ADR's opt-in cargo-leaf-prepare build-time plan for determinism-sensitive deployments. ZERO hot-path cost: the frozen container is read lock-free at steady state, and the engine never runs after seal().

## API / type sketch

// ═══ leaf-core: the auto-config discovery channel + descriptor ABI ═══
// (built on cross-crate-discovery's Descriptor/ContractId/SequenceKey/SourceTag/RegistryBuilder
//  and conditional-strategy's CondExpr/DefinitionProbe — reused verbatim, NOT re-minted)

// ── SECOND discovery channel, beside ::leaf_core::COMPONENTS ──
#[linkme::distributed_slice] pub static AUTO_CONFIGS: [AutoConfigDescriptor] = [..];

pub struct AutoConfigDescriptor {
    pub contract: ContractId,          // stable identity (cross-crate-discovery); keys exclude/before/after
    pub source:   SourceTag,           // owning crate, for the expected-vs-found self-check
    pub seq:      SequenceKey,          // (epoch, contract) — deterministic batch tie-break, NEVER link order
    pub guard:    &'static CondExpr,    // conditional-strategy algebra: all(on_property(..), on_missing_bean(..), ..)
    pub order:    OrderHint,            // auto-config-ordering: order(i32) + before/after [ContractId]
    pub seeds:    &'static [BeanSeed],  // lite-config #[bean] factory seeds (boxed-future async Factory at dyn edge)
    pub role:     CandidateRole,        // Fallback by default ⇒ a user bean transparently supersedes
}
pub enum CandidateRole { Normal, Primary, Fallback }   // auto-config defaults register as Fallback

// ═══ leaf-boot: the engine (the codegen-boundary ADR's R1 deferred batch, made concrete) ═══
pub struct AssemblyCtx<'a> {
    pub env:        &'a SealedEnv,               // sealed BEFORE this pass (5f fence done)
    pub defs:       &'a mut DefinitionView<'a>,  // GROWING set; incremental register so OnMissingBean sees predecessors
    pub probe:      &'a DefinitionProbe<'a>,     // conditional-strategy: no-instantiation would_resolve_unique
    pub exclusions: &'a ExclusionSet,            // cross-crate-discovery: ContractId-keyed, typo-guarded
}

/// ONE cold synchronous pass in App<Resolve>, AFTER user defs + AFTER the eager user-config batch.
/// Ladder: exclude(ContractId) > user-bean back-off(OnMissingBean) > auto-config default.
pub fn run_autoconfig(ctx: &mut AssemblyCtx, candidates: &[&AutoConfigDescriptor])
    -> Result<(), AssemblyError>
{
    if ctx.env.flag("leaf.enable-autoconfiguration") == Some(false) { return Ok(()); } // kill-switch
    let ordered = auto_config_ordering::order_batch(candidates, ctx.exclusions)?;        // delegated 3-pass sort
    for c in ordered {                                                                   // SequenceKey-tie-broken
        if ctx.exclusions.contains(c.contract) { report.record(Exclusion(c.contract)); continue; } // mints NO BeanId
        match eval(c.guard, &cond_ctx(ctx)) {                       // conditional-strategy CondExpr eval
            Match  => ctx.defs.register_lite(c.seeds, c.role),       // visible to LATER candidates' OnMissingBean
            NoMatch(reason) => report.record(Negative(c.contract, reason)),   // silent, recorded — NOT an error
        }
    }
    Ok(())
}

## Forces on other concepts
- **Cross-Crate Discovery + Anti-DCE + Exclusions (upstream cross-crate-discovery ADR)**: Honored and EXTENDED by exactly one new channel: AUTO_CONFIGS is a second #[distributed_slice] that inherits force-link (Defense 0), section-retention (Defense B), the expected-vs-found self-check (Defense MANIFEST + per-crate provides_descriptors anchor), ContractId keying, and SequenceKey ordering UNCHANGED. Exclusions key on ContractId (never TypeId); a starter feature auto-adds its integration crates to the force-link participating set AND the ExpectedManifest; a DCE-vanished starter is AntiDceError::SourceVanished. This ADR owns the AUTO_CONFIGS channel + its descriptor row; that ADR owns the substrate.
- **Conditional Strategy + Profiles + Auto-Config Back-off (upstream conditional-strategy ADR)**: Honored and CONSUMED unchanged: each AutoConfigDescriptor.guard is that ADR's const CondExpr; back-off IS its DefinitionProbe.would_resolve_unique (OnMissingBean ⇐ !Unique, OnSingleCandidate ⇐ Unique), evaluated in its REGISTER sub-pass; the precedence exclude > back-off > default is sequenced here. A Cfg-tier OnClass/OnCapability guard pairs with this ADR's starter force-link (a feature gates compilation not linkage). The engine adds NO new condition kind — it is the headline consumer of the condition algebra's runtime tier.
- **Metadata / Codegen Boundary (upstream codegen-boundary ADR)**: Honored: the engine IS that ADR's R1 runtime deferred-batch (default); the build-frozen autoconfig plan is its opt-in cargo-leaf-prepare accelerator whose residual lands in the SAME App<Resolve> slot. #[leaf::auto_config] stays a thin macro emitting one const AutoConfigDescriptor (hand-writable). The umbrella/binary-crate build.rs owns force-link + ExpectedManifest for starters (the binary-crate responsibilities that ADR fixed).
- **@Primary / @Fallback & Candidate Resolution**: Auto-config beans register with CandidateRole::Fallback by DEFAULT so a user bean transparently supersedes a present auto-config default even when both exist (softer than all-or-nothing OnMissingBean back-off). The candidate-resolver MUST treat a Fallback candidate as losing to any non-fallback candidate of the same contract; the per-type expected-count self-check subtracts excluded/backed-off candidates. Co-bound with the @Primary/@Fallback count-arithmetic ADR.
- **Crate Topology & Distribution (charter §6, owner decision #3)**: FORCES the inward DAG with a new starter tier: leaf-macros -> leaf-core -> integration crates -> leaf-starter-* aggregators -> leaf umbrella. Starter/integration crates depend on leaf-core, NEVER the umbrella (cycle/hard-error). BOM = [workspace.dependencies] single-copy internally (resolver='3') + version-pinned umbrella as the downstream coordination point + a leaf-bom doc-table for non-umbrella users. A starter = a dep:-hidden umbrella feature over an aggregator crate; STACK starters bundle a curated multi-crate set.
- **Container Refresh Lifecycle & Compile-vs-Runtime Split**: FORCES the engine into a fixed slot: seal_environment -> collect user defs + eager user-config batch -> apply_exclusions -> run_autoconfig (incremental back-off batch over AUTO_CONFIGS in SequenceKey order) -> freeze. It outputs DEFINITION order/contents ONLY; the async instantiation scheduler stays dependency-driven (doc-1234 boundary). Post-freeze edits are a typestate error; the kill-switch short-circuits the whole batch.
- **Unified Error / Diagnostics Model (register §6) + Condition Report**: FORCES leaf doctor / the startup report to keep FOUR absence states distinct and reconcile them into ONE account: AntiDceError::SourceVanished (loud, names crate), Exclusion(ContractId), condition-report Negative(reason, silent-recorded), CompiledOutByCfg(SourceTag). An enabled-starter-zero-contribution is a loud self-check warning. A later NoSuchBean enriches from the frozen report by ContractId ('backed off because OnMissingBean found user Foo' / 'starter redis not enabled').
- **Bootstrap / Run-Participant Registries (07-bootstrap)**: The auto-config AUTO_CONFIGS channel is one more typed distributed_slice sharing the cross-crate-discovery substrate, alongside the run-participant streams (RunListeners, ContextInitializers, Runners, FailureAnalyzers, AppTypeDeducers). A starter that ships a Runner/Initializer/FailureAnalyzer force-links it via the same umbrella-feature mechanism; the expected-vs-found self-check + leaf doctor listing cover all channels uniformly. No bespoke per-stream activation path.
- **WASM / Dynamic Second-Source Layer (dropped for v1)**: A dynamic/WASM-contributed auto-config has no cargo feature and no Rust type, so it CANNOT be a starter and is gated ONLY at the conditional-strategy Runtime tier; it enters via cross-crate-discovery's reopen_epoch second epoch (epoch >= 1), registering an origin-agnostic AutoConfigDescriptor keyed on ContractId, sorted AFTER all static rows by SequenceKey. This ADR specifies only that the engine is origin-agnostic (a WASM auto-config backs off and orders indistinguishably); the WASM runtime is Phase-3 WASM-local.

## Open sub-questions (→ Phase 3)
- The exact STACK-starter curation policy and its semver surface: which integration crates a leaf-starter-web bundles (router + tokio + json + validation?), whether a stack starter pins peers via the workspace or re-exports feature toggles, and how a user opts OUT of one crate within a stack starter (a per-crate disable feature vs an exclude= on its auto-config). Couples to the additive-only feature discipline. Phase 3.
- Whether leaf-bom ships as JUST a documented version-tuple table or ALSO as a thin crate exposing the tuple as compile-time consts for tooling (e.g. a `cargo leaf check-versions` that warns when a hand-picked non-umbrella set drifts from the tested tuple). Measure whether non-umbrella usage is common enough to justify the tooling. Phase 3, measured before built.
- The precise @Fallback-role interaction with OnMissingBean: does an auto-config bean register as BOTH Fallback role AND carry an OnMissingBean guard (belt-and-suspenders), or does the Fallback role REPLACE OnMissingBean for the common 'user overrides default' case (reserving OnMissingBean for the all-or-nothing case)? Couples to the @Primary/@Fallback count arithmetic and the DefinitionProbe semantics. Phase 3.
- Test-slice mode mechanics: how slice!{ WebStack } resolves a named curated subset (a const list of ContractIds emitted by a #[slice] macro vs an explicit AutoConfigDescriptor array), how optional-if-absent entries interact with force-link (an optional entry whose crate isn't linked is skipped, but does that need its own self-check class?), and whether a slice runs the SAME exclusion/back-off ladder. Cross-cuts testing ergonomics. Phase 3.
- The kill-switch granularity: is leaf.enable-autoconfiguration=false the only off-switch, or is there a per-starter leaf.autoconfigure.<starter>.enabled=false (finer than exclude= by ContractId)? And whether the kill-switch composes with the exclude property (kill-switch short-circuits the whole batch before exclusions are even read). Config-tier detail. Phase 3.
- Whether the umbrella's internal version pins use exact '=1.4.0' or caret '^1.4' for re-exported integration crates: exact pins give the strongest BOM-like alignment but force a leaf umbrella release for every integration patch; caret allows independent patching but weakens the 'one version coordinates all' guarantee. Measure against release cadence and the single-copy-of-leaf-core invariant. Phase 3.
