# Phase 2 — Coherence Review

**Verdict:** COHERENT WITH BOUNDED, FIXABLE SEAMS. The 12 ADRs compose into ONE architecture with a genuinely consistent spine: a single erased shared handle (ownership-model) carried through one origin-agnostic Provider seam, one BeanId-keyed frozen registry, one linkme/anti-DCE discovery substrate, one CondExpr tier-map, one @Order comparator, one LeafError chain, and one phase machine (Define->Resolve->Wired->Running) with seal() as the freeze fence. Every ADR's forcesOnOtherConcepts is actually honored by the forced ADR — I checked each forward reference (ownership->registry/async/scopes/deferral/config; registry->codegen/annotation/autowiring; codegen->registry/cross-crate; cross-crate->codegen/conditional/autoconfig; compile-runtime->registry/codegen; container-shape->refresh/factory-bean/deferral; proxy->ownership/registry/async/deferral; injection->all clusters; error->codegen/compile-runtime/conditional; conditional->compile-runtime/cross-crate; autoconfig->cross-crate/conditional/codegen) and found no case where a forced concept contradicts the forcing constraint. The register's eight 'resolve-first' tensions are all settled and the resolutions agree with each other. Charter invariants hold: async-first single-mode (2.3), reactive-hot/poll-cold (2.4/5f), thin macros emitting hand-writable const data (2.10), unification-over-bespoke (2.11), stable-by-default (2.8, no nightly gating any public API), rich-early-aggregated diagnostics (2.9 via the App<Wired> AssemblyReport + FailureAnalyzer), and the inward-pointing crate DAG (6). The four contradictions I list are all SEAM-LEVEL precision gaps at the dynamic-WASM-epoch / advised-prototype / advised-concrete-injection boundaries — none invalidates the core model, each has a concrete one-ADR fix, and three of them already have the resolving principle stated elsewhere in the same ADR set (it just needs to be made explicit at the specific seam). The single most material one to close before Phase 3 is the singleton-OnceCell-carry-forward-across-epochs gap, because silently constructing two singleton instances would violate the Once multiplicity contract that the whole shared-handle model rests on. Recommend Phase 3 record these four as explicit co-decided seam notes; no Phase 2 ADR needs to be reopened wholesale.

## Contradictions found & fixes

- **compile-runtime-split (singleton store) vs registry-substrate / cross-crate-discovery (dynamic epoch model)** — registry-substrate fixes the singleton store as `singletons: Box<[OnceCell<ErasedBean>]>` indexed by BeanId.0 (a fixed-length array sized at freeze). compile-runtime-split and cross-crate-discovery then adopt FRESH-SNAPSHOT-PER-EPOCH: reopen_epoch() builds a NEW FrozenContainer with NEW BeanIds appended and the live pointer is arc_swapped. These are in tension on a concrete point the ADRs never reconcile: after a dynamic re-freeze, singletons ALREADY instantiated in epoch N (live Arcs in epoch N's OnceCell array) must remain valid and not be re-instantiated in epoch N+1's fresh array. The ownership-model says the Arc strong-count governs the real free and in-flight resolutions hold their epoch Arc — but a SINGLETON resolved in epoch N and then resolved again via the epoch N+1 pointer would hit a DISTINCT, empty OnceCell slot and construct a SECOND instance, violating the Once multiplicity contract (singleton = one instance per container). No ADR states how already-initialized singleton cells are carried forward (migrated/shared) into the new epoch's array.
  - **Fix:** cross-crate-discovery / compile-runtime-split YIELD on the detail (registry-substrate owns storage, charter-style 'this ADR owns keying+storage'): specify that reopen_epoch() carries forward the prior epoch's initialized OnceCell<ErasedBean> contents for retained BeanIds — i.e. epoch N+1's singleton array is seeded from epoch N's filled cells (BeanId is append-only/stable for static rows per the 'static rows always precede dynamic rows in BeanId order' invariant), so a static singleton keeps ONE instance across epochs and only newly-appended dynamic BeanIds start empty. State this explicitly as a registry-substrate consequence so the Once contract holds across the atomic swap.
- **compile-runtime-split (typestate: post-seal edit is a COMPILE error) vs itself + cross-crate-discovery (reopen_epoch runtime mutation)** — compile-runtime-split states `seal(self) -> App<Wired>` consumes mutable access so 'a post-seal edit is a COMPILE error on the static path', while ALSO exposing `App<Running>::reopen_epoch() -> App<Define>'` for the dynamic/WASM lane. The same ADR both forbids post-seal definition mutation at the type level AND provides a typed method to mutate definitions post-seal. This is not a true contradiction (reopen_epoch is described as the ONLY sanctioned mutation, producing a fresh epoch rather than editing the frozen one) but the wording 'post-seal edit is a compile error' is unqualified and reads as conflicting with the existence of reopen_epoch. A reader cannot tell whether reopen_epoch is a typestate-legal transition or an escape that breaks the stated invariant.
  - **Fix:** compile-runtime-split clarifies (no real yield needed, only precision): the invariant is 'you cannot MUTATE a sealed epoch in place' — reopen_epoch consumes App<Running> and returns a fresh App<Define>' whose RegistryBuilder is a NEW mutable builder pre-seeded (copy-on-write) from the prior FrozenContainer. The typestate is preserved because mutability is only ever available in App<Define|Resolve>; reopen_epoch re-enters that state rather than editing App<Running>. Reword the consequence to 'in-place edit of a sealed epoch is unrepresentable; the only path to new definitions is a fresh epoch via reopen_epoch'.
- **proxy-substrate (service-trait mandatory for advised beans) vs autoconfig-starters-bom + container-shape (dogfooded infrastructure as advisors)** — proxy-substrate makes a service trait MANDATORY for any advised/scoped bean (no subclass-a-concrete-class in Rust) and hard-errors on an advised concrete-only bean. It also says leaf's own tx/cache/validation/async/retry are Role::Infrastructure advisors over the SAME mechanism. But the auto-proxy-creator installs a proxy at the after_init seam by swapping the ErasedBean for a generated newtype that implements the bean's service trait and upcasts to Arc<dyn Svc>. The unresolved seam: a bean injected by CONCRETE type (Ref<Concrete>, which registry-substrate explicitly supports as 'concrete handle matches by exact TypeId only') CANNOT be transparently proxied, because the generated OrderProxy implements dyn Order, not Concrete — it has no Concrete TypeId to match. So a concrete-typed injection of an advised bean either (a) gets the un-advised raw target (advice silently bypassed — the exact silent-failure the anti-DCE doctrine forbids) or (b) must be a hard error. proxy-substrate says concrete-only beans are 'un-advisable' but does not address a bean that HAS a service trait, is advised, yet is ALSO injected somewhere by its concrete type.
  - **Fix:** proxy-substrate + injection-mechanics jointly state the rule explicitly: if a bean is advised, every injection of it MUST go through a declared service-trait view; a concrete-type injection point for an advised bean is a startup-validation-pass error (AssemblyReport NoSuchBean/ScopeMismatch-class), surfaced by the App<Wired> pass, NOT a silent raw-target handoff. This is a small addition to injection-mechanics' Selector (a Concrete TypeMatch against an advised bean is rejected) and to proxy-substrate's eligibility gate. It preserves 'advice is never silently bypassed.'
- **ownership-model (get_owned<T> requires T: 'static, prototype = owned move) vs proxy-substrate (prototype beans + advice)** — ownership-model says prototype publishes Published::Owned(Box<dyn Any + Send>) and get_owned<T> returns an owned T; prototype 'registers no teardown' and is exempt from refcount. proxy-substrate's TargetSource seam is built around ErasedBean = Arc (FixedTarget(ErasedBean) / ScopeTarget yielding ErasedBean). There is no stated story for advising a PROTOTYPE bean: the proxy holds a TargetSource that returns ErasedBean (an Arc), but a prototype's publication is an owned Box, not an Arc, and is supposed to be a fresh move per resolution. A transparent advice proxy over a prototype would either have to (a) wrap each owned instance in an Arc (contradicting 'never refcounted, owned move') or (b) not support advice on prototypes. The ADRs do not say which.
  - **Fix:** proxy-substrate states the scope of advisability: transparent advice applies to shared-shape beans (singleton + context-scoped, whose publication is Published::Shared(ErasedBean)); a prototype that is advised gets a per-resolution proxy whose TargetSource constructs-and-owns a fresh instance, with the proxy itself being the returned value (the proxy is Arc-shaped but wraps a freshly-built-per-call target via an Owned-aware TargetSource variant). Alternatively, declare advised-prototype out of scope for Phase 2 with a diagnostic. Either way, add an explicit Published::Owned lane to the TargetSource contract so the 'owned move, no refcount' prototype invariant is not silently violated.

## Reconciliations
- OWNERSHIP-MODEL IS THE CONSISTENT KEYSTONE. ErasedBean = Arc<dyn Any+Send+Sync> + Ref<T> + the Published{Shared,Owned} enum + the dyn Provider->BoxFuture seam are referenced verbatim and HONORED unchanged by every downstream ADR (registry-substrate's REGISTRY VALUE, container-shape's Engine::create, proxy-substrate's TargetSource, injection-mechanics' Provider::provide, async-context-model's 5b). No ADR introduces a second shared-handle type. The 'WASM-ness stops at the proxy / origin-agnostic Provider' invariant is preserved end-to-end (registry-substrate's Origin is diagnostic-only and never read on a resolution path; container-shape's Engine 'cannot tell origins apart').
- THE Send+Sync+'static BOUND RIDES ErasedBean/Bean IN ONE PLACE and is not re-expressed. ownership-model places it on the handle; async-context-model 5e explicitly says it 'rides the upstream Bean/ErasedBean bound'; the concurrency contract is satisfied by the handle type. The #[diagnostic::on_unimplemented] doctrine message has a single owner (async-context-model owns the spawn/shared-bound message; error-model owns the macro emission of it) — no duplication, charter 2.11 satisfied.
- THE ANTI-DCE SELF-CHECK IS COHERENTLY SINGLE-SOURCED. ownership-model FORCES it ('erased handle makes this non-negotiable'), registry-substrate locates it at freeze(), cross-crate-discovery OWNS its concrete shape (ExpectedManifest + SourceTag + provides_descriptors anchor + AntiDceError), compile-runtime-split places it at the App<Define>->App<Resolve> edge, container-shape runs it in Context::refresh(), error-model renders it (Tier 1) and autoconfig adds the AUTO_CONFIGS slice to the expected set. Every ADR points at ONE check, not a reinvented per-feature one. This is the cleanest cross-ADR thread.
- THE ONE @Order/@Priority COMPARATOR IS GENUINELY UNIFIED. injection-mechanics (B) mints leaf_core::order::cmp_order over OrderKey{value,source} and explicitly enumerates ALL consumers: autowiring tie-break, collection/map ordering, Lookup::ordered_stream, runtime-lifecycle phases, advisor chain. proxy-substrate CONSUMES it at 'correctness grade' with RoleTier-first; cross-crate-discovery layers the SequenceKey(epoch,ContractId) deterministic tiebreak BELOW it (not a competing comparator). No site defines its own. Resolves register Clusters 10 and 11's 'ONE comparator' demand.
- THE FOUR ABSENCE STATES ARE KEPT CRISPLY DISTINCT ACROSS ADRs. registry-substrate (NullBean = NULL_BEAN slot vs map-miss), cross-crate-discovery (DCE-vanished=loud / excluded=no BeanId / condition-not-met=silent / compiled-out-by-cfg), conditional-strategy (CondExpr report classes), error-model (ConditionOutcome is NEVER a LeafError; NoSuchBean enriches FROM the frozen ConditionReport) and autoconfig (the four-state leaf doctor account) all agree on the same taxonomy and the same silent-now/loud-later bridge. This was register 6's central tension and it is settled consistently.
- THE CONSUMER-SIDE NAME COLLISION IS RESOLVED COHERENTLY. container-shape renames the jakarta-Provider family to Lookup<T>/LazyRef<T>/Inject<T> to free 'Provider' for the single construction seam; injection-mechanics binds to exactly those renamed names through the one resolve(Strictness,Cardinality) driver; proxy-substrate explicitly defers the honest-visible-handle shapes (ScopedRef/Provider/@Lazy) to the deferral family and keeps advice transparent. The transparent-vs-honest split ('advice = transparent dyn Svc; deferral = honest handle') is stated identically in both proxy-substrate and injection-mechanics.
- THE CONDITION TIER-MAP IS A SINGLE KERNEL ABI WITH ONE OWNER CHAIN. codegen-boundary fixes the per-kind tier-map (cfg/const-fold/runtime) as kernel ABI; compile-runtime-split EXECUTES it in App<Resolve>; conditional-strategy SUPPLIES the concrete CondExpr enum + DefinitionProbe + phase()=max-over-leaves inference; autoconfig CONSUMES it as the headline back-off engine. Profiles-always-runtime is stated identically in three ADRs. The 'runtime tier is the mandatory floor because dynamic/WASM defs have no cfg' rationale is repeated consistently and is load-bearing in all four.
- TEARDOWN-WITHOUT-async-Drop IS CONSISTENT ACROSS THE ASYNC/PROXY/INJECTION TRIO. async-context-model 5d (TeardownLedger, LIFO drain, no async Drop, sync RAII for Cx/Permit), ownership-model (TeardownLedger, Arc strong-count governs free, prototype registers no teardown), proxy-substrate (around-advice cancel = sync RAII Drop, async finalization = ledger), injection-mechanics (cancelling a prototype build leaks nothing, owned-move) and compile-runtime-split (shutdown().await reverse-wave) all describe the SAME mechanism with no contradiction. The 'Arc strong-count, not a slot index, governs the real free' is the shared soundness argument for in-flight scoped work.
