# AOP & Interception

> The five designs (aop-proxying substrate, auto-proxy-creator decision/collection layer, aspect-model authoring layer, scoped-proxies sibling, async-execution thin advisor) form a coherent, well-stratified three-tier AOP stack plus two consumers, and they correctly absorb the Spring reference's load-bearing intent: advice-added-around-not-compiled-in, the single-proxy-carries-the-union-of-advice model, role-hint isolation (infra vs aspect), the after_init wrap-and-swap seam, the early-reference/circular hook, the self-invocation honesty boundary, and the proxyMode/ProxyConfig surface. The tier boundaries (who-wraps-one-call vs who-decides-which-advisors-match vs who-authors) are drawn sharply and consistently, and the deferred Phase-2 keystones (proxy mechanism, transparent-vs-honest handle, erased-vs-typed args, collection substrate) are flagged identically across features so they will be resolved once rather than divergently. The gaps are concentrated in (1) Spring AOP capabilities the reference doc itself under-covers — introductions/mixins, the TargetSource per-invocation abstraction, lookup-method injection as the third scope-mismatch cure, and proxy identity/introspection semantics — none of which any design surfaces, and (2) the separate-but-adjacent @Configuration full-mode inter-bean-call interception, which is proxying machinery living in this area but owned by no design. The main coherence risks are not contradictions but THREE undecided cross-feature keystones (transparent-vs-honest handle, the proxy mechanism, ordering-model grade) whose resolution must be forced to be identical across aop-proxying/scoped-proxies/auto-proxy-creator/aspect-model, plus one genuine duplication candidate (the scope-resolution wrap vs the advice-chain wrap, which both features already nominate for unification).

_5 feature designs. Implementation-agnostic — candidate realizations for Phase 2/3, not decisions._

## Scoped Proxies  `[scoped-proxies]`

A scoped proxy lets a LONGER-lived bean (a singleton) hold ONE stable, injected reference to a SHORTER-lived bean (request/session/prototype/custom) while each method call transparently resolves and dispatches to the LIVE shorter-lived target in the currently-active scope. It is the sanctioned cure for the singleton-holds-shorter gotcha (spring doc lines 822-824, 1757): inject a shorter scope directly and the singleton freezes one instance forever, silently defeating the narrower scope. Spring's load-bearing intent (850-854, 1755-1762): DI wires the graph ONCE; rather than leak scope-lookup logic into business code (calling getBean per request) Spring interposes a transparent proxy of the SAME public type so the long-lived bean codes against a stable reference while the proxy re-resolves the live short-lived target PER INVOCATION — preserving IoC and keeping business code scope-agnostic. The mechanics to replicate at the INTENT level: (1) the injected stand-in is type-compatible with the target (the holder never knows it holds a proxy); (2) on EACH call the proxy resolves the live instance from the active scope (NOT once at injection), so it is the per-call re-resolution that distinguishes it from a frozen direct reference; (3) configured as metadata pairing scope with a proxy mode (@Scope(proxyMode=TARGET_CLASS|INTERFACES|NO|DEFAULT) / @RequestScope etc. defaulting proxyMode to a proxy); (4) the per-call resolution rides the SAME scope-resolution path scopes defines (scopes owns multiplicity/lifetime + the get entry point; scoped-proxies owns the per-call indirection that calls it); (5) it is the SIBLING of the deferral-handle fix (ObjectProvider/Provider, 787-796) — both re-resolve per call against current scope state; the proxy hides the indirection behind the target's own type, the deferral handle makes it visible in the type signature (the spring doc explicitly frames ObjectProvider as giving the same lifecycle safety as a scoped proxy but with an honest, debuggable call site, line 2567). I must replicate this INTENT — a transparent, IoC-preserving, per-call-re-resolving stand-in for a narrower-scoped bean injected into a wider-scoped holder — WITHOUT importing the JVM mechanisms Spring sheds: no CGLIB runtime subclassing, no JDK dynamic-proxy synthesis, and CRUCIALLY no ThreadLocal-bound active-scope lookup (RequestContextHolder), which silently goes stale across .await/thread-hops in async Rust. The per-call resolution must bind to an async-propagated ambient (the deferred context-propagation keystone), the stand-in must be origin-agnostic (a WASM-module-sourced or test-double scoped target re-resolves identically), and the proxy mechanism is shared with aop-proxying's wrap substrate (a transparent stand-in routing each call) differing only in WHAT proceed() does — route to a freshly-resolved live target rather than through an advice chain to a fixed target. I do NOT pick the proxy mechanism, the ambient/async-context model, the scope-instance ownership model, or the cross-crate substrate — those are Phase 2.

### Candidate realizations

#### R1 — Re-resolving delegating stand-in (a degenerate AOP proxy whose 'advice' is scope re-resolution): reuse aop-proxying's wrap-and-swap + call-routing substrate, swapping the fixed target for a per-call scope lookup
Realize the scoped proxy as the SAME transparent stand-in aop-proxying (R1/R2) produces — a thing that is-a the target's service trait, injected at the DI seam in place of the target — but where the call-routing primitive, instead of walking an advice chain to a FIXED target, performs a per-call SCOPE RESOLUTION and dispatches to the freshly-resolved live target. Concretely: for a request-scoped bean `Cart` exposing `dyn Cart`, the holder (a singleton) is injected `Arc<dyn Cart>` that is actually a `ScopedProxy<dyn Cart>` (a generated delegating newtype OR a runtime handle, per whichever aop-proxying mechanism Phase 2 picks). Each method `fn add(&self, i: Item)` is `{ let live = self.resolver.resolve_live(self.scope_id, self.target_key, &self.ambient()).await?; live.add(i).await }`. The substrate scoped-proxies OWNS is thin: a `LiveResolver` seam — `async fn resolve_live(&self, scope: ScopeId, key: TargetKey, ambient: &Ambient) -> Result<ScopedRef<dyn T>, ResolveError>` — which is a DIRECT call into scopes' `Scope::get`/`InstanceStore::get_or_init` (the upstream scopes design's resolution entry point), passing the SAME erased origin-agnostic Factory from bean-definition. There is no advice chain, no JoinPoint mutation, no interceptor ordering — proxyMode just selects 'wrap this injection with a re-resolving stand-in instead of injecting the raw target'. The proxy is materialized at the post-processor-spi after_init seam exactly as aop-proxying's: an InstancePostProcessor sees the holder's injection point is typed `dyn Cart`, the target's definition is request-scoped + proxyMode!=NO, and substitutes a `ScopedProxy<dyn Cart>` ErasedBean. Self-invocation honesty falls out identically (the live target's own self-call never re-enters the proxy). The 'scope' bound to the proxy (singleton-shaped: one stand-in per holder, created once) NEVER holds the live target — it holds the resolver + key + a way to reach the ambient, so it is itself safely a singleton even though it yields request-scoped values.

**Pros:** Maximal unification (charter §2.11): a scoped proxy IS an AOP proxy whose single 'advice' is re-resolve-then-delegate, so leaf builds ONE wrap-and-swap + call-routing substrate (aop-proxying) and scoped-proxies adds only the `LiveResolver` seam — no second proxy machine, no duplicate after_init swap, no duplicate self-invocation story; the two siblings the upstream aop-proxying design explicitly flagged as 'a strong argument for unifying the wrap primitive' are unified here.; Inherits aop-proxying's transparency: because the stand-in is-a `dyn Cart` and upcasts (trait-upcasting 1.86), the holder injects it as the SAME type the target satisfies — Spring's invisibility preserved, autowiring-resolution untouched (the holder's field is still `Arc<dyn Cart>`), no ripple into injection-styles.; The per-call re-resolution is structurally LITERAL: each delegating method awaits `resolve_live` then forwards — the 'resolve the live target per invocation' intent is the method body, plainly visible in `cargo expand` (charter §2.10), the cleanest possible mapping of Spring's 'routes each call to the currently-active scoped instance' (1762).; Origin-agnostic for free: `resolve_live` calls scopes' erased Factory, so the live target may be a native bean, a test double, or a future WASM-module product — the proxy never assumes how the live instance is built, only that scopes can yield one (honors the brief's origin-agnostic mandate by delegating construction entirely to scopes).; Cleanly composes with REAL advice: if the request-scoped target is ALSO @Transactional, the one-proxy-many-concerns model (aop-proxying line 1770) means the SAME proxy chain can carry both the scope-re-resolution step AND the advice chain — scope re-resolution is simply the innermost 'advisor' that produces the target the rest of the chain wraps.

**Cons:** Inherits ALL of aop-proxying's unresolved tensions undiluted: the no-CGLIB/no-trait-less-bean problem (a scoped bean with no service trait cannot get a transparent stand-in), the erased-vs-typed args bridge, and the boxed-future-per-call dyn tax — so scoped-proxies cannot be settled before aop-proxying's proxy-mechanism keystone is, making it the MOST mechanism-coupled realization.; The 'advice' framing is slightly forced: scope re-resolution is not really cross-cutting behavior wrapped AROUND a call — it CHOOSES the receiver. Modeling 'pick the target' as a degenerate around-advisor (one that resolves a target and immediately delegates without a fixed inner) may distort aop-proxying's JoinPoint/proceed contract (whose proceed() assumes a FIXED inner target), forcing a 'late-bound target' variant of the contract that aop-proxying must accommodate.; Per-call resolution adds a `resolve_live` await on EVERY method call on the hot request/prototype path — even for the common request-scoped case where the live instance already exists in the active request store (a cache hit), so the cost is a store lookup + ambient read + (for the dyn path) a boxed future per call — strictly more than a direct Arc deref the holder would have if it weren't scope-mismatched (§2.5 tension, flagged for Phase 2).; Couples to BOTH the still-undecided aop-proxying mechanism AND the still-undecided ambient/async-context model (the proxy must read the active scope from an async-propagated ambient on each call) — two deferred keystones at once.; If aop-proxying's chosen mechanism is the visible-handle variant (Proxied<dyn T>), scoped proxies LOSE their defining transparency (the holder would see `Proxied<dyn Cart>`), which is a worse divergence for scoped proxies than for AOP because the WHOLE point of a scoped proxy is that the holder is scope-agnostic.

**Mechanisms considered:** scoped proxy = aop-proxying's transparent stand-in (generated delegating newtype OR runtime ProxyHandle, per Phase 2) with a single 'resolve-live-then-delegate' step replacing the advice chain; a thin `LiveResolver` seam over scopes' `Scope::get`/`InstanceStore::get_or_init` entry point, passing bean-definition's erased origin-agnostic Factory; late-bound-target variant of aop-proxying's JoinPoint/proceed contract (proceed() resolves the receiver per call rather than dispatching to a fixed inner); proxyMode as bean-definition metadata (TARGET_CLASS/INTERFACES/NO/DEFAULT analogue) selecting 'wrap injection with re-resolving stand-in' vs 'inject raw'; materialization at the post-processor-spi after_init seam, identical to aop-proxying's wrap-and-swap; trait-upcasting (1.86) to inject as dyn Trait; composition with a real advice chain when the scoped target is also advised (scope-resolution as the innermost step of the unified chain)

**T1 — cross-crate/tooling:** Inherits aop-proxying's cross-crate profile wholesale: if the chosen mechanism is a generated stand-in TYPE (aop-proxying R1, fits Architecture C build-time codegen) the scoped proxy is a generated type per scoped+proxied target and must be emitted where the target's service trait is visible; if it is a runtime handle over a metadata method-table (aop-proxying R2, fits Architecture A/B link-time descriptors) the scoped proxy is the SAME handle parameterized to re-resolve. The proxyMode + scope is metadata ON bean-definition (a thin `&'static` descriptor field), discovered via whatever substrate Phase 2 picks (linkme/inventory/codegen/explicit — CANDIDATES). Scoped proxies add NO new discovery surface beyond what scopes (the scope itself) and aop-proxying (the wrap) already need — a scoped proxy is just a bean-definition flag, not a separately-discovered contribution, which SHRINKS the silent-DCE surface (no scoped-proxy-specific descriptor to be DCE'd). The one cross-crate subtlety: the proxy for a request-scoped bean in crate X must call the SCOPE registered (possibly) in crate Y and read the ambient bound (possibly) by a web integration crate Z — so the `LiveResolver` must be origin/crate-agnostic (a runtime call through scopes' registry, not a compile-time link to a specific scope), keeping the deps-point-inward rule (charter §6: core defines the proxy + LiveResolver shape, web crate provides only the ambient binding). The anti-DCE self-check inherits from scopes (an expected scope-contributing crate producing nothing → a scoped proxy resolving under a missing scope → a loud ResolveError, not a silent wrong-instance). Generics: a scoped proxy over `Repo<User>` needs a concrete instantiation (generics-can't-register), macro hard-errors with a per-instantiation escape, inherited from both upstreams.

**T2 — execution context:** Materialization (wrap-and-swap) is cold-path at the holder's after_init inside the holder's per-bean creation guard (no global lock) — the stand-in is built once, cheaply (it captures resolver + key + ambient-accessor, NOT a live target), and the holder is typically a singleton so this happens at refresh. The HOT path is per-method-call on the holder's reference: each call runs `resolve_live(scope, key, ambient).await` then delegates. `resolve_live` runs in the CALLER's execution context (the holder's caller, typically a request-handling task) and is the only async point — it awaits scopes' `Scope::get`/`get_or_init`, which on a request-store HIT is cheap (a keyed lookup) and on a MISS awaits the factory (cold, first-touch in this request). The active-scope lookup MUST read an async-propagated ambient (task-local / structured-scope handle), NOT a ThreadLocal — because the holder's caller migrates across worker threads at every .await and a ThreadLocal active-scope reference would silently resolve the WRONG request's instance or none (this is THE execution-context clash for scoped proxies, §2.3/§2.4, and it is identical to the clash scopes flagged for web-scope binding — the proxy is the per-call CONSUMER of that ambient). Reactive on the hot path (§2.4): the resolution is a notification-driven await of scopes' store, never a poll. Cancellation: if the holder's caller future is dropped mid-call, the in-flight `resolve_live` and the delegated call see standard async cancellation; the proxy holds no resources across the call (it does not own the live target — scopes' store does), so cancellation drops cleanly with nothing to unwind (no async-Drop concern for the proxy itself; teardown of the resolved scoped instance is scopes' ledger's job at scope-end, NOT the proxy's). Errors: a failed `resolve_live` (no active scope, scope torn down, factory failed) surfaces as a rich ResolveError carrying the holder identity + the target's scope + key + what the factory needed (§2.9) — strictly richer than Spring's silent stale-reference or NPE.

#### R2 — Proxy-free: the holder injects a typed re-resolving handle (ScopedRef<dyn T> / a deferral-primitive cousin) — make the per-call re-resolution HONEST in the type, not hidden behind a fake target type
Question whether a transparent proxy OBJECT is needed at all, exploiting the rust-clash insight (from scopes) that the singleton-holds-shorter mismatch is, in Rust, often a near-untypeable error rather than Spring's silent freeze — so the 'inject directly and pretend it's the same type' shortcut Spring tolerates is ALREADY gone, which removes much of the reason to fake the target's type. Instead, the holder declares a VISIBLE re-resolving handle at the injection point: `cart: ScopedRef<dyn Cart>` (not `cart: Arc<dyn Cart>`). `ScopedRef<dyn T>` is a small handle holding `{ resolver, scope_id, key, ambient_accessor }` whose single method `async fn get(&self) -> Result<ScopedGuard<dyn T>, ResolveError>` re-resolves the live target from the active scope on each call; the holder writes `self.cart.get().await?.add(item).await`. This is DIRECTLY the deferral-primitives sibling (ObjectProvider/Provider) specialized to scoped targets — and the spring doc itself (line 2567, 794-796) frames ObjectProvider as giving 'the same lifecycle safety as a scoped proxy but with an honest, debuggable call site instead of hidden bytecode', i.e. Spring documents the deferral handle as the HONEST alternative to the proxy. R2 makes that honest alternative the PRIMARY shape: no proxy type to generate, no after_init wrap-and-swap, no trait-less-bean problem (you never need a stand-in that is-a the trait), no self-invocation subtlety (the holder explicitly calls `.get()`). proxyMode becomes 'INTERFACES/TARGET_CLASS' → 'inject a ScopedRef (or the R1 transparent proxy if transparency is required)' vs 'NO' → 'inject raw (and fail-fast if scope-mismatched)'. The substrate is shared with deferral-primitives: `ScopedRef<T>` is the same live-handle machinery as `ObjectProvider<T>`, just bound to a specific scope + key at the injection point rather than re-querying the whole registry.

**Pros:** Zero proxy codegen and zero runtime type-synthesis: no stand-in type that must be-a the target trait, so it COMPLETELY sidesteps aop-proxying's hardest clashes — the no-CGLIB / no-trait-less-bean problem, the erased-vs-typed args bridge (the holder calls typed methods on the resolved guard directly, no chain erasure), and the per-target codegen volume; the least mechanism-coupled of the three.; Maximal honesty and best diagnostics (§2.9, DX #1 in its truth-telling sense): the per-call re-resolution is VISIBLE in the type (`ScopedRef<dyn Cart>`) and at the call site (`.get().await?`), so the holder author SEES that this dependency is re-resolved per use and that resolution can FAIL — exactly the 'honest, debuggable call site' the spring doc prefers (2567); a stale/missing scope is a typed `Result`, not a silent wrong instance.; Unifies hard with deferral-primitives (charter §2.11): `ScopedRef` and `ObjectProvider` are the same live-re-resolving-handle primitive, so leaf builds ONE deferral substrate serving both 'optional/multiple/lazy registry lookup' and 'scoped per-call re-resolution' — no proxy machine needed at all for the scope-mismatch cure.; Cancellation/ownership is trivial and async-clean: `ScopedRef` owns no live target (it holds a resolver + key), `get()` yields a `ScopedGuard` whose lifetime is the call; dropping it mid-call is standard async cancellation with nothing to unwind, and the scoped instance's teardown stays entirely scopes' concern — no async-Drop pressure anywhere on the proxy side.; Aligns with the AOT/build-time lineage and the upstream scopes design's fail-fast wiring diagnostic: because the deferral is in the TYPE, a build-time or startup pass can SEE that a singleton holds a `ScopedRef<RequestScoped>` (correct) vs a raw `Arc<RequestScoped>` (the mismatch) and validate it before any request runs.

**Cons:** DIVERGES from Spring's defining intent: the WHOLE point of a scoped proxy is that the holder is scope-AGNOSTIC and codes against the target's plain type as if it were a normal collaborator — R2 makes the holder scope-AWARE (it must name `ScopedRef` and call `.get().await?`), which is the exact 'leak scope-lookup logic into business code' that Spring's proxy intent (854) sets out to PREVENT. This is the sharpest parity-of-feel break of the three.; Changes the holder's injection-point type and call sites, rippling through autowiring-resolution (it must resolve `ScopedRef<dyn T>` as a deferral target, like ObjectProvider) and into every method body that uses the dependency (`.get().await?.method()` instead of `.method()`) — more ceremony at every use, cutting against low-ceremony DX.; Provides NO migration path for code that genuinely needs the transparent shape (e.g. handing a request-scoped collaborator to a third-party trait that expects `Arc<dyn Cart>`): R2 alone cannot satisfy an API that wants the bare trait object, so it must fall back to R1's transparent proxy for that case — meaning R2 is incomplete as a SOLE realization and really wants R1 as a companion for the transparency-required path.; The `.get().await?` per call is two awaits deep on the hot path (resolve, then the actual method) and the fallibility is in the caller's face on every use — ergonomically heavier than R1's hidden re-resolution for the common 'I just want to call a method' case (DX cost weighed against the honesty benefit).; Re-resolution-per-call semantics differ subtly from Spring's: Spring's proxy re-resolves per METHOD CALL transparently; R2 re-resolves per `.get()` and the holder might cache a `ScopedGuard` across several calls (correct within one scope activation, a bug if held across scope boundaries) — so R2 must document the guard's scope-bounded lifetime, a discipline Spring's transparent proxy didn't expose to the user.

**Mechanisms considered:** a visible `ScopedRef<dyn T>` injection-point handle (deferral-primitives cousin) with `async fn get() -> Result<ScopedGuard<dyn T>, ResolveError>` re-resolving from the active scope per call; shared live-handle substrate with deferral-primitives' ObjectProvider/Provider — ScopedRef = ObjectProvider bound to a specific scope+key at the injection point; proxyMode mapped to 'inject ScopedRef' (honest) vs 'inject R1 transparent proxy' (when the bare trait type is required) vs 'NO' (inject raw + fail-fast on mismatch); ScopedGuard with a call-bounded (or scope-activation-bounded) lifetime borrowing from scopes' active store; no ownership of the live target by the handle; build-time / startup fail-fast validation that a singleton holds a ScopedRef (correct) not a raw shorter-scoped Arc (the mismatch), leveraging the type being explicit; autowiring-resolution treats ScopedRef<dyn T> as a deferral target (resolved like ObjectProvider, not as the bean itself)

**T1 — cross-crate/tooling:** Lightest cross-crate footprint of the three: NO proxy type and NO after_init wrap-and-swap, so no generated-type force-link concern and no proxy-specific descriptor to be DCE'd. `ScopedRef<dyn T>` is a plain library type in leaf-core (the same crate hosting the deferral-primitives handle), parameterized at the holder's injection point — its construction is wiring code emitted where the holder is wired, referencing scopes' registry by identity at runtime. The proxyMode metadata still rides bean-definition (a thin descriptor field discovered via the Phase-2 substrate — CANDIDATE). The cross-crate subtlety is the SAME as R1: the ScopedRef's `get()` calls a scope registered possibly in another crate and reads an ambient bound possibly by a web crate, so the resolver path is a runtime call through scopes' registry (origin-agnostic, crate-agnostic), keeping deps-point-inward (core defines ScopedRef + the resolver shape; web crate provides only the ambient binding). Because R2 changes the holder's PUBLIC injection-point type to `ScopedRef<dyn T>`, consumers in OTHER crates that wire such a holder must name `ScopedRef` — a cross-crate API-surface coupling (mild, since ScopedRef is a leaf-core type on the umbrella's stable path). Generics inherit the can't-register hard-error (a `ScopedRef<Repo<User>>` needs the concrete `Repo<User>` scoped registration). No new anti-DCE surface beyond scopes'.

**T2 — execution context:** No materialization phase to speak of — the `ScopedRef` is constructed as part of the holder's normal wiring (cold-path, at the holder's creation, capturing resolver + scope_id + key + ambient-accessor; holds no live target). The HOT path is the holder calling `self.cart.get().await?` then a method on the guard — `get()` runs in the caller's execution context and awaits scopes' `Scope::get`/`get_or_init` against the ACTIVE scope read from the async-propagated ambient (task-local/structured-scope, NOT a ThreadLocal — same §2.3/§2.4 clash and same resolution as R1; the ambient is read at `get()` time and must be the current request/session for the serving task even after thread-hops). Store HIT is cheap; MISS awaits the factory (cold, first-touch this scope). Reactive on the hot path (notification-driven await, never poll, §2.4). Cancellation: dropping the holder's future mid-`get()` is standard async cancellation; the `ScopedGuard` borrows from scopes' store and owns no teardown — scope-end teardown is scopes' ledger's job, so the proxy side has NO async-Drop concern. The fallibility is explicit: `get()` returns a `Result` so 'no active scope' / 'scope torn down mid-call' is a typed error the holder handles (§2.9), and because it is in the holder's face, the holder can choose its own recovery — strictly more controllable than a transparent proxy that must decide error behavior itself. Cross-task hops: if the holder hands work to an executor (task-execution/async-execution), the moved work's `ScopedRef.get()` finds NO active scope unless context-propagation moved the ambient — an explicit seam (the same one scopes flagged), and BECAUSE the handle is visible, the failure is a legible ResolveError at the moved call site rather than a mysterious wrong-instance.

#### R3 — Late-bound provider indirection: the proxy is a thin newtype over an origin-agnostic LiveTargetProvider, decoupling 'what re-resolves per call' from 'how the stand-in dispatches' — a two-piece design that can present EITHER a transparent face (R1) or an honest handle (R2) over ONE re-resolution core
Decompose the scoped proxy into the two orthogonal things it actually does, so the re-resolution CORE is shared and the PRESENTATION (transparent vs honest) is a swappable facade — directly answering the upstream aop-proxying phase2Question 'transparent dyn Trait vs visible handle' WITHOUT picking one for scoped proxies. Piece 1, the CORE: an origin-agnostic `LiveTargetProvider<T>` — `async fn current(&self, ambient: &Ambient) -> Result<TargetRef<T>, ResolveError>` — that, given the active ambient, yields the live scoped target for one call. It is built ONCE per (scope_id, key) and holds no instance; internally it just calls scopes' `Scope::get`/`InstanceStore::get_or_init` with bean-definition's erased Factory, so it is the SINGLE place per-call re-resolution lives, reused by BOTH the transparent and the honest face, AND reusable by deferral-primitives (an ObjectProvider for a scoped bean IS a LiveTargetProvider with a registry-wide key) and by aop-proxying (a late-bound advisee). Piece 2, the FACADE: (a) a TRANSPARENT facade — a generated/handle stand-in (aop-proxying's substrate) whose every method does `self.provider.current(ambient).await?.method(args).await`, injected as `dyn T` (Spring-faithful); OR (b) an HONEST facade — `ScopedRef<dyn T>` (R2) wrapping the same provider, exposing `.get()`; OR (c) NO facade — the provider itself injected for advanced users. proxyMode chooses the facade (TARGET_CLASS/INTERFACES → transparent stand-in; a new leaf-native mode → honest ScopedRef; NO → raw+fail-fast). The KEY divergence from R1: R1 fuses re-resolution INTO the proxy's call-routing (scope-lookup as degenerate advice); R3 keeps re-resolution as a SEPARATE provider the facade merely calls — so the transparent facade does not need a 'late-bound-target' variant of aop-proxying's proceed() contract (it just calls `provider.current()` then dispatches normally), keeping aop-proxying's JoinPoint/proceed contract clean and un-distorted, and letting a transparent scoped proxy that is ALSO advised compose as 'provider.current() yields the target, THEN the normal advice chain wraps it'.

**Pros:** Refuses the transparent-vs-honest dilemma at Phase 1 (correct for an implementation-agnostic phase): ONE re-resolution core, TWO facades chosen by proxyMode, so Phase 2/3 can pick the default presentation (or offer both) without redesigning the core — strictly more optionality than R1 (transparent-only) or R2 (honest-only).; Cleanest unification across THREE features (charter §2.11): `LiveTargetProvider` is reused by scoped-proxies (both facades), deferral-primitives (ObjectProvider over a scoped bean), AND aop-proxying (late-bound advisee) — the per-call-re-resolve-the-live-target primitive is built once and serves the whole scope-mismatch + deferral + late-binding family, the maximal-shared-machinery the charter demands.; Keeps aop-proxying's JoinPoint/proceed contract UNDISTORTED (unlike R1): re-resolution is a separate `provider.current()` call the facade makes BEFORE normal dispatch, not a degenerate advisor that bends proceed()'s fixed-inner assumption — so a scoped+advised bean composes as 'resolve target, then run the unchanged advice chain on it', the most honest composition.; Origin-agnostic by construction at the core (the brief's mandate): `LiveTargetProvider::current` delegates entirely to scopes' erased Factory, so native / test-double / WASM-module live targets all re-resolve through the same provider — and because the provider is the ONLY origin-touching piece, the facades stay completely construction-agnostic.; Best diagnostics surface (§2.9): the provider is the single chokepoint where a re-resolution can fail, so rich ResolveError context (holder, scope, key, ambient state, what the factory needed) is produced in ONE place and surfaces identically through whichever facade — no duplicated error logic between a transparent and an honest path.

**Cons:** Most machinery of the three (core + two facades + the proxyMode→facade mapping) — a coherence burden: the transparent and honest facades must produce IDENTICAL re-resolution + error + cancellation semantics, and proxyMode now selects among MORE shapes than Spring's enum, so the metadata surface and the docs are wider (a judgment call on whether the optionality earns its complexity).; The transparent facade STILL needs aop-proxying's stand-in mechanism (generated type or runtime handle) and thus STILL inherits the no-trait-less-bean / erased-args / boxed-future clashes for the transparent path — R3 isolates those to the transparent facade but does not eliminate them, so it is not strictly simpler than R1 where transparency is required.; Two-piece indirection (facade → provider → scopes' store) is one hop deeper than R1's fused design on the hot path; the extra `provider.current()` indirection must be made allocation-free/inlinable for the transparent facade or it regresses §2.5 vs R1's direct delegation (likely fine since it is a method call into a held provider, but it is a real extra layer to keep cheap).; Introduces a leaf-native proxyMode (the 'honest ScopedRef' facade) with NO Spring analogue — a divergence from the ScopedProxyMode enum parity (TARGET_CLASS/INTERFACES/NO/DEFAULT) that must be justified as an honest-Rust improvement rather than gratuitous; risks confusing porting Spring users who expect the four modes.; Leans on BOTH deferred keystones (aop-proxying mechanism for the transparent facade AND the ambient/async-context model for `current()`) plus the scope-instance ownership model (what does `TargetRef<T>` hand out — Arc/borrow/guard) — so while it isolates the unknowns to the right pieces, it still cannot be FULLY settled in Phase 1 (correctly surfaced as phase2Questions).

**Mechanisms considered:** a shared origin-agnostic `LiveTargetProvider<T>` core (`async fn current(ambient) -> Result<TargetRef<T>, ResolveError>`) over scopes' Scope::get/InstanceStore::get_or_init + bean-definition's erased Factory; two swappable facades over the core: transparent stand-in (aop-proxying substrate, injected as dyn T) and honest ScopedRef<dyn T> (R2), plus a raw-provider escape; proxyMode mapped to facade choice (TARGET_CLASS/INTERFACES→transparent; leaf-native mode→honest; NO→raw+fail-fast); provider reused by deferral-primitives (ObjectProvider over a scoped bean) and aop-proxying (late-bound advisee), composing 'current() yields target THEN advice chain wraps it'; single ResolveError chokepoint in the provider, surfaced identically through any facade; TargetRef<T> as the per-call yield whose ownership shape (Arc / borrow / scope-bounded guard) is deferred to the scope-instance ownership keystone

**T1 — cross-crate/tooling:** Combines R1 and R2's profiles cleanly because the pieces map to different substrates: the `LiveTargetProvider` core is a plain leaf-core library type (no discovery, no DCE risk — like R2's ScopedRef), while the TRANSPARENT facade inherits aop-proxying's mechanism-dependent cross-crate profile (generated type → Architecture C; runtime handle → Architecture A/B) and the HONEST facade is a plain leaf-core type (R2's profile). proxyMode + scope ride bean-definition descriptors via the Phase-2 substrate (CANDIDATE). No scoped-proxy-specific discovery contribution (it is bean-definition metadata, like R1/R2), so no added silent-DCE surface; the anti-DCE self-check inherits from scopes (missing scope crate → loud ResolveError). The core's origin-agnosticism is the strongest of the three (the provider is the SOLE construction-touching piece, delegating to scopes' Factory), so a future dynamic WASM-module source slots in behind `current()` with zero facade changes — the two-source duality (static link-time scope registry as default 'classpath' PLUS optional dynamic directory-scanned source) is captured as 'the provider's `current()` calls whichever scope/store scopes resolved, regardless of origin'. Deps point inward: core defines provider + both facade shapes; web crate provides only the ambient binding; scope crates register scopes. Generics: per-instantiation registration hard-error inherited.

**T2 — execution context:** The provider core is built cold (at holder wiring, capturing resolver + scope_id + key + ambient-accessor, no live target). HOT path: the facade calls `provider.current(ambient).await?` then dispatches — `current()` runs in the caller's execution context, reads the ACTIVE scope from the async-propagated ambient (task-local/structured-scope, NOT ThreadLocal — same §2.3/§2.4 clash, same resolution as R1/R2; ambient read per call so thread-hops are tolerated), and awaits scopes' store (HIT cheap, MISS awaits factory cold). The transparent facade hides this await inside each method (Spring-faithful); the honest facade exposes it as `.get().await?`. Reactive on the hot path (§2.4, notification-driven await of scopes' store, never poll). Cancellation: dropping the caller future mid-`current()` is standard async cancellation; the provider and facades own NO live target (scopes' store does), so nothing to unwind — no async-Drop concern on the proxy side (teardown of the resolved scoped instance is scopes' ledger at scope-end). The single ResolveError chokepoint in `current()` produces rich context (holder, scope, key, ambient state, factory expectation, §2.9) surfaced identically through either facade. Cross-task hops: `current()` on moved work finds no active scope unless context-propagation moved the ambient — explicit seam (shared with scopes/task-execution); the honest facade makes this a legible error at the moved call site, the transparent facade surfaces it as a thrown ResolveError from the method — both honest, neither a silent wrong-instance.

### Rust clashes
- ThreadLocal-bound active-scope lookup vs .await/thread-hops — THE clash for this feature. Spring's scoped proxy resolves the live target by reading the active request/session from a ThreadLocal (RequestContextHolder), exploiting the servlet invariant that a request stays on one thread (spring doc 2091-2112). In async Rust the holder's caller (a request-handling task) migrates across worker threads at every .await, so a ThreadLocal active-scope reference read PER CALL would silently resolve the wrong request's instance, or none — a silent correctness bug, the exact failure Spring itself flags for virtual threads (2162-2164). leaf's scoped proxy MUST read the active scope from an async-propagated ambient (task-local / structured-concurrency scope value), integrated with context-propagation's capture/restore at every task hop (2120-2164) — never a raw ThreadLocal. The proxy is the per-call CONSUMER of the same ambient scopes binds; this is unresolvable in Phase 1 because it depends on the deferred async-context model.
- No CGLIB / no JDK dynamic proxy for the TRANSPARENT face — inherited from aop-proxying. Rust cannot synthesize at runtime a type that is-a the target's trait. So a transparent scoped proxy (the Spring default for @RequestScope etc.) must be EITHER a compile-time-generated delegating type OR a runtime handle over a metadata method-table (aop-proxying's mechanisms), and a scoped bean with NO service trait cannot get a transparent stand-in at all (Spring's CGLIB-subclass-a-concrete-class case has no Rust equivalent). This forces leaf either to require scoped+proxied beans to expose a service trait, or to offer the honest-handle face (R2/R3) where transparency is impossible — a real divergence Spring did not face.
- The singleton-holds-shorter mismatch is near-untypeable in Rust rather than a silent freeze — an ADVANTAGE that reshapes the feature. In Spring, injecting a session-scoped bean directly into a singleton compiles and silently freezes one instance (the latent bug the proxy cures). In Rust a singleton holding a direct Arc<RequestScopedBean> is often un-typeable (the request bean has no stable identity outside a request) or a loud wiring error — so the 'inject directly and it silently breaks' footgun is mostly disarmed, which means the scoped proxy and the deferral handle (deferral-primitives) are LOAD-BEARING, first-class cures rather than optional niceties, and the honest-handle shape (R2) is more defensible in Rust than in Spring because the mismatch is already not silent.
- No async Drop for teardown of the per-call-resolved target — but the proxy SIDESTEPS it. The resolved scoped instance needs eventual destruction (request/session teardown), and Rust has no async Drop (nightly doc 273-281, a first-class constraint). Crucially the proxy/handle must NOT own the live target's teardown: it holds a resolver + key, not the instance — so teardown stays entirely scopes' container-driven async ledger at scope-end, and the proxy has nothing to unwind on cancellation. The clash is real for scopes; the design discipline for scoped-proxies is to keep the proxy a non-owning re-resolver so it inherits scopes' teardown rather than reimplementing it (all three realizations encode this: the proxy never caches the live instance across calls/scopes).
- Per-call boxed-future tax on the transparent face — inherited from aop-proxying's dyn chain. A transparent stand-in that is-a dyn T must, at the dyn boundary, return boxed futures (AFIT/RPITIT not dyn-compatible, nightly doc 160-182, 260), so each delegated call boxes a future PLUS the re-resolution await — strictly more per-call cost than a direct Arc deref the holder would have absent the scope mismatch. Acceptable on the request hot path (a request already costs allocations) but a §2.5 concern the transparent face owns; the honest face (R2) avoids the proxy-dispatch box (the holder calls the resolved guard's methods directly, monomorphized) at the cost of visible .get().await? ceremony.
- proxyMode enum parity vs Rust's transparent/honest split. Spring's ScopedProxyMode is TARGET_CLASS (CGLIB) / INTERFACES (JDK) / NO / DEFAULT — both proxy modes assume a transparent stand-in differing only in JVM synthesis mechanism, which Rust lacks. leaf cannot map TARGET_CLASS vs INTERFACES to a meaningful Rust distinction (there is no class-vs-interface proxy choice), so the proxyMode metadata must be RE-INTERPRETED: 'produce a transparent stand-in (when a trait exists)' vs 'NO' (inject raw + fail-fast), with an optional leaf-native 'honest handle' mode (R2/R3) that has no Spring analogue — an intent-not-mechanism re-expression that must be justified, not a 1:1 enum port.
- Re-resolution-per-call semantics depend on the scope-instance ownership model (the deferred keystone, charter §3, viewed through the proxy). Spring's proxy re-resolves and the GC handles the rest; in Rust, what does each call's resolution HAND OUT — an Arc<dyn T> clone (cheap, shareable, but the holder could stash it and defeat re-resolution), a borrow/guard bound to the call (prevents stashing, enforces re-resolution, but lifetime-entangles the call), or a store index? The transparent face wants something dispatchable per method; the honest face yields a guard whose lifetime must be call-or-scope-bounded so the holder can't accidentally cache a stale target. This is unresolvable here and dictates the per-call yield type across all three realizations.

### Interactions
- **scopes** — TIGHTEST UPSTREAM. scopes defines the multiplicity/lifetime AND the per-call resolution entry point (Scope::get / InstanceStore::get_or_init over bean-definition's erased Factory); scoped-proxies is THE sanctioned cure for the singleton-holds-shorter gotcha scopes defines, providing the per-call indirection that CALLS that entry point. The proxy holds the scope identity + target key and re-resolves the live instance from scopes' active store on each call. The proxyMode metadata pairs a scope with a proxy. scoped-proxies adds no instance storage — scopes owns the store; the proxy is a non-owning re-resolver, so the live target's teardown stays scopes' ledger's job.
- **deferral-primitives** — SIBLING CURE + shared substrate. The other sanctioned fix for the scope-mismatch gotcha is an ObjectProvider/Provider-analogue (a live handle re-resolving per call). The spring doc explicitly frames ObjectProvider as 'the same lifecycle safety as a scoped proxy but with an honest, debuggable call site instead of hidden bytecode' (2567). R2 IS this — a ScopedRef<dyn T> that is an ObjectProvider bound to a specific scope+key; R3 shares one LiveTargetProvider core between scoped proxies and ObjectProvider. The two features want a UNIFIED live-re-resolving-handle primitive: the proxy is the transparent face, the deferral handle the honest face, over one re-resolution core.
- **aop-proxying** — SHARED PROXY SUBSTRATE (the transparent face). aop-proxying owns the transparent stand-in: wrap-and-swap at the post-processor-spi after_init seam, the call-routing primitive, trait-upcasting injection, the self-invocation boundary. A transparent scoped proxy IS that stand-in whose per-call step is scope re-resolution instead of an advice chain (R1 fuses re-resolution into the routing; R3 keeps re-resolution as a separate provider the facade calls before normal dispatch, leaving aop-proxying's JoinPoint/proceed contract undistorted). The upstream aop-proxying design flagged scoped-proxies as a sibling and 'a strong argument for unifying the wrap primitive' — this design honors that. A scoped+advised bean composes as one proxy: resolve the live target, then run the advice chain on it.
- **autowiring-resolution** — How the proxy reaches the holder. For the transparent face (R1/R3a) the stand-in is-a dyn T and upcasts, so autowiring injects it wherever the target's type is wanted, transparently — autowiring-resolution untouched (Spring's invisibility). For the honest face (R2/R3b) the holder's injection point is ScopedRef<dyn T>, which autowiring-resolution must resolve as a DEFERRAL target (like ObjectProvider, not as the bean itself) — a divergence autowiring-resolution must account for. The transparent-vs-honest choice ripples directly through this feature.
- **context-propagation** — CRITICAL ENABLER. The proxy's per-call active-scope lookup must read an async-propagated ambient that survives .await and thread/task hops — exactly context-propagation's capture/restore SPI job (the rust-native replacement for Spring's RequestContextHolder ThreadLocal, 2120-2164). When the holder hands work to an executor (task-execution/async-execution), context-propagation must move the ambient or the proxy's next .current()/.get() finds no active scope. This is the per-call CONSUMER side of the same ambient scopes binds; the deepest dependency.
- **locale-context** — Sibling ambient. 'The current request/session' the scoped proxy keys its per-call resolution on is the same per-execution ambient locale-context exposes (current locale/request attributes via thread-bound holders in Spring, 2093-2112). Both are inputs context-propagation moves across boundaries; the proxy reads the scope-store ambient, locale-context reads the locale ambient, from one async-context substrate.
- **bean-definition** — UPSTREAM metadata carrier. proxyMode (the TARGET_CLASS/INTERFACES/NO/DEFAULT analogue, re-interpreted as transparent/honest/raw) is metadata ON the definition alongside scope (child-wins on merge, like scope). bean-definition's origin-agnostic erased Factory is the construction recipe the proxy's re-resolution (via scopes) awaits — the proxy never assumes native construction.
- **bean-instantiation** — Drives transparent-proxy materialization. For the transparent face, the wrap-and-swap runs at the holder's after_init (the proxy is built once, capturing resolver+key+ambient-accessor, NOT a live target) inside the holder's per-bean creation guard. bean-instantiation publishes the proxy as the holder's collaborator. The honest face (R2) needs no materialization phase — the ScopedRef is part of normal wiring.
- **circular-references** — Early-reference integration (transparent face only). If a scoped proxy must be exposed early into a cycle (getEarlyBeanReference analogue, spring 1766-1768, 952-954), the early reference handed to mid-cycle collaborators must be the proxy. Since a scoped proxy holds no live target (only a resolver), materializing it early is trivial (no not-yet-initialized target to wrap, unlike an AOP early-proxy over a partially-built bean) — the proxy is inherently early-safe. The honest face (R2) sidesteps cycles entirely (it is a plain handle).
- **concurrency-contract** — Realizes scope as the concurrency lever across the lifetime gap. concurrency-contract frames narrowing scope (singleton→request) as the answer to mutable per-interaction state; the scoped proxy is what makes that narrowing USABLE from a singleton holder without the holder reverting to a frozen reference — it preserves the narrower scope's per-interaction freshness while the holder stays a safely-published singleton. The proxy itself must be safely shareable (it is a singleton-shaped non-owning re-resolver).
- **lifecycle-callbacks** — Teardown gating delegated, not owned. The proxy never triggers destruction callbacks — the live targets it resolves are torn down by scopes at scope-end (request/session) or never (prototype), per scopes' ledger. lifecycle-callbacks defines init/destroy ordering; scoped-proxies must NOT own any of it (keeping the proxy a non-owning re-resolver is what makes prototype's hand-off-and-forget and request/session teardown stay scopes' concern, sidestepping the no-async-Drop problem on the proxy side).
- **transaction-management** — Composes within one proxy chain. A request-scoped bean that is also @Transactional gets ONE proxy: in R1 scope-re-resolution is the innermost step and the tx interceptor wraps it; in R3 the LiveTargetProvider yields the target THEN the unchanged advice chain (incl. tx) wraps it. Ordering matters (one-proxy-many-concerns, spring 1770-1772) — scope resolution must happen before advice runs so advice wraps the live target, not a stale one.
- **container-hierarchy** — Scope-resolution must respect hierarchy. The proxy's per-call resolution goes through scopes/bean-registry's resolution path, which is hierarchy-aware (child sees parent). A scoped proxy in a child context resolving a scope/target visible only in the parent must resolve upward consistently — the proxy delegates this to the resolution path it calls, not reimplementing lookup.

### Open questions for Phase 2
- Transparent stand-in vs honest re-resolving handle as the DEFAULT (or offer both): R1/R3a inject a transparent dyn-T proxy (Spring-faithful invisibility, holder is scope-agnostic, but inherits aop-proxying's no-trait-less-bean / boxed-future clashes); R2/R3b inject a visible ScopedRef<dyn T> (honest, debuggable, zero proxy codegen, the spring-documented honest alternative — but the holder becomes scope-AWARE, the exact 'leak scope-lookup into business code' Spring's proxy intent prevents). This is the DX keystone for this feature and ripples through autowiring-resolution and every holder call site; it interacts with whether the no-trait-less-bean problem makes transparency impossible for some beans. Held unresolved.
- What is the rust-native ambient the proxy reads per call to find the ACTIVE scope — a tokio task-local, a structured-concurrency scope value, an explicit leaf context object, or a context-propagation capture/restore handle? SHARED with scopes (which binds it), context-propagation, locale-context, task-execution. MUST be resolved before the per-call resolution can be designed; it determines whether ThreadLocal is truly banished and whether the proxy resolves the correct request's instance after .await/spawn/cancellation. The single deepest unknown, identical to scopes' web-scope-binding keystone.
- The scope-instance ownership model viewed through per-call yield (charter §3 keystone): what does each re-resolution HAND OUT — an Arc<dyn T> clone (cheap, shareable, but the holder could stash it and silently defeat re-resolution), a call-or-scope-bounded borrow/guard (enforces re-resolution, prevents staleness, but lifetime-entangles the call), or a store handle/index? Dictates the per-call API of all three realizations and whether a holder can accidentally cache a stale target across scope boundaries. Unresolvable per-realization.
- Does leaf inherit aop-proxying's keystone (proxy mechanism: generated type vs runtime handle vs resolution-seam middleware) for the TRANSPARENT scoped-proxy face, or can scoped proxies be settled INDEPENDENTLY via the honest handle (R2) that needs no proxy mechanism at all? If transparency is required for any bean, scoped-proxies is gated on aop-proxying's mechanism choice; if the honest handle is acceptable as the default, scoped-proxies decouples from it. The dependency direction between these two features' Phase-2 resolutions is itself open.
- How is proxyMode's ScopedProxyMode enum (TARGET_CLASS/INTERFACES/NO/DEFAULT) re-expressed in Rust given there is no class-vs-interface proxy distinction? Candidates: collapse TARGET_CLASS+INTERFACES into one 'transparent' mode + NO + a leaf-native 'honest handle' mode; or keep the four for porting familiarity and document the mechanism re-mapping. An intent-not-mechanism call that interacts with the transparent-vs-honest default and with parity-of-feel for porting Spring users.
- Where is the singleton-holds-shorter MISMATCH detected and how is the cure prescribed — build-time static analysis (possible if the graph is AOT-analyzed and the holder's type reveals a raw shorter-scoped Arc, the §3 compile-vs-runtime split), a startup validation pass, or first-resolution runtime? Because the honest face makes the mismatch TYPE-visible, R2/R3 enable richer/earlier detection; couples to autowiring-resolution and to how much graph is frozen at build time. Decides whether the gotcha is caught with the richest diagnostic before any request runs.
- Teardown-on-cancellation interaction (delegated but must be verified): the proxy is designed to be non-owning so teardown stays scopes' ledger — but when the holder's serving task is cancelled mid-call, is there any window where a freshly-resolved-but-not-yet-used scoped instance is created and then orphaned (a store entry with no corresponding teardown)? Shared with scopes' teardown-on-cancellation question and the async-context model; leaving it open risks scope-store leaks on the common cancellation path even though the proxy itself owns nothing.

## Proxy-Based AOP Substrate  `[aop-proxying]`

This feature is the SUBSTRATE for all declarative interception: the mechanism by which a target bean is wrapped in a stand-in object that intercepts external method calls, routes them through an ordered interceptor chain (before/around/after the real method), and is returned IN PLACE OF the target at the DI seam — so every collaborator that injects the bean gets the proxy and DI never sees the raw target. Spring's load-bearing intent (lines 1633-1639, 1693-1699): advice is something the container ADDS AROUND a bean when it hands it to collaborators, NOT something compiled into the class; it attaches at the seam between beans (the DI-injected reference), not to internal control flow. The mechanics to replicate at the SUBSTRATE level: (1) proxying is realized by returning a DIFFERENT object from an instance-decoration hook (the post-processor-spi after_init seam from upstream — 'a BPP returns a different object IS how AOP swaps in a proxy', lines 918, 1697), so there is no separate proxy-registration step and proxying composes with all other instance decoration automatically; (2) the proxy is transparent to callers — they program against the bean's type/contract and never know they hold a wrapper; (3) the self-invocation boundary (lines 1714-1729) is the HONEST consequence of the wrapper model — `this.method()` from inside the target bypasses the proxy, because advice lives on the external reference, not in the target's own control flow (leaf must reproduce this boundary honestly, not paper over it); (4) one proxy carries the UNION of all matching advice in one ordered chain (line 1770-1772 — that union/ordering is auto-proxy-creator's job; THIS feature only provides the wrap-and-swap seam and the call-routing primitive); (5) early-proxy exposure into cycles (getEarlyBeanReference, lines 1766-1768, 777) — if a cycle forces an early reference, that reference must be the proxy, or some collaborators bypass advice; (6) a proxy-configuration surface (proxy-target-class / @Proxyable / context-wide ProxyConfig, lines 1655-1691) controlling HOW the stand-in is shaped (which exposed surface). I must replicate the INTENT (a transparent, non-invasive, DI-seam-attached interception wrapper that is just another lifecycle decoration) WITHOUT importing the JVM mechanisms it sheds: there is no CGLIB runtime subclassing and no JDK dynamic-proxy (java.lang.reflect.Proxy) in Rust — Rust cannot synthesize, at runtime, a NEW type that implements arbitrary traits and overrides methods. The proxy 'identity' must be an origin-agnostic erased handle (so a WASM-module-sourced bean or a test double proxies identically), the call-routing primitive must be defined WITHOUT prejudging how the stand-in is generated, and the proxy/early-reference mechanism is the explicit cross-cutting question this feature surfaces but does NOT resolve. I do NOT pick the codegen-vs-runtime split, the scope/ownership model the proxy handle rides on, the registry substrate, or whether the proxy is a generated type, a vtable shim, or a trait-object wrapper — those are Phase 2.

### Candidate realizations

#### R1 — Per-target generated delegating newtype (compile-time 'CGLIB analogue'): one stand-in struct per proxied bean type, emitting the same delegation a user could hand-write
Realize the proxy as a CONCRETE, per-target stand-in struct GENERATED at build/codegen time — the rust-native analogue of CGLIB's runtime subclass, moved to compile time (charter §3 metadata/codegen lineage; thin-macro §2.10 — the macro emits ONLY what a user could hand-write). For a bean type `OrderSvc` exposing service trait `dyn Order`, leaf-codegen emits `struct OrderSvcProxy { target: <handle to OrderSvc>, chain: Arc<AdviceChain> }` and `impl Order for OrderSvcProxy { async fn place(&self, a: Args) -> R { self.chain.invoke(JoinPoint::new::<{place}>(&self.target, a, |t,a| t.place(a)).await } ... }` — one delegating method per advisable method, each routing through the interceptor chain then calling the real method via a captured tail closure. The substrate this feature OWNS is the call-routing primitive: `trait Interceptor { async fn invoke(&self, jp: &mut JoinPoint<'_>) -> Result<Outcome, AdviceError>; }` and `JoinPoint` carrying the target handle, the typed-but-erased args, the method identity, and a `proceed()` that advances the chain (next interceptor, or finally the real method). The proxy struct is just the thin generated delegation; the chain/JoinPoint/Interceptor is hand-writable library code. Wrap-and-swap happens at the instance-decoration seam (post-processor-spi after_init): an InstancePostProcessor — installed by auto-proxy-creator — receives the ErasedBean for `OrderSvc`, asks 'does any advice match this type?', and if so returns the generated `OrderSvcProxy` ErasedBean in its place. Because the proxy `impl Order`s, it upcasts to `Arc<dyn Order>` (trait-upcasting, stable 1.86) and is injected wherever `dyn Order` is wanted — DI never sees the raw target. Self-invocation falls out honestly: the target's own `self.place()` calls its inherent method, never the proxy — exactly Spring's boundary, for the same reason (the wrapper is external). Proxy shape (@Proxyable / proxy-target-class analogue) becomes 'which trait(s) the generated proxy implements' — a codegen input: expose all service traits, or a chosen subset (the v7 `interfaces=` encapsulation control, line 1685). The early-reference hook is a generated `early_proxy()` that builds the same stand-in around a not-yet-initialized target handle.

**Pros:** Zero runtime type-synthesis and the closest honest analogue to Spring's intent: a real stand-in OBJECT that IS-A `dyn Order`, injected transparently — DI genuinely never sees the raw target, and self-invocation is honestly the same boundary for the same structural reason (advice is external to the target's control flow).; Lowest steady-state overhead of the three (§2 low-overhead #2): the delegation is monomorphized generated code, the only indirection is the chain walk a user opted into; no per-call reflection, no per-call boxing of the dispatch itself (only the boxed-future tax inherent to any async dyn chain). The hot per-call path is just 'walk N interceptors then call the real method'.; Maximally thin-macro / hand-writable (§2.10): the generated proxy is literally the delegating impl a developer could write by hand; `cargo expand` shows an ordinary newtype + trait impl + chain walk — nothing hidden, escape hatch is 'write the proxy yourself'.; The call-routing primitive (Interceptor / JoinPoint / proceed) is the clean, reusable substrate every declarative feature (transaction-management, caching, validation, async-execution, retry-resilience, exception-translation) composes onto as one ordered chain — exactly the 'one proxy, many concerns' union (line 1770), with this feature owning only the seam and the chain, not the advice.; Origin-agnostic at the handle level: the generated proxy wraps an ErasedBean target handle, so a WASM-module-sourced target or a test double is wrapped identically; the proxy doesn't assume native construction of its target.; Stable-buildable: relies only on trait-upcasting (1.86), AFIT/RPITIT (1.75), and codegen — no nightly. The proxy is a normal struct, so it carries Send/Sync exactly when the target+chain do (no special vtable surgery).

**Cons:** Requires per-method delegation codegen for every advisable method of every potentially-proxied type — codegen volume, and it pre-judges the metadata/codegen boundary (§3, a Phase-2 keystone): presenting a build-time-generated proxy as THE shape biases that call, so it must stay a CANDIDATE.; The proxy can only intercept methods reachable through a TRAIT the proxy implements — a bean with no service trait (Spring's CGLIB-only case, line 1646) has no interface to stand in for. Rust has no subclassing, so there is NO equivalent of CGLIB's 'subclass a concrete class with no interface': leaf would have to require advisable beans to expose a trait (a real divergence — 'program to traits' becomes mandatory for AOP, not just preferred), or generate an inherent-method wrapper that callers must hold by the concrete proxy type (defeating transparency).; Generics can't be proxied without a concrete instantiation (the cross-crate generics-can't-register constraint applies to codegen too): `Repo<T>` needs a generated proxy per concrete `Repo<User>` — the macro must hard-error on generic targets with a 'proxy this concrete instantiation' escape hatch.; Cross-crate reach: the proxy for a bean defined in crate X must be generated where X's trait set is known; if advice is contributed in crate Y, codegen must see both — couples to whichever cross-crate substrate Phase 2 picks (Architecture C build-time codegen fits best; A/B link-time fits worse because the proxy is a TYPE, not just a descriptor).; JoinPoint must carry args/return ERASED (the chain is heterogeneous and dyn) yet the real-method tail closure needs them TYPED — bridging erased-at-the-chain to typed-at-the-call requires either per-method generated glue (more codegen) or a downcast at the proceed boundary (a runtime failure surface if an interceptor mutates args off-type).; Pre-commits to 'the proxy is a generated type' before the proxy/early-reference question (and the scope/ownership model the target handle rides on) is decided — if Phase 2 chooses a runtime registry / dynamic source as primary, a TYPE-per-target shape is awkward for dynamically-sourced beans that have no compile-time presence.

**Mechanisms considered:** build/codegen-time per-target delegating newtype (CGLIB-subclass analogue moved to compile time) implementing the bean's service trait(s); Interceptor / JoinPoint / proceed() chain as the hand-writable call-routing library primitive; proxy struct is thin generated delegation over it; wrap-and-swap at the post-processor-spi after_init ErasedBean seam (proxy returned in place of target); trait-upcasting (1.86) to inject the proxy as dyn Trait; @Proxyable/proxy-target-class realized as a codegen input: which/which-subset of traits the generated proxy implements (the v7 interfaces= encapsulation control); generated early_proxy() over a not-yet-initialized target handle for the getEarlyBeanReference / cycle case; per-method generated glue to bridge erased-chain args/return to the typed real-method tail closure

**T1 — cross-crate/tooling:** Best fit for Architecture C (build-time codegen, the sqlx/portable model): the proxy is a generated TYPE, and code that walks the resolved graph (leaf-codegen) can emit `OrderSvcProxy` wherever the bean's trait set AND the matching advice are both visible — naturally on every target incl. wasm/embedded where link-time slices fail. leaf-core defines the substrate (Interceptor, JoinPoint, AdviceChain, the after_init seam this rides) and stays advice-agnostic; leaf-codegen owns proxy emission. The wrap-DECISION (does any advice match this type?) is still a runtime question over collected advisor descriptors (auto-proxy-creator's job), but the proxy TYPE must exist at compile time — so the generator must emit a proxy for any type that COULD be advised (every type carrying an advisable trait + an opt-in), or codegen must see the advisor set. This couples proxy generation to the cross-crate advisor-discovery substrate: if advisors are contributed via link-time descriptors (Architecture A/B) but proxies are types (codegen C), the two halves must rendezvous at the binary/app-root — a real tension flagged for Phase 2. Generics hard-error in the proxy macro (no monomorphization to generate against) with a concrete-instantiation escape hatch, mirroring the generics-can't-register rule. Origin-agnostic: the generated proxy wraps an ErasedBean target handle, so a WASM-module-sourced or test-double target is wrapped by the SAME generated delegation — but a target that exists ONLY dynamically (no compile-time type) cannot get a compile-time-generated proxy, so dynamic-source beans need R2/R3's runtime shim (the two-source duality surfaces here as a hard split, not a smooth blend — a Phase-2 question). The anti-DCE self-check applies to the advisor descriptors, not the proxy types (types are force-linked by being referenced in generated wiring).

**T2 — execution context:** The wrap-and-swap runs in the instance-decoration phase driven by bean-instantiation / refresh-lifecycle: cold-path, single bean at a time, inside the per-bean creation guard (NOT a global lock) — the after_init hook may .await (building the proxy is cheap/sync, but installing the chain could touch async config). The early_proxy() path runs inside cycle resolution and is SYNC (no .await) — building the stand-in must not re-enter the async creation machinery. The HOT path is per-advised-call at steady state: `chain.invoke(jp)` walks N interceptors then calls the real method. Each interceptor's invoke is async at the dyn boundary → boxed future (AFIT not dyn-safe; the §2 reactive-on-hot-paths constraint means the chain itself must be waker-driven, not polling — it is, since it's just awaiting the next interceptor's future). Execution-context discipline: an interceptor that blocks (a sync cache lookup, a blocking tx resource) MUST spawn_blocking — the substrate documents that a JoinPoint runs on the CALLER's execution context (advice like @Transactional binds resources to that context — context-propagation territory, Phase 2). Cancellation: if the caller's future is dropped mid-chain, the in-flight interceptor and the real method see standard async cancellation — the substrate must document that around-advice holding resources across proceed() must be cancel-safe (release on drop), since there is no async Drop (stable). Errors propagate as AdviceError up the chain as a normal Result; the substrate adds the JoinPoint context (which bean, which method, which interceptor) to the causal chain (§2.9). Self-invocation is a compile-time/structural fact, not a runtime cost — the target's inherent call simply never enters the chain.

#### R2 — Runtime interceptor wrapper around an erased handle (JDK-dynamic-proxy analogue): no generated type, a single dispatching wrapper drives a metadata-described method table
Realize the proxy WITHOUT generating a type per target — the rust-native analogue of JDK dynamic proxies (one generic InvocationHandler driving a reflective dispatch, lines 1645). A single library type `ProxyHandle { target: ErasedBean, methods: &'static MethodTable, chain: Arc<AdviceChain> }` stands in for ANY proxied bean. The bean's advisable surface is described by a metadata table the macro emits ALONGSIDE the bean (thin: just data — `&'static MethodTable` of `MethodEntry { id, name, fn erased_invoke(&ErasedBean, ErasedArgs) -> BoxFuture<ErasedRet> }`), where each entry's fn pointer is a thin generated thunk that downcasts the target handle and the erased args, calls the real method, and re-erases the return. Interception is then UNIFORM and type-free: the consumer calls a method through a typed facade that resolves to `chain.invoke(JoinPoint{ entry, target, args })`, and proceed() either advances to the next interceptor or calls `entry.erased_invoke`. The transparency seam is the harder problem (Rust has no runtime type-synthesis): the proxy can't BE a `dyn Order` for free. Two sub-mechanisms, both kept as candidates: (a) a thin generated facade `impl Order for ProxyHandle`-by-newtype that forwards each trait method to `chain.invoke` via the method id — i.e. the typed shell is generated (small, one forwarding line per method) but the DISPATCH/CHAIN machinery is shared runtime code (vs R1 where the whole delegation including the real-method tail is generated); or (b) the consumer holds an explicit `Proxied<dyn Order>` smart-pointer whose `Deref`/call-sites go through the handle — making proxying VISIBLE in the type (a divergence from Spring's invisibility, but honest and zero-codegen for the dispatch). Wrap-and-swap is the same after_init ErasedBean seam. The substrate owns: the MethodTable metadata shape, the ProxyHandle, the chain/JoinPoint/proceed, and the erased-args bridging convention. @Proxyable/shape becomes 'which MethodEntries the table exposes'. Early-reference: the ProxyHandle wraps a not-yet-initialized target handle — trivial since the handle is just a pointer + table.

**Pros:** No type-per-target codegen for the DISPATCH machinery: one ProxyHandle + one chain walker serves every proxied bean, dramatically less generated code than R1 and a smaller cross-crate footprint (the macro emits DATA — a method table — not a full delegating type), fitting link-time descriptor substrates (Architecture A/B) cleanly since a `&'static MethodTable` is exactly the thin const descriptor those favor.; The runtime method table is naturally origin-agnostic AND dynamic-source-friendly (the two-source duality): a native bean's entries are generated thunks; a WASM-module bean's entries are bridge fns into the module — the ProxyHandle can't tell them apart, so the optional dynamic 'classpath' slots in with NO new proxy path (the place R1 splits hard, R2 blends smoothly).; Closest to JDK-dynamic-proxy intent: 'one handler, a method table, dispatch by method identity' — and the v7 `interfaces=` subset control (line 1685) is just 'expose fewer MethodEntries', a natural fit. Also handles the no-extra-codegen case for beans whose proxy is built purely from runtime metadata.; Decouples the proxy from the target's concrete type entirely at the dispatch layer — so a single chain implementation, JoinPoint, and proceed() are written ONCE as testable library code (§2.10), and adding an advisable method is a metadata-table entry, not a regenerated type.; Early-reference and cycle exposure are trivial (wrap a handle that isn't initialized yet — the table doesn't care), and the proxy's Send/Sync is a property of the handle+table+chain, uniform across all proxied beans.; Sub-mechanism (b) (`Proxied<dyn T>` visible wrapper) is a genuinely rust-honest option that makes the self-invocation boundary and the proxy's existence VISIBLE in the type signature — arguably better DX-diagnostics (§2.9) than Spring's invisible proxy, and zero codegen for dispatch.

**Cons:** The transparency problem is unsolved without SOME generated typed shell: sub-mechanism (a) still generates a forwarding `impl Order for ProxyHandle` per trait (less than R1 but not zero), and sub-mechanism (b) BREAKS Spring's invisibility (consumer sees `Proxied<dyn Order>`, not `dyn Order`) — a real DX divergence (§2 DX #1) that changes injection-point types and ripples into autowiring-resolution.; Per-call erased-args bridging is the hot-path cost: every advised call boxes args into ErasedArgs and the return into ErasedRet (alloc + downcast per call), strictly more per-call overhead than R1's monomorphized delegation — directly hits §2 low-overhead and §2.5 (avoid gratuitous boxing); the downcast is also a runtime failure surface (an interceptor that swaps an arg off-type fails at proceed()).; The MethodTable + erased_invoke thunk convention is a hand-rolled mini-reflection layer — exactly the JVM reflection capability Rust lacks and the charter warns against re-importing for its own sake (§2.2); it must earn its keep vs R1's no-reflection delegation.; Method identity is by id/index in a table — refactoring the bean's methods reorders the table; the macro must derive stable ids (name-hash, the cross-crate ordering-as-data discipline) or interceptors keyed by method break silently.; Around-advice that needs the TYPED args (a cache key from a specific parameter, a validation constraint on a typed field) must downcast out of ErasedArgs — pushing type-safety to runtime and making advice authoring less ergonomic than R1's typed JoinPoint glue.; Still cannot proxy a bean with no advisable trait under sub-mechanism (a) (nothing to `impl`), and sub-mechanism (b) leaks proxy-ness everywhere it's injected — neither cleanly reproduces 'any bean, transparently proxied' that CGLIB gave on the JVM.

**Mechanisms considered:** single library ProxyHandle (JDK-dynamic-proxy/InvocationHandler analogue) driving a `&'static MethodTable` of erased-invoke thunks — no type-per-target for dispatch; thin generated forwarding `impl Trait for ProxyHandle` (sub-mechanism a, small per-method) VS a visible `Proxied<dyn T>` smart-pointer (sub-mechanism b, zero dispatch codegen but proxy-visible); erased args/return bridging convention (ErasedArgs/ErasedRet) with generated downcasting thunks at the real-method boundary; method identity as stable name-hash ids in the table (ordering-as-data) so interceptors key methods reliably across refactors; same post-processor-spi after_init ErasedBean wrap-and-swap; table-described shape for @Proxyable/interfaces= subset control; ProxyHandle over a not-yet-initialized handle for early-reference/cycle exposure; dynamic-source (WASM) entries as bridge-fn thunks

**T1 — cross-crate/tooling:** Best fit for link-time descriptor substrates (Architecture A/B): the per-bean MethodTable is a thin `&'static` const descriptor — exactly the shape `linkme` favors — and the dispatch machinery (ProxyHandle, chain) is shared kernel code with no per-target type to force-link. This is also the ONLY one of the three that handles the dynamic-source half of the two-source duality smoothly: a WASM-module bean contributes a MethodTable whose entries bridge into the module, and the ProxyHandle dispatches identically — the static link-time 'classpath' and the optional dynamic directory-scanned source feed ONE proxy mechanism (CANDIDATES, not a decision). leaf-core defines ProxyHandle, MethodTable/MethodEntry, the erased-args convention, AdviceChain/JoinPoint, and the after_init seam; the macro emits only the const table + thunks (thin, §2.10). Generics still can't register a concrete table (no monomorphized thunk symbol) → macro hard-errors with a per-instantiation shim. The transparency shell (sub-mechanism a) IS a generated type and reintroduces R1's force-link concern for that shell; sub-mechanism (b) avoids it entirely (no generated type) at the DX cost. Anti-DCE self-check applies to the method-table descriptors and the advisor descriptors uniformly (one descriptor stream), and the kernel must expose a proxied-types-by-table introspection surface so the check is writable.

**T2 — execution context:** Wrap-and-swap is cold-path at after_init inside the per-bean creation guard (no global lock), same as R1. The HOT per-call path is heavier than R1: `chain.invoke` walks N boxed-future interceptors (waker-driven await, satisfying §2.4 reactive-on-hot-paths) and proceed() finally calls `entry.erased_invoke`, which BOXES the future and erases args/return — an alloc per advised call beyond R1's. Execution context: the JoinPoint runs on the caller's context (advice binds resources there); a blocking interceptor MUST spawn_blocking. The erased thunk must preserve Send across the boxed future (the chain spans the async executor) — so the kernel's erased-invoke signature is `fn(&ErasedBean, ErasedArgs) -> Pin<Box<dyn Future<Output=Result<ErasedRet, AdviceError>> + Send + '_>>`, fixing a Send bound on all advisable methods (consistent with the dyn-bean Send story, nightly-doc line 260). Cancellation: dropping the caller future cancels mid-chain; around-advice must release resources on drop (no async Drop). Early-reference path is sync (wrap an uninitialized handle). Errors carry JoinPoint context (bean/method/interceptor) into the §2.9 causal chain. The runtime downcast in erased_invoke / arg access is a typed failure point that must produce a rich diagnostic (which method, expected vs actual erased type), not a panic.

#### R3 — Pure interceptor middleware over the resolution seam: no proxy OBJECT at all — interception is a chain the CONTAINER applies when handing a bean to a collaborator
Question whether leaf needs a proxy OBJECT at all. Spring needs one because the JVM injects a reference and has no other interception point — once A holds B, the container is out of the loop, so the only way to intercept B's calls is to make the reference itself a wrapper. But leaf controls the RESOLUTION/HANDING-OUT seam (autowiring-resolution, deferral-primitives) and could model interception as MIDDLEWARE the container composes at the point a dependency is handed over, rather than as a standalone wrapper type. Two candidate shapes, both proxy-OBJECT-free for the common case: (a) `Advised<T>` injected handle — instead of injecting `Arc<dyn Order>`, advised injection points receive an `Advised<dyn Order>` (a deferral-primitive cousin) that, on each method call, routes through the chain; the chain is held by the container and looked up by target id, so there is no per-target generated type AND no standalone proxy — the 'proxy' is the typed handle the resolution seam chose to hand over, parameterized by the advice that matched. (b) call-site interception via a generated typed facade that the INJECTION POINT receives: leaf knows, at wiring time, that `dyn Order` here is advised, so it hands a `OrderAdvised` (a thin tower::Service-style middleware stack the macro emits as `impl Order` forwarding through `tower::Layer`-like advisors). The substrate this feature owns is then NOT 'a proxy type' but an INTERCEPTION TOWER primitive — `trait Advisor: Layer<TargetCall>` composing into a `Stack` (the tower model, idiomatic async-Rust middleware), with the container deciding at the resolution seam which tower wraps which handed-out reference. Self-invocation is STILL honest and for the same reason: `self.method()` inside the target never enters the tower (the tower lives on the handed-out reference, exactly as Spring's proxy lives on the injected reference). Wrap-and-swap becomes 'the resolution seam selects an advised handle variant' rather than 'after_init returns a different object' — though it can ALSO be expressed as the after_init swap to keep parity with the post-processor-spi seam. This reframes 'proxy' as a property of the EDGE (the A→B dependency) rather than a property of B.

**Pros:** Most idiomatic async-Rust: an interception TOWER (tower::Layer/Service-style, the ecosystem's lingua franca for middleware) is exactly how Rust composes cross-cutting behavior around async calls — leaf's advice authors write `Layer`s they already understand, and the chain is the well-trodden `Stack` composition (DX #1 for the Rust-native, vs R1/R2 which transplant a JVM proxy mental model).; Can avoid a standalone proxy OBJECT entirely for the common case: interception is the container's choice at the resolution edge, so there is no 'a bean is now a different type than it was' identity puzzle, no per-target generated delegation for dispatch, and the scope/ownership model is unburdened by a wrapper type (defers cleanly to the Phase-2 ownership keystone).; Models 'advice attaches to the SEAM between beans' (Spring's own stated mental model, line 1729) LITERALLY: the seam IS the A→B edge, and the tower lives there — so self-invocation being uninterceptable is structurally obvious (there's no edge for an internal call), arguably the most honest realization of the intent.; The tower/Layer primitive is reused FAR beyond AOP: the same middleware-stack substrate naturally serves event-multicaster dispatch decoration, task-execution decoration, and any 'compose async behavior around a call' need — maximal unification (§2.11), one interception primitive for the whole framework.; One-proxy-many-concerns and ordering (line 1770-1772) are native to the tower model: advisors are Layers composed in a Stack in declared order, and ordering-as-data (the cross-crate discipline) maps directly to Layer ordering — misordering is at least inspectable as a Stack, not hidden in object identity.; Cancellation/errors are tower-native (Service::call returns a future that's cancel-safe and error-typed), aligning with §2.3/§2.4 with no bespoke chain machinery; the substrate inherits a mature async-cancellation story.

**Cons:** Biggest divergence from Spring's invisibility: if the advised handle is a DIFFERENT type at the injection point (`Advised<dyn Order>` / `OrderAdvised`), the consumer's field type changes, rippling into autowiring-resolution and injection-styles — the bean is NOT transparently 'still a dyn Order', cutting against DX #1's Spring-grade transparency promise (the very thing proxies buy on the JVM).; If instead the handle MUST be a plain `dyn Order` for transparency, R3 collapses back toward R1/R2 (you still need SOMETHING that is-a `dyn Order` and routes through the tower) — so the 'no proxy object at all' claim only fully holds for shape (a) with a visible handle; the invisible case still needs a generated facade, eroding the distinction.; 'Interception is a property of the EDGE not the bean' breaks a Spring guarantee: in Spring EVERY external caller of a proxied bean gets advice (the bean is wrapped once, globally); an edge-based model risks advising SOME injection points and not others unless the container guarantees every handed-out reference is advised — re-introducing a per-target wrapping decision and the 'is this reference advised?' ambiguity Spring eliminated by wrapping the bean itself.; Pre-judges the scope/ownership and resolution model (Phase-2 keystones, §3/§5): making interception a resolution-seam concern entangles it with how dependencies are handed over (borrow vs Arc vs handle) — presenting it as a realization biases that signal; must stay a CANDIDATE.; Early-reference/cycle exposure (getEarlyBeanReference) is awkward: if the 'proxy' only exists at the resolution edge, what does a mid-cycle collaborator inject? The tower must be materializable as a handle BEFORE the target is initialized — pushing back toward a handle-that-exists-early (R2's ProxyHandle), so the cycle case may force a proxy object after all.; tower's `Service` is request/response-shaped (one call type per Service); a bean has MANY methods with different signatures, so mapping a multi-method trait onto Layer/Service needs an enum-of-calls or per-method Service — a real impedance mismatch that either bloats the call type or generates per-method machinery, partially reintroducing R1/R2's codegen.

**Mechanisms considered:** interception as container-applied middleware at the resolution/handing-over seam (no standalone proxy object for the common case); tower::Layer/Service-style Advisor + Stack composition as the async-native interception-tower primitive (reused beyond AOP); Advised<T> / generated typed-facade handle chosen by autowiring-resolution at the injection point (visible-handle vs invisible-facade sub-shapes); 'advice on the edge (A→B dependency) not on the bean' framing — self-invocation uninterceptable becomes structurally obvious; still expressible as the post-processor-spi after_init swap to keep seam parity; ordering-as-data maps to Layer order in the Stack; enum-of-calls or per-method Service to bridge tower's single-call shape onto a multi-method bean trait (the impedance point)

**T1 — cross-crate/tooling:** Advisors-as-Layers are contributed cross-crate as descriptors (a `&'static AdvisorDescriptor { order, matches_fn, make_layer_fn }`) over whatever substrate Phase 2 picks (linkme/codegen/explicit) — origin-agnostic, since a Layer is just async middleware regardless of where its target bean came from (a WASM-module bean's calls go through the same Stack). leaf-core defines the Advisor/Layer/Stack primitive, the JoinPoint-equivalent call context, and the resolution-seam hook; advice crates contribute Layer descriptors. The big T1 question this shape raises: if interception is chosen at the RESOLUTION seam, the decision 'is this edge advised?' must be made consistently for every handed-out reference to a given target — which needs the FULL advisor set known when wiring happens (a runtime question over collected descriptors, same as auto-proxy-creator). The dynamic-source half of the two-source duality fits well (a dynamically-sourced Advisor Layer composes into the Stack like any other) — but the VISIBLE-handle sub-shape changes injection-point types, which is a compile-time cross-crate concern (consumers in other crates must name `Advised<dyn Order>`), creating a cross-crate API-surface coupling the invisible/proxy shapes avoid. Anti-DCE self-check applies to advisor descriptors. Generics: a generic advisor Layer needs concrete instantiation to register (generics-can't-register), same hard-error discipline.

**T2 — execution context:** The interception tower runs on the CALLER's execution context at each advised call — the most async-native of the three, since tower Services are built for exactly this (await the inner Service through composed Layers). Hot path is waker-driven await through the Stack (§2.4 satisfied natively, no polling). Each Layer's Service::call returns a boxed future at the dyn boundary (AFIT/dyn tax, line 260) — same boxed-future cost as R1/R2, but the cancellation/error model is tower-mature (drop-to-cancel, typed errors), so around-advice cancel-safety is the established tower discipline rather than bespoke. A blocking advisor MUST spawn_blocking (execution-context discipline). The resolution-seam wrap decision is cold-path at wiring; if interception is materialized lazily (first-call), the first call pays a one-time Stack-build cost (cold) then steady-state is hot — must ensure the build is not a per-call cost. Cycle/early-reference: materializing a tower handle before the target is initialized is the awkward case (T2 re-entrancy) — the handle must defer target resolution to call time (a deferral-primitive trait the cycle case already needs), and must NOT .await during cycle exposure. Errors fold JoinPoint/call context into §2.9. context-propagation across the tower (a tx/locale bound for the duration of the call) is Phase-2 territory but the tower is the natural place to capture/restore it (a Layer that wraps the inner call in a context scope).

### Rust clashes
- No CGLIB and no JDK dynamic proxy — Rust cannot synthesize a NEW type at runtime that implements arbitrary traits/overrides methods. This is THE clash for this feature: Spring's entire proxy substrate rests on the JVM's ability to either subclass a concrete class (CGLIB) or build an interface-implementing object from a list of interfaces + an InvocationHandler (java.lang.reflect.Proxy) AT RUNTIME. Rust has neither. So leaf's 'stand-in that is-a the bean's type' must be EITHER a compile-time-generated delegating type (R1), OR a single runtime wrapper driving a metadata method-table with a thin generated typed shell (R2), OR re-framed as resolution-seam middleware that may not be a standalone type at all (R3). The pure-runtime, zero-codegen, transparent-as-the-original-type proxy that CGLIB gives simply has no Rust equivalent — every realization pays in either codegen or transparency.
- No subclassing means a bean with NO trait cannot be transparently proxied — Spring's CGLIB can proxy a concrete class that implements no interface (line 1646, 1651). Rust has no subclassing, so a stand-in can only stand in THROUGH a trait the stand-in implements. This forces a divergence: leaf either makes 'expose a service trait' MANDATORY for advisable beans (turning Spring's preferred 'program to interfaces' into a hard requirement for AOP), or hands out a concrete proxy type callers must name (defeating transparency). There is no 'proxy any concrete class as itself' option.
- The 'returns a different object' identity swap relies on GC'd reference aliasing Rust doesn't have — on the JVM the proxy and target are independently GC'd heap objects and the proxy freely aliases/replaces the target reference. In Rust the bean is an owned/Arc'd erased handle with explicit lifetime/ownership; swapping the target for a proxy at after_init means the proxy must OWN or borrow the target handle under whatever Phase-2 ownership model applies — and the target must NOT also be injected raw elsewhere (Spring's allowRawInjectionDespiteWrapping mismatch, line 779-781, becomes a Rust ownership question: who holds the raw target once the proxy exists?). The substrate must keep the swap origin-agnostic (ErasedBean in, possibly-different ErasedBean out) and defer the ownership of target-inside-proxy to Phase 2.
- Async fn in traits isn't dyn-safe, but the interceptor/advisor chain MUST be dyn (heterogeneous, cross-crate, origin-agnostic incl. WASM-module advisors) — so every advice invoke()/Service::call at the dyn boundary returns a boxed future (Pin<Box<dyn Future + Send>>), an alloc per advised call on the HOT path. This is the per-call AOP tax and it's unavoidable on stable for the dyn chain; it's the §2.4/§2.5 tension the feature must own (the chain stays waker-driven/reactive — it's just awaiting the next future — but each hop boxes). A monomorphized zero-cost chain can't coexist with the heterogeneous dyn chain on the same surface (Phase-2 async-execution-model call).
- Erased args/return at a heterogeneous chain vs typed real-method call — the chain is type-free (it dispatches any method of any bean) but the real method and typed around-advice (a cache key from a param, a validated typed arg) need the concrete types. Bridging erased-at-the-chain to typed-at-the-method needs either per-method generated glue (R1, more codegen) or an erased-args downcast at proceed() (R2, a runtime failure surface). The JVM hides this with reflection + autoboxing; Rust forces the choice between codegen volume and runtime-typed-failure, neither free.
- No async Drop on stable for around-advice teardown — around-advice that acquires a resource before proceed() and releases after (a transaction, a lock, an exposeProxy-style context binding) cannot rely on async Drop if the call is cancelled mid-proceed(). Around-advice must be cancel-safe (release synchronously on drop, or the container's explicit teardown owns it) — mirroring the bean-teardown story (nightly-doc line 273-281). 'The advice owns its scope across proceed()' is a non-syntactic discipline the substrate must document, and cancellation mid-chain must unwind partially-applied advice explicitly.
- Early-proxy exposure into cycles needs a proxy that exists BEFORE the target is initialized — Spring's getEarlyBeanReference (line 1766-1768, 777) hands the proxy into a cycle so collaborators don't inject the raw target. In Rust the early stand-in must wrap a not-yet-initialized target handle and must NOT .await during cycle exposure (re-entrancy into the async creation machinery). This couples the proxy mechanism to bean-registry's early-reference cache AND to the (undecided) proxy story: whether this early-proxy sub-tier even earns its keep depends on the cycle/ownership decisions still open — it is the most speculative part of the substrate (and AOT contexts disallow cycles entirely, line 785, so it may be runtime-only).
- Self-invocation is uninterceptable in Rust too — and for the IDENTICAL structural reason (the wrapper is external to the target's control flow), so leaf reproduces the boundary honestly rather than as a bug. But Rust adds NO new escape hatch: there is no AopContext.currentProxy() ThreadLocal analogue without a context-propagation holder (Phase-2), and 'inject a self-reference (the proxy)' requires the bean to be able to hold a handle to its own proxy — an ownership/cycle puzzle (the proxy owns the target, the target wants the proxy) that is cleaner in GC than in Rust ownership. The substrate must document the boundary and steer to the same Spring-sanctioned workarounds (refactor the self-call out, or inject the proxy handle) rather than synthesizing a hidden one.
- Method identity for advice matching has no runtime reflection — Spring matches pointcuts against java.lang.reflect.Method objects discovered reflectively. Rust has no such method metadata at runtime, so the advisable-method set must be DECLARED as data (a generated method table / per-method delegation), and pointcut matching (aspect-model's job) operates over that declared metadata, not a runtime method walk. Method identity must be stable across refactors (name-hash ids, ordering-as-data) or method-keyed advice breaks silently — a stricter discipline than Spring's reflective matching needs.

### Interactions
- **auto-proxy-creator** — auto-proxy-creator is the PRIMARY consumer and the layer directly above this substrate: it is the single per-context instance-tier processor that collects ALL matching advisors into ONE ordered chain per bean and DECIDES whether a given bean is eligible (does any advisor match its type?). This feature provides the substrate auto-proxy-creator drives: the wrap-and-swap seam (return a proxy in place of the target at after_init), the call-routing primitive (Interceptor/JoinPoint/proceed chain or tower Stack), and the early-reference hook. The division is sharp: THIS feature owns 'how a stand-in wraps a target and routes one call through one chain'; auto-proxy-creator owns 'which advisors match, their union, their order, and the one-creator-wins infrastructure-vs-aspect isolation'.
- **aspect-model** — aspect-model is the user-facing layer two above: it lets users declare pointcuts + before/after/around advice (@Aspect beans) that become advisors. Those advisors' advice bodies are the Interceptors/Layers that populate the chain THIS feature routes calls through. This feature is advice-AGNOSTIC — it defines the JoinPoint/proceed contract advice authors program against (the around-advice 'proceed() to continue the chain or the real method' primitive) without knowing what any specific advice does; aspect-model turns declarative @Before/@Around into concrete chain entries.
- **post-processor-spi** — This feature is built ON the upstream post-processor-spi's instance-decoration seam: the wrap-and-swap IS the after_init hook returning a different ErasedBean (the canonical 'a BPP returns a different object IS how AOP swaps in a proxy', the keystone use case the SPI was kept proxy-mechanism-neutral for). The early-proxy exposure uses the SPI's EarlyRefPostProcessor / early_reference sub-tier. This feature plugs the actual proxy mechanism into the SPI's deliberately-neutral seam, so leaf-core's post-processor-spi never depends on AOP — it just offers 'a processor may replace the handle' and AOP supplies the replacement.
- **circular-references** — The early-proxy-exposure hook (getEarlyBeanReference analogue) is the integration point: if a singleton in a setter/field cycle gets proxied, the early reference handed to mid-cycle collaborators must be the PROXY, not the raw target, or some collaborators bypass advice (Spring's raw-vs-wrapped mismatch, line 779-781). This feature must materialize an early stand-in over a not-yet-initialized target without re-entering async creation; circular-references defines the tolerate-setter/field-cycles-via-early-ref machinery and the raw-vs-wrapped consistency check this hook participates in. Whether this sub-tier is even needed depends on the still-open proxy/cycle story (and AOT disallows cycles, line 785), making it the most speculative coupling.
- **bean-instantiation** — bean-instantiation drives the wrap-and-swap at the right point: after_init (proxy returned AFTER init callbacks, so the proxy wraps a fully-initialized target) and the early_reference hook when a cycle demands it, all inside the per-bean creation guard (not a global lock). bean-instantiation calls the substrate's seam and publishes the possibly-different returned handle as the bean's identity — every subsequent injection of that bean gets the proxy.
- **bean-registry** — bean-registry provides the singleton cache the proxy is published into (the proxy, not the raw target, becomes the cached/injected instance) and the early-reference cache slot the early-proxy hook targets. The substrate's 'who owns the raw target once the proxy wraps it' is a bean-registry ownership question (the raw target must not also be handed out raw — Spring's allowRawInjectionDespiteWrapping). Whether the registry even has a three-level early-proxy cache slot is part of the undecided proxy story this feature surfaces.
- **scoped-proxies** — scoped-proxies is a SIBLING use of the same substrate for a different intent: instead of wrapping for cross-cutting advice, it wraps a longer-lived bean's reference so each call resolves the live shorter-lived (request/session/prototype) target. It uses THIS feature's stand-in + call-routing primitive but routes proceed() to a freshly-RESOLVED target per call rather than through an advice chain to a fixed target. The two share the proxy substrate (a transparent stand-in that intercepts calls) and diverge only in what the interception does — a strong argument for unifying the wrap primitive (§2.11).
- **transaction-management** — transaction-management is the canonical declarative cross-cutting concern built on this substrate: its TransactionInterceptor is one entry in the chain THIS feature routes through (opens/commits/rolls-back a transaction AROUND the real method via proceed()). It depends on the substrate's around-advice contract and on context-propagation to bind the transaction to the call's execution context — and on the substrate documenting cancel-safety (a cancelled call mid-transaction must roll back, since there is no async Drop).
- **caching** — caching's CacheInterceptor is another chain entry: it consults/populates a cache AROUND the call (may short-circuit proceed() entirely on a cache hit — a key around-advice capability this substrate must support: an interceptor that returns WITHOUT calling proceed()). Ordering within the shared chain relative to transaction advice is the classic 'cache outside vs inside the tx' correctness concern (line 1772) that auto-proxy-creator orders but THIS substrate's single-ordered-chain shape makes possible.
- **validation** — validation's method-validation interceptor is a chain entry that inspects typed arguments/return before/after the real call — the prime example of around-advice needing TYPED access to the JoinPoint's args (the erased-vs-typed clash): it must reach the concrete argument types to apply JSR-303-style constraints, which is cheap in R1's typed glue and a downcast in R2's erased table.
- **retry-resilience** — retry-resilience's retry interceptor wraps proceed() in a retry/backoff loop (calling proceed() multiple times) and concurrency-limiting — exercising the substrate's around-advice contract where a single advised call drives MULTIPLE invocations of the real method, and the cold-path retry/backoff polling (permitted on cold paths per §2.4) lives in this chain entry, not the substrate.
- **async-execution** — async-execution runs a method on a TaskExecutor via this substrate (an interceptor that dispatches proceed() onto a chosen executor rather than running it inline) — the clearest case where the substrate's 'JoinPoint runs on the caller's execution context' default is deliberately broken by advice, and where the self-invocation/void-return caveats (a self-called @Async silently runs inline) are exactly this feature's self-invocation boundary.
- **exception-translation** — exception-translation weaves a translating advisor onto @Repository-equivalent beans (converts vendor exceptions to a uniform hierarchy) — a concrete after/around chain entry that catches the real method's error result and re-maps it. It is one of leaf's own dogfooded advisors riding this substrate's wrap-and-swap, advising only beans whose metadata marks them as repository-boundary beans (the natural pointcut).
- **autowiring-resolution** — autowiring-resolution is how the proxy reaches collaborators: because the proxy is-a the bean's service trait (R1/R2) and upcasts to dyn Trait, autowiring injects the proxy wherever the raw bean's type is wanted — transparently. R3's visible-handle sub-shape would CHANGE the injection-point type (Advised<dyn Order>), rippling into this feature's resolution logic — a key tension. The substrate must keep the proxy injectable as the SAME type the target satisfies, or accept a divergence autowiring-resolution must account for.
- **container-core** — container-core's engine-vs-façade split realizes the 'AOP silently does nothing in a bare engine' parity: the bare engine installs no auto-proxy-creator, so no wrap-and-swap happens and advice is inert (exactly Spring's plain-BeanFactory behavior); the context façade installs the auto-proxy-creator that drives this substrate. This feature provides the substrate the façade's installed creator uses; container-core fixes the boundary that makes AOP an opt-in-by-using-the-context capability.
- **context-propagation** — context-propagation is the Phase-2 mechanism the substrate needs for advice that binds ambient context across the call (transaction sync, locale, tracing) and ESPECIALLY across thread hops introduced by async-execution advice: an around-advisor that captures context before proceed() and restores it after relies on context-propagation's capture/restore SPI. The substrate is the natural place to wrap a call in a context scope (a Layer/interceptor), but the holder/restore mechanism itself is context-propagation's (and Phase-2's) to define.

### Open questions for Phase 2
- What is the proxy MECHANISM — a compile-time-generated delegating type per target (R1), a single runtime wrapper over a metadata method-table with a thin typed shell (R2), or resolution-seam middleware that may not be a standalone object (R3)? This is the keystone for THIS feature and it cross-cuts the metadata/codegen boundary (§3): R1/R2-shell pre-judge toward codegen-generated proxy types; R2-table/R3 lean runtime; the choice determines per-call overhead (monomorphized delegation vs erased-args boxing), codegen volume, and whether dynamic-source beans can be proxied at all. Held unresolved.
- Does leaf REQUIRE every advisable bean to expose a service trait (since Rust has no CGLIB subclass-a-concrete-class), making 'program to traits' MANDATORY for AOP rather than preferred — or does it offer a concrete-proxy-type path that sacrifices transparency for trait-less beans? This is a real DX (#1) divergence from Spring with no clean rust-native middle and cross-cuts injection-styles and component-stereotypes.
- Is the proxy injectable as the SAME type the target satisfies (transparent, dyn Trait — preserving Spring's invisibility and leaving autowiring-resolution untouched), or as a VISIBLE handle type (Advised<dyn T>/Proxied<dyn T> — honest and zero-dispatch-codegen but changing injection-point types and rippling through autowiring-resolution, injection-styles, deferral-primitives)? The transparency-vs-honesty call is a DX keystone that ripples cross-crate (consumers in other crates must name the injection type).
- How are heterogeneous-chain ERASED args/return bridged to the TYPED real method and typed around-advice (cache keys from a param, validation on a typed arg)? Per-method generated glue (R1, codegen volume) vs an erased-args downcast at proceed() (R2/R3, a runtime-typed failure surface). This is the rust-native re-expression of the JVM's reflection+autoboxing and it interacts with the async-execution-model and the §2.9 diagnostics for the downcast-failure case.
- Is the early-proxy-exposure sub-tier (getEarlyBeanReference analogue) needed at all, and what does it materialize? It depends on the still-open circular-references AND scope/ownership decisions (who owns the raw target inside the proxy; does the registry have a three-level early-proxy cache slot), and AOT contexts disallow cycles entirely (line 785) — so this may be a runtime-only sub-tier or unnecessary. The most speculative part of the substrate. Cross-cuts circular-references, bean-registry, the ownership keystone.
- What is the async-across-dyn boxing standard for the interceptor/advisor chain? The chain MUST be dyn (heterogeneous, cross-crate, origin-agnostic incl. WASM-module advisors) so every hop boxes a future on the HOT per-call path (§2.4/§2.5 tension) — is boxed-future-per-hop the accepted standard (with the chain staying waker-driven), or does the hot advised-call path need a monomorphized zero-cost escape for the common single-advisor case? The async-execution-model decision viewed through the AOP chain.
- How does around-advice own its scope across proceed() without async Drop — the cancel-safety contract for a call cancelled mid-chain (a transaction to roll back, a lock to release, a context binding to unwind)? This mirrors the bean-teardown 'no async Drop' discipline and must define whether advice releases synchronously on drop, or the container's explicit teardown owns it, and how partial advice unwinds on cancellation. Cross-cuts context-close, transaction-management, context-propagation.
- How does the two-source duality (static link-time 'classpath' PLUS optional dynamic directory-scanned WASM-module source) reconcile with the proxy mechanism? R1 (generated type) splits HARD — a dynamic-only bean has no compile-time type to generate a proxy against, so it needs a runtime shim; R2 (runtime table) and R3 (tower) blend smoothly (a dynamic bean contributes a method table / Layer like any other). The mechanism choice determines whether dynamic-source beans are proxyable uniformly or only through a fallback — a candidate, not a decision.
- Where does the wrap-DECISION live and how is it kept consistent — at after_init (Spring's per-bean global wrapping: every external caller gets advice) vs at the resolution seam (R3's per-edge model, which risks advising some injection points and not others)? Spring's guarantee is 'wrap the bean once, globally'; an edge-based model must reproduce that or re-introduce the 'is this reference advised?' ambiguity. Cross-cuts autowiring-resolution and the resolution model.

## Auto-Proxy Creator & Advisor Chain  `[auto-proxy-creator]`

This feature is the DECISION-and-COLLECTION layer that sits directly ABOVE the aop-proxying substrate and directly BELOW aspect-model. The aop-proxying substrate owns "how one stand-in wraps one target and routes one call through one chain" (Interceptor/JoinPoint/proceed, wrap-and-swap at the post-processor-spi after_init seam, early-reference hook). aspect-model owns "users declare pointcuts + before/after/around advice that become advisors." THIS feature owns the middle: a SINGLE per-context instance-tier processor that (1) collects ALL contributed advisors visible in this context, (2) for each freshly-built bean asks "which advisors MATCH this bean?" (eligibility), (3) computes the UNION of matching advisors and ORDERS them deterministically into ONE interceptor chain, (4) drives the substrate's wrap-and-swap to install that one chain as the bean's proxy, and (5) provides infrastructure-vs-aspect ISOLATION via role hints.\n\nSpring's load-bearing intent (lines 1701-1772, grounded): (a) ONE-CREATOR-WINS — most developers assume one AOP creator; there are several families but only ONE runs per context, and the single proxy it produces carries the UNION of all matching advice (line 1703). The two Spring families are InfrastructureAdvisorAutoProxyCreator (considers ONLY advisors marked @Role(ROLE_INFRASTRUCTURE) — how @EnableTransactionManagement/@EnableCaching/@EnableAsync/@Validated keep their advice isolated, line 1705) and AnnotationAwareAspectJAutoProxyCreator (does everything the infra variant does AND additionally discovers user @Aspect beans — it 'upgrades' the creator, line 1706). The capability lattice means the more-capable creator subsumes the less-capable one; they never both run. (b) ROLE-HINT ISOLATION (lines 612-618, 1708) — @Role is purely informational metadata (ROLE_APPLICATION=0, ROLE_SUPPORT=1, ROLE_INFRASTRUCTURE=2) that does NOT cascade and changes no bean behaviour; the auto-proxy creator CONSUMES it to keep tx/cache/async advice from being entangled with or accidentally matched by application aspects. (c) ONE-PROXY-MANY-CONCERNS-ORDERING (lines 1770-1772) — a bean that is @Transactional AND @Cacheable AND targeted by custom aspects gets a SINGLE proxy whose chain contains all matching advice ordered by advisor order (@Order/Ordered/@Priority); because there is no per-feature proxy, misordering (cache outside-vs-inside the tx) is INVISIBLE in object identity — ordering only matters and only applies WITHIN one chain. (d) ELIGIBILITY-AS-MATCHING (line 1695) — in after_init it asks 'does any advisor match this freshly-initialized bean?', and only then builds the proxy; un-advised beans pass through unwrapped (so the bare engine / no-creator case leaves advice inert — container-core parity). (e) THE EARLY-REFERENCE GATE (line 1768) — the creator implements getEarlyBeanReference so a cycle-exposed bean is wrapped once, early, and recorded so after_init does NOT double-wrap (line 2531 — earlyProxyReferences dedup so early-ref and final singleton are the SAME object).\n\nI must replicate this INTENT in rust-native terms WITHOUT importing the JVM mechanisms: no runtime classpath to scan for @Aspect beans, no runtime reflection to match pointcuts against java.lang.reflect.Method, and (deferred upstream) no runtime-synthesized proxy type. The auto-proxy creator's matching must operate over DECLARED method/advisor metadata (data, not a reflective method walk), advisor COLLECTION must be cross-crate and origin-agnostic (a WASM-module-sourced advisor or a test advisor collects identically), and the 'which creator wins' lattice must be re-expressed as a capability/role-filtered policy over a single collected advisor stream rather than mutually-exclusive BPP beans racing to register. I do NOT decide the proxy mechanism (aop-proxying's deferred keystone), the collection substrate (linkme vs codegen vs explicit — Phase 2 T1), the scope/ownership model the chain rides on, or where the wrap-decision ultimately lives — those are Phase 2. I own only: the advisor-COLLECTION contract, the MATCH/eligibility model, the UNION+ORDER policy, and the role-hint isolation policy — all proxy-mechanism-neutral.

### Candidate realizations

#### R1 — Single per-context AdvisorRegistry + ProxyAdvisor processor: one instance-tier post-processor that collects, matches, orders, and drives the substrate (the direct AbstractAutoProxyCreator analogue)
Realize the auto-proxy creator as ONE per-context infrastructure processor (the literal AbstractAutoProxyCreator analogue) installed by the context façade (container-core), driving the upstream aop-proxying wrap-and-swap at the post-processor-spi after_init seam. The processor HAS-A an `AdvisorRegistry` built ONCE at context refresh: it gathers every `AdvisorDescriptor { id, order: i32, role: Role, pointcut: &'static dyn Pointcut, make_interceptor: fn(&Container) -> Arc<dyn Interceptor> }` collected via whatever Phase-2 substrate provides (origin-agnostic: native, WASM-module, or test advisors collect into the SAME stream), plus the @Aspect-derived advisors aspect-model contributes. On each bean's after_init the processor: (1) MATCHES — for the bean's declared method-metadata table, asks each advisor's `pointcut.matches(bean_type, method_meta)` (a pure predicate over DECLARED metadata, never a reflective method walk), collecting matching advisors; (2) if none match, returns the bean UNWRAPPED (eligibility gate — un-advised beans are zero-cost); (3) if any match, computes the UNION, SORTS by (role-tier, order, stable-tiebreak-id) into one `AdviceChain`, and calls the substrate's wrap-and-swap to install that ONE chain as the bean's proxy. The 'two families / only one wins' lattice becomes a single processor parameterized by a `CreatorPolicy` (a capability set): the context installs ONE processor whose policy is the UNION of enabled capabilities — Infrastructure-only (consider only Role::Infrastructure advisors) is the floor; enabling user aspects (the @EnableAspectJAutoProxy analogue) UPGRADES the same processor's policy to also consider Role::Application advisors — never two racing processors. Role-hint isolation (lines 612-618, 1708) is a MATCH-TIME filter: the policy decides which role-tiers are eligible, and ordering keys on role-tier FIRST so infrastructure advice (tx) is positioned deterministically relative to application aspects. getEarlyBeanReference is a second entry point on the same processor that builds the SAME chain early and records the bean id in an `early_proxied` set so after_init skips re-wrapping (the dedup that makes early-ref and final singleton identical). The processor is advice-AGNOSTIC and substrate-AGNOSTIC — it only knows AdvisorDescriptor + Pointcut + the substrate's install-chain seam.

**Pros:** Cleanest intent parity (§2.2): it IS the AbstractAutoProxyCreator — one per-context processor, one collected advisor set, one match-union-order-install path, one proxy per bean carrying the union. The 'one creator wins / one proxy many concerns / ordering within one chain' triad maps 1:1 onto a single processor + single AdviceChain, so the Spring mental model transfers exactly.; Sharp, honest division of labour with neighbours: this processor owns COLLECTION+MATCH+UNION+ORDER+ROLE-ISOLATION; aop-proxying owns the wrap+route primitive; aspect-model owns advice authoring. Each is independently testable — the matcher is a pure fn over (bean_type, method_meta, advisor_set), no container or async needed (§2.10).; The 'only one wins' lattice is re-expressed SAFELY for Rust's no-classpath reality: instead of mutually-exclusive BPP beans racing to register (a runtime-ordering hazard the cross-crate doc warns against — ordering is unspecified, line 154/355), ONE processor's policy is the UNION of enabled capabilities computed deterministically at refresh. Eliminates Spring's 'which creator registered first wins' fragility entirely.; Role-hint isolation falls out as a match-time predicate + an ordering key, exactly mirroring @Role's 'metadata-not-enforcement, doesn't-cascade' semantics (lines 616-618) — the role is data on the AdvisorDescriptor, the processor is the sole consumer, and infra-vs-aspect entanglement is prevented structurally without changing any bean behaviour.; Eligibility gate makes the un-advised path zero-cost and the bare-engine/no-creator path inert (container-core parity, line 49/110): a context that installs no ProxyAdvisor processor simply never matches anything, so advice is silently inert exactly as Spring's plain BeanFactory.; Origin-agnostic by construction: AdvisorDescriptor is a flat descriptor with a fn-pointer interceptor factory and a &'static dyn Pointcut, so a WASM-module advisor or a test double contributes identically — no assumption of native construction, slotting in the optional dynamic source later.

**Cons:** The single AdvisorRegistry built at refresh assumes ALL advisors are known at one point — fine for the static link-time source, but for an optional dynamic (WASM-module-scanned) source the registry must be refreshable or the dynamic advisors miss beans already proxied (re-proxying an existing bean is impossible without re-running after_init). Couples the 'when is the advisor set frozen' question to the (undecided) two-source-duality and refresh-lifecycle.; Matching cost is O(beans × advisors) at refresh on the cold path; cheap for typical advisor counts but the pointcut-match predicate over declared method metadata must be efficient — and the metadata table it matches against is generated by the (deferred) proxy/codegen mechanism, so this realization implicitly leans on method-metadata existing as DATA, pre-judging part of the metadata/codegen boundary (§3).; Pre-commits to the wrap-DECISION living at after_init (Spring's per-bean global wrapping: every external caller gets advice). If Phase 2 chooses a resolution-seam/per-edge model (aop-proxying's R3), this processor's 'decide once at after_init' shape is awkward — it would have to decide per-edge instead, re-introducing the 'is THIS reference advised?' ambiguity Spring eliminated.; The CreatorPolicy capability-lattice (infra-floor upgraded to include aspects) must be assembled deterministically from cross-crate capability contributions — which Phase-2 substrate computes the union, and is it the binary/app-root's job (the only place the full feature set is visible, per topology line 260)? Couples the policy assembly to the cross-crate composition model.; getEarlyBeanReference dedup needs a per-context `early_proxied` set keyed by bean id — a small piece of shared mutable state on the hot-ish cycle path; must be a fine-grained per-context structure, not a global lock (§2.5), and its necessity depends on the still-open cycle/early-proxy story (aop-proxying's most speculative sub-tier).; Concentrates all AOP policy in one processor type — if a future need wants genuinely DIFFERENT creators for different bean populations (beyond role filtering), the single-processor model must grow more policy knobs rather than compose, slightly cutting against unification-by-shared-primitive (§2.11).

**Mechanisms considered:** one per-context infrastructure post-processor (AbstractAutoProxyCreator analogue) at the post-processor-spi after_init seam, HAS-A an AdvisorRegistry built at refresh; AdvisorDescriptor { id, order, role, &'static dyn Pointcut, make_interceptor fn-ptr } collected via the Phase-2 cross-crate substrate (linkme/codegen/explicit — candidates), origin-agnostic; pointcut.matches(bean_type, &method_meta_table) as a pure predicate over DECLARED method metadata (the rust-native replacement for reflective Method matching); CreatorPolicy as a capability lattice: Infrastructure-only floor, UPGRADED in-place to also consider Application-role advisors when user-aspect support is enabled (the @EnableAspectJAutoProxy 'upgrade', never two racing processors); role-tier as the PRIMARY ordering key + match-time eligibility filter (the @Role(ROLE_INFRASTRUCTURE) isolation, lines 612-618/1708); union + stable sort by (role-tier, order:i32, tiebreak-id) into one AdviceChain, installed via the upstream aop-proxying wrap-and-swap; getEarlyBeanReference entry point building the same chain early + an early_proxied dedup set so after_init won't double-wrap (line 2531)

**T1 — cross-crate/tooling:** The advisor COLLECTION is the T1 crux and it is origin-agnostic: AdvisorDescriptor is a thin const descriptor (id, order:i32, role, &'static dyn Pointcut, make_interceptor fn-ptr) — exactly the shape Architecture A/B (linkme distributed slice) favours, AND emittable explicitly by Architecture C/D, so the collection substrate stays a Phase-2 choice. leaf-core (or leaf-aop kernel) defines AdvisorDescriptor, Pointcut, AdviceChain, the CreatorPolicy, and the after_init seam; advice crates (tx/cache/validation) and aspect-model contribute descriptors. The DCE hazard applies directly here: a 'tx advisor crate' the binary never path-references contributes NO descriptors and silently disables @Transactional — so the anti-DCE self-check (cross-crate doc line 152/345) MUST cover the advisor descriptor stream: a startup self-check asserting expected advisor counts / a manifest cross-check, and the creator must expose an 'advisors discovered, by role/crate' introspection surface so the check is writable and `leaf doctor` can report it. Ordering is encoded AS DATA on each descriptor (order:i32 + role-tier + stable tiebreak), resolved at runtime by the creator's sort — NEVER link/section order (line 154/355). The CreatorPolicy capability-union must be assembled where the full enabled-feature set is visible (the binary/app-root per topology line 260) — which Phase-2 cross-crate mechanism computes that union is open. The two-source duality (line 362): the static link-time 'classpath' contributes descriptors at link time; an optional dynamic directory-scanned (WASM-module) source contributes descriptors at runtime — this realization handles the static source cleanly but a dynamic source must feed the SAME AdvisorRegistry and forces the 'when is the advisor set frozen / can we re-proxy' question (a Phase-2 candidate, not a decision). Generics: a generic advisor (Pointcut over Repo<T>) cannot register a concrete descriptor (generics-can't-register, line 153/349) → the contributing macro hard-errors with a per-instantiation shim. Matching operates over DECLARED method-metadata tables (no runtime reflection), so method identity must be stable-as-data (name-hash ids, ordering-as-data) or method-keyed pointcuts break silently across refactors.

**T2 — execution context:** Advisor COLLECTION + AdvisorRegistry build is COLD-PATH at context refresh (refresh-lifecycle), single-threaded-conceptually, before eager singletons — descriptor walk + topo/priority sort, no .await needed for the registry itself (descriptors are const data; make_interceptor fn-ptrs are NOT called yet). MATCH + UNION + ORDER + wrap-and-swap runs per-bean at after_init inside the per-bean creation guard (NOT a global lock, §2.5) on the cold instantiation path; building the chain may .await only if an interceptor factory touches async config (make_interceptor(&Container) could await a dependency) — execution-context discipline: the creator must document that interceptor construction runs on the refresh/creation context and a blocking factory MUST spawn_blocking. The getEarlyBeanReference path runs INSIDE cycle resolution and MUST be SYNC (no .await) — building the early chain must not re-enter the async creation machinery (re-entrancy hazard); the early_proxied dedup set is a fine-grained per-context structure read/written under the per-bean guard. The HOT path (per advised call) is OWNED by aop-proxying's AdviceChain walk, NOT this feature — this feature only ASSEMBLES the chain; once installed, per-call dispatch (boxed-future per dyn interceptor hop, §2.4 waker-driven) is the substrate's concern. Errors at match/assembly time (a pointcut that can't resolve, an interceptor factory that fails) propagate as a startup/refresh error carrying full context (which bean, which advisor, which role, the candidate advisors considered) into the §2.9 causal chain — preferring the richest diagnostic at refresh over a cryptic runtime surprise. Cancellation of refresh aborts registry build cleanly (no partial proxies published, all-or-nothing per refresh-lifecycle).

#### R2 — Precomputed bean→advisor binding plan (build-time match): matching is resolved at codegen/refresh into a static plan, the runtime only installs precomputed chains
Question whether matching needs to happen at runtime at all. Spring matches at after_init because it discovers @Aspect beans and Method objects reflectively at runtime. But leaf knows the bean types and their declared method metadata, AND the advisor pointcuts, as DATA at build/codegen time (or at the latest, at the start of refresh before any bean is built) — so MATCH can be precomputed into a `ProxyPlan`: a static map `bean_type_id -> &[AdvisorRef]` (already unioned, already ORDERED, already role-filtered). The runtime auto-proxy creator then does NO per-bean matching; at after_init it looks up the bean's precomputed advisor list (O(1)) and, if non-empty, calls the substrate's wrap-and-swap to install the chain (only the interceptor INSTANCES are built lazily at install time via make_interceptor; the WHICH-and-ORDER is frozen). Two sub-shapes, both candidates: (a) build-time codegen (Architecture C, the sqlx model): leaf-codegen walks the resolved graph + advisor descriptors + bean method-metadata and emits a checked-in/OUT_DIR `proxy_plan.rs` mapping each advisable bean type to its ordered advisor refs — fully deterministic, inspectable, --check-able, and the match diagnostics are COMPILE/BUILD-time (the richest, earliest per §2.9). (b) refresh-time plan: the same match runs once at the START of refresh (after descriptors collected, before beans built) producing an in-memory ProxyPlan, then bean creation just consumes it. Role-hint isolation and the capability-lattice are applied DURING plan computation (the plan only includes advisors the CreatorPolicy admits, ordered role-tier-first). The 'one creator wins' becomes 'one plan' — there is exactly one ProxyPlan per context, the union of enabled capabilities. getEarlyBeanReference consults the SAME plan (the early chain is the planned chain), so early-ref/after_init consistency is automatic (same plan, same answer).

**Pros:** Moves the match-and-order work from per-bean runtime to ONCE-per-context (sub-shape b) or COMPILE-TIME (sub-shape a), and makes the result a single inspectable artifact — the strongest §2.9 diagnostics story of the three: 'why is bean X proxied with chain [tx, cache, audit]?' is answerable by reading the plan, and (sub-shape a) match failures surface at build time, the earliest legible point.; Eliminates the O(beans × advisors) per-after_init match cost entirely — the after_init hot-ish cold path becomes an O(1) plan lookup, lowest startup overhead of the three (§2 low-overhead #2) for large bean graphs.; getEarlyBeanReference / after_init consistency is FREE: both consult the same frozen plan, so early-ref and final proxy carry the identical chain by construction — the double-wrap/earlyProxyReferences dedup problem (line 2531) dissolves because the answer is precomputed, not recomputed.; Sub-shape (a) is the most platform-portable and the cleanest fit for the wasm/embedded path (Architecture C, cross-crate doc line 315/232): the plan is plain generated code, no link-time slice walk, no DCE hazard for the MATCH (though advisor descriptors still need force-linking if collected via linkme) — and it dovetails with the AOT lineage (compile-vs-runtime split, §3).; The ProxyPlan is a clean, origin-agnostic data structure (type-id → ordered advisor-refs) that a dynamic source COULD extend by appending entries before freeze — keeping the two-source duality expressible (static plan + dynamic addenda) as a candidate.; Determinism + reproducibility: a checked-in --check-able plan (sub-shape a) turns 'silently lost an advisor' into a CI failure (the sqlx-prepare discipline, line 228/235), directly attacking the cross-crate doc's #1 silent-DCE failure mode for AOP wiring.

**Cons:** Build-time matching (sub-shape a) pre-judges the metadata/codegen boundary AND the compile-vs-runtime split (§3, two Phase-2 keystones) HARD — presenting a precomputed plan as THE shape biases those decisions, so it must stay strictly a CANDIDATE. The refresh-time variant (b) is less committal but still assumes the full advisor+bean-metadata set is known before any bean is built.; Conflicts with RUNTIME conditions (conditions/condition-family, profiles): whether a bean EXISTS and whether an advisor is ACTIVE can depend on runtime container state / properties / active profiles — so a build-time plan must either be conditional itself (plan-per-profile-combination, explosive) or fall back to runtime match for conditionally-present advisors, eroding the 'fully precomputed' benefit. The cross-crate doc is explicit that link-time collection answers 'what COULD contribute' but runtime conditions answer 'what DOES' (line 209) — the plan straddles that line uncomfortably.; Dynamic-source advisors (WASM-module) discovered AFTER the plan freezes cannot retroactively proxy already-built beans — strictly worse than R1's refreshable registry for the dynamic half of the two-source duality; a late advisor either misses beans or forces a re-plan + re-proxy (impossible once injected).; A precomputed bean→advisor map requires the match input (bean types + their method-metadata tables + all pointcuts) to be fully available to the planner — for sub-shape (a) that means leaf-codegen must see across crates (the cross-crate-codegen reach problem), and for both shapes it deepens the dependence on method-metadata-as-data being the chosen mechanism (pre-judging aop-proxying's deferred proxy/metadata keystone).; Pointcuts that are inherently DYNAMIC (match on a runtime argument value, not just type/method — Spring's dynamic pointcuts) cannot be fully precomputed: the plan can only freeze the STATIC (type/method) match; any per-call dynamic residue must still be evaluated at call time by the interceptor, so the plan is necessarily a STATIC over-approximation, not the whole answer.; Less faithful to Spring's 'decide at after_init' mental model — a developer debugging 'why isn't my advice firing' looks at the bean lifecycle in Spring; here they must look at a plan artifact, a different (arguably better, but divergent) mental model that changes the debugging surface.

**Mechanisms considered:** precompute MATCH+UNION+ORDER into a static ProxyPlan: bean_type_id -> &[ordered AdvisorRef], computed once; sub-shape (a) build-time codegen (Architecture C / sqlx model): leaf-codegen emits a checked-in/OUT_DIR proxy_plan.rs, --check-able, compile-time match diagnostics; sub-shape (b) refresh-time plan: match runs once at start of refresh (after descriptor collection, before bean build) into an in-memory plan; CreatorPolicy capability-lattice + role-tier filter applied DURING plan computation (plan only contains admitted advisors, role-tier-first ordered); runtime after_init reduced to O(1) plan lookup + lazy make_interceptor instantiation; WHICH/ORDER frozen, only interceptor INSTANCES built late; getEarlyBeanReference consults the SAME plan (early-ref/after_init consistency by construction, dissolving the earlyProxyReferences dedup); static plan + dynamic addenda (append before freeze) as the two-source-duality candidate; static-match-only with dynamic-pointcut residue evaluated at call time

**T1 — cross-crate/tooling:** Best fit for Architecture C (build-time codegen, the deterministic/portable path, cross-crate doc line 305-317/232): the ProxyPlan is a generated artifact, fully platform-independent incl. wasm/embedded, avoiding the MATCH-side DCE hazard entirely (the plan is plain code, force-linked by being referenced in generated wiring). The advisor DESCRIPTORS still need a collection substrate (linkme/codegen/explicit) and still carry the anti-DCE self-check obligation, but the MATCH RESULT is frozen, so a missing advisor manifests as a MISSING PLAN ENTRY that a checked-in --check-able plan (sqlx-prepare discipline, line 228/235) turns into a loud CI failure rather than silent runtime no-op — the strongest answer to the #1 silent-failure mode. CRITICAL T1 tension: the planner must see, across crates, BOTH the bean types + their method-metadata AND all advisor pointcuts — leaf-codegen in the binary crate can do this via cargo metadata / a manifest convention (it walks DECLARED inputs, not a true classpath, line 228/317), but advisors contributed by sibling crates must surface their descriptors to the planner — coupling plan generation to the cross-crate composition model. The capability-lattice (which creator-policy is active) is itself a binary/app-root decision (the only place the full feature set is visible, line 260). The two-source duality splits: static link-time advisors are fully plannable; a dynamic directory-scanned (WASM-module) source either appends to the plan before freeze (a candidate) or is unplannable and needs runtime match (R1's path) — so R2 favours the static source and degrades on the dynamic one. Generics-can't-register (line 153/349) applies to advisor descriptors AND to bean types in the plan — a generic bean needs per-instantiation plan entries, the macro hard-errors otherwise. Conditional presence (conditions/profiles) is the hard T1/T2 boundary: the plan is 'what could be advised' and must reconcile with runtime 'what is active' (line 209).

**T2 — execution context:** Plan computation is COLD-PATH: sub-shape (a) at BUILD time (no runtime execution context at all — the strongest 'fail before the program even runs' per §2.9); sub-shape (b) ONCE at the very start of refresh (refresh-lifecycle), before any eager singleton, single-conceptual-thread, no .await (it consumes const descriptors + method-metadata, produces a plan structure). At after_init the per-bean work is an O(1) plan lookup (no match cost) + lazy make_interceptor instantiation — the interceptor-factory call MAY .await (touching async config/dependencies) and runs on the creation context inside the per-bean guard (no global lock, §2.5); a blocking factory MUST spawn_blocking (execution-context discipline). getEarlyBeanReference consults the frozen plan SYNC (no .await, no re-entrancy into async creation) — and because the plan is the single source of truth, the early chain == the final chain with no dedup machinery. The HOT per-call path is again aop-proxying's chain walk, not this feature. Cancellation: build-time plans have no runtime cancellation; refresh-time plan build is part of all-or-nothing refresh (cancel aborts cleanly, no partial state). The reconcile-with-runtime-conditions case (T2-relevant): conditionally-present advisors that the plan can't statically resolve must defer to a runtime gate — that residual runtime decision runs on the refresh/creation context and must carry rich diagnostics if it diverges from the plan (a §2.9 'plan said X, runtime condition said Y' explanation). Errors at plan-build carry full context (bean, advisor, role, why-matched/why-not) — earliest-legible-diagnostic per §2.9.

#### R3 — Advisor-as-first-class-bean + per-injection-edge chain assembly (no central creator; collection via ObjectProvider, assembly at the resolution seam)
Question whether a CENTRAL creator should own collection and matching at all. In Spring the central creator exists because the BeanPostProcessor seam is the only interception point. But leaf controls the resolution seam (autowiring-resolution, deferral-primitives) and could model advisors as ordinary first-class beans collected via the SAME collection-injection / ObjectProvider machinery leaf already has — and assemble the per-bean chain WHERE the proxy decision is made, rather than in a bespoke central processor. Shape: an `Advisor` is just a bean implementing `trait Advisor { fn order(&self) -> i32; fn role(&self) -> Role; fn pointcut(&self) -> &dyn Pointcut; fn interceptor(&self) -> Arc<dyn Interceptor>; }`. The set of advisors is resolved as a `List<dyn Advisor>` via collection-injection (already ordered by @Order/@Priority — line: ordering is a consumer concern, separate from startup order) — so the auto-proxy logic REUSES collection-injection's ordering machinery rather than re-implementing a sort. The 'creator' shrinks to a thin policy bean (`ProxyAdvisorPolicy`) that holds the injected ordered advisor list and exposes `chain_for(bean_type, method_meta) -> Option<AdviceChain>`; the actual wrap-and-swap is still driven at the aop-proxying after_init seam (to preserve per-bean global wrapping), but the MATCH+ORDER logic is a pure function over the collection-injected advisor list — so the auto-proxy creator becomes a THIN consumer of existing container primitives (collection-injection for the set+order, conditions for which advisors are active, role for isolation) rather than a special subsystem. Role-isolation reuses candidate-resolver/qualifier semantics: Role::Infrastructure is modeled as an implicit qualifier, and the policy's capability set selects which role-qualified advisors are collected (the infra-floor-vs-aspect-upgrade lattice expressed as a qualifier filter on the collection-injection point). The 'only one creator wins' becomes 'there is only one ProxyAdvisorPolicy bean, and it collects the union of all active Advisor beans' — capability lattice as bean-presence (more capability = more advisor beans active via conditions/profiles), no racing processors. getEarlyBeanReference still needs the policy to answer early, so the policy must be resolvable before the cycle resolves (an infrastructure bean, eagerly available).

**Pros:** Maximal unification (§2.11, the entire reason Phases 2-4 exist): advisor collection REUSES collection-injection (set + @Order/@Priority ordering), advisor activation REUSES conditions/profiles, role-isolation REUSES candidate-resolver/qualifier matching, and the creator shrinks to a thin policy bean — no bespoke AdvisorRegistry, no bespoke sort, no special collection substrate. The auto-proxy creator stops being a subsystem and becomes a thin composition of primitives leaf already has.; Advisors-as-beans is the most Spring-faithful at the conceptual level — in Spring an Advisor IS a bean (BeanFactoryTransactionAttributeSourceAdvisor is a @Bean), and the infrastructure creator literally collects Advisor beans by role; this realization makes that explicit rather than hiding advisors in a separate descriptor stream, so the user mental model 'advice is contributed by beans' is preserved and inspectable via normal bean introspection.; Conditions/profiles compose for FREE: whether an advisor participates is just whether its Advisor bean is active (a @Conditional/@Profile decision, evaluated by the existing condition machinery) — directly answering the cross-crate doc's 'runtime conditions answer what DOES wire up' (line 209) WITHOUT a bespoke plan-vs-condition reconciliation (R2's hard problem dissolves here).; Role-isolation-as-qualifier reuses the candidate-resolver/qualifier semantics already designed, so @Role(ROLE_INFRASTRUCTURE) becomes 'an implicit qualifier on the Advisor type' and the infra-floor / aspect-upgrade lattice is a qualifier-filtered collection-injection point — no special role machinery, consistent with primary-fallback/candidate-resolver patterns.; Ordering inherits collection-injection's already-decided @Order/@Priority semantics (a consumer concern, separate from startup order), so the one-proxy-many-concerns ORDER (lines 1770-1772) is the SAME ordering users already understand from List<T> injection — one fewer ordering model to learn (§2 DX #1).; The thin-policy-bean is trivially overridable (§2.10 escape hatch): a user can supply their own ProxyAdvisorPolicy bean (primary/fallback) to fully customize matching/ordering without touching a framework subsystem — the most open extension surface of the three.

**Cons:** Advisors-as-beans means advisors must be RESOLVED (built) before they can match — but the auto-proxy creator runs DURING bean instantiation, creating a chicken-and-egg / bootstrap-ordering problem: the advisor beans (and their dependencies) must be fully built BEFORE any advisable bean's after_init, yet they're beans in the same graph. Spring solves this by treating BPPs+their references as a special early phase NOT eligible for auto-proxying (line 950: 'neither BPP instances nor beans they directly reference are eligible for auto-proxying') — R3 must reproduce that early-phase carve-out, a real complexity it inherits.; collection-injection ordering is a CONSUMER concern by design (separate from startup order) — but advisor ordering here is load-bearing for CORRECTNESS (cache-outside-vs-inside-tx, lines 1770-1772), not mere preference; conflating 'consumer ordering' with 'correctness-critical advice ordering' risks under-specifying the guarantee the auto-proxy creator must make. The ordering must be deterministic and validated, which collection-injection may treat more loosely.; getEarlyBeanReference needs the policy + its advisor set resolvable BEFORE the cycle resolves — but if advisors are beans that might themselves be in cycles or depend on advisable beans, 'resolve the advisor set early' re-enters the bootstrap-ordering problem at the worst time (mid-cycle, must be sync, no .await). The early-proxy story (already aop-proxying's most speculative sub-tier) gets MORE fragile here.; Reusing candidate-resolver/qualifier for role-isolation entangles AOP role semantics with the autowiring/qualifier subsystem — a change to qualifier matching could silently change which advisors are infrastructure-isolated, a coupling that cuts against the clean separation R1 gives (role as plain data on a descriptor, consumed only by the creator).; Pre-judges the resolution/ownership model (§3 keystones): 'collect advisors via ObjectProvider, assemble the chain as a thin policy over container primitives' entangles auto-proxy with HOW dependencies are handed over and HOW collection-injection materializes a List<dyn Advisor> — presenting it as a realization biases the ownership/resolution signal; must stay a CANDIDATE.; Origin-agnostic-but-heavier for dynamic sources: a dynamic (WASM-module) advisor must be registered as a BEAN (go through full bean-definition/instantiation) to participate, not just contribute a descriptor — heavier than R1/R2's flat descriptor for the dynamic source, though it does mean dynamic advisors get full lifecycle/conditions uniformly (a pro and a con).

**Mechanisms considered:** Advisor as a first-class bean (trait Advisor { order, role, pointcut, interceptor }) collected via collection-injection as List<dyn Advisor>, reusing @Order/@Priority ordering; the 'creator' shrinks to a thin ProxyAdvisorPolicy bean: holds the injected ordered advisor list, exposes chain_for(bean_type, method_meta) -> Option<AdviceChain>; wrap-and-swap still at aop-proxying after_init (preserving per-bean global wrapping); MATCH+ORDER as a pure fn over the collection-injected advisor list; Role::Infrastructure as an IMPLICIT QUALIFIER, infra-floor/aspect-upgrade lattice as a qualifier-filtered collection-injection point (reusing candidate-resolver); advisor ACTIVATION via conditions/profiles (an advisor participates iff its bean is active) — reusing the condition machinery, dissolving plan-vs-condition reconciliation; early-phase carve-out (advisors + their direct refs NOT auto-proxied, line 950) to break the advisor-must-exist-before-matching bootstrap cycle; user-overridable ProxyAdvisorPolicy bean (primary/fallback) as the escape hatch

**T1 — cross-crate/tooling:** Advisors-as-beans means advisor CONTRIBUTION reuses the SAME cross-crate bean-discovery substrate as every other bean (whatever Phase 2 picks for component-scanning/registration) — no SEPARATE advisor descriptor stream, which is a unification win BUT means the anti-DCE self-check for advisors is just the general bean-discovery self-check (a missing advisor crate = a missing bean, same silent-DCE failure mode, line 152/345, same mitigation). The ordering machinery is collection-injection's, already cross-crate-aware (ordering-as-data on each advisor bean: @Order/@Priority, resolved at runtime, never link order, line 154/355). Role-as-qualifier rides candidate-resolver's qualifier matching, also cross-crate. The capability-lattice (infra-floor vs aspect-upgrade) is expressed as WHICH advisor beans are active (conditions/profiles) + the policy's qualifier filter — computed at runtime over the collected bean set, naturally consistent with 'runtime conditions answer what DOES wire' (line 209), and assembled where the full feature set is visible (binary/app-root, line 260). The two-source duality (line 362): a static link-time advisor and a dynamic (WASM-module) advisor are BOTH just beans — the dynamic one registers as a bean via the dynamic source and participates through the SAME collection-injection point, the SMOOTHEST blend of the three for the dynamic half (no separate path), at the cost that dynamic advisors must go through full bean lifecycle. Generics-can't-register (line 153/349) applies as it does to any bean — a generic Advisor bean needs per-instantiation registration. The chicken-and-egg (advisors must be built before matching) is a cross-crate-AND-lifecycle concern: the early-phase carve-out (line 950) must hold across crates so an advisor in crate Y is built before an advisable bean in crate X is proxied — coupling to the (Phase-2) startup-ordering/refresh model.

**T2 — execution context:** The defining T2 challenge: advisors are BEANS, so they must be INSTANTIATED (async, on the creation context) BEFORE any advisable bean's after_init can match against them — this forces an explicit EARLY PHASE in refresh-lifecycle (the line-950 carve-out: build advisor beans + their direct dependencies first, and exempt THEM from auto-proxying to avoid infinite regress). That early phase is cold-path async (advisor beans may .await during their own construction), single all-or-nothing per refresh. Once advisors exist, MATCH+chain-assembly at after_init is a pure sync fn over the resolved List<dyn Advisor> (no .await for matching; interceptor() may return a pre-built Arc<dyn Interceptor> so no late async). getEarlyBeanReference is the sharpest hazard: it runs SYNC mid-cycle and needs the advisor set already resolved — so the early advisor phase MUST complete before any cycle that could trigger early-proxy exposure; if an advisor bean is itself cyclic this is a re-entrancy trap (must be detected and errored with rich §2.9 context, not deadlocked). Cancellation: advisor early-phase is part of all-or-nothing refresh (cancel aborts cleanly). The HOT per-call path is aop-proxying's chain walk (boxed-future per dyn hop, §2.4 waker-driven), unchanged. Execution-context discipline: advisor-bean construction and interceptor() run on the refresh/creation context; a blocking advisor factory MUST spawn_blocking. Reusing conditions/profiles for activation means condition evaluation (which advisors are active) runs at its normal refresh point and feeds the collection-injection set — no separate evaluation context. Errors (advisor failed to build, cyclic advisor, ambiguous role-qualifier) propagate as refresh errors with full causal chain (§2.9).

### Rust clashes
- No runtime classpath / no runtime reflection to DISCOVER advisors — Spring's AnnotationAwareAspectJAutoProxyCreator scans the running container for @Aspect beans and reflectively discovers their @Pointcut/@Before/@Around advice (line 1706); InfrastructureAdvisorAutoProxyCreator queries the BeanFactory for Advisor beans by role at runtime. Rust has neither a classpath to scan nor runtime reflection to enumerate annotated methods. So advisor COLLECTION must be either link-time descriptor registration (linkme — with its silent-DCE hazard), build-time codegen (a generated plan/registry), explicit registration, or advisors-as-beans riding the general bean-discovery substrate. There is NO 'scan the context for things annotated @Aspect at runtime' option — the discovery is structural/declared, decided by whatever cross-crate substrate Phase 2 picks, and carries the anti-DCE self-check obligation (a missing advisor crate silently disables @Transactional, never erroring).
- No reflective pointcut matching against java.lang.reflect.Method — Spring's pointcuts match against reflectively-discovered Method objects (signatures, annotations, modifiers) at runtime. Rust has no such runtime method metadata, so the auto-proxy creator's MATCH must operate over DECLARED method metadata (a generated method table — the data the deferred aop-proxying proxy mechanism produces). This means: (a) the matchable surface is fixed at build/codegen time, not reflectively open; (b) method identity for method-keyed pointcuts must be stable AS DATA (name-hash ids, ordering-as-data) or advice silently mis-matches across refactors; (c) annotation-based pointcuts (@Transactional on a method) require the annotation presence to be captured AS DATA in the method metadata at macro-expansion time, not read reflectively. The matcher is a pure predicate over data, which is cleaner and more testable than Spring's reflection — but the matchable vocabulary is whatever the metadata captures, not 'anything reflection can see'.
- The 'two families, only one wins' lattice has no clean Rust analogue as RACING BPP beans — Spring registers different auto-proxy creator beans and hardens registration so 'the most capable creator wins' (line 954: 'registration hardened so the most capable creator wins'), relying on runtime bean-registration ordering and override semantics. In Rust, link/registration ordering is UNSPECIFIED and may be deliberately randomized (cross-crate doc line 154/355) — so 'whichever creator registered first/last wins' is a non-deterministic hazard, NOT a reliable mechanism. leaf must re-express 'only one wins' as ONE creator whose CAPABILITY/POLICY is the deterministically-computed UNION of enabled capabilities (infra-floor UPGRADED to include aspects), assembled where the full feature set is visible (the binary/app-root, topology line 260) — never as competing beans whose link order decides the winner.
- @Role is metadata-not-enforcement in Spring (lines 612-618) and DOESN'T CASCADE — it is purely informational (consumed by the creator and by tooling), changes no bean behaviour, and a @Role on a @Configuration class does NOT propagate to its @Bean methods. Replicating this honestly in Rust means role must be plain DATA on each advisor descriptor/bean (set per-advisor, never inherited), consumed ONLY by the auto-proxy creator's match-filter and ordering — leaf must resist the temptation to make role a type-level marker or an enforced trait bound (which would make it enforcement, not metadata, and would cascade through type relationships), keeping it as orthogonal informational data exactly as Spring intends.
- One-proxy-many-concerns ORDERING is correctness-critical yet INVISIBLE in object identity (lines 1770-1772) — cache-outside-vs-inside-tx is a real bug you cannot see by inspecting the proxy. In Spring, ordering is by @Order/Ordered/@Priority and resolved at runtime. In Rust, ordering MUST be encoded as DATA (an order:i32 + role-tier + stable tiebreak on each advisor) and resolved by the creator's deterministic sort — NEVER link/section/constructor order (which is unspecified, line 154/355). Because the misordering is invisible (no per-feature proxy to inspect), leaf must make the assembled chain ORDER inspectable/diagnosable (a §2.9 'this bean's chain is [tx, cache, audit] in this order, because...' surface) — the rust-native answer to an invisibility Spring shares but leaf can do better on by surfacing the order as queryable data.
- The advisor-COLLECTION-before-MATCH timing collides with async bean construction — Spring builds the creator early and matches at after_init, but advisors-as-beans (R3) or interceptor-factories that depend on the container (make_interceptor(&Container)) mean some advisor machinery must be RESOLVED (possibly async) before matching. Spring's answer is the special early phase where BPPs + their direct references are NOT eligible for auto-proxying (line 950) — leaf must reproduce this carve-out to avoid infinite regress (the creator proxying its own advisors' dependencies), and the carve-out interacts with leaf's async-first construction: the advisor early-phase is cold-path async, but getEarlyBeanReference needs the advisor set resolved SYNC mid-cycle, a re-entrancy tension with no JVM equivalent (the JVM resolves this with synchronous lazy init under a lock; leaf cannot block the async creation machinery).
- getEarlyBeanReference dedup relies on GC'd reference identity and a mutable earlyProxyReferences set (line 2531) — Spring records that it already produced an early proxy so after_init returns the SAME object, relying on freely-aliased GC'd references. In Rust the early proxy and the final proxy must be the SAME owned/Arc'd handle, and the dedup set is shared mutable state on the cycle path that must be a fine-grained per-context structure (no global lock, §2.5). Whether this sub-tier is even needed depends on the still-open cycle/early-proxy/ownership story (aop-proxying's most speculative coupling) — and AOT contexts disallow cycles entirely (aop-proxying note re: line 785), so the early-proxy dedup may be runtime-only or unnecessary.
- Conditional advisor presence straddles the compile-vs-runtime line — link-time/build-time collection answers 'what advisors COULD participate', but whether an advisor is ACTIVE can depend on runtime conditions/properties/active profiles (conditions/condition-family/profiles), and the cross-crate doc is explicit that 'runtime conditions answer what DOES wire up' (line 209). Spring evaluates this uniformly at runtime. In Rust, a build-time precomputed match plan (R2) cannot fully resolve conditionally-present advisors without either plan-per-condition-combination (explosive) or a runtime fallback — so leaf must decide where the advisor-activation decision lives, and that decision is entangled with the (Phase-2) conditional strategy (cfg vs codegen vs runtime gates) — there is no single Rust mechanism that is both compile-time-deterministic AND runtime-condition-aware.

### Interactions
- **aop-proxying** — DIRECT downstream dependency — aop-proxying is the SUBSTRATE this feature drives. aop-proxying owns the wrap-and-swap seam (return a proxy in place of the target at after_init), the call-routing primitive (Interceptor/JoinPoint/proceed or the AdviceChain walk), and the early-reference hook; THIS feature is its PRIMARY consumer. The division is sharp: aop-proxying provides 'install this one ordered chain as the bean's proxy'; this feature DECIDES which advisors match, computes their union, and orders them into that one chain. This feature must NOT redefine the proxy mechanism — it stays mechanism-neutral and only assembles the chain the substrate installs.
- **aspect-model** — DIRECT upstream consumer — aspect-model is the user-facing layer that lets users declare @Aspect beans with @Pointcut/@Before/@After/@Around advice, which become advisors (an Advisor = a pointcut + an interceptor + order + role). aspect-model PRODUCES the AdvisorDescriptors (or Advisor beans in R3) that THIS feature COLLECTS, MATCHES, and ORDERS. The 'AnnotationAwareAspectJAutoProxyCreator additionally discovers @Aspect beans' relationship (line 1706) is realized as: this feature's CreatorPolicy, when upgraded to include user aspects, considers the Role::Application advisors that aspect-model contributes. This feature is advice-AGNOSTIC — it never knows what an advice does, only its pointcut/order/role.
- **post-processor-spi** — This feature IS installed as one of the post-processor-spi's instance-tier processors (the InstancePostProcessor whose after_init returns a possibly-different ErasedBean — the canonical 'a BPP returns a different object IS how AOP swaps in a proxy', line 1697). The early-reference hook uses the SPI's EarlyRefPostProcessor / getEarlyBeanReference sub-tier. CRITICALLY, this feature inherits the SPI's line-950 carve-out: the auto-proxy creator (itself a post-processor) and the advisors/interceptors it references must be in the early infrastructure phase and NOT eligible for auto-proxying themselves, or infinite regress results. leaf-core's post-processor-spi stays AOP-neutral; this feature plugs the collection+match+order policy into the neutral after_init seam.
- **refresh-lifecycle** — refresh-lifecycle drives WHEN this feature's collection happens: the AdvisorRegistry/ProxyPlan is built at a fixed point during refresh (after advisor descriptors/beans are available, before eager singletons are instantiated — analogous to Spring registering all BPPs before instantiating singletons). The advisor early-phase (the line-950 carve-out, especially load-bearing in R3 where advisors are beans) is an ordered refresh step this feature requires. The all-or-nothing refresh guarantee means a match/assembly failure aborts refresh cleanly with no partially-proxied beans published — a failure-as-early-as-richest-diagnostic-allows (§2.9) integration point.
- **bean-instantiation** — bean-instantiation calls this feature's match+wrap at the right point — after_init (so the proxy wraps a fully-initialized target) and the getEarlyBeanReference hook when a cycle demands it — all inside the per-bean creation guard (NOT a global lock). bean-instantiation publishes the possibly-different returned handle (the proxy) as the bean's identity, so every subsequent injection gets the proxy. This feature only DECIDES and ASSEMBLES; bean-instantiation owns the actual creation/publication lifecycle into which the decision is woven.
- **bean-registry** — bean-registry provides the singleton cache the proxy (not the raw target) is published into, and (if the early-proxy sub-tier survives Phase 2) the early-reference cache slot the getEarlyBeanReference dedup targets. This feature reads bean type/method metadata from the registry to perform matching. The 'who owns the raw target once the proxy wraps it' (Spring's allowRawInjectionDespiteWrapping mismatch) is a bean-registry ownership question this feature's wrap-decision surfaces but does not resolve — the raw target must not also be handed out raw, or some collaborators bypass advice.
- **circular-references** — The getEarlyBeanReference path is the integration point: if a singleton in a setter/field cycle is advised, the early reference handed to mid-cycle collaborators must be the SAME proxy (with the SAME assembled chain), not the raw target, or collaborators bypass advice (line 1768). This feature must answer 'what is this bean's chain?' consistently for both the early-ref and the after_init call — R2 (frozen plan) makes this consistency automatic; R1/R3 need the early_proxied dedup set. The early answer must be produced SYNC (no .await, no re-entry into async creation), which is the sharpest re-entrancy hazard, and AOT-disallows-cycles may make this sub-tier runtime-only.
- **transaction-management** — transaction-management contributes a Role::Infrastructure advisor (the TransactionInterceptor + its pointcut, the BeanFactoryTransactionAttributeSourceAdvisor analogue, line 1705/1712) that THIS feature collects, matches against @Transactional-marked methods, and orders into the chain. The infrastructure-vs-aspect isolation (@Role(ROLE_INFRASTRUCTURE)) keeps tx advice from being accidentally matched by application aspects — this feature's role-filter is what enforces that isolation. The classic cache-outside-vs-inside-tx ordering correctness (lines 1770-1772) is decided by this feature's deterministic role-tier+order sort.
- **caching** — caching contributes a Role::Infrastructure advisor (CacheInterceptor + pointcut) collected and ordered by this feature. Its ordering RELATIVE to the transaction advisor within the single shared chain is the canonical correctness concern (line 1772) that this feature's union+order policy makes possible and must surface as inspectable (since misordering is invisible in object identity). caching's ability to short-circuit proceed() (cache hit) is the substrate's (aop-proxying) concern, not this feature's — this feature only positions caching's interceptor in the chain.
- **validation** — validation contributes a Role::Infrastructure method-validation advisor (the @Validated analogue, line 1705/1712) that this feature collects, matches against validation-marked methods, and orders. Validation's need for TYPED argument access is the substrate's (aop-proxying) erased-vs-typed concern — this feature only contributes validation's interceptor to the chain at the role-isolated position the policy dictates.
- **async-execution** — async-execution contributes a Role::Infrastructure advisor (the AsyncAnnotationAdvisor analogue, dispatching proceed() onto a TaskExecutor, line 1712) collected and ordered by this feature. The @EnableAsync 'registers the auto-proxy-creator + the advisor' wiring (line 1712) maps onto: enabling async-execution contributes its advisor descriptor into the stream this feature collects AND ensures the CreatorPolicy considers infrastructure advisors. Its self-invocation/void-return caveat is aop-proxying's self-invocation-boundary concern, not this feature's.
- **retry-resilience** — retry-resilience contributes a Role::Infrastructure advisor (the RetryOperationsInterceptor analogue, line 1712) that wraps proceed() in a retry/backoff loop — collected and ordered by this feature. Its cold-path retry/backoff polling (permitted on cold paths, §2.4) lives in the interceptor, not in this feature's collection/match path. This feature positions it in the chain at its role/order-determined slot.
- **exception-translation** — exception-translation contributes a translating advisor (the @Repository-boundary advisor analogue) collected by this feature, with a pointcut matching repository-boundary-marked beans. It is one of leaf's own dogfooded infrastructure advisors riding this feature's collect+match+order pipeline — a concrete example of how a framework-supplied advisor uses the same collection contract as a user @Aspect.
- **scoped-proxies** — SIBLING use of the aop-proxying substrate, but mostly BYPASSES this feature's advisor-collection. scoped-proxies wraps a longer-lived bean's reference to resolve a live shorter-lived target per call — it does NOT collect cross-cutting advisors; it installs a single fixed scope-resolution interceptor. The interaction is at the SUBSTRATE level (both produce a proxy via aop-proxying), and the open question is whether scoped-proxies' wrap participates in the SAME single-proxy-per-bean as advice (one proxy carrying both scope-resolution AND cross-cutting advice) — a unification candidate (§2.11) this feature's single-chain model could absorb.
- **container-core** — container-core's engine-vs-façade split realizes the 'AOP silently does nothing in a bare engine' parity (line 49/110): the bare engine installs NO auto-proxy creator, so no collection/match/wrap happens and all advice is inert; the context façade installs THIS feature's creator (with its CreatorPolicy) that drives the aop-proxying substrate. container-core is where the single per-context creator and its capability-lattice policy are installed, fixing the boundary that makes AOP an opt-in-by-using-the-context capability — and the only place the full enabled-capability set (which determines the policy union) is visible.
- **collection-injection** — In R3 specifically, this feature REUSES collection-injection to gather advisors as an ordered List<dyn Advisor> (inheriting @Order/@Priority ordering), making the auto-proxy creator a thin consumer of collection-injection rather than a bespoke registry. Even in R1/R2 the relationship is conceptual: advisor ordering (@Order/Ordered/@Priority, lines 1770-1772) uses the SAME ordering vocabulary collection-injection defines, so leaf should share ONE ordering model (§2.11) rather than invent a second for advisors. The tension: collection-injection treats ordering as a consumer preference, but advisor ordering is correctness-critical.
- **conditions** — conditions/condition-family/profiles decide which advisors are ACTIVE (an advisor participates iff its condition holds — the @Conditional/@Profile on an advisor or its @Enable* selector). This feature must reconcile link/build-time advisor COLLECTION ('what could contribute') with runtime condition evaluation ('what does wire up', cross-crate doc line 209). In R3 this is FREE (an advisor is a conditional bean); in R2 (build-time plan) it is the hard plan-vs-condition reconciliation tension. The conditional strategy (cfg vs codegen vs runtime gate, §3) directly shapes where advisor activation is decided — a Phase-2 entanglement.
- **candidate-resolver** — In R3, role-hint isolation (@Role(ROLE_INFRASTRUCTURE)) is modeled by REUSING candidate-resolver's qualifier matching — Role::Infrastructure as an implicit qualifier on the Advisor type, the infra-floor/aspect-upgrade lattice as a qualifier-filtered collection point. This is a unification candidate (one qualifier mechanism serving both autowiring and advisor-role-isolation) but also a coupling risk (a qualifier-matching change could silently alter advisor isolation). In R1/R2, role stays plain data on a descriptor, decoupled from candidate-resolver — the cleaner separation.

### Open questions for Phase 2
- WHERE does advisor MATCHING happen — per-bean at runtime after_init (R1, the AbstractAutoProxyCreator analogue), or precomputed into a static bean→advisor PLAN at build-time/refresh-start (R2)? This cross-cuts the metadata/codegen boundary AND the compile-vs-runtime split (§3, two keystones): R2-build-time pre-judges hard toward codegen; R1/R2-refresh lean runtime. The choice determines startup overhead (O(beans×advisors) match vs O(1) lookup), diagnostic timing (build-time vs refresh-time vs runtime), and how cleanly conditional/dynamic advisors are handled. Held unresolved.
- Are advisors FIRST-CLASS BEANS collected via the general bean-discovery + collection-injection machinery (R3, maximal unification — advisor activation = conditions, ordering = collection-injection, role = qualifier), or a SEPARATE flat descriptor stream (R1/R2, cleaner separation but a bespoke collection path)? Advisors-as-beans maximizes §2.11 unification but inherits the chicken-and-egg bootstrap problem (advisors must be built before they can match, forcing the line-950 early-phase carve-out) and entangles AOP with the resolution/ownership model (§3). The unification-vs-bootstrap-complexity tradeoff is a Phase-2 keystone.
- WHERE does the wrap-DECISION live and is it per-bean-global or per-edge? Spring decides once at after_init (every external caller gets advice — the bean is wrapped once, globally). aop-proxying's R3 (resolution-seam middleware) would push the decision per-injection-edge, risking 'advise some injection points and not others'. This feature's collect+match+order policy assumes a single per-bean decision; if the proxy mechanism is per-edge, the creator must reconcile per-edge consistency. Cross-cuts aop-proxying's mechanism keystone and autowiring-resolution.
- How is the 'two families, only one wins' capability-LATTICE assembled deterministically across crates? The infra-floor-upgraded-to-aspects policy must be the UNION of enabled capabilities, computed where the full feature set is visible (the binary/app-root, topology line 260) — but WHICH Phase-2 cross-crate mechanism computes that union (a scan!/with list, a metadata sweep, a generated app-root)? And how does it avoid the link-order non-determinism (line 154/355) that makes 'whichever creator registered last wins' unreliable in Rust? Cross-cuts the cross-crate composition model.
- How is advisor-activation reconciled between link/build-time COLLECTION ('what could contribute') and runtime CONDITIONS ('what does wire up', cross-crate doc line 209)? A build-time match plan (R2) cannot fully resolve conditionally-present advisors (conditions/profiles) without plan-per-condition-combination explosion or a runtime fallback; advisors-as-beans (R3) get this free but pay the bootstrap cost. This is the conditional-strategy keystone (cfg vs codegen vs runtime gate, §3) viewed through advisor activation.
- Does the getEarlyBeanReference early-proxy sub-tier (and its earlyProxyReferences dedup, line 2531) earn its keep, and what does the creator answer EARLY? It depends on the still-open circular-references + scope/ownership decisions (who owns the raw target inside the proxy; does the registry have an early-proxy cache slot), must be produced SYNC mid-cycle (no .await, a re-entrancy hazard), and AOT-disallows-cycles may make it runtime-only or unnecessary. The most speculative coupling — cross-cuts circular-references, bean-registry, aop-proxying, the ownership keystone.
- How does the two-source duality (static link-time advisor 'classpath' PLUS optional dynamic directory-scanned WASM-module source) reconcile with each collection model? R1's refreshable registry handles dynamic advisors but raises 'when is the advisor set frozen / can we re-proxy already-built beans'; R2's frozen plan favours static and degrades on dynamic (a late advisor misses built beans); R3's advisors-as-beans blends smoothest (a dynamic advisor is just a bean) but forces dynamic advisors through full bean lifecycle. A candidate-comparison, not a decision.
- Is advisor ORDERING (correctness-critical: cache-outside-vs-inside-tx, lines 1770-1772) the SAME ordering model as collection-injection's @Order/@Priority (a consumer-preference concern), or a distinct correctness-grade ordering? Unifying them (§2.11) is attractive but collection-injection treats ordering loosely (a consumer concern separate from startup order) whereas advisor misordering is an invisible correctness bug. And since misordering is invisible in object identity, where does the assembled-chain ORDER get surfaced as an inspectable §2.9 diagnostic?
- What is the matchable METHOD-METADATA vocabulary, given no runtime reflection? Pointcut matching operates over DECLARED method metadata (the data the deferred aop-proxying proxy mechanism produces), so the matchable surface (signatures, annotation-presence, modifiers, return types) is fixed at macro-expansion/codegen time, NOT reflectively open. Method identity must be stable-as-data (name-hash, ordering-as-data) or method-keyed pointcuts silently mis-match across refactors. This is entangled with aop-proxying's metadata mechanism keystone — what the proxy generates determines what the creator can match on.

## Pointcut / Advice / Aspect Model  `[aspect-model]`

This feature is the USER-FACING AUTHORING layer for cross-cutting behavior — the top of the three-tier AOP stack. Below it: auto-proxy-creator (the COLLECT+MATCH+UNION+ORDER+role-isolation layer whose design I was handed and build on concretely) and, below that, aop-proxying (the wrap-and-swap + AdviceChain/JoinPoint/proceed substrate — proxy mechanism DEFERRED to Phase 2). aspect-model owns ONLY: (1) the POINTCUT vocabulary — a reusable, composable predicate that matches JOIN POINTS (which beans/methods a concern applies to); (2) the ADVICE-KIND vocabulary — before / after-returning / after-throwing / after (finally) / AROUND, each a way to interpose behavior relative to proceed(); (3) the @Aspect grouping — a reusable bean that bundles one or more (pointcut, advice) pairs with shared state/config; and (4) the LOWERING from those authored aspects into the flat AdvisorDescriptor/Advisor stream that auto-proxy-creator collects. It does NOT collect, match-against-beans, union, order, role-filter, wrap, or route calls — those are downstream.\n\nSpring's load-bearing intent I must replicate (grounded, lines 1635-1772, 1705-1712): (a) ADVICE-IS-ADDED-AROUND, NOT-COMPILED-IN (line 1639) — an aspect declares behavior the container interposes at the bean seam; the user's class stays a plain POJO unaware it is advised; the same mental model must hold in Rust (a #[component] gains advice without editing its body). (b) A POINTCUT IS A REUSABLE NAMED PREDICATE over join points (line 1706: @Pointcut/@Before/@Around) — Spring uses AspectJ expression strings (execution(* com..*Service.*(..)), @annotation(Transactional), within(..)); the predicate is decoupled from the advice so one pointcut feeds many advices. (c) AROUND IS THE GENERAL FORM — before/after are conveniences over an around that controls whether/when proceed() runs and can transform args/return/exceptions (lines 1635, 1712: the interceptor is what runs around the real call). (d) @Aspect BUNDLES + IS ITSELF A BEAN (line 1706: discovers user @Aspect beans, turning their advice into Advisors) — an aspect can hold injected collaborators (a metrics client, a tx manager) and shared pointcut definitions; it is wired by the container like any bean, then its advices become advisors. (e) AN ASPECT LOWERS TO ADVISORS (advisor = pointcut + interceptor + order + role) — the single unit auto-proxy-creator consumes; @Order on the aspect/advice sets chain position (lines 1770-1772, correctness-critical, invisible in object identity). (f) THE SELF-INVOCATION BOUNDARY (lines 1714-1737) — advice attaches to the inter-bean seam, never to internal this.method() control flow; this is an honest property of the wrapper model that aspect-model must communicate to users (a this-call won't be advised), NOT paper over. (g) INERT-MARKER → WIRING → BEHAVIOR (line 1712) — Spring's own cross-cutting features (tx/cache/validation/async/retry, exception-translation) are just framework-shipped aspects/advisors using THIS SAME authoring substrate; aspect-model must be the one vocabulary both users and leaf's own features author against (unification, §2.11).\n\nI must replicate this INTENT in rust-native terms WITHOUT the JVM mechanisms: no AspectJ expression LANGUAGE evaluated by a runtime weaver, no reflective java.lang.reflect.Method to match against, no runtime discovery of @Aspect beans by classpath scan. A pointcut becomes a predicate over DECLARED metadata (a generated method/type descriptor table), authored either as a typed Rust predicate, a parsed-at-macro-time expression mini-language, or attribute-driven markers — all candidates. An aspect becomes a normal leaf bean carrying advice closures/methods; its lowering to AdvisorDescriptors is THIN-macro codegen emitting exactly the descriptor shape auto-proxy-creator already defined. I do NOT decide: the proxy mechanism (aop-proxying keystone), the advisor-COLLECTION substrate (linkme vs codegen vs explicit — Phase 2 T1, already flagged by auto-proxy-creator), how matching is staged (runtime vs precomputed plan — auto-proxy-creator's open question), or the scope/ownership the chain rides on. I own only: the pointcut authoring + composition model, the advice-kind model, the @Aspect bean model, and the aspect→advisor lowering — all collection/match/proxy-mechanism-neutral.

### Candidate realizations

#### R1 — Typed trait-based aspects: Advice as async trait methods, Pointcut as a typed predicate fn, @Aspect as a #[aspect] bean lowering to AdvisorDescriptors
Author aspects as ordinary Rust: a #[aspect] bean is a normal leaf component implementing one or more advice traits, and a pointcut is a TYPED predicate value, not a parsed string. Core shapes (placeholder names): `trait Pointcut: Send + Sync { fn matches(&self, jp: &JoinPointMeta) -> bool; }` where JoinPointMeta is the DECLARED method/type metadata auto-proxy-creator already matches over (type-id, method-name-hash, attribute-presence flags, arg/return type-ids). Advice is expressed as async trait methods on the aspect bean, one trait per kind so the kind is in the type: `trait AroundAdvice { async fn around(&self, jp: JoinPoint<'_>) -> AdviceOutcome; }` (jp.proceed().await is the call-through), plus thinner `trait BeforeAdvice { async fn before(&self, jp: &JoinPointInfo<'_>); }`, `AfterReturningAdvice`, `AfterThrowingAdvice`, `AfterAdvice`. before/after are sugar the lowering wraps into an around interceptor (around is the general form). A #[aspect] struct declares its (pointcut, advice-method) bindings via thin attributes: `#[advice(around, pointcut = svc_pc, order = 100)] async fn time_it(&self, jp: JoinPoint<'_>) -> AdviceOutcome { ... }` where `svc_pc` is a const/fn pointcut value (a typed predicate built from composable combinators: `pointcut::within::<dyn OrderService>() & pointcut::annotated::<Transactional>() & pointcut::method_returns::<Result<_,_>>()`). The #[aspect] macro (THIN) emits, per advice binding, one `AdvisorDescriptor { id, order, role: Role::Application, pointcut: &'static dyn Pointcut, make_interceptor: |c| { let aspect = c.resolve::<Self>(); Arc::new(AdviceInterceptor::around(aspect, &Self::time_it)) } }` into whatever collection substrate Phase 2 picks — i.e. it lowers exactly to auto-proxy-creator's already-defined descriptor. The aspect bean itself is resolved by the container (so it can inject collaborators); the interceptor closes over an Arc to it. Pointcut combinators are pure values composable with &/|/! and reusable across advices (the @Pointcut-reuse intent). leaf's own tx/cache/validation features author the SAME way but emit Role::Infrastructure descriptors.

**Pros:** Most idiomatic-Rust and most type-checked: a pointcut is a value with combinators (composable like iterator adapters), advice is a plain async fn, the advice KIND is encoded in which trait/attribute is used — the compiler checks signatures, `cargo expand` shows exactly the AdvisorDescriptor a user could have hand-written (thin macros, §2.10/§10). No expression-string parser to build, test, or diagnose.; Pointcut-as-typed-predicate makes the matchable vocabulary HONEST about the no-reflection reality (rustClash): you can only match on what JoinPointMeta exposes as data, and that surface is a normal Rust API (within::<T>, annotated::<A>, returns::<R>) — discoverable, autocompletable, refactor-safe (a renamed type breaks the within::<T> at compile time rather than silently un-matching a stale string).; Aspects-are-beans falls out for free: a #[aspect] is just a #[component], so it injects collaborators, participates in conditions/profiles (an aspect is active iff its bean is active), and is introspectable as a normal bean — directly honoring line 1706 (@Aspect IS a bean) and unifying with the whole container model (§2.11).; around-as-general-form with before/after as lowering-sugar matches Spring's interceptor reality (line 1712) exactly and keeps the substrate (aop-proxying) seeing only ONE interceptor kind — the lowering collapses the five advice kinds into one around-shaped AdviceInterceptor, so aop-proxying's AdviceChain stays uniform.; Lowers cleanly to auto-proxy-creator's EXISTING AdvisorDescriptor contract (id, order, role, &'static dyn Pointcut, make_interceptor fn-ptr) with zero new collection machinery — aspect-model becomes purely the authoring + lowering layer, no overlap with the downstream feature, sharp division of labour.; Origin-agnostic: the emitted AdvisorDescriptor is a flat const with a fn-ptr factory and a &'static dyn Pointcut — a WASM-module-sourced aspect or a test aspect lowers to the identical shape, so dynamic sources slot in unchanged.

**Cons:** A &'static dyn Pointcut requires the predicate to be const-constructible (for the linkme path) — composable combinators (within & annotated) that allocate or capture can't be const, forcing either a const combinator algebra (restrictive) or an inventory/runtime-built pointcut (non-const, pre-judging the collection substrate toward inventory and away from zero-cost linkme). Couples to the (Phase-2) collection-substrate choice.; Less faithful to Spring's pointcut EXPRESSION ergonomics (line 1706: execution(* ..Service.*(..))) — a one-line AspectJ string becomes a combinator chain; powerful and typed but more verbose for the common cases, and migrating Spring users must learn a new vocabulary rather than reusing muscle memory (DX cost on the parity-feel axis, §2.1).; Encoding advice-kind as distinct traits/attributes multiplies the surface (five traits or five attribute variants) and the lowering must wrap each into the around shape — more codegen paths to keep thin and consistent; an around that forgets to call proceed() (drops the call) is a logic bug the type system can't fully prevent.; JoinPoint<'_> with proceed() that returns the typed result is hard to make both dyn-safe AND typed: the substrate erases the call, so around advice likely sees an erased AdviceOutcome (boxed/Any) rather than the concrete return type — typed argument/return access (the @Around(value) intent) is limited unless aop-proxying provides typed shims, pre-judging that deferred keystone.; The pointcut predicate runs against JoinPointMeta whose CONTENTS are produced by the deferred proxy/metadata mechanism — this realization implicitly assumes method metadata exists as rich-enough DATA (attribute-presence, type-ids) to express useful pointcuts, leaning on aop-proxying's metadata keystone before it's decided.; async advice methods on a dyn-dispatched aspect inherit RPITIT/AFIT-not-dyn-safe (nightly doc §RPITIT): the around() future must be boxed (Pin<Box<dyn Future>>) at the dyn interceptor boundary, an unavoidable per-hop alloc on the (substrate-owned) hot path; aspect-model must shape its advice trait so this boxing lives at the lowering seam, not leak into authoring.

**Mechanisms considered:** Pointcut as `&'static dyn Pointcut` predicate over JoinPointMeta (declared metadata), built from composable typed combinators (within::<T>, annotated::<A>, returns::<R>) with &/|/! operators; advice-kind as distinct async traits (AroundAdvice/BeforeAdvice/AfterReturning/AfterThrowing/After) OR a single #[advice(kind=...)] attribute on an aspect method; before/after lowered into around; #[aspect] = #[component] + thin codegen that emits one AdvisorDescriptor per advice binding (Role::Application), make_interceptor closing over Arc<the resolved aspect bean>; JoinPoint<'_> { proceed().await, info } passed to around advice; before/after get a read-only JoinPointInfo; aspect bean resolved by the container so advice can inject collaborators; conditions/profiles gate aspect activation via normal bean activation; lowering collapses all five kinds to one around-shaped AdviceInterceptor so aop-proxying's AdviceChain sees a uniform interceptor; leaf's own tx/cache/validation/exception-translation author via the SAME traits but emit Role::Infrastructure descriptors

**T1 — cross-crate/tooling:** An aspect is authored in any crate (user or framework) and the #[aspect] macro emits AdvisorDescriptor(s) into the SAME collection substrate auto-proxy-creator consumes — so aspect-model inherits ALL of that feature's T1 obligations and adds none of its own collection machinery. The macro is THIN (emits one const descriptor + an interceptor-factory closure per advice binding, fully hand-writable, absolute paths ::leaf_aop::… per proc-macro hygiene). The DCE hazard applies: an aspect crate the binary never path-references contributes NO descriptors and its advice silently never fires — so the same anti-DCE force-link + startup self-check that auto-proxy-creator mandates must cover authored aspects, and `leaf doctor` should list "aspects discovered, by crate, with their pointcut summaries" so a vanished aspect is loud not silent. CONST-POINTCUT tension (the sharp T1 point unique to this realization): a &'static dyn Pointcut built from combinators must be const-constructible for the linkme/distributed-slice path (zero-cost, no pre-main); a combinator algebra that allocates pushes toward inventory (life-before-main, non-const) — so the pointcut combinator design is coupled to the Phase-2 substrate choice and the macro must hard-error if a user writes a non-const pointcut on the linkme path. Generics-can't-register (cross-crate doc line 153/349): an aspect targeting Repo<T> can't emit a concrete descriptor — the macro hard-errors and asks for a per-instantiation register_aspect!(Repo<ConcreteT>) shim. Method/type identity in pointcuts must be stable-as-data (type-id, name-hash) or a refactor silently changes matches — the typed within::<T>/annotated::<A> form mitigates this (compile error on rename) better than a string. The two-source duality (line 362): a static link-time aspect and a dynamic WASM-module aspect both lower to the same descriptor shape; the dynamic one needs its pointcut and interceptor-factory to be expressible without native const-construction (a runtime-built Pointcut), so R1's typed-const pointcut favours the static source and needs a runtime-pointcut variant for the dynamic one.

**T2 — execution context:** Authoring/lowering is COMPILE-TIME (macro expansion) — no runtime execution context; the emitted AdvisorDescriptor is const data. The aspect BEAN is constructed on the COLD-PATH at refresh like any component (async, on the creation context — it may .await to inject collaborators); make_interceptor(&Container) resolves the aspect Arc and builds the AdviceInterceptor, also cold-path at the auto-proxy-creator's match/assembly point (a blocking aspect constructor MUST spawn_blocking — execution-context discipline). The HOT path is per advised call: around()'s future is driven by the substrate's AdviceChain walk; because AFIT/RPITIT are not dyn-safe (nightly doc §RPITIT/§AFIT), each around hop is a boxed future (Pin<Box<dyn Future + Send>>) — waker-driven, never polled busy (§2.4); aspect-model must keep this boxing at the dyn interceptor seam so authoring stays plain `async fn`. jp.proceed().await is the cancellation/error propagation point: cancelling the outer call drops the around future, which must drop its proceed() future cleanly (no async-Drop available, nightly doc §AsyncDrop — so an around holding a resource across proceed() must release it via explicit RAII/sync Drop, not async cleanup); an error from proceed() surfaces as the typed AdviceOutcome the around can observe/transform (after-throwing semantics) or propagate. Errors at lowering/authoring time (a pointcut referencing an unknown type, a non-const pointcut on the linkme path) are COMPILE errors with full context (§2.9 earliest-legible). A mis-authored around that never calls proceed() is a runtime logic bug — diagnosable by a debug-mode chain tracer but not preventable at compile time.

#### R2 — Expression-string pointcuts parsed at macro-expansion time into typed predicates (AspectJ-feel authoring, zero runtime parser)
Preserve Spring's pointcut-EXPRESSION ergonomics but move ALL parsing to compile time. Authors write a pointcut as a string in an attribute — `#[pointcut("execution(* ..service.*Service.*(..)) && @annotation(Transactional)")]` or a named `#[pointcut(name = "services", "within(..service..) && !@annotation(NoAudit)")]` reusable across advices — and the leaf-aop proc-macro PARSES that mini-language at expansion time into the SAME typed Pointcut predicate R1 builds with combinators, emitting it as generated code. There is NO runtime expression evaluator (unlike Spring's AspectJ weaver): the string is a compile-time DSL whose grammar (execution/within/@annotation/args/this/target/this-returns, &&/||/!) lowers to a const/generated Pointcut over JoinPointMeta. Advice is still authored as async methods on a #[aspect] bean: `#[around("services")] async fn audit(&self, jp: JoinPoint<'_>) -> AdviceOutcome`. The macro resolves the named pointcut, parses/validates the expression against the KNOWN metadata vocabulary (it can check that `@annotation(Transactional)` refers to a real attribute type in scope, that a within type path resolves), and emits the AdvisorDescriptor exactly as R1 does. Unparseable or unresolvable expressions are COMPILE errors with a caret pointing into the string (rich diagnostic, §2.9). Two sub-shapes as candidates: (a) the grammar lowers to R1's typed combinators (so R1 and R2 share the predicate runtime, R2 is just a front-end); (b) the grammar lowers to a flat interpreted-at-runtime PointcutProgram (a small bytecode over JoinPointMeta) — still no string parsing at runtime, just a data-driven matcher, which is more uniform for a dynamic (WASM-module) source that ships pre-compiled pointcut programs.

**Pros:** Highest Spring-parity FEEL (§2.1, DX #1): execution(..)/within(..)/@annotation(..) is exactly the muscle memory Spring users have; a migrating user copies their pointcut expression nearly verbatim, which is the single biggest authoring-DX win of the three for the parity audience.; Compile-time parsing keeps it stable and zero-runtime-cost (DX #2 overhead): no expression evaluator ships in the binary, the string never exists at runtime, and a malformed expression fails at COMPILE time with a precise span — strictly better diagnostics than Spring's runtime AspectJ parse errors (§2.9 earliest-legible-richest).; Pointcut reuse (the @Pointcut-as-named-definition intent, line 1706) is natural: a named pointcut string is defined once and referenced by many advices across the aspect — and (sub-shape b) a pointcut program is a portable data artifact a dynamic source can ship, the cleanest two-source-duality story of the three.; Front-ends onto R1 (sub-shape a): the string grammar and the typed combinators can be the SAME predicate runtime with two authoring surfaces — users pick expression-string (familiar) or typed-combinator (refactor-safe), unifying rather than competing (§2.11). aspect-model offers both feels over one matcher.; The compile-time parser can VALIDATE against the declared metadata vocabulary — flag an @annotation(Foo) where Foo isn't a known attribute, or a within(path) that resolves to nothing — turning Spring's silent never-matches into a loud compile error (a diagnostic Spring cannot give because it parses lazily at runtime).; Thin-macro-compatible in spirit: the parser lives in a NORMAL crate (leaf-codegen/a parsing lib), not in the proc-macro's hidden logic; the proc-macro calls it and emits hand-writable typed-predicate code — `cargo expand` shows the lowered combinators, so nothing is hidden in the macro (§10).

**Cons:** A pointcut mini-language is a real GRAMMAR to design, parse, version, and document — the heaviest authoring-surface of the three, and a string DSL cuts against "thin macros emit only hand-writable code" unless the lowered output (R1 combinators) is exactly hand-writable and shown by cargo expand. Risk of the macro becoming a logic-heavy mini-compiler (§10 tension).; String→type resolution at macro time is LIMITED by proc-macro single-invocation scope (cross-crate doc lines 88-92): the macro sees only its invocation tokens, so `within(::other_crate::Foo)` or `@annotation(SomeAttr)` can only be NAME-checked syntactically, not type-resolved across crates — so the "validate against the vocabulary" pro is partial; some errors still defer to a later pass, weakening the compile-time-diagnostic claim for cross-crate references.; Expression matching surface is bounded by what JoinPointMeta exposes (same no-reflection rustClash as R1) — but a string LOOKS like full AspectJ, so it over-promises: users will write execution patterns or args(..) binding that the metadata can't support, and the gap between AspectJ-the-language and leaf's-subset is a documentation/expectation hazard Spring doesn't have (Spring really does have reflective Method access).; Sub-shape (b)'s runtime PointcutProgram interpreter is a per-match data-driven evaluation — cheap, but it IS a small runtime matcher (vs R1's monomorphized predicate), a slight overhead and a second matching path to maintain; and it pre-judges toward the runtime-match (not precomputed-plan) staging that auto-proxy-creator left open.; Refactor-safety regresses vs R1: a renamed type or moved method silently breaks a within(..)/execution(..) string (it just stops matching) unless the compile-time resolver is strong enough to catch it — and per the cross-crate scope limit it often isn't, re-introducing exactly the stale-string-silent-mismatch hazard the typed form avoided.; Still inherits the const-pointcut / collection-substrate coupling (sub-shape a) and the dyn-async boxing of R1 — R2 adds an authoring front-end, not a different runtime, so all of R1's lower-level T1/T2 tensions remain.

**Mechanisms considered:** pointcut expression mini-language (execution/within/@annotation/args/this/target + &&/||/!) PARSED at macro-expansion time, never at runtime; named #[pointcut(name=.., "...")] definitions reusable by many #[around("name")]/#[before("name")] advices on a #[aspect] bean; sub-shape (a): grammar lowers to R1's typed combinator predicate (shared runtime, expression-string is a front-end); sub-shape (b): grammar lowers to a flat data-driven PointcutProgram matched over JoinPointMeta at runtime (portable, dynamic-source-friendly); compile-time validation of the expression against the known declared-metadata vocabulary (loud error on unknown attribute/unresolved within), within proc-macro single-invocation limits; parser lives in a normal leaf-codegen/parsing crate; proc-macro calls it and emits hand-writable lowered predicate code (cargo expand legible); emits the SAME AdvisorDescriptor (Role::Application; Role::Infrastructure for framework features) auto-proxy-creator consumes

**T1 — cross-crate/tooling:** Parsing is fully compile-time, so no runtime expression artifact crosses crate boundaries — the emitted output is the same AdvisorDescriptor + Pointcut as R1, inheriting auto-proxy-creator's collection/DCE/force-link/self-check obligations identically. The DISTINCTIVE T1 limit is proc-macro single-invocation scope (cross-crate doc lines 88-92): the expression parser runs inside leaf-aop's proc-macro during expansion of ONE aspect and sees only that invocation's tokens — it cannot enumerate types in sibling crates or resolve a cross-crate type path, so within(::other::Foo) / @annotation(OtherAttr) can be syntax-checked and recorded as a type-id-to-be-resolved but not fully validated until a later (runtime or generated-aggregator) pass; this splits diagnostics: local references fail at compile time (rich), cross-crate references may defer (still must carry full context per §2.9). Sub-shape (b)'s PointcutProgram is a portable data artifact that a dynamic WASM-module source can ship and the runtime matches uniformly — the smoothest two-source-duality fit (static aspects compile their program at build time, dynamic aspects ship a pre-built program), at the cost of a runtime matcher. Generics, stable-method-identity, force-link, and the const-pointcut/substrate coupling are all as R1 (R2 is a front-end). Crucially, the GRAMMAR and parser belong in a normal crate (leaf-codegen), NOT the proc-macro's hidden logic, to honor thin-macros (§10) — the proc-macro must emit code a user could have written against the typed API.

**T2 — execution context:** Expression parsing + lowering is COMPILE-TIME (macro/codegen) — no runtime context; malformed expressions are compile errors with a span into the string (§2.9 strongest authoring-time diagnostic). Sub-shape (a) has the identical runtime profile to R1 (monomorphized predicate, cold-path match, hot-path boxed-future around). Sub-shape (b) adds a COLD-PATH (or precomputed) PointcutProgram evaluation over JoinPointMeta at the auto-proxy-creator match point — pure data interpretation, no .await, no blocking; the HOT advised-call path is unchanged (substrate AdviceChain, boxed around futures, waker-driven §2.4). Aspect-bean construction and interceptor assembly are cold-path async as in R1 (spawn_blocking for a blocking aspect ctor). Cancellation/error propagation through jp.proceed().await is identical to R1 (drop-on-cancel, AdviceOutcome carries the error for after-throwing); no async-Drop (nightly §AsyncDrop) so around-held resources release via sync RAII. The only added T2 surface is the (cold-path or build-time) parse step, which has no runtime execution-context cost.

#### R3 — Aspect-as-composed-interceptor-bean: no distinct pointcut/advice vocabulary; an aspect is a bean that contributes a (matcher, around-middleware) pair, reusing collection-injection + conditions, with before/after as combinators
Question whether aspect-model needs a bespoke pointcut LANGUAGE and advice-KIND taxonomy at all, or whether an aspect is just a bean that contributes an ordered (matcher, middleware) pair — the tower/middleware shape, expressed as a first-class leaf bean. Shape: the unit of authoring is `trait Aspect: Send + Sync { fn matcher(&self) -> &dyn Pointcut; fn order(&self) -> i32; fn role(&self) -> Role; fn intercept(&self) -> Arc<dyn Interceptor>; }` — i.e. an aspect IS (already) the around-middleware + its pointcut + its position, with NO separate before/after trait surface. before/after/after-returning/after-throwing are AUTHORED as ergonomic COMBINATORS that build the around interceptor: `Advice::around(|jp| async { ... })`, `Advice::before(|info| async {...})`, `Advice::after_returning(...)`, composed with `.and_then(..)` — a small fluent builder, all plain functions (thin/no macro needed; a #[aspect] attribute is pure sugar that calls the builder). Crucially, an Aspect is a NORMAL bean collected via the SAME machinery leaf already has: aspects are gathered as a `List<dyn Aspect>` via collection-injection (inheriting @Order/@Priority ordering), activated via conditions/profiles (an aspect participates iff its bean is active), and role-isolated via the role() hint — so aspect-model contributes NO new collection/match/order/registry; it rides auto-proxy-creator's consumption of Aspect-beans-as-advisors. The pointcut is whatever the bean returns from matcher() — could be a typed predicate (R1), a parsed program (R2), or a hand-written closure — aspect-model stays AGNOSTIC about pointcut authoring and just requires "an aspect yields a matcher". This is the maximal-unification, minimal-vocabulary realization: an aspect is a bean that says "match these join points, run this middleware, at this order/role."

**Pros:** Maximal unification (§2.11): aspect COLLECTION = collection-injection (List<dyn Aspect>, @Order/@Priority ordering reused), aspect ACTIVATION = conditions/profiles (an aspect is active iff its bean is — dissolving the build-time-vs-runtime-activation problem auto-proxy-creator R2 struggled with), role-isolation = the role() data hint; aspect-model adds NO bespoke machinery and becomes a thin trait + a combinator builder over primitives leaf already has.; Smallest authoring vocabulary and zero mandatory macro: an aspect is a bean implementing one trait; advice kinds are library combinators (before/after/around as functions), so the hand-writable API IS the whole feature and the #[aspect] macro is pure optional sugar — the strongest §10 (thin-macros/everything-hand-writable) and §2.10 (escape hatch at every level) story of the three.; around-as-the-only-real-kind is the most honest model of the substrate (aop-proxying sees one interceptor): before/after are explicitly DERIVED combinators, so there's no five-trait surface to keep consistent and no lowering ambiguity — the builder is the single source of truth for how a before becomes an around.; Aspects-as-beans is the most Spring-conceptually-faithful at the wiring level (line 1706: @Aspect IS a bean discovered and turned into advisors) AND the most rust-framework-idiomatic (tower Layer/Service, axum middleware) — it meets both the Spring parity feel and the Rust ecosystem's mental model, easing adoption from either direction.; Pointcut-authoring-agnostic: because aspect-model only requires matcher() -> &dyn Pointcut, R1's typed combinators and R2's expression strings can BOTH be offered as pointcut front-ends without aspect-model committing — it cleanly separates "how you write a pointcut" (a sub-decision) from "what an aspect is" (this feature), reducing what Phase 2 must entangle.; Dynamic-source-smoothest: a WASM-module aspect is just another Aspect bean registered through the dynamic source and collected via the same List<dyn Aspect> point — no separate descriptor path, the cleanest two-source-duality blend (matching auto-proxy-creator's R3 finding).

**Cons:** Inherits the chicken-and-egg bootstrap problem auto-proxy-creator R3 flagged: aspects-as-beans must be BUILT before they can match/advise other beans, forcing the line-950 early-phase carve-out (aspect beans + their direct dependencies NOT themselves auto-proxied) — a real lifecycle complexity aspect-model's authoring choice pushes onto refresh-lifecycle and bean-instantiation.; Collapsing the advice-KIND taxonomy into combinators loses some declarative legibility: Spring's @AfterThrowing(throwing="ex") is a self-documenting annotation; a `Advice::after_throwing(|e| ...)` closure is equivalent but the KIND is in a method name not a declaration, so tooling/introspection ("what kinds of advice target this bean?") reads less cleanly than R1/R2's explicit kind attributes (a minor DX/diagnosability cost on §2.9 introspection).; Pushing pointcut authoring out of this feature (agnostic matcher()) means aspect-model under-specifies the single thing users most associate with @Aspect — the pointcut DX (the execution(..) feel) is deferred to a sub-decision, so on its own R3 doesn't deliver the Spring-parity authoring feel R2 does; it relies on R1/R2 being layered in as the matcher front-end.; Heavier per-aspect runtime identity: each aspect is a full bean (lifecycle, conditions, injection) rather than a flat const descriptor — richer (a pro) but more allocation/indirection than R1's const AdvisorDescriptor, and a generic Aspect bean still can't auto-register (generics-can't-register, cross-crate doc line 153/349), needing a per-instantiation shim.; Reusing collection-injection's ordering (a CONSUMER-preference concern, separate from startup order) for advice ordering — which is CORRECTNESS-critical (cache-outside-vs-inside-tx, lines 1770-1772) — conflates a loose ordering model with a strict one (the same tension auto-proxy-creator flagged); aspect-model must demand a correctness-grade deterministic order from a machinery designed for preference, or surface the order as an inspectable diagnostic.; Entangles aspect authoring with the resolution/ownership model (§3 keystone): "an aspect is a bean collected via collection-injection and resolved as List<dyn Aspect>" pre-judges how dyn beans are stored/handed over — presenting it as the authoring shape biases the ownership/registry signal, so it must stay a CANDIDATE.

**Mechanisms considered:** Aspect as a first-class bean: trait Aspect { matcher() -> &dyn Pointcut, order(), role(), intercept() -> Arc<dyn Interceptor> }; before/after/after-returning/after-throwing/after as fluent COMBINATORS (Advice::before(..).and_then(around(..))) building one around interceptor — no separate advice-kind traits; aspects collected as List<dyn Aspect> via collection-injection (reusing @Order/@Priority ordering), activated via conditions/profiles, role-isolated via role(); #[aspect] attribute as pure OPTIONAL sugar that calls the hand-writable builder; the trait + builder is the full feature; matcher() pointcut-authoring-AGNOSTIC: accepts a typed predicate (R1), a parsed program (R2), or a hand-written closure — aspect-model doesn't pick; line-950 early-phase carve-out: aspect beans + direct deps built first and exempt from auto-proxying to break the bootstrap cycle; dynamic (WASM-module) aspect = just another Aspect bean through the dynamic source, collected at the same List<dyn Aspect> point; tower/axum-middleware mental model (Layer/Service) as the rust-idiomatic anchor for the around-middleware shape

**T1 — cross-crate/tooling:** Aspects-as-beans means aspect CONTRIBUTION reuses the SAME cross-crate bean-discovery substrate as every other bean (whatever Phase 2 picks for component-scanning) — NO separate advisor-descriptor stream, a unification win, but the anti-DCE self-check for aspects is just the general bean-discovery self-check (a missing aspect crate = a missing bean, same silent-DCE failure mode line 152/345, same force-link + leaf-doctor mitigation). Ordering rides collection-injection's @Order/@Priority (ordering-as-data, resolved at runtime, never link order, line 154/355). Activation rides conditions/profiles, naturally answering "runtime conditions decide what DOES advise" (line 209) without a build-time plan-vs-condition reconciliation. The matcher() being pointcut-authoring-agnostic means the const-pointcut/substrate coupling and the cross-crate type-resolution limits live in the chosen pointcut FRONT-END (R1/R2), not in aspect-model itself — a clean separation. Generics-can't-register applies to a generic Aspect bean (needs a per-instantiation register_aspect! shim). The two-source duality (line 362) is smoothest here: a static link-time aspect and a dynamic WASM-module aspect are BOTH just Aspect beans collected at the same List<dyn Aspect> point, the dynamic one registering through the dynamic source — but a dynamic aspect must go through full bean lifecycle (heavier than a flat descriptor) and its matcher() must be expressible without native const-construction (a runtime-built pointcut). The bootstrap carve-out (line 950) must hold ACROSS crates: an aspect bean in crate Y must be built before an advisable bean in crate X is proxied — coupling aspect authoring to the (Phase-2) startup-ordering/refresh model.

**T2 — execution context:** The defining T2 challenge (shared with auto-proxy-creator R3): aspects are BEANS, so they must be INSTANTIATED (async, on the creation context, possibly .awaiting to inject collaborators) BEFORE any advisable bean's after_init can match against them — forcing an explicit EARLY PHASE in refresh-lifecycle (the line-950 carve-out: build Aspect beans + their direct deps first, exempt THEM from auto-proxying to avoid infinite regress). That early phase is cold-path async, all-or-nothing per refresh. Once aspects exist, matcher() is a pure/cheap call and intercept() returns a pre-built Arc<dyn Interceptor> (no late async at match time). The before/after COMBINATORS build async closures whose futures, at the dyn interceptor boundary, are boxed (Pin<Box<dyn Future + Send>>, AFIT-not-dyn-safe, nightly §RPITIT/§AFIT) — waker-driven on the hot path (§2.4), boxing confined to the builder's lowering. getEarlyBeanReference (auto-proxy-creator's sub-tier) needs the aspect set resolved SYNC mid-cycle — so the aspect early-phase MUST complete before any cycle that triggers early-proxy exposure; a cyclic aspect bean is a re-entrancy trap to detect-and-error with rich context (§2.9), never deadlock. proceed() cancellation/error propagation: cancelling the call drops the around future which drops its proceed() future (no async-Drop, §AsyncDrop — around-held resources release via sync RAII); proceed() errors flow into the after_throwing combinator. Conditions/profiles activation runs at its normal refresh point and feeds the List<dyn Aspect> set — no separate evaluation context. Errors (aspect failed to build, cyclic aspect, ambiguous order) propagate as refresh errors with the full causal chain (§2.9). The HOT per-call path is aop-proxying's AdviceChain walk, unchanged.

### Rust clashes
- No runtime AspectJ expression LANGUAGE and no reflective java.lang.reflect.Method to match against — Spring's pointcuts are AspectJ strings (execution(* ..Service.*(..)), @annotation(Transactional), within/args/this/target) evaluated by a runtime weaver against reflectively-discovered Method objects (signatures, annotations, modifiers, arg/return types). Rust has neither a runtime method-metadata mirror nor a shipped expression evaluator. So a pointcut must be a predicate over DECLARED metadata (a generated JoinPointMeta table — the data aop-proxying's deferred proxy mechanism produces), authored as either a typed combinator predicate (R1), a compile-time-parsed expression string lowering to that predicate (R2), or a hand-written matcher closure (R3). The matchable VOCABULARY is fixed at macro-expansion/codegen time to whatever the metadata captures (type-id, method-name-hash, attribute-presence, arg/return type-ids) — NOT the open-ended reflective surface AspectJ enjoys. An expression string therefore over-promises (it LOOKS like full AspectJ but can only match the captured subset), and method/type identity must be stable-as-data or pointcuts silently mis-match across refactors (the typed within::<T>/annotated::<A> form turns a rename into a compile error; a string just stops matching).
- No runtime classpath to DISCOVER @Aspect beans — Spring's AnnotationAwareAspectJAutoProxyCreator scans the running container for beans annotated @Aspect and reflectively enumerates their @Before/@Around/@Pointcut methods (line 1706). Rust has no classpath scan and no method enumeration. So an aspect's discovery is structural: either link-time descriptor registration (linkme — with its silent-DCE hazard), build-time codegen, explicit registration, or aspects-as-beans riding the general bean-discovery substrate (R3) — whatever Phase 2 picks for component-scanning. There is NO 'find everything @Aspect at runtime' option; an aspect crate the binary never path-references silently contributes nothing (the anti-DCE self-check obligation applies to authored aspects exactly as to advisors).
- around-advice's proceed() cannot cheaply give TYPED argument/return access across a dyn-erased call — Spring's ProceedingJoinPoint exposes getArgs()/proceed(newArgs) and the typed return because reflection carries the Method and boxed args. In leaf the call is routed through an erased AdviceChain (aop-proxying), so an around sees erased arguments/return (boxed/Any-typed AdviceOutcome) unless the substrate generates typed shims per advised method. This bounds the @Around-transforms-args/return intent: aspect-model can offer typed access only as far as the deferred proxy mechanism materializes typed shims — so the richest around ergonomics pre-judge aop-proxying's metadata keystone and may be a typed-monomorphized-path-only capability with an erased fallback.
- async advice on a dyn-dispatched aspect forces boxed futures at the dyn boundary — RPITIT/AFIT are NOT dyn-compatible (nightly doc §RPITIT/§AFIT/line 162), and leaf's beans/interceptors are dyn-erased, so an `async fn around(&self, jp)` must be boxed (Pin<Box<dyn Future + Send>>) at the interceptor seam regardless of nightly features. This is unavoidable on the per-advised-call hot path (the substrate's concern), but aspect-model must SHAPE its advice trait/combinators so the boxing lives at the lowering seam and authors still write plain `async fn` — and must not pretend a zero-cost static around exists across the dyn chain. Send-ness of the future is invisible to callers without trait_variant/RTN (nightly doc §RTN ownerless), so the advice trait likely needs a trait_variant Send variant for tokio multi-thread.
- No async Drop means around-advice holding a resource ACROSS proceed() cannot auto-release asynchronously — Spring's around can wrap proceed() in try/finally that does async-ish cleanup on the JVM freely. Rust has no AsyncDrop (nightly §AsyncDrop, experimental with a known tokio-shutdown crash), and leaf's beans are dyn-erased (no dyn async-drop story). So an around that opens something before proceed() and must close it after — especially on the CANCELLATION path where the around future is dropped mid-proceed() — must release via SYNC RAII/explicit Drop, or via an explicit after()/finally combinator the framework awaits, NOT via async Drop. aspect-model must steer authors toward explicit finally-combinator cleanup and document that cancellation drops the around future (sync Drop runs, async cleanup does not).
- The self-invocation boundary (lines 1714-1737) is a HARD honest limit, not a paper-over — advice attaches to the inter-bean seam (the injected reference), so a target's internal this.method() call bypasses the proxy and the advice silently does not run. In Rust the same is true: leaf wraps the handle callers hold, and a method body calling self.other() goes straight to the concrete impl with no chance to re-enter the wrapper (no bytecode rewriting, by design). aspect-model must COMMUNICATE this (a #[component]'s self-calls aren't advised) rather than hide it — Spring's sanctioned workarounds (refactor to a collaborator, inject a self-proxy reference) map directly, and AopContext.currentProxy() has no clean rust-native analogue worth importing (it couples business code to the framework). This is intent-parity: replicate the honest boundary, not a leaky escape.
- @Order on advice/aspects is correctness-critical yet INVISIBLE in object identity (lines 1770-1772) — cache-outside-vs-inside-tx is a real bug you can't see by inspecting the proxy. Spring orders by @Order/Ordered/@Priority resolved at runtime. In Rust, advice/aspect order MUST be encoded as DATA (order:i32 + role-tier on the lowered AdvisorDescriptor / Aspect bean) and resolved by auto-proxy-creator's deterministic sort — NEVER link/section/constructor order (unspecified, line 154/355). Because misordering is invisible, aspect-model should surface each authored aspect's order AS queryable data and let auto-proxy-creator's chain-order diagnostic show the resolved position — the rust-native improvement over Spring's invisible-ordering footgun. R3's reuse of collection-injection ordering (a consumer-preference concern) for this correctness-grade ordering is a specific tension to resolve.
- A pointcut predicate over declared metadata must be const-constructible for the zero-cost link-time path — for the &'static dyn Pointcut shape (R1/R2 sub-shape a) the predicate must be a const expression to ride linkme's distributed slice (no life-before-main, zero-cost); composable combinators that allocate or capture (within & annotated) can't be const, pushing toward inventory (life-before-main, non-const) or a runtime-built pointcut. So the pointcut authoring algebra is COUPLED to the Phase-2 collection-substrate choice (linkme vs inventory vs codegen), and the macro must hard-error on a non-const pointcut on the linkme path. This is a clash between Spring's freely-constructed-at-runtime pointcuts and Rust's const/link-time constraints — there is no single rust mechanism that is both freely-composable-at-author-time AND zero-cost-const-registered.
- Generics can't auto-register an aspect — Spring discovers a generic-method aspect reflectively; Rust's link/codegen registration needs a concrete monomorphized 'static value (cross-crate doc line 153/349). An aspect whose pointcut or advice is generic over T (advising Repo<T>) cannot emit one descriptor — the #[aspect] macro must hard-error and require a per-instantiation register_aspect!(Repo<ConcreteT>) shim, exactly as auto-proxy-creator flagged for advisors. Spring's generic-aspect ergonomics have no free rust-native analogue.

### Interactions
- **auto-proxy-creator** — DIRECT downstream consumer — auto-proxy-creator (whose design I build on) is the COLLECT+MATCH+UNION+ORDER+role-isolation layer that CONSUMES what aspect-model produces. aspect-model lowers each authored aspect into the AdvisorDescriptor stream (R1/R2: a flat const descriptor with id/order/role/&'static dyn Pointcut/make_interceptor; R3: an Aspect bean collected as advisor) that auto-proxy-creator gathers. The 'AnnotationAwareAspectJAutoProxyCreator additionally discovers @Aspect beans' relationship (line 1706) is realized as: aspect-model contributes Role::Application descriptors that auto-proxy-creator's upgraded CreatorPolicy admits. aspect-model OWNS authoring; auto-proxy-creator OWNS what-matches-which-bean, union, ordering, role-filter. I do NOT re-collect, re-match, or re-order — I only produce the advisor units and the pointcut predicate they carry.
- **aop-proxying** — INDIRECT (two-tier) downstream dependency — aop-proxying is the SUBSTRATE: the wrap-and-swap seam, the AdviceChain/JoinPoint/proceed call-routing primitive, and the erased-vs-typed argument access. aspect-model's advice (the around interceptor its lowering produces) is what ultimately runs inside aop-proxying's AdviceChain, and aspect-model's JoinPoint<'_>/proceed() ergonomics are bounded by what aop-proxying's deferred proxy mechanism materializes (erased vs typed args/return). aspect-model must stay mechanism-neutral: it authors interceptors and pointcuts, never the proxy or the chain walk.
- **transaction-management** — SIBLING-as-FIRST-CLASS-CONSUMER — @Transactional is one of leaf's own cross-cutting features that AUTHORS against aspect-model's exact substrate (line 1712: the marker is inert; the wiring registers the advisor + interceptor). tx is a framework-shipped aspect: a pointcut matching @Transactional-marked methods + an around interceptor opening/committing/rolling-back, emitting a Role::Infrastructure descriptor. This is the dogfooding unification (§2.11): tx must use the SAME pointcut + advice + lowering vocabulary users do, differing only by role hint. If tx needs something aspect-model can't express, aspect-model is under-powered.
- **caching** — SIBLING framework consumer — @Cacheable/@CacheEvict author as a framework aspect (Role::Infrastructure): a pointcut matching cache-marked methods + an around interceptor that consults/populates the cache and can SHORT-CIRCUIT proceed() (a cache hit returns without calling through). The short-circuit is precisely aspect-model's around-is-general-form intent (an around decides whether proceed() runs at all). Its ordering RELATIVE to the tx aspect within one chain (line 1772) is auto-proxy-creator's concern, but cache's @Order is authored via aspect-model's order hint.
- **validation** — SIBLING framework consumer — @Validated method validation authors as a framework aspect (Role::Infrastructure): a pointcut matching validation-marked methods + an around interceptor validating args before proceed() and the return after. Its need for TYPED argument access is the rustClash (erased proceed()) that aspect-model surfaces but aop-proxying must answer — validation is the canonical case where the typed-vs-erased JoinPoint tension bites.
- **async-execution** — SIBLING framework consumer — @Async authors as a framework aspect (Role::Infrastructure): a pointcut matching @Async methods + an around interceptor dispatching proceed() onto a TaskExecutor. The @EnableAsync 'register the creator + the advisor' wiring (line 1712) maps onto: async-execution contributes its aspect/advisor via aspect-model's lowering. Its self-invocation/void-return caveat is the self-invocation-boundary rustClash aspect-model must communicate (an internal call to an @Async method runs synchronously).
- **retry-resilience** — SIBLING framework consumer — @Retryable authors as a framework aspect (Role::Infrastructure): a pointcut matching retry-marked methods + an around interceptor wrapping proceed() in a retry/backoff loop. Its retry/backoff is cold-path POLLING permitted by §2.4 (a retry loop is not a hot-path busy-wait), living inside the around interceptor aspect-model produces — a clean example of an around that calls proceed() multiple times.
- **exception-translation** — SIBLING framework consumer + the @Repository-pointcut exemplar — exception-translation authors as a framework aspect whose POINTCUT matches @Repository-stereotype beans (the natural pointcut, line 471) and whose around/after-throwing advice converts vendor exceptions to the DataAccessException analogue. It is leaf's clearest demonstration that a STEREOTYPE is a pointcut target (component-stereotypes interaction) and that a framework advisor uses the identical authoring vocabulary as a user @Aspect.

Stereotype presence must be in the declared metadata the matcher reads.
- **annotation-model** — POINTCUT-PREDICATE-INPUT source — annotation-model treats annotations as composable, metadata-readable-without-loading data with attribute forwarding/merging. aspect-model's @annotation(..)/attribute-presence pointcuts read exactly this metadata; the typed annotated::<A>() combinator (R1) and the @annotation(A) expression (R2) both query annotation-model's captured attribute data. The richness of what a pointcut can match on annotations is bounded by what annotation-model captures as data at macro time.

The tension (also flagged by auto-proxy-creator): collection-injection treats ordering as a consumer PREFERENCE, but advice ordering is correctness-critical (lines 1770-1772) — aspect-model needs a correctness-grade guarantee from preference-grade machinery.

- **context-propagation** — AROUND-ADVICE-CROSSES-THREADS coupling — when an around interceptor moves work across an .await/thread boundary (async dispatch, executor hop), ambient context (request/locale/tx-sync, locale-context holders) must survive via context-propagation's capture/restore SPI. aspect-model's around shape is a primary site where execution context hops, so its T2 discipline must declare that an around crossing a boundary participates in context-propagation — otherwise advice silently loses ambient context (the rust-native answer to ThreadLocal-across-.await).

### Open questions for Phase 2
- HOW are pointcuts AUTHORED — typed predicate combinators (R1, refactor-safe, compiler-checked, but a new vocabulary), a compile-time-parsed expression mini-language (R2, Spring-parity feel, but a grammar to build and bounded by the metadata vocabulary), or left agnostic behind an Aspect.matcher() (R3)? This is the single biggest authoring-DX decision and cross-cuts the metadata vocabulary keystone: whichever way, the matchable surface equals what aop-proxying's deferred proxy/metadata mechanism captures as DATA (type-id, method-name-hash, attribute-presence, arg/return type-ids) — so pointcut authoring can't be designed before that metadata surface is. Held unresolved.
- Is an aspect a FLAT CONST DESCRIPTOR (R1/R2 — lowers to auto-proxy-creator's AdvisorDescriptor, zero-cost link-time, but const-pointcut-constrained and a separate stream) or a FIRST-CLASS BEAN collected via collection-injection (R3 — maximal §2.11 unification: activation=conditions, ordering=collection-injection, but inherits the line-950 bootstrap carve-out and entangles with the resolution/ownership model)? This mirrors auto-proxy-creator's open advisor question and must be answered consistently across both features — the unification-vs-bootstrap-complexity tradeoff is a shared keystone.
- Does the pointcut predicate need to be CONST-constructible, and how does that couple to the Phase-2 collection substrate? A &'static dyn Pointcut on linkme demands const (restricting the combinator algebra); a non-const composable pointcut pushes to inventory (life-before-main) or a runtime-built/codegen'd pointcut. So the pointcut authoring algebra and the collection-substrate choice (linkme vs inventory vs build-time codegen, cross-crate doc Architectures A-D) are entangled — there is no freely-composable-AND-zero-cost-const pointcut. Held as a candidate-comparison.
- How much TYPED argument/return access can around-advice get through the dyn-erased AdviceChain? Spring's ProceedingJoinPoint gives typed getArgs()/proceed(newArgs)/return via reflection; leaf's erased substrate gives boxed/Any unless aop-proxying generates typed per-method shims. So the richest @Around-transforms-args/return ergonomics pre-judge aop-proxying's metadata/proxy keystone — is typed access a monomorphized-path-only capability with an erased fallback, or a uniform (boxed) surface? Cross-cuts aop-proxying's deferred mechanism.
- Where does ADVICE ORDERING live and is it correctness-grade? Advice/aspect order is correctness-critical and invisible in object identity (lines 1770-1772), yet R3 would reuse collection-injection's consumer-PREFERENCE ordering. Is advice ordering the SAME model as collection-injection's @Order/@Priority (unification, §2.11) or a distinct correctness-grade ordering? And since misordering is invisible, where is the resolved chain order surfaced as an inspectable §2.9 diagnostic? Shared with auto-proxy-creator and collection-injection.
- How is the SELF-INVOCATION boundary communicated and is any mitigation offered? Spring documents the limit and steers toward refactor/self-injection over AopContext (lines 1731-1737). leaf has the same hard limit (no bytecode rewriting). Should aspect-model (a) only document it, (b) offer a self-proxy-injection ergonomic (inject the advised handle into the bean), or (c) emit a compile-time lint/diagnostic when a #[component] self-calls an advised method? The AopContext.currentProxy() analogue is deliberately undesirable (couples business code to the framework) — is there a cleaner rust-native nudge? Unresolved.
- How does an around-advice that crosses a thread/.await boundary participate in context-propagation, and what is the cancellation/cleanup contract? An around that dispatches proceed() to an executor (async-execution/task-execution) hops execution context — ambient context must survive via context-propagation's capture/restore, and on cancellation the dropped around future must release resources via SYNC RAII (no async-Drop, nightly §AsyncDrop). What contract does aspect-model demand of an around so context-propagation plugs in uniformly and resource cleanup is honest across the cancellation path? Cross-cuts context-propagation, task-execution, and the async-execution model (§2.3).
- How does the two-source duality (static link-time aspects PLUS an optional dynamic directory-scanned WASM-module source) reconcile with the authoring/lowering model? R1/R2's typed-const pointcut favours static aspects and needs a runtime-built-pointcut variant (or R2 sub-shape b's portable PointcutProgram) for dynamic ones; R3's aspect-as-bean blends smoothest (a dynamic aspect is just a bean) but forces dynamic aspects through full bean lifecycle. Keep the descriptor/pointcut format origin-agnostic so a WASM-sourced aspect lowers identically — a candidate-comparison, not a decision.
- Should leaf's own cross-cutting features (tx/cache/validation/async/retry/exception-translation) author against the EXACT user-facing aspect-model vocabulary (maximal dogfooding §2.11), or against a lower-level advisor-descriptor API directly (decoupling framework internals from user authoring ergonomics)? If they share the vocabulary, aspect-model's expressiveness is gated by their hardest requirement (e.g. typed args for validation, short-circuit for caching, retry-loop for retry); if they don't, leaf risks two divergent authoring surfaces. Cross-cuts every framework cross-cutting feature.

### Candidate realizations

#### R1 — `AsyncDispatchAdvisor`: a Role::Infrastructure advisor contributing one dispatch interceptor into the auto-proxy-creator chain (the literal AsyncAnnotationAdvisor analogue)
Realize @Async exactly as Spring does relative to its neighbours: a single Role::Infrastructure ADVISOR contributed into the already-designed auto-proxy-creator's advisor stream. The `#[async]` attribute macro (thin, charter §2.10) emits ONLY metadata: it marks the method in the bean's declared method-metadata table with an `AsyncMarker { executor_qualifier: Option<&'static str>, return_shape: AsyncReturn }` where `AsyncReturn ∈ {Unit (fire-and-forget), Handle (awaitable JoinHandle/impl Future), IllegalSync (a non-Future, non-unit return → compile-time hard error)}`. The `enable_async!`-analogue (the @EnableAsync wiring, line 1712) contributes an `AdvisorDescriptor { role: Role::Infrastructure, order, pointcut: AsyncPointcut (matches any method carrying AsyncMarker), make_interceptor: |c| Arc::new(AsyncDispatchInterceptor::new(c)) }` into the auto-proxy-creator's collection stream — it does NOT register a separate proxy creator (that's the auto-proxy-creator's single-per-context job). At call time the `AsyncDispatchInterceptor::around(join_point)` does NOT proceed() inline; instead it (1) resolves the target executor via the selection policy, (2) captures the invocation as a future (the join_point's proceed() IS the target method's future), (3) calls `executor.spawn(fut)` for the Unit shape or `executor.submit(fut)` for the Handle shape, and (4) returns immediately — unit for fire-and-forget, the join handle for the awaitable shape. Executor selection (line 2033) is a small policy struct resolved per-interceptor-construction (cold path, cached): qualifier from AsyncMarker → an `#[async_config]`-designated async-default executor bean → the container's primary execution facility; FORCE-MODE is a container flag that, when set, makes the policy IGNORE the qualifier and always pick the application executor (line 2069). Async-exception routing (line 2032): for the Unit shape the spawned future is wrapped so an Err/panic is delivered to an injected `AsyncUncaughtFailureHandler` bean (container-level, the AsyncUncaughtExceptionHandler analogue) rather than dropped; for the Handle shape the error rides the join handle to the awaiting caller. Self-invocation (line 1716): because this is woven at the inter-bean seam via the proxy, an internal `self.foo().await` bypasses the interceptor and runs inline — the macro emits a diagnostic note and (candidate) a lint-style warning; the sanctioned fix is refactor-to-collaborator or self-injection, exactly as Spring.

**Pros:** Tightest intent parity AND tightest neighbour-fit: this IS the AsyncAnnotationAdvisor — one infrastructure advisor, one interceptor, contributed into the auto-proxy-creator stream exactly as transaction-management/caching/validation do (the catalog's stated relationship). The Spring mental model and the leaf AOP machinery both transfer 1:1; @Async, @Transactional, @Cacheable are siblings sharing one chain.; Reuses ALL the already-designed upstream machinery with zero new substrate: advisor collection (auto-proxy-creator), the run-somewhere seam (task-execution), role-isolation + chain ordering (auto-proxy-creator's role-tier sort). This feature adds only a pointcut + an interceptor + a selection policy — the thinnest possible surface, maximal unification (charter §2.11).; Ordering within the shared chain falls out for free and is correctness-relevant in the same way tx/cache ordering is: @Async's dispatch interceptor's position relative to @Transactional matters (dispatch-then-transact vs transact-then-dispatch are different semantics — a tx that spans the async hop is almost always wrong), and the auto-proxy-creator's deterministic role-tier+order sort already governs it; this feature just supplies the order hint and surfaces it as an inspectable diagnostic.; The self-invocation caveat is preserved HONESTLY (charter §2.2 intent parity) because the mechanism genuinely is wrap-at-the-seam — leaf does not pretend to intercept internal calls, matching Spring's honest boundary; and the macro can emit a richer diagnostic than Spring (a compile-time note pointing at the self-call) since it has the metadata.; Origin-agnostic: the advisor is a flat descriptor with a fn-ptr interceptor factory, so a WASM-module or test async-advisor collects identically; the interceptor depends only on `Arc<dyn TaskExecutor>` (task-execution's dyn seam), so the backing substrate is swappable per charter §2.6 without touching this feature.

**Cons:** Inherits the full per-call dyn cost stack: the interceptor is one boxed-future hop in the auto-proxy-creator chain (AFIT-not-dyn-safe wall, nightly doc lines 160-181), the executor's spawn takes a boxed future (task-execution's dyn boundary), and the AsyncReturn=Handle path boxes a join handle — three allocations on what is often a hot fan-out path (charter §2.5 tension). For genuinely hot async dispatch this is heavier than a monomorphized spawn the user could hand-write.; The return-shape discipline collides with Rust's type system harder than Spring's void/Future trichotomy: an `async fn` ALREADY returns a future when called, so the interceptor must distinguish 'the method's own returned future that the caller will await' (Handle) from 'fire-and-forget, the caller wants unit back' (Unit) — but a fire-and-forget `#[async] async fn foo() -> ()` STILL produces a `Future<Output=()>` the proxy must spawn-and-drop, and the proxy's signature must therefore differ from the target's (return unit vs return a handle), which is awkward to express through one uniform interceptor signature over erased values (see T2).; Executor SELECTION at interceptor-construction time assumes the executor set is resolvable when the interceptor is built (cold path at refresh) — but a `@Async("name")` qualifier referencing an executor that doesn't exist is a wiring error whose best diagnostic timing (compile vs refresh) is entangled with the deferred conditional/codegen strategy; R1 leans toward a refresh-time validation pass.; Self-invocation-bypasses-proxy is MORE surprising in Rust than in Spring: a Rust dev calling `self.foo().await` gets a future that runs inline on the current task — there is no visible 'it ran on the same thread' tell (unlike Spring where the missing thread-name in a log hints at it), so the silent-sync-fallback footgun is sharper and the diagnostic burden higher.; Pre-commits to the wrap-at-after_init / per-bean-global interception model (the auto-proxy-creator's R1 shape). If aop-proxying ultimately chooses a per-edge resolution-seam model, 'which references get the async-dispatch proxy' becomes ambiguous and this advisor's single-decision shape needs rework.

**Mechanisms considered:** thin `#[async]` macro emitting only an AsyncMarker into the bean's declared method-metadata table (return_shape + optional executor qualifier); an AdvisorDescriptor { Role::Infrastructure, order, AsyncPointcut, make_interceptor fn-ptr } contributed into the auto-proxy-creator's advisor-collection stream (NOT a separate proxy creator); AsyncDispatchInterceptor that spawns/submits the join_point's future onto a selected Arc<dyn TaskExecutor> instead of proceeding inline; executor-selection policy struct: qualifier → async-default bean → primary facility, with a container force-mode flag overriding the qualifier (lines 2033/2069); AsyncUncaughtFailureHandler bean for routing fire-and-forget errors/panics (the AsyncUncaughtExceptionHandler analogue, line 2032); return-shape compile-time discipline: Unit | Handle(impl Future/JoinHandle) | IllegalSync→hard error (the void/Future/CompletableFuture rule, line 2029); macro-emitted self-invocation diagnostic note (richer than Spring's silent bypass, lines 1714-1737)

**T1 — cross-crate/tooling:** unused

**T2 — execution context:** This feature's T2 is the crux — it MOVES work across an execution boundary. The interceptor's `around` runs on the CALLER's task (the cold-or-hot path of whoever invoked the proxied method); instead of awaiting proceed() inline, it SPAWNS proceed()'s future onto the selected executor and returns. CRITICAL execution-context discipline (charter §2.3): the target method's future is spawned onto the EXECUTOR's context (multi-thread pool / current-thread / bounded), which generally requires it to be `Send + 'static` (task-execution's R1/R2 Send tension surfaces here exactly) — so a `#[async]` method capturing non-Send state is a compile error against a multi-thread executor, and the rustClash about Send/'static-vs-swap-by-config lands directly on this feature. CANCELLATION: the Unit (fire-and-forget) shape detaches — the spawned task outlives the caller, and on caller-drop the task is NOT cancelled (it must run to completion or be drained at shutdown), which is the honest fire-and-forget semantic and must be documented; the Handle shape ties cancellation to the returned handle — dropping the handle drops/aborts the spawned task per the executor's policy. ERRORS: fire-and-forget Unit has nowhere to return an error/panic — it MUST route to the AsyncUncaughtFailureHandler on the executor's uncaught-failure path (NOT swallowed, line 2032); the Handle shape carries Err through the awaited handle to the caller (the checked-exception-wrapping analogue, line 1871). SELF-INVOCATION (line 1716): an internal `self.foo().await` does NOT hit the interceptor and runs INLINE on the caller's task — the future executes on the wrong context with the wrong cancellation/error semantics, silently; this is the honest wrapper boundary and the sharpest async footgun (no thread-name tell as in Spring). CONTEXT PROPAGATION: ambient context (request/locale/tracing) does NOT survive the spawn unless task-execution's TaskDecorator chain captures-and-restores it across the hop — this feature DECLARES the dependency on that seam (charter §3 deferred ambient-context model) and cannot implement propagation itself; OFF by default (Spring's opt-in, line 1871, data-bleed avoidance). The interceptor's executor-selection + construction is cold-path at refresh (resolving the Arc<dyn TaskExecutor>, no .await for selection); per-call dispatch is the hot path and must be a cheap spawn (one boxed-future hop). Lifecycle: in-flight fire-and-forget tasks are drained by the executor's container-driven shutdown().await (task-execution / context-close), not by this feature.

#### R2 — Macro-rewrite to an explicit spawn at the call site's body (no runtime advisor): the `#[async]` macro emits the dispatch directly, the proxy only routes selection
Question whether @Async needs a runtime advisor + interceptor at all, given charter §2.10 (thin macros emit only hand-writable code) and §2.2 (realize intent, not the JVM mechanism — Spring needs the proxy ONLY because it cannot rewrite the method; leaf's macro CAN rewrite the method body into exactly what a user would hand-write). Shape: the `#[async]` attribute macro transforms `#[async] async fn foo(&self, x: X) -> ()` into a wrapper that, instead of running the body inline, resolves the bean's injected executor handle and SPAWNS the original body onto it — i.e. the macro emits the spawn that a user could have written by hand (`self.__executor.spawn(async move { /* original body */ })`). The original body is moved into a private `__foo_impl` method; the public `foo` becomes the dispatcher. Executor selection is resolved by giving the bean an injected executor field (or a small `AsyncDispatch` handle) chosen at the bean's construction per the same policy (qualifier → async-default → primary), so selection is DI-time, not interceptor-time. This means there is NO per-method runtime advisor for the dispatch behaviour — the dispatch is compiled into the method. The @EnableAsync-analogue shrinks to 'ensure the executor + the selection policy + the uncaught-failure handler are wired into the context' (a wiring concern, not an advisor). Async-exception routing: the macro-emitted spawn wraps the body's Result/panic and forwards to the injected uncaught-failure handler for the Unit shape, or returns the JoinHandle for the Handle shape — emitted code, fully visible in `cargo expand`. Force-mode and qualifier resolution still consult the container at the bean's construction to pick which executor the field binds to. CRUCIALLY this changes the self-invocation story: because the dispatch is compiled INTO the method, `self.foo()` ALSO dispatches (no proxy to bypass) — leaf can ELIMINATE Spring's most infamous gotcha entirely (charter §2.2: prove a JVM workaround unnecessary). Self-invocation runs async correctly.

**Pros:** KILLS the self-invocation gotcha (charter §2.2 'prove it unnecessary'): because the dispatch is compiled into the method body, internal `self.foo()` calls dispatch onto the executor just like external ones — the single most infamous @Async trap (lines 1714-1737, 2031) simply does not exist in leaf. This is a genuine DX win over Spring (charter §1 DX) impossible in the proxy model.; Lowest per-call overhead (charter §2.5): no auto-proxy-creator chain hop, no interceptor boxed-future indirection, no advisor matching — the spawn is a direct monomorphized call the compiler can inline; only the executor's own dyn-spawn boxing remains. Hottest async-dispatch paths pay the least.; Maximally honors thin-macros (charter §2.10): the macro emits EXACTLY the hand-writable spawn; `cargo expand` shows a literal `self.executor.spawn(async move { ... })` — nothing hidden, the escape hatch is 'write the spawn yourself', and there is no opaque runtime advisor to reason about.; No advisor-collection DCE hazard for the dispatch BEHAVIOUR (cross-crate doc line 149) — the dispatch is in the method body, force-linked by the bean being used; a missing advisor crate cannot silently disable @Async dispatch (though the executor itself is still a Layer-0 hazard). The silent-sync-fallback failure mode of R1 is eliminated.; Return-shape handling is cleaner: the macro sees the real return type at expansion and can emit the correct spawn-and-drop (Unit) vs return-handle (Handle) code with a precise compile-time error on an illegal return — no erased-value gymnastics through a uniform interceptor signature.

**Cons:** DIVERGES from the auto-proxy-creator unification (charter §2.11): @Async stops being 'one more infrastructure advisor in the shared chain' and becomes a bespoke macro-rewrite — so it no longer composes in the SAME ordered chain as @Transactional/@Cacheable, and the cross-cutting ORDERING relative to tx/cache (which the catalog explicitly wants in one chain) is lost or must be re-established by hand. This is the sharpest cost: @Async leaves the shared interception model.; Ordering vs @Transactional becomes ill-defined / dangerous: if @Async is a body-rewrite-spawn but @Transactional is a chain interceptor, their relative order (does the tx wrap the spawn or run inside it?) is determined by where the macro injects the spawn vs where the proxy wraps — opaque and easy to get wrong, exactly the invisible-misordering bug the shared-chain model was designed to prevent.; Self-invocation now ALWAYS dispatches — which is a pro for the gotcha but a SEMANTIC DIVERGENCE from Spring (charter §2.2 intent parity tension): code ported from Spring that RELIED on self-calls running sync (a real pattern) silently changes behaviour. The 'fix' is arguably better but it is not parity.; Force-mode central override (line 2069) is harder: the macro bakes the dispatch into the body at compile time, but force-mode is a RUNTIME container decision to reroute all @async to the application executor — so the macro must emit code that consults a runtime selection handle anyway, partially reintroducing the indirection it tried to avoid, and a runtime force-mode flip cannot un-bake the compiled dispatch.; The bean now has a hidden dependency (the injected executor field/handle) that the macro adds — which must be reconciled with the bean's declared injection points and constructor (injection-styles), and a bean with no async methods must not pay for it; the macro-added field couples @Async to the bean's construction shape more invasively than an external advisor.; Pre-judges the metadata/codegen boundary AND the macro-vs-runtime split (charter §3 keystones) toward heavy compile-time rewrite — must stay a CANDIDATE; presenting body-rewrite as the shape biases those deferred decisions.

**Mechanisms considered:** `#[async]` macro rewrites the method: body → private `__impl`, public method → `self.__executor.spawn(async move { __impl(..).await })` (the hand-writable spawn); executor resolved as an injected field/handle at bean construction (DI-time selection: qualifier → async-default → primary), not at interceptor time; @EnableAsync shrinks to wiring the executor + selection policy + uncaught-failure handler (no per-method advisor for dispatch); macro-emitted Result/panic forwarding to the injected AsyncUncaughtFailureHandler (Unit shape) or JoinHandle return (Handle shape); force-mode + qualifier as a runtime selection handle the emitted body consults (partial reintroduction of indirection); self-invocation dispatches correctly because the spawn is in the body — eliminating Spring's gotcha (charter §2.2)

**T1 — cross-crate/tooling:** Radically different and LIGHTER cross-crate profile for the dispatch behaviour: there is NO advisor descriptor to collect, so the auto-proxy-creator's advisor-collection substrate and its DCE hazard do NOT apply to async dispatch — the dispatch code lives in the bean's own method body and is force-linked by the bean being referenced (a bean nobody uses contributes nothing, but then its @async methods are never called either, so no silent-failure). What STILL crosses crates and STILL carries the Layer-0 hazard (cross-crate doc line 149): the MANDATORY runtime-backed executor the emitted spawn targets — `self.__executor.spawn(...)` needs a real `Arc<dyn TaskExecutor>` (task-execution), so the execution-facility self-check + force-link obligation is unchanged and remains catastrophic if missed. The `@Async("name")` qualifier is resolved at the bean's construction via autowiring-resolution's cross-crate qualifier matching (a named executor in another crate). Force-mode is a runtime container flag the emitted body must consult, assembled at the binary/app-root (topology line 260) — so even this compile-time-rewrite model retains a runtime selection seam that crosses crates. Generics: a generic `#[async]` method monomorphizes naturally with the bean (better than R1 here — no per-instantiation descriptor shim needed since there's no descriptor; the spawn is emitted into each monomorphization), though the spawned future's Send bound must hold for every instantiation. The macro stays thin (charter §2.10) emitting absolute-path references to the executor handle API in leaf-core. Because the dispatch isn't a collected contribution, there is no ordering-as-data concern for async itself — but that is exactly the lost-shared-chain-ordering cost (it can't be ordered relative to tx/cache advisors that ARE collected).

**T2 — execution context:** Execution context is MORE explicit at the source (charter §2.3) because the spawn is visible in `cargo expand`: the macro-emitted `self.__executor.spawn(async move { ... })` shows precisely where work crosses onto the executor. Self-invocation: `self.foo()` runs the SAME emitted spawn, so internal calls cross the boundary correctly (the gotcha is gone) — but this means an internal call that the Spring-porting dev expected to run inline now hops threads, a context-change they must be aware of. CANCELLATION: identical to R1 — Unit fire-and-forget detaches (task outlives caller, drained at shutdown), Handle ties cancellation to the returned join handle. ERRORS: the emitted code forwards Unit-shape errors/panics to the injected AsyncUncaughtFailureHandler and rides Handle-shape errors back through the JoinHandle — but because the routing is emitted code, it is inspectable and the diagnostic is a compile-time-visible code path rather than an opaque interceptor. CONTEXT PROPAGATION: same hard dependency on task-execution's TaskDecorator capture/restore seam across the spawn (charter §3 deferred ambient-context model) — the emitted spawn must run through the executor's decorator chain, not bypass it, or context silently drops; this feature declares the seam, doesn't implement it. Construction-time executor SELECTION runs on the bean's construction context (cold path, may .await if resolving an executor that itself awaits config). Lifecycle/shutdown: in-flight fire-and-forget spawns are the executor's drain concern (task-execution/context-close). The key T2 divergence from R1: there is no interceptor task-hop in a shared chain, so the per-call execution path is shorter, but the ordering relative to other cross-cutting concerns (tx) is no longer governed by a single chain walk — a correctness-of-composition risk.

#### R3 — `Async<T>` deferral primitive + a return-type-driven advisor: model async-execution as an explicit deferred-result type the container materializes, unifying with deferral-primitives
Push unification (charter §2.11) hardest by questioning whether @Async should be a method ANNOTATION at all, vs an explicit RETURN-TYPE that declares 'this result is produced off the current execution context' — folding async-execution into the deferral-primitives family (ObjectProvider/lazy/0..1/0..N already make deferral explicit in the type signature). Shape: a method declares its async intent in its TYPE — `fn foo(&self, x: X) -> Async<()>` or `fn report(&self) -> Async<Report>` — where `Async<T>` is a container-aware deferral handle (the rust-native AsyncTaskExecutor.submit result, the Future/CompletableFuture analogue, line 2029). The advisor (still Role::Infrastructure, still in the auto-proxy-creator chain like R1) matches methods RETURNING `Async<T>` (a TYPE-driven pointcut, not a marker annotation) and its interceptor materializes the `Async<T>` by spawning the body onto the selected executor; awaiting the `Async<T>` yields the result, dropping it detaches (Unit case: `Async<()>` can be dropped fire-and-forget or awaited for completion). This makes the return-type discipline (line 2029 void/Future/CompletableFuture) STRUCTURAL: you literally cannot write an illegal async return because the only legal async return is `Async<T>` — no compile-time check needed, the type IS the contract. Executor selection: an optional `#[async_executor("name")]` attribute OR a typed `Async<T, Q>` qualifier phantom carries the selection; default is async-default→primary; force-mode overrides. The `Async<T>` handle unifies with deferral-primitives' ObjectProvider/lazy machinery — it is 'the result deferred onto another execution context' alongside 'the dependency deferred/optional/lazy', sharing the same deferred-access vocabulary and possibly the same cancellation/error-surfacing patterns. Async-exception routing: an unwrapped/dropped `Async<()>` whose body failed routes to the uncaught-failure handler; an awaited `Async<T>` surfaces Err to the awaiter (the handle IS the error channel — cleaner than the void/Future split). Self-invocation: still a proxy-woven advisor, so internal calls bypass it (same caveat as R1) UNLESS combined with R2's body-rewrite — a hybrid is a candidate.

**Pros:** Makes the return-type discipline (line 2029) STRUCTURALLY UNFORGEABLE rather than a checked rule: the only legal async return is `Async<T>`, so 'illegal sync return on an @async method' is not a diagnostic to emit but a type that cannot be written — the strongest charter §2.9 'fail as early as legible' outcome (it fails at the type level, earlier and clearer than any validation pass).; Maximal unification with deferral-primitives (charter §2.11): 'a result produced on another execution context' joins 'a dependency that is optional/lazy/multiple' as ONE family of explicit-in-the-type-signature deferral, sharing vocabulary, cancellation, and error-surfacing machinery instead of inventing async-specific ones — exactly the 'unify, don't bespoke' mandate, and the deferral family is the natural home for 'deferred onto a thread'.; The error channel is unified and cleaner: `Async<T>` IS the result-or-error handle, so the awkward void-cannot-propagate / Future-can-propagate split (lines 2032, 1871) collapses into one model — await to get Result<T>, or drop to fire-and-forget-with-handler-routing; the caller's intent (await vs drop) cleanly selects the error semantics.; Self-documenting at the call site (charter §1 DX): a caller seeing `-> Async<Report>` KNOWS the work is deferred onto another context without reading an annotation on the definition — the type signature carries the execution-context fact to every call site, which annotations (R1) and body-rewrites (R2) do not.; Still rides the auto-proxy-creator shared chain (like R1) so ordering relative to @Transactional/@Cacheable is preserved and governed by the same deterministic sort — keeping the cross-cutting composition R2 sacrificed.; Origin-agnostic and dyn-friendly: `Async<T>` is a concrete container type wrapping an erased join handle; a WASM-module async method returning the WIT-async analogue can be adapted to `Async<T>` at the provider proxy, slotting into the same model.

**Cons:** DIVERGES MOST from Spring's surface (charter §2.2 intent-parity tension): Spring users know `@Async void foo()`; `fn foo() -> Async<()>` is a different idiom and the 'just annotate the method' DX is lost — porting requires changing signatures, not just adding an attribute. The intent (run off-context) is preserved but the felt experience diverges more than R1/R2.; Introduces a new first-class container type (`Async<T>`) whose relationship to plain `impl Future`/`tokio::JoinHandle`/the executor's own submit-handle must be carefully defined — risks being a redundant wrapper over the executor's existing join handle (task-execution already returns a handle from submit), so the unification-with-deferral-primitives win must be real and not just a rename, or it adds a concept for little gain.; The fire-and-forget Unit case is awkward in a return-type model: `Async<()>` that you must explicitly drop-or-await is more ceremony than `#[async] fn foo()` returning nothing — and an accidentally-dropped `Async<()>` whose body silently failed (routed to the handler) vs an intentionally-fire-and-forget one are indistinguishable at the type level, blurring intent.; Still inherits the self-invocation caveat (it's a proxy-woven advisor like R1) — so unlike R2 it does NOT kill the gotcha; an internal call to a `-> Async<T>` method... actually returns an Async<T> that the self-caller would still have to await, which is LESS surprising than R1's silent-sync but introduces a new confusion (does self-calling materialize the dispatch or not? depends on whether the proxy is in the path).; Couples async-execution to the deferral-primitives design (charter §3 — the deferral/ObjectProvider model is itself a Phase-1 neighbour, not a settled foundation), so this realization's viability depends on a sibling feature's shape — presenting it biases that coupling; must stay a CANDIDATE.; Force-mode and qualifier selection on a TYPE-driven model need either a phantom type parameter `Async<T, Q>` (heavy, viral through signatures) or an out-of-band attribute (reintroducing the annotation R3 tried to replace) — neither is clean.

**Mechanisms considered:** async intent declared in the RETURN TYPE: `fn foo() -> Async<T>` (the void/Future/CompletableFuture discipline made structural, line 2029); a Role::Infrastructure advisor with a TYPE-driven pointcut (matches methods returning Async<T>) contributed into the auto-proxy-creator chain (like R1); `Async<T>` as a deferral handle unified with deferral-primitives (ObjectProvider/lazy family) — 'deferred onto another execution context'; await Async<T> → Result<T> (error rides the handle); drop Async<()> → fire-and-forget with uncaught-failure-handler routing — unifying the void/Future error split; executor selection via `#[async_executor("name")]` attribute or `Async<T,Q>` phantom qualifier; default async-default→primary; force-mode override; hybrid candidate: combine with R2's body-rewrite to also kill self-invocation while keeping the typed handle

**T1 — cross-crate/tooling:** Cross-crate profile is essentially R1's (a Role::Infrastructure advisor descriptor collected via the auto-proxy-creator substrate, same DCE hazard, same anti-DCE self-check obligation: a missing async-advisor crate silently leaves `-> Async<T>` methods un-materialized — though here the failure is LOUDER than R1 because an un-materialized `Async<T>` is a never-completing handle the caller awaits forever, which a self-check/timeout surfaces, vs R1's silent-sync-run). The TYPE-driven pointcut matches over DECLARED method metadata — the return type `Async<T>` must be captured AS DATA at macro/codegen time (entangled with aop-proxying's metadata mechanism), and `Async<T>` being a known container type makes the match a stable structural key (more refactor-robust than R1's marker-annotation, which a rename could drop). The MANDATORY executor Layer-0 hazard is identical (task-execution force-link + self-check). Qualifier resolution (attribute or phantom) crosses crates via autowiring-resolution. The deep T1 coupling unique to R3: `Async<T>` is a core type in leaf-core that the deferral-primitives family also lives in, so its ABI must be co-designed across those features and ultra-stable (emitted method signatures hard-code it workspace-wide). Generics: `Async<T>` over a generic T monomorphizes with the method; a generic async method on a generic bean still can't register a concrete advisor descriptor (generics-can't-register, line 153/349) → per-instantiation shim, same as R1. Force-mode assembled at binary/app-root (topology line 260). WASM-origin: a WASI-0.3-async guest method (wasm exploration doc) adapts to `Async<T>` at the host-proxy, keeping the model origin-agnostic.

**T2 — execution context:** Execution context is carried in the TYPE to every call site (charter §2.3 strongest of the three): `-> Async<T>` declares 'produced on another execution context' structurally. The advisor's interceptor materializes `Async<T>` by spawning the body's future onto the selected executor (same Send + 'static requirement and rustClash as R1/R2). CANCELLATION is unified with the deferral handle: dropping the `Async<T>` aborts/detaches per policy (Unit `Async<()>` dropped = fire-and-forget detach, drained at shutdown; awaited = caller controls cancellation by dropping the await) — this is cleaner than R1's split because cancellation rides the one handle type. ERRORS ride the `Async<T>` handle to the awaiter (Result<T>), or route to the uncaught-failure handler if a failed `Async<()>` is dropped — the single error channel collapses the void/Future split (lines 2032/1871). SELF-INVOCATION: a proxy-woven advisor like R1, so an internal self-call may bypass materialization — BUT because the method returns `Async<T>`, a self-caller gets an Async<T> they must still await, which surfaces the boundary in the type rather than silently running sync; this is a softer footgun than R1 but introduces 'is this Async materialized via the proxy or constructed inline?' ambiguity (the hybrid-with-R2 candidate removes it). CONTEXT PROPAGATION: identical hard dependency on task-execution's TaskDecorator capture/restore across the spawn (charter §3 deferred); the `Async<T>` materialization must run through the decorator chain. Construction/selection cold-path at refresh; per-call materialization is the hot path (one chain hop + one spawn + one Async<T> handle allocation — same cost class as R1). Lifecycle: in-flight Async<()> tasks drained by the executor's shutdown().await (task-execution/context-close).

### Rust clashes
- No proxy-only interception, so the self-invocation gotcha is NEGOTIABLE rather than inherent — a fork in the road. Spring's self-invocation limitation (lines 1714-1737, 2031) is a FORCED consequence of the JVM wrapper model: it cannot rewrite the target's bytecode, so internal `this.foo()` calls bypass advice. In Rust, leaf's macros CAN rewrite the method body (R2), which means leaf can CHOOSE to either (a) preserve the gotcha honestly by staying proxy-woven (R1/R3 — intent parity, charter §2.2) or (b) eliminate it by compiling the dispatch into the body (R2 — 'prove the JVM workaround unnecessary', charter §2.2). This is not a clash to resolve mechanically but a deliberate DX-vs-parity decision: killing the gotcha is a real DX win (charter §1) but a semantic divergence from Spring (code relying on self-calls-run-sync changes behaviour). There is no JVM analogue to this choice because the JVM had no choice.
- The void/Future/CompletableFuture return discipline (line 2029) collides with the fact that an `async fn` ALWAYS returns a future. In Spring, void/Future/CompletableFuture are distinct, statically-distinguishable return types and 'void means fire-and-forget' is unambiguous. In async Rust, a `#[async] async fn foo() -> ()` produces a `Future<Output=()>` exactly like an awaitable one — the difference between 'fire-and-forget, give me unit' and 'give me a handle to await' is NOT in the return type but in the CALLER's intent. So leaf must invent the distinction: either a return-shape classification the macro reads (Unit vs Handle, R1/R2) or a structural type (`Async<T>`, R3). The proxy's emitted signature must differ from the target's (return unit vs return a join handle for the Unit shape), which has no clean Spring parallel.
- Send + 'static bounds fracture 'run this method off-thread', with no Spring analogue. Spring's @Async method has no Send constraint — any method can be dispatched to a thread pool. In Rust, spawning the method's future onto a multi-thread executor (task-execution's default) requires that future be `Send + 'static` (nightly doc lines 130-136, 160-181: AFIT futures' Send-ness is invisible to callers, RTN unstable). So a `#[async]` method that captures non-Send state (an Rc, a non-Send borrow) is a COMPILE ERROR against a multi-thread executor but would be fine against a current-thread one — meaning the executor-swap-by-config (task-execution's intent, line 2049-2058) can BREAK @Async at compile time, directly contradicting Spring's 'virtual threads don't change the contract' (line 2058). This is the deepest clash and it is inherited from task-execution but lands on every @Async method.
- async fn in traits is not dyn-compatible (nightly doc lines 160-181), so the dispatch interceptor pays a boxed future at the auto-proxy-creator dyn chain boundary AND the executor's spawn takes a boxed future — the @Async dispatch path is double-boxed on the hot fan-out path where Spring's interceptor was a cheap virtual call onto an already-pooled thread. The zero-cost monomorphized spawn is only available off the dyn surface (R2's body-rewrite), trading the shared-chain unification for it.
- Fire-and-forget has nowhere to send an error, sharpened by no thread-interrupt and cooperative cancellation. Spring routes void-@Async uncaught exceptions to AsyncUncaughtExceptionHandler (line 2032) on the pooled thread. In Rust a detached spawned task that errors/panics has its result dropped silently unless explicitly routed — so leaf MUST wrap the spawned future to forward Err/panic to a container-level handler, AND because there is no thread-interrupt, a fire-and-forget task cannot be force-stopped: cancellation is cooperative (the future is dropped at an await point) and a detached @Async task that ignores cancellation can outlive the caller and the drain window. The 'what happens to in-flight @Async work at shutdown' question has no Spring-thread-interrupt answer — it is cooperative-drain-with-timeout (task-execution/context-close).
- ThreadLocal-based context propagation is actively wrong across the @Async spawn (lines 1871, 2089-2099 vs nightly task-hop reality). Spring's @Async caveat 'ThreadLocal/MDC/Observation not propagated unless you add a ContextPropagatingTaskDecorator' (line 1871) assumes ThreadLocals that a decorator can capture-and-restore on the borrowed thread. In Rust, async tasks have no inherent thread-locals surviving an .await/task move (a task migrates threads under work-stealing), so the @Async dispatch's context-propagation MUST go through task-execution's TaskDecorator capture/restore seam over a rust-native ambient-context mechanism (charter §3 deferred) — this feature can only DECLARE the dependency on that seam, not implement propagation, and ThreadLocal-shaped solutions are a footgun.
- Executor selection's qualifier resolution and force-mode straddle compile-vs-runtime with no classpath. Spring resolves @Async("name") and force-mode at runtime against the bean factory (lines 2033, 2069). In Rust there is no runtime classpath to look up a named executor reflectively — the qualifier must resolve via autowiring-resolution over the link/build-time-collected executor beans, and a qualifier naming an executor in an unlinked crate is a silent-DCE failure (cross-crate doc line 149) surfaced (at best) at a refresh validation pass, not at runtime lookup. Force-mode, a runtime container flag in Spring, must be assembled where the full feature set is visible (the binary/app-root, topology line 260) and reconciled with whatever compile-time dispatch R2 baked in.
- The inert-marker→advisor wiring (line 1712) has no runtime-registration analogue and inherits the auto-proxy-creator's DCE hazard with a catastrophic signature. In Spring, @EnableAsync's ImportSelector registers the advisor + interceptor at runtime. In Rust the async advisor is a link/build-time-collected descriptor (auto-proxy-creator's substrate) vulnerable to dead-code elimination — and a dropped async advisor does not error: every `#[async]` method silently runs SYNCHRONOUSLY inline (R1/R3) or, if the executor itself is missing, the spawn target does not exist. This silent-wrong-thread failure is worse than a missing ordinary bean (the code still 'works'), mandating the anti-DCE self-check (cross-crate doc line 152) specifically for the async advisor AND the mandatory-executor force-link.

### Interactions
- **task-execution** — HARD downstream dependency — task-execution is the run-somewhere seam this feature targets. The AsyncExecutionInterceptor→AsyncTaskExecutor relationship (line 2029) is realized as: this feature's dispatch interceptor (R1/R3) or macro-emitted spawn (R2) calls task-execution's injected executor abstraction (its R1 dyn TaskExecutor / R2 Spawner capability / R3 ExecutorHandle — whichever ships). This feature OWNS the @async marker, the dispatch decision, executor SELECTION (qualifier→async-default→primary + force-mode, lines 2033/2069), and async-exception routing; task-execution OWNS spawn/spawn_blocking/concurrency-bounding/decorator-chain/lifecycle-drain. The Send+'static-vs-swap-by-config tension and the mandatory-runtime Layer-0 hazard are INHERITED from task-execution and land on every @async method.
- **auto-proxy-creator** — DIRECT upstream — in R1/R3 this feature contributes ONE Role::Infrastructure AdvisorDescriptor (AsyncAnnotationAdvisor analogue, lines 1705/1712) into the auto-proxy-creator's advisor-collection stream; the creator MATCHES the @async pointcut against beans, computes the UNION with other advisors (tx/cache), ORDERS by role-tier+order into one chain, and drives the wrap-and-swap. The catalog-stated 'async-execution contributes a Role::Infrastructure advisor collected and ordered by this feature' is exactly this relationship. This feature supplies the pointcut + interceptor factory + order hint; the creator does collection/match/union/order/install. In R2 (body-rewrite) this feature LEAVES the shared chain — its sharpest cost, sacrificing cross-cutting ordering.
- **aop-proxying** — INDIRECT (transitive via auto-proxy-creator) — aop-proxying is the substrate that wraps the bean and routes the call through the chain into which this feature's interceptor is installed. This feature does NOT touch the proxy mechanism; it only contributes one interceptor the substrate runs. The self-invocation caveat (lines 1714-1737) is a DIRECT consequence of aop-proxying's wrap-at-the-seam model — R1/R3 inherit it honestly; R2 sidesteps the substrate entirely by compiling dispatch into the body, which is why R2 alone can kill the gotcha.
- **transaction-management** — SIBLING infrastructure advisor sharing the chain — the ORDERING of @async's dispatch interceptor relative to @Transactional's interceptor within the single shared chain (auto-proxy-creator's deterministic sort) is CORRECTNESS-CRITICAL: dispatching onto another thread BEFORE the tx interceptor means the transaction does not span the async work (almost always intended); dispatching INSIDE the tx would bind the transaction across the thread hop (almost always wrong, and tx resources are thread/context-bound). This feature must supply an order hint that places dispatch correctly relative to tx, and the auto-proxy-creator must surface the assembled order as an inspectable diagnostic since misordering is invisible in object identity. R2's body-rewrite makes this ordering ill-defined (the dispatch is in the body, not the chain) — a real composition risk.
- **context-propagation** — Provides the capture/restore the @async spawn needs across the thread hop. Spring's caveat (line 1871) 'ThreadLocal/MDC/Observation not propagated unless you add a ContextPropagatingTaskDecorator' is realized as: this feature's dispatch must run through task-execution's TaskDecorator chain, where context-propagation wires capture-on-the-caller-task / restore-in-the-spawned-task. This feature DECLARES the dependency on that seam; context-propagation defines WHAT is captured. OFF by default (Spring's opt-in, data-bleed avoidance). HARD dependency: the rust-native ambient-context-across-.await model is charter §3 deferred, so this feature exposes the seam but cannot implement propagation.
- **deferral-primitives** — Strongest in R3 — async-execution as 'a result deferred onto another execution context' is modeled as a member of the deferral family (ObjectProvider/lazy/0..1/0..N), reusing the explicit-in-the-type-signature deferral vocabulary via an `Async<T>` handle. This unifies cancellation and error-surfacing across 'deferred dependency' and 'deferred-onto-a-thread result' (charter §2.11). In R1/R2 the relationship is conceptual only. The coupling is a risk: deferral-primitives is itself a Phase-1 neighbour, so R3's viability depends on its shape.
- **event-multicaster** — PEER consumer of the same dispatch capability — the multicaster's opt-in async event delivery (line 1850/1868) routes listener invocations through an executor, and a per-listener `@Async` event listener (line 1869) is THIS feature applied to a listener method. The async-listener caveats (line 1824: no return-value re-publishing on @Async listeners because the result is produced after the publisher returned; line 1871: exceptions to a handler not the caller) are this feature's fire-and-forget/uncaught-failure semantics applied to events. Shared intent: both leave the caller's task and inherit the same context-propagation + error-routing concerns.
- **scheduling** — SIBLING declarative-execution feature — @Scheduled (cron/fixedRate/fixedDelay) is the time-triggered cousin of @Async's on-call dispatch; you can enable scheduling WITHOUT @Async (line 2041). Both contribute infrastructure behaviour that runs work on an execution facility (TaskScheduler vs TaskExecutor). A Phase-2 question is whether they share the dispatch+selection machinery (a @Scheduled method's body running on an executor is structurally like an @Async dispatch with a trigger). The single-threaded-default scheduler caveat (line 2045) is scheduling's analogue of @Async's executor-selection consequence.
- **retry-resilience** — SIBLING infrastructure advisor + shared concurrency primitive — @ConcurrencyLimit (lines 2077-2081) caps concurrent @Async invocations via a SimpleAsyncTaskExecutor concurrency limit, i.e. the bounded-permit / ConcurrencyGate primitive task-execution carries; limit-of-1 as a declarative instance lock (line 2081) applies to @async methods. Retry's interceptor and @async's dispatch interceptor are ordered siblings in the auto-proxy-creator chain (do we retry the dispatch or dispatch-then-retry-inside?). Both are 'Framework 7 resilience/async' annotations enabled via an @Enable* selector (line 2077).
- **runtime-lifecycle** — In-flight @Async work participates in the running-state machine via the executor (task-execution IS the Lifecycle bean). This feature does not itself manage lifecycle, but its detached fire-and-forget tasks must be gracefully drained on stop() through the executor's drain — so @Async's fire-and-forget semantics define WHAT must be drained, and runtime-lifecycle/context-close define HOW (cooperative cancellation + timeout, no async Drop, nightly doc lines 276-280).
- **context-close** — Symmetric teardown — because no async Drop (nightly doc lines 86, 276-280), in-flight @Async tasks are drained by the container-driven `executor.shutdown().await` that context-close orchestrates (stop accepting, drain to timeout, abort remainder). A detached fire-and-forget @async task that outlives the drain window is aborted; this feature must document that fire-and-forget tasks are NOT guaranteed to complete on shutdown (the honest cooperative-cancellation semantic).
- **container-core** — The @EnableAsync-analogue wiring (line 1712) is installed through container-core's engine-vs-façade split: the bare engine with no async advisor/executor leaves @async inert (parity with line 49/110 — though for @async 'inert' dangerously means 'runs sync', mandating the self-check); the context façade installs the async advisor + ensures the mandatory executor + assembles force-mode where the full feature set is visible (topology line 260). container-core is also where the catastrophic mandatory-executor + async-advisor self-check (anti-DCE, cross-crate doc line 152) is enforced at refresh.
- **autowiring-resolution** — Executor SELECTION rides autowiring-resolution — the @Async("name") qualifier (line 2033) resolves the named executor via the same qualifier/primary machinery, and 'no qualifier → primary execution facility' uses primary resolution. A qualifier naming an executor in an unlinked crate is a silent-DCE failure surfaced as a refusal-to-resolve at refresh (autowiring-resolution's fail-fast-on-ambiguity/absence). Force-mode (line 2069) is a deliberate override of normal qualifier resolution — the container central-reroutes regardless of the qualifier, overridable only by the explicit async-config (a documented exception to autowiring's normal rules).
- **conditions** — Gates async wiring — the async advisor + default executor selection back off / activate conditionally (@ConditionalOnMissingBean for a user-supplied executor; an @ConditionalOnThreading(VIRTUAL)-analogue picking the virtual-thread executor, lines 2051-2056). Whether @Async is enabled at all is the @EnableAsync-analogue, itself an import/condition decision. The conditional strategy (cfg vs codegen vs runtime gate, charter §3) shapes where async-enablement and executor-choice are decided.
- **component-stereotypes** — Lightweight — `#[async]` is a method-level marker on methods of managed beans (typically @Service/@Component stereotypes); the bean must be container-managed and injected-by-reference for the proxy-woven realizations (R1/R3) to intercept external calls. A non-bean or a directly-constructed object gets no async dispatch (parity with Spring's 'only managed beans are proxied').

## Area coherence notes

- TRANSPARENT-VS-HONEST HANDLE is the single biggest cross-feature consistency risk and it is currently presented as four PARALLEL open questions rather than one. aop-proxying (Q3), scoped-proxies (Q1), auto-proxy-creator (Q3 per-edge), aspect-model (injection-type), and async/autowiring all hinge on whether the proxy injects as the SAME type the target satisfies (dyn Trait, Spring-faithful, autowiring untouched) or as a VISIBLE handle (Advised<dyn T>/ScopedRef<dyn T>, rippling injection-point types cross-crate). Each design honestly flags its own ripple, but nothing forces ONE answer. If scoped-proxies picks the honest ScopedRef while aop-proxying keeps transparent dyn-Trait, a bean that is BOTH scoped AND advised (a case scoped-proxies' transaction-management interaction explicitly promises composes into ONE proxy) becomes incoherent — you cannot wrap a visible-handle and a transparent proxy as one object. This MUST be resolved as a single area-wide decision.
- THE PROXY MECHANISM keystone (R1 generated type vs R2 runtime table vs R3 resolution-seam middleware) is deferred identically by aop-proxying, inherited by scoped-proxies, aspect-model, and async — but auto-proxy-creator's R3 (per-injection-EDGE chain assembly) and aop-proxying's R3 (resolution-seam middleware) introduce a SHARP latent contradiction with Spring's load-bearing 'wrap once, globally, every external caller gets advice' guarantee. auto-proxy-creator names this (its Q3: per-bean-global vs per-edge) and aop-proxying names it (its final phase2Question), but the two designs do not state whether they will resolve it the SAME way. If aop-proxying lands on per-edge middleware while auto-proxy-creator's collect+match+order policy assumes a single per-bean decision, the 'is this reference advised?' ambiguity Spring eliminated re-appears. The per-bean-global-vs-per-edge call must be made ONCE for the whole substrate, not per-feature.
- ORDERING MODEL has a grade-mismatch flagged by THREE features but unresolved: auto-proxy-creator (Q on @Order/@Priority unification), aspect-model (Q5 correctness-grade vs preference-grade), and the interactions with collection-injection all observe that advisor/advice ordering is CORRECTNESS-CRITICAL and invisible-in-object-identity (cache-outside-vs-inside-tx), whereas collection-injection treats @Order as a loose consumer PREFERENCE. async-execution makes this concrete and urgent: its dispatch interceptor's position relative to @Transactional is correctness-critical (dispatch-before-tx vs dispatch-inside-tx changes whether the tx spans the async hop), and async R2 (body-rewrite) makes the ordering 'ill-defined' because the dispatch leaves the shared chain entirely. So the same ordering vocabulary is being asked to serve a preference concern AND a correctness concern AND (under async R2) to govern a step that isn't in the chain. The area needs one decision: is there a single correctness-grade ordering model, and where is the resolved chain order surfaced as an inspectable §2.9 diagnostic (a requirement THREE designs assert but none owns)?
- DUPLICATED WRAP MECHANISM — scope-resolution wrap vs advice-chain wrap — is correctly identified by BOTH scoped-proxies and aop-proxying as a unification candidate (§2.11: 'a strong argument for unifying the wrap primitive'), and scoped-proxies' R3 (LiveTargetProvider core feeding the unchanged advice chain) is the cleanest reconciliation. This is a healthy convergence, NOT a clash — but it is currently a SHARED ASPIRATION with no owner. The risk is that aop-proxying ships a call-routing primitive shaped only for fixed-target advice chains, and scoped-proxies then bolts re-resolution on as a special interceptor (R1) rather than the substrate exposing the per-call target-supplier seam (the TargetSource concept missing above). Whichever feature defines the call-routing primitive must build in the 'where does the target come from on this call' seam from the start, or the unification both designs want becomes a retrofit.
- SELF-INVOCATION BOUNDARY is treated consistently (honest, not papered over) by aop-proxying, aspect-model, and async — GOOD — but async-execution surfaces a coherence wrinkle the cluster must reconcile: async R2 (macro body-rewrite) deliberately KILLS the self-invocation gotcha (dispatch compiled into the body fires even on this.foo()), while aspect-model and aop-proxying PRESERVE it as honest parity. If async eliminates the boundary for @Async but tx/cache/validation preserve it, leaf ships INCONSISTENT self-call semantics across cross-cutting concerns that all claim to use the same substrate (aspect-model's dogfooding §2.11). A user's this.foo() would silently run-sync-but-advised for @Async and run-unadvised for @Transactional on the same method shape — a worse trap than Spring's uniform 'self-calls bypass advice'. The cluster must decide whether the self-invocation boundary is uniform across ALL advisors or per-advisor.
- CATASTROPHIC-SILENT-FAILURE / anti-DCE self-check is raised by async-execution (Q8) and container-core ('AOP silently does nothing in a bare engine') but the semantics DIVERGE in a way the area should reconcile: for ordinary advice 'inert' means advice-does-nothing (matches Spring's plain-BeanFactory parity, benign), but for @Async 'inert' means runs-SYNCHRONOUSLY-on-the-wrong-thread (dangerous), and for a DCE'd auto-proxy-creator it means EVERY advisor (tx/cache/validation) silently vanishes. The designs correctly note container-core installs the creator and async needs a mandatory-executor self-check, but there is no single owner for 'what does leaf doctor / the refresh self-check assert about the WHOLE AOP wiring being present.' This anti-DCE assertion is a cross-cutting requirement (auto-proxy-creator, async, container-core all gesture at it) that needs one home.
- EARLY-PROXY / getEarlyBeanReference sub-tier is consistently flagged as the MOST speculative coupling by aop-proxying, auto-proxy-creator, scoped-proxies, and circular-references — and the designs are coherent that (a) it may be runtime-only since AOT disallows cycles, and (b) scoped proxies are inherently early-safe (no live target to wrap) while AOP early-proxies over a not-yet-initialized target are the hard case. One latent inconsistency to watch: auto-proxy-creator requires the early answer and the after_init answer to be the SAME chain (earlyProxyReferences dedup), and asserts R2 (frozen plan) makes this automatic while R1/R3 need a dedup set — but aop-proxying's mechanism choice (which the creator must not pre-judge) determines whether an early stand-in CAN be materialized over a partial target at all. The dedup-consistency guarantee and the mechanism choice are coupled; the designs should state that the early-ref sub-tier's existence is GATED on the mechanism decision, not designed independently.
