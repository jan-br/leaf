# Phase 2 Register — Cross-cutting Tensions & Open Questions

This register consolidates the cross-cutting tensions and open questions surfaced across all Phase 1 feature designs. Raw questions are deduped and clustered under meta-concept headings. Each cluster names the core tension and lists the constituent questions with their originating features. The final section ranks the highest-leverage decisions that gate the rest.

---

## 1. Scope / Lifetime / Ownership Model (THE keystone)

**Core tension:** What is the type of a stored/handed-out bean — `Arc<dyn Any+Send+Sync>` (commits to refcount, forces Send+Sync everywhere, fights the no-needless-Arc memory discipline), an arena+index slot (no per-bean Arc but breaks for prototype/scoped), or container-owned-handed-out-as-borrow (lifetimes leak into every resolution API)? Singleton, prototype, and request/session scopes have genuinely different ownership shapes, and almost every other decision in the framework hangs off the answer. The handle MUST stay origin-agnostic so a WASM-module provider publishes the identical handle.

- Singleton-cache handle type: `Arc<dyn Any+Send+Sync>` vs arena+DefIdx slot vs container-owned-borrow; drives `get<T>` return type — *IoC Container Core; Bean Definition Registry & Engine; Bean Instantiation & Singleton Caching*
- The keystone viewed through PUBLICATION: prototype publishes an owned move cleanly, singleton/scoped need a shared/stored handle; what does the publish step's signature look like — *Bean Instantiation & Singleton Caching*
- Who OWNS a scoped instance and how is it handed out (singleton container-owned-shared vs prototype caller-owned-move vs request store-owned-dropped-at-boundary, but possibly referenced by in-flight work) — *Bean Scopes & Scope SPI*
- Ownership/lifetime of stored vs merged BeanDefinition: owned-and-handed-as-`&`, `Arc`, or arena-indexed; drives whether merge produces owned/Arc/index and whether Vec/heap fields can be interned/arena'd — *BeanDefinition Metamodel*
- Environment read handle: `Arc<dyn PropertyResolver+Send+Sync>` vs scope-bound borrow vs cheap-clone `Env(Arc<EnvCore>)` newtype — *Environment Abstraction; @Value Injection*
- PropertySource stack representation: in-place mutable Vec+index vs persistent Arc-layered cons-chain vs mutable-Vec-frozen-to-precomputed-index — *PropertySource Stack & Precedence*
- Origin representation: owned-inline vs interned `OriginId`+side-table vs lazy source-delegated lookup — *Origin Tracking*
- Resolved message return type: owned `String` vs `Arc<str>` vs borrow into catalog — *MessageSource / i18n*
- Cached value ownership: Arc-shared vs Clone-per-read; who drops entries; eviction-policy home — *Caching Abstraction*
- Parent-link lifetime: `Arc<dyn ParentView>` vs borrowed `&'a Engine` chain; queryable-property vs ambient-in-request — *Parent/Child Container Hierarchies*
- Container's OWN handle stored without an Arc reference cycle (weak / interior-mutability split / borrow-scoped) — *resolvableDependencies registration*
- Container back-reference every deferral handle holds (`Arc<Container>`? scoped borrow? arena handle?); can providers be stored on beans at all — *Deferral Primitives*
- `LazyValue<T>` capturing env without forcing `Arc<dyn PropertyResolver>` — *@Value Injection*
- ApplicationArguments handle ownership: `Arc` vs leaked-`'static` vs per-consumer clone; dual typed/stringly representation shares one backing — *Command-line argument property source*
- Qualifying-set output shape: Vec-of-borrows vs Arc'd slice vs bitset over registry index vs arena SmallVec; does filtering allocate per resolution — *Autowire Candidate-Resolution; Autowiring Resolution Algorithm*
- `resolvableDependencies` infrastructure handles stored as the SAME erased handle as cached singletons — *Autowiring Resolution Algorithm*
- hierarchy_depth labeling of parent-context candidates without forcing Arc — *@Primary / @Fallback*
- Conversion to a consuming-move (re-seed before sharing) constrains the "first share point" of the environment — *EnvironmentConverter*
- Config-properties object as first-class bean vs inert projection entangles this keystone — *@ConfigurationProperties*
- Diagnostic artifact (condition report) ownership: `Box<[Record]>` owned by context vs `Arc` handle readable from a hot path — *Condition Evaluation Report*
- Where the destroyer captures the singleton handle (one global Arc-registry vs per-scope arenas with independent lifetimes; avoid double-retention) — *Context Close & Destruction Ordering*
- Safe-publication bound physically rides the handle if `Arc<dyn Trait+Send+Sync>`; else must be re-expressed — *Container Concurrency Contract*

## 2. Type-Erasure & Registry Substrate (keep ORIGIN-AGNOSTIC)

**Core tension:** How are `dyn` beans keyed and stored — primarily TypeId-keyed with a name layer, or name-keyed with a type index (Spring's model)? What is the canonical erased handle shape, and how does trait-upcasting (stable 1.86) participate? This substrate must admit native, hand-registered, test-double, AND WASM-module origins identically.

- Registry substrate: singleton cache `Arc<dyn Any>`-by-TypeId vs arena-handle vs other; drives whole engine API shape — *IoC Container Core*
- Name-vs-type primary identity and the coherence contract between the two keying schemes (aliases, collisions, name-keyed Map injection, `&` deref) — *IoC Container Core; Bean Definition Registry & Engine*
- "Injectable as any supertype": must every injectable `dyn Trait` view be explicitly macro-emitted as a type-index entry; interaction with trait-upcasting — *Bean Definition Registry & Engine*
- `BeanName` as interned `Symbol`/`u32` vs heap `String` vs arena-handle vs type-level key; the interner's existence; cross-registry name identity in hierarchies — *Bean Naming, Aliases & Collisions*
- Annotation vocabulary as string-keyed records vs types (marker traits); determines whether dynamic/WASM origins can contribute annotations — *Meta-Annotation / Merged-Annotation Model; Component Stereotypes*
- Stereotype vocabulary as DATA (MarkerId descriptors) vs TYPES (marker traits + supertrait lattice) — *Component Stereotypes*
- One shared metadata primitive (AnnotationMetadata analogue) consumed by annotation-model, bean-definition, config-metadata — *Meta-Annotation / Merged-Annotation Model*
- One shared descriptor/seed (CandidateDescriptor/RawBeanDefinition) consumed by scanning, stereotypes, bean-definition, registry — *Component Scanning; Component Stereotypes*
- ElementId the condition report keys by must match what autowiring/registry query by; must handle generic guarded elements with no single symbol — *Condition Evaluation Report*
- Conversion as compile-time trait vs runtime ConversionService registry; orphan-rule escape hatch (`Conv<T>` newtype / derive / registry-only) — *Type Conversion & Formatting*
- Event store keyed by TypeId (typed channels) vs erased Object bus vs hybrid; runtime ResolvableType analogue for generic/WASM events — *Application Event System; Event Multicaster*
- Canonical erased handle shape for an infrastructure service; concrete-struct handles can't upcast (exact-TypeId match) while trait handles get upcast match — *resolvableDependencies registration*
- NullBean: present-but-null as a variant of the stored handle vs side-table vs map-presence; must settle WITH the substrate — *NullBean sentinel*
- Heterogeneous named cache: one physical `Box<dyn Any>`+TypeId store vs family of typed views — *Caching Abstraction*
- Factory↔product registry shape: two slots / one Provider / one definition with derived product metadata — *FactoryBean Indirection*
- Canonical stable cross-crate identity for auto-config names (module_path-derived vs author-assigned id) as a semver surface — *Auto-Configuration Ordering & Exclusions*
- Stable callback IDENTITY for dedup (fn-pointer / method ident / step-id / symbol) — *Bean Lifecycle Callbacks*
- "Does this erased bean implement Aware X?" via trait-upcast from common `Bean` supertrait vs macro-emitted flag-set vs `Any` downcasts — *Aware Callbacks*
- Dense bean-index space for bitsets, stable across static+dynamic registration, extended without a steady-state lock (COW swap) — *Autowire Candidate-Resolution*

## 3. Metadata / Codegen Boundary

**Core tension:** Are descriptors/merged-annotations/definitions emitted by proc-macro into a link-time slice, by build.rs codegen over the resolved graph, or assembled at runtime — and where does merge/flatten/freeze happen (compile-time frozen vs runtime late-overlay-capable)? This trades zero-overhead + earliest diagnostics against openness to dynamic/profile/WASM overlays. A proc-macro sees only its own crate; cross-crate metadata reuse forces build-time codegen.

- Descriptor emission mechanism (proc-macro link-slice vs build.rs vs runtime); auto-detection as link-collected set vs generated table vs runtime scan — *IoC Container Core*
- BeanDefinition SEED: flat `const RawBeanDefinition` vs builder attachment (forced by open attribute map); const-friendliness vs codegen — *BeanDefinition Metamodel*
- Annotation merge at compile-time (flattened, frozen) vs runtime (engine merges on demand) — *Meta-Annotation / Merged-Annotation Model; Component Stereotypes*
- `@Value` lowered to typed monomorphized resolve-and-convert vs interpreted by a discoverable runtime post-processor — *@Value Injection*
- Placeholder grammar parsed separately (AST, compile-time malformed-template errors) vs fused with evaluation — *Property Resolution & Placeholders*
- Config-metadata boundary: proc-macro co-emission vs build.rs/`cargo leaf metadata` codegen vs link-time reflection catalog; cross-module `@ConfigurationPropertiesSource` needs build-time codegen — *Configuration Metadata Generation; @ConfigurationProperties*
- BindTarget self-description: `#[derive(BindTarget)]` const SCHEMA vs serde adapter vs per-type DataObjectBinder — *Binder API*
- Lifecycle-callback discovery: macro-emitted fn-pointer vs forced trait impl vs runtime probe vs build.rs — *Bean Lifecycle Callbacks*
- Stereotype-minting DX and its extensibility cost matching annotation-model's realization — *Component Stereotypes*
- Precise Rust "simple name" spec (aliases, re-exports, nested types, declared ident vs canonical path; `proc_macro::Span` involvement) — *Bean Naming, Aliases & Collisions*
- NameGenerator policy locus: per-seed flag vs context-wide strategy vs build-time-frozen table — *Bean Naming, Aliases & Collisions*
- Compile-time span hand-off: macro→runtime threading of `proc_macro::Span` source locations into the unified Origin — *Origin Tracking*
- `@Configuration` as runtime managed bean vs pure compile/codegen grouping dissolved into per-`@Bean` defs — *@Configuration / @Bean*
- Const-descriptor (statically inspectable) vs fn/trait seam (runtime/dynamic-capable) as the DEFAULT for declarative source contribution — *@PropertySource-equivalent*
- Matchable method-metadata vocabulary is fixed at macro/codegen time, not reflectively open; method identity must be stable-as-data — *Auto-Proxy Creator; Pointcut/Advice/Aspect Model*
- Cron parse home (macro / build.rs / startup pass); how much schedule is compile-time-legible — *Scheduling*

## 4. Compile-vs-Runtime Resolution Split (AOT lineage)

**Core tension:** How much of merge / wiring validation / collapse / planning / collection-sort / condition-evaluation moves to build or freeze time versus runtime eager-instantiate? An immutable-layered / const-seed / frozen-plan model collapses work to build time (earliest-richest diagnostic, wave-parallel no-guard instantiation, fast startup, wasm-friendly) but forbids mid-bootstrap dynamism and cannot see dynamic/WASM sources; the recursive/mutable runtime model is dynamic-edge-faithful but defers diagnostics.

- Eager-instantiate fail-fast vs build/startup validation pass over descriptors — *IoC Container Core; Bean Definition Registry & Engine; Bean Instantiation & Singleton Caching*
- Recursive-lazy creation vs plan-then-execute wave parallelism — *Bean Instantiation & Singleton Caching*
- Mutable-concurrent-map registry vs immutable-snapshot-at-freeze registry (AOT-lineage freeze-early) — *Bean Definition Registry & Engine*
- How much definition validation/merge/collapse moves to build time; frozen effective definition (AOT) — *BeanDefinition Metamodel*
- Freeze/seal boundary (markBeanAsCreated analogue): runtime state flag vs type-state consuming mutable access vs append-only-then-collapse; precise post-freeze-edit diagnostic — *BeanDefinition Metamodel*
- Whole-graph wiring validation at plan time vs surfaced during eager pass — *Bean Instantiation & Singleton Caching; Autowiring Resolution Algorithm*
- Scope-mismatch (singleton-holds-shorter) detection: build-time static analysis vs startup pass vs first-resolution — *Bean Scopes; Scoped Proxies*
- Ambiguity fail-fast timing (compile / startup-validation-pass / eager-instantiate); consistency across single/collection/scoped — *Autowiring Resolution Algorithm; Autowire Candidate-Resolution*
- Qualification validation/matching timing in the fail-fast ladder — *Autowire Candidate-Resolution*
- `@Bean` inter-factory edge resolution at build vs runtime; intra-crate freeze vs cross-crate runtime degrade — *@Configuration / @Bean*
- Import-graph resolution (edge-walk + group-merge + ordering + conditions) at build vs runtime; intra-vs-cross-crate asymmetry — *@Import Family*
- Auto-config assembly/ordering/filtering build-frozen vs runtime; build-frozen/runtime-residual seam legibility — *Auto-Configuration Engine; Auto-Configuration Ordering & Exclusions*
- Two-phase build-time metadata fast-path (Boot's ASM pre-filter analogue) — *Auto-Configuration Engine*
- Collection injection: aggregate-then-sort-per-resolution vs pre-sorted-frozen-group at freeze — *Collection / Array / Map Injection*
- Advisor matching per-bean-runtime vs precomputed static plan at build/refresh-start — *Auto-Proxy Creator*
- `@Value` / `@ConfigurationProperties` failure-timing consistency (malformed grammar / missing placeholder / conversion error / lazy.get) — *@Value Injection; @ConfigurationProperties*
- Conversion's position in the fail-fast ladder, consistent across `@Value` vs `@ConfigurationProperties` — *Type Conversion & Formatting*
- TxPlan / manager-selection precomputed at build vs runtime-conditional — *Transaction Management*
- Expression evaluation's fail-fast timing (compile / startup / per-event / per-call); bean-reference sub-case irreducibly runtime — *Expression Language*
- Statically-named resources hoisting missing-resource error to compile/codegen — *Resource Loading*
- Compile-time `Valid<..>` parameters vs always-runtime validation — *Validation Abstraction*
- Mandatory-dependency fail-fast: type-state compile error on hand/test path vs startup AssemblyError on container path — *Injection Styles*

## 5. Async Execution & Context-Propagation Model (T2 keystone)

**Core tension:** What is leaf's rust-native answer to ThreadLocal-holders-across-`.await` and to running work off the caller's task? This single decision (a) the ambient task-local primitive, (b) the dyn-boxing-vs-monomorphized standard, (c) the executor shape, (d) cancellation/teardown without async Drop, (e) Send+Sync bounds vs runtime-swap — gates almost every cross-cutting feature and several container internals.

### 5a. The ambient context primitive (ThreadLocal-across-`.await`)
- Canonical task-local: tokio task_local vs runtime-agnostic AmbientStore trait vs structured-concurrency scope; soft-tokio dependency for the default backing — *Locale/Request Context Holders; Context Propagation*
- THE keystone three shapes: accessor-registry+RAII (R1), poll-wrapping combinator over one framework task-local (R2), explicit Context threading + ambient bridge at seams (R3); per-poll reinstall cost & skip-when-unchanged — *Context Propagation*
- Propagate-by-default vs isolate-by-default across hops; per-key policy (locale/trace MUST propagate; tx connection MUST NOT) — *Locale/Request Context Holders; Context Propagation*
- Ambient-everywhere (DX parity) vs explicit-`&Ctx` (correctness); where the explicit-vs-ambient line falls given AOP/event/scheduling seams have no place to thread `&Ctx` — *Locale/Request Context Holders; Context Propagation*
- One opaque bundle vs N independently-registered accessors; scope-based contexts (tracing open/close, Null masking) in a value model — *Context Propagation*
- What ambient does the scoped proxy read per call to find the active scope — *Scoped Proxies*
- Rust-native ambient request/session/websocket scopes bind to — *Bean Scopes*
- EvalContext ambient (locale, request, tx state) entering without a thread-local — *Expression Language; Type Conversion; Validation; Transaction-Bound Event Listeners*
- ConvertCtx (strict/lenient, unit defaults, locale) threaded without `.await`-unsafe thread-local — *Type Conversion*
- Marshalling propagation context across the WASM provider boundary (R3 explicit marshals cleanest) — *Context Propagation; Transaction Management*

### 5b. Async-across-dyn boxing standard
- Boxed-future-on-dyn (alloc per resolution) vs monomorphized zero-cost typed primary with dyn escape — *IoC Container Core; Bean Definition Registry & Engine; Bean Instantiation & Singleton Caching; MessageSource; Resource Loading; Container Extension SPIs; Parent/Child Hierarchies; Bean Lifecycle Callbacks; Aware Callbacks; @Configuration/@Bean; Programmatic Registration; @Bean; Lookup-method injection*
- Origin-agnostic factory/Provider MUST be dyn → boxed-future fn-pointer; singletons cheap, prototype/scoped hot cost — *BeanDefinition Metamodel; Bean Instantiation*
- Boxed-future-per-hop on the advised-call HOT path vs monomorphized escape for the single-advisor case — *Proxy-Based AOP Substrate; SmartInitializingSingleton*

### 5c. Executor shape & scheduling unification
- Executor as one fat dyn trait vs capability traits vs config-driven dispatcher+handle — *TaskExecutor Abstraction*
- TaskScheduler separate vs time-aware capability of the same execution facility — *TaskExecutor Abstraction; Scheduling*
- Single global timer-wheel/driver shared with retry backoff & periodic refresh vs per-scheduler — *Scheduling*
- Concurrency-ceiling primitive (ConcurrencyGate / bounded-permit) owned by task-execution vs retry vs a shared lower primitive — *TaskExecutor Abstraction; Scheduling; Retry/Resilience*
- Permit held across retry backoff sleeps vs re-acquired per attempt — *Retry/Resilience*
- Sequential vs concurrent construction of N collection elements (join, with cancellation/error-aggregation) — *Collection / Array / Map Injection*
- Shared "declarative off-context dispatch" primitive across `@Async`, `@Scheduled`, async events, retry attempts — *Declarative Async; Event Multicaster*
- Per-listener async ownership (here / async-execution / events) and one coherent contract — *Event Multicaster*
- Sync-by-default-on-caller's-task dispatch; ambient propagation to inline-sync vs spawned-async listeners — *Application Event System; Event Multicaster*
- Provider's first-`.await` integrating with background-bootstrap readiness as a structured-concurrency join — *Deferral Primitives; Background Bean Initialization*

### 5d. Cancellation, drain & teardown without async Drop
- Framework-wide cancellation contract: abort vs detach default, cancel tokens, drain timeouts, tasks outliving the drain — *TaskExecutor Abstraction; Scheduling; Retry/Resilience; Runtime Lifecycle; Context Propagation*
- Where uncaught task/scheduled-body/async failure goes (single container-level handler); compose with the error model — *TaskExecutor Abstraction; Scheduling; Declarative Async; Event Multicaster*
- Partial-construction cleanup on failed/cancelled build: scope-guard vs cleanup-ledger vs drop-bomb — *Bean Instantiation; Container Refresh Lifecycle; Background Bean Initialization; Circular References; Bean Lifecycle Callbacks; Aware Callbacks; SmartInitializingSingleton*
- Teardown-on-cancellation for context scopes (request/session destruction callbacks still run) — *Bean Scopes; Scoped Proxies*
- Transaction finalization (rollback) under future-drop with no async-Drop RAII — *Transaction Management*
- Around-advice scope ownership / cancel-safety across proceed() mid-chain — *Proxy-Based AOP Substrate; Pointcut/Advice/Aspect Model*
- Per-bean graceful-destroy timeout + error-aggregation substrate for the destroy walk — *Bean Lifecycle Callbacks; Context Close; Runtime Lifecycle*
- Per-phase graceful-drain timeout + rust-native `stop(Runnable)` analogue — *Runtime Lifecycle*
- Single-flight (`sync=true`) shared-future wait, async-cancellation of the shared computation — *Caching Abstraction; Retry/Resilience*
- Sync restore rides Drop cleanly, but does any context concern need async cleanup on scope exit — *Context Propagation*

### 5e. Send/Sync bounds vs runtime-swap
- Mandate all spawned/shared work Send+'static (loses current-thread flexibility) vs two surfaces vs accept compile-breaking swaps; ties to RTN/trait_variant — *TaskExecutor Abstraction; Context Propagation; Background Bean Initialization; Declarative Async; Transaction Management*
- Send/Sync discipline for inheritable ambient values; strictly-task-local non-inherited values could avoid Send — *Locale/Request Context Holders*
- Universal vs scope-graded bound; single-threaded-runtime carve-out from Send+Sync — *Container Concurrency Contract*

### 5f. The async pre-materialization pipeline (BootstrapContext analogue)
- async-fetch → in-memory-snapshot → sync-use, framework-wide, for remote/async-backed sources and customizers — *Environment; PropertySource Stack; Property Resolution; Origin Tracking; @Value Injection; Relaxed Binding; Binder API; Externalized Config; @ConfigurationProperties; ConfigurationPropertySource adapter; @PropertySource-equivalent; RandomValuePropertySource*
- Where the async config-IO phase ends and the synchronous bootstrap-resolution phase begins (the fence) — *ConfigurationPropertySource adapter; spring.main.* self-binding; EnvironmentConverter*
- Retry/backoff poll loop as a sanctioned cold-path activity; refresh primitive for re-snapshotting — *Externalized Config*
- RNG interior-mutability / per-resolution-context RNG handle as part of the execution-context model — *RandomValuePropertySource*

## 6. Unified Error / Diagnostics Model

- The diagnostics/observability SPINE: `tracing` as core vs bespoke recorder with optional bridge — *Startup Instrumentation*
- Pull/handle recorder vs push event-stream; shared with anti-DCE self-check & phase-order assertions — *Startup Instrumentation*
- Aggregate-all-failures startup-validation pass vs fail-on-first (placeholders, binding, conversion, constraints) — *Property Resolution; Binder API; @ConfigurationProperties; Relaxed Binding*
- One "collect-all-the-things-that-went-wrong-across-this-typed-boundary" accumulator shared by binder, conversion, validation — *Validation Abstraction*
- Lenient vs strict DEFAULT for unresolved placeholders / `@Value`; per-consumer vs one global knob — *Property Resolution; @Value Injection; Expression Language; Placeholder escaping; ${...}-vs-#{} dispatch*
- `@AliasFor` / stereotype misconfiguration: compile_error! vs build-time pass vs startup — *Meta-Annotation; Component Stereotypes*
- Condition outcomes threading into the report so silent-now vs loud-later stay distinct & time-separated; cross-tier coherent account — *@Conditional Condition SPI; Conditional-On-X Family; Auto-Configuration Engine; Condition Evaluation Report*
- Abstain-vs-veto three-valued fold (total, canonical) + per-candidate veto reason retention — *Autowire Candidate-Resolution*
- Per-resolution diagnostic trace cost vs richness: always-materialized vs only-on-error vs behind-a-flag; crate-provenance for cross-module NoSuchBean — *Autowiring Resolution; Deferral Primitives; @Primary/@Fallback*
- Loud collision vs anti-DCE missing-bean: unified present-and-clashing vs absent-but-expected diagnostic — *Bean Naming, Aliases & Collisions*
- Relaxed COLLISION policy (`first-name` AND `firstName`): hard error vs warning vs last-wins-with-diagnostic — *Relaxed Binding*
- Contract-violation diagnosis: compile-time bound with `#[diagnostic::on_unimplemented]` doctrine message vs startup pass — *Container Concurrency Contract*
- `leaf doctor`-style lint for scope-choice smell (guarded vs un-guarded mutable singleton state) — *Container Concurrency Contract*
- Provenance depth for collision diagnostics (Spans, TypeIds, alias chain); always vs diagnostic-mode-only — *Bean Naming, Aliases & Collisions; BeanDefinition Metamodel; Autowiring Resolution*
- Report delivery channels (env-toggled dump / structured serde / tracing bridge) and no-op default — *Condition Evaluation Report*
- Resolved feature/cfg snapshot mechanism to trustworthily label "feature X not enabled" — *Condition Evaluation Report*
- Report taxonomy additions (CompiledOutByCfg, BuildFoldedFalse) frozen in core ABI; machine-parseable reason strings — *Condition Evaluation Report*
- Assembled advisor/cache/tx ORDER surfaced as an inspectable diagnostic (misordering is invisible in object identity) — *Auto-Proxy Creator; Caching Abstraction; Retry/Resilience; Pointcut/Advice/Aspect Model*
- Imperative registrar branch outcomes made accountable to the condition report — *Programmatic Registration*
- Conflict policy: config-overrides-programmatic-setter silent-with-diagnostic vs hard error — *spring.main.* self-binding*
- Rollback matching reuses the unified error-matching vocabulary (TypeId/downcast/predicate/trait) — *Transaction Management*
- DataAccessError shape: single enum with `kind` vs `dyn DataAccessError` trait family; cross-crate extensibility — *Persistence Exception Translation*
- Silent-untranslated-error self-check (worse than a loud missing-bean) — *Persistence Exception Translation*

## 7. Cross-Crate Composition & Tooling Model (static registry + optional WASM "classpath") + Anti-DCE

**Core tension:** One substrate for ALL discovery features — linkme distributed-slice (zero-overhead, server-target-fragile, DCE-prone) vs inventory (life-before-main) vs build-time codegen (portable, no silent-empty) vs explicit `with()`/`enable()` — PLUS the optional dynamic directory-scanned/WASM "second classpath." Must be decided ONCE, kept scoped-not-global (predictability), and carry a mandatory expected-vs-found self-check so a DCE-dropped crate fails LOUDLY.

### 7a. The default substrate & scoping
- Default cross-crate substrate (A/B/C/D) and how "which crates participate" is expressed; façade auto-detect/force-link/self-check binds to it — *IoC Container Core; Resource Loading*
- linkme vs build-codegen as default 'classpath', co-decided across EVERY discovery feature — *Component Scanning*
- Base-package SCOPING re-expressed natively (= binary crate + explicitly enabled integration crates); `scan!`/`enable!` DX — *Component Scanning*
- Discovery substrate shared (or not) across env-customizer, source-contributors, scanning, auto-config, holders, catalogs, scopes, processors, listeners, converters, advisors, runners, transports, generators — *Environment; PropertySource Stack; Locale Holders; MessageSource; Type Conversion; Bean Scopes; Container Extension SPIs; Application Event System; Aware Callbacks; Component Stereotypes; Profiles; Relaxed Binding; Persistence Exception Translation; Validation; SPRING_APPLICATION_JSON loader; spring.main.* self-binding; RandomValuePropertySource; @PropertySource-equivalent; Command-line argument source*
- Const-constructibility coupling: `&'static dyn Pointcut` on linkme demands const (restricting the algebra) → no freely-composable-AND-zero-cost-const pointcut — *Pointcut/Advice/Aspect Model*

### 7b. The anti-DCE / silent-empty self-check
- Mandated expected-vs-found self-check; manifest location (façade vs generated app-root); rich per-source diagnostic; differs across substrates — *Container Extension SPIs; Component Scanning; Container Refresh Lifecycle; SmartInitializingSingleton; Auto-Configuration Engine; Runtime Lifecycle; Declarative Async*
- Mandatory-runtime-crate force-link + self-check (safety-critical: a binary must never silently lack its execution facility / dispatch engine / async advisor / scheduler / cache backend) — *TaskExecutor Abstraction; Event Multicaster; Declarative Async; Caching Abstraction; Background Bean Initialization; Scheduling*

- Collection injection's expected-count + per-crate provenance (is "expected count" even knowable without a build-time manifest) — *Collection / Array / Map Injection*
- `@Fallback` / `@Primary` count-dependent arithmetic precondition: per-type-key expected-count self-check before the decision point — *@Primary/@Fallback*
- Report doubling as the DCE self-check vs a separate mechanism — *Condition Evaluation Report*
- Distinguishing "missing candidate" (DCE'd) from "typo'd exclusion/before_name" and from "genuine backoff" — *Auto-Configuration Ordering & Exclusions; Auto-Configuration Engine*

### 7c. Determinism & ordering-as-data
- Two-phase parse-vs-register ORDERING with unspecified link/section/constructor order → deterministic topo/phase sort over descriptor DATA — *@Conditional Condition SPI; Conditional-On-X Family*
- Source-contributor ordering via position-hint data (first/last/before/after) applied in a runtime sort — *PropertySource Stack*

### 7d. The two-source (static + dynamic/WASM) duality
- A dynamically-discovered definition has no compile-time cfg and no Rust type → can ONLY be gated/typed at runtime, forcing a mandatory runtime layer for origin-agnosticism — *@Conditional Condition SPI; Conditional-On-X Family; Profiles; Auto-Configuration Engine; Condition Evaluation Report*

- Enabling a dynamic source explicit/scoped vs implicit/global — *Resource Loading*

## Cross-feature tensions to resolve first

The following are the highest-leverage decisions: each one gates a large fraction of the rest, and several features explicitly state they "cannot be settled until" these land. They should be sequenced roughly top-to-bottom, as the lower ones partly depend on the upper ones.

1. **Scope / lifetime / ownership model + the type-erasure & registry substrate (§1 + §2), decided jointly.** This is THE keystone almost every other cluster defers to. It fixes the singleton-handle type, `get<T>`'s return type, name-vs-type keying, the erased handle shape, and whether handles can be stored on beans — and it must be origin-agnostic so a WASM provider publishes identically. Nearly every Container & Beans feature, the env handle, provenance, message return types, and the concurrency contract's safe-publication bound all hang off this.

2. **The async execution & context-propagation model (§5), as a single coordinated decision.** Sub-decisions (a) the ThreadLocal-across-`.await` primitive, (b) the async-across-dyn boxing standard, (c) the executor/scheduler shape + concurrency-gate, (d) the cancellation/teardown-without-async-Drop contract, and (e) Send/Sync-vs-runtime-swap are deeply entangled and must not be decided piecemeal. This gates context-propagation, locale, task-execution, scheduling, async events, tx, scopes, every dyn boundary, and all partial-construction/teardown cleanup.

3. **The cross-crate composition & tooling substrate + mandatory anti-DCE self-check (§7).** Pick ONE default substrate (linkme vs build-codegen vs explicit) and the optional dynamic/WASM "second classpath," kept scoped-not-global, with a canonical expected-vs-found self-check. Every discovery feature (scanning, stereotypes, conditions, auto-config, listeners, processors, converters, holders, catalogs, scopes, advisors, runners, transports) shares this and must not invent bespoke registries. The "two-source duality timing vs the freeze boundary" reconciliation lives here.

4. **The metadata/codegen boundary + compile-vs-runtime split (§3 + §4), which together imply the conditional strategy (§8) and dependency-based-auto-config y/n.** Whether descriptors/merges/definitions/plans freeze at build time or assemble at runtime determines diagnostic timing, mid-bootstrap dynamism, and whether a mutable-registry or immutable-snapshot engine is admissible. The per-kind condition tier-map is a load-bearing kernel ABI that the macros must agree on; profiles being always-runtime is the coherence constraint.

5. **The proxy / interception substrate (§10).** The no-CGLIB mechanism choice (generated type vs runtime table vs resolution-seam middleware) and transparent-vs-visible-handle decision gate AOP, scoped-proxies, lazy, tx, cache, validation, retry, exception-translation, the early-proxy cache tier, and the self-invocation contract — and ripple cross-crate through injection-point types.

6. **The unified error/diagnostics spine (§6) and the one ordering-primitive (the @Order/@Priority comparator + phase/topo sort) (§11 + §12).** Whether `tracing` is core, whether failures aggregate at a startup pass, how silent-now stays distinct from loud-later, and whether one ordering primitive serves autowiring tie-break, collection ordering, lifecycle phases, advisor order, and destruction order. These are pervasive but mostly downstream of 1–5.

7. **The config read/mutate seam, precedence-rung policy owner, and the canonicalization/escape/absence pipeline (§14), decided once for the whole config family.** One seam, one precedence ladder owner, one canonical key-segment/array-index primitive, one absence model, and one `${...}`-vs-`#{...}` dispatch owner — so the ~15 config features stop each re-answering them. Largely downstream of 1, 2f (the async pre-materialization fence), and 3.

8. **Container shape (BeanFactory/ApplicationContext; one-engine-vs-pluggable-kernel) and FactoryBean-as-Provider (§9).** A unification call that needs the whole catalog; deciding it too pluggably risks pre-empting the meta-architecture and widening the stable ABI, so it should follow 1–4.

### Bootstrap mechanics flagged as possibly-missing
- **Process keep-alive (spring.main.keep-alive / KeepAlive listener)** — The spring.main.keep-alive VALUE is stored in extra-11's BootstrapSettings (02-configuration-environment.md line 1762/1771), but NO design owns the BEHAVIOR. exit-code-shutdown owns arming/firing the shutdown hook and the termination/exit-code coordination, but explicitly models the steady-state as 'return-code-to-caller' — it never owns the keep-alive WINDOW (await-until-signal) that prevents an immediate exit for an app whose only 'work' is bean side-effects. spring-application-run's run() ends at 'if running: ready' and hands back; runners run before ready and return. So a leaf app with no auto-started runtime-lifecycle component (no server/scheduler holding a task) would complete run() and exit — silently dropping the keep-alive intent. This is sharper in async Rust than on the JVM (no ambient non-daemon-thread notion), and it is the natural seam between spring-application-run's tail, exit-code-shutdown's hook, and runtime-lifecycle's 'is anything actually running' — currently unowned.
- **ApplicationStartup recorder wiring at boot (context.setApplicationStartup) + the boot-level root step** — startup-instrumentation (05-events-lifecycle.md line 541) defines its root as 'leaf.context.refresh' and frames itself purely as the OBSERVER of refresh-lifecycle — it never mentions a boot-level root span, nor the act of INSTALLING the recorder into the context (Spring's setApplicationStartup), nor how the boot root becomes the parent of the refresh root across the boot->refresh handoff. spring-application-run's interaction note claims 'the run carries parent linkage explicitly into the refresh fan-out (the §2.3 fix)' and that leaf.application.run is the root — but NO design owns the actual seam: who creates the boot recorder, when it is set on the context (the prepare_context/setApplicationStartup moment), and how the refresh root's parentId points at the boot root. This is a concrete ownership gap between spring-application-run and startup-instrumentation, not just a naming question, and it is the mechanism that makes boot-failure forensics (run-phase + nested refresh-phase in one tree) actually work.
